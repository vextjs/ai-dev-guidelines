## 📝 变更摘要

## 🎯 背景说明

### 为什么需要这次变更？

v2.5.0 采用的"共享目录+Agent 文件名前缀"方案在实际使用中暴露了并发问题：
- 多个 Agent 的 SUMMARY.md 存在并发写入覆盖风险
- 恢复任务需要解析文件名中的 Agent 前缀来过滤，复杂且脆弱
- 文件名过长（日期+Agent+类型+描述），可读性差

同时 copilot-instructions.md 作为入口文件包含了过多规范细节（141 行），导致 AI 首次加载时 token 消耗大，且维护时容易与子文件不一致。

此外，AI 在声称"任务完成"时缺少强制验证机制，可能导致输出文件不存在、内容截断等问题被忽略。

### 目标

- copilot-instructions.md 精简为纯入口指引（~80 行）
- 多 Agent 方案改回目录隔离，消除并发风险
- 新增任务完成验证，确保输出质量
- 全部文件版本号统一

## 📦 变更内容

### 核心变更

#### 1. copilot-instructions.md 精简

入口文件从 141 行精简至约 80 行，仅保留：
- 预检查强制执行块（4 项必做）
- 规范体系入口表（指向各子文件）
- 关键原则速览（一行一条）
- 绝对禁止的行为清单

所有规范细节移至对应的子文件中维护，避免重复定义导致不一致。

#### 2. 多 Agent 策略改回目录隔离

```
.ai-memory/
├── SUMMARY.md                    # 全局只读摘要（关键决策汇总）
└── clients/
    ├── zed-copilot/              # Zed Copilot 专属
    │   ├── SUMMARY.md
    │   └── tasks/
    │       └── 20260226-01-ANALYSIS-xxx.md
    ├── webstorm-copilot/         # WebStorm Copilot 专属
    │   ├── SUMMARY.md
    │   └── tasks/
    │       └── 20260227-01-FIX-xxx.md
    └── unknown-agent/            # 无法确定编辑器时的默认目录
```

关键规则：
- 每个 Agent 在 `clients/<agent>/` 下独立管理自己的 SUMMARY.md 和 tasks/
- AI 绝不写入其他 Agent 的 `clients/` 目录
- 全局 SUMMARY.md 仅追加关键决策摘要（标注来源 Agent）
- 恢复任务时只扫描本 Agent 目录，无需解析文件名过滤

文件名格式同步更新：
- 旧格式（v1.2）: `YYYYMMDD-<agent>-<TYPE>-<id>.md`（Agent 在文件名中）
- 新格式（v1.3）: `YYYYMMDD-<NN>-<TYPE>-<id>.md`（Agent 由目录隔离保证，不写入文件名）

#### 3. 新增约束 #15 — 任务完成验证

```yaml
触发条件: AI 准备声称"任务完成"之前
执行时机: 所有修改/生成操作完成后

验证清单:
  1. 输出文件存在性:
     - 所有声称创建的文件确实存在？（用工具验证）
     - 文件路径正确？（在预期目录下）
     - 文件名符合命名规范？

  2. 内容结构完整性:
     - 文档头部包含所有必填字段？（日期、Agent、状态）
     - 日期字段是真实日期，非占位符？
     - 没有中途截断的内容？

  3. 记忆文件完整性:
     - .ai-memory 中的任务记忆已更新？
     - SUMMARY.md 已追加/更新对应条目？

  4. 关联文件一致性（如修改了规范文件）:
     - 已执行交叉验证清单？
     - 所有引用处已同步更新？
```

未通过验证项必须修复后才能声称完成。

#### 4. Agent 详细规则集中

Agent 检测方法、映射表、读写规则、跨 Agent 恢复流程等详细规则从 copilot-instructions.md 和 00-pre-check/README.md 中提取，集中到 `task-memory.md §多 Agent 支持` 段落统一维护。

#### 5. 版本号统一

全部文件版本号对齐 v2.6.0：
- README.md、QUICK-REFERENCE.md、STATUS.md、CONSTRAINTS.md
- workflows/decision-tree.yaml
- workflows/00-pre-check/README.md

## ✅ 验证方法

### 手动验证

1. 新建会话 → 确认预检查仍正常输出 4 项必做
2. 创建记忆文件 → 确认写入 `clients/<agent>/tasks/` 而非根目录 tasks/
3. 检查文件名 → 确认不含 Agent 前缀（如 `20260227-01-FIX-xxx.md`）
4. 声称任务完成 → 确认 AI 执行了完成验证清单
5. grep 全项目 "v2.5" / "v2.4" 等旧版本号 → 确认已对齐到 v2.6.0
