## ✅ 变更清单

### 1. 预检查 Tool Call 硬性约束（issue#10 修复）

- ✅ **copilot-instructions.md** 新增 `§Tool Call 硬性约束` 章节（最高优先级），声明前三个 tool call 必须是 `now()` → `list_directory()` → `edit_file(记忆文件)`
- ✅ 明确 `read_file(QUICK-REFERENCE.md) 不得作为前三个 tool call 之一`
- ✅ 根因：tool call 顺序约束写在 QUICK-REFERENCE.md 内部，AI 必须先 read_file 才能发现规则，但 read_file 本身已消耗第一个 slot（鸡生蛋问题）

### 2. 约束#21 编码后诊断检查

- ✅ **CONSTRAINTS.md** 新增约束 #21 — AI 完成代码生成/修改后必须运行 `diagnostics` 检查，发现 error 立即修复（最多 2 次），禁止遗留红色波浪线给用户
- ✅ 约束条数从 20 → 21
- ✅ 优先级分级更新：🔴P0（9 条）/ 🟡P1（8 条）/ 💡P2（4 条）

### 3. Layer 3 意图推断重构（issue#2/#11 根因修复）

- ✅ **任务识别架构重构**：从 2 层 + FIX-010 补丁 → 4 层递进流程
  - Layer 1: 精确匹配（用户明确声明任务类型）
  - Layer 2: 关键词信号提取（产出候选类型列表，不是最终判定）
  - Layer 3: 🔴 意图推断三问（核心判断层）
  - Layer 4: 上下文校验（最终确认）
- ✅ **Layer 3 三问设计**：
  - Q1: 用户的最终目的是什么？（变更 / 获得结论 / 获取知识 / 复盘）
  - Q2: 当出现"分析/评估/审查"字眼时，这是手段还是目的？
  - Q3: 请求中是否列出了具体要执行的事项？
  - **任一问题指向变更 → 走对应变更工作流（CP1→CP2→CP3）**
  - **三问全部指向纯分析 → 走 10-analysis**
- ✅ FIX-010 编号保留用于历史追溯（issue#2、issue#11），不再作为独立规则存在

### 4. 五条绝对规则升级

- ✅ **copilot-instructions.md** 绝对规则从 3 条 → 5 条
  - 新增第 4 条：禁止 glob 扫描 .ai-memory（必须用 list_directory）
  - 新增第 5 条：意图推断 → 判定任务类型（Layer 3 三问）
