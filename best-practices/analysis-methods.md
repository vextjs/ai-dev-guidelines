# 项目分析方法论

> 针对不同复杂度项目的分析策略

---

## 📋 复杂度评估

### 评估标准

| 复杂度 | 文件数 | 调用层次 | 分析策略 |
|--------|-------|---------|---------|
| 简单 | < 5 | < 3 层 | 完整阅读 |
| 中等 | 5-15 | 3-5 层 | 层次追踪 |
| 复杂 | 15-50 | > 5 层 | 层次追踪 + 分批 |
| 超大 | > 50 | 复杂 | 必须确认计划 |

---

## 🔵 方法 A: 完整阅读

### 适用场景

```yaml
适用:
  - 简单项目（< 5 文件）
  - 改造范围明确
  - 调用层次简单
```

### 执行步骤

```yaml
步骤:
  1. 根据全局搜索结果，确定需要改造的文件
  2. 使用 read_file 逐个读取文件
  3. 理解现有实现逻辑:
     - 函数/类的职责
     - 参数和返回值
     - 调用关系和依赖
     - 错误处理方式
  4. 记录关键发现

输出格式:
  📖 相关代码分析（完整阅读）
  
  已读取文件:
  1. app/service/subscription.ts (L50-L200)
     - getExpiryDate(): 当前逻辑使用 moment().add(30, 'days')
     - 配置读取: 从 config.subscription.expiry_days
     - 调用方: 3 处（订阅创建/续费/积分兑换）
  
  关键发现:
  - ⚠️ 测试模式没有独立配置
  - ⚠️ 时间间隔是硬编码
```

## 🟡 方法 B: 层次追踪

### 适用场景

```yaml
适用:
  - 中等复杂度项目（5-15 文件）
  - 需要理解调用链路
  - 可能存在统一机制
```

### 核心思想

```yaml
思想:
  - 不是读所有文件
  - 追踪调用链路
  - 快速发现统一机制（runner/middleware/hook）
```

### 执行步骤（5 步）

```yaml
步骤 1 - 识别入口点:
  - 从全局搜索结果找到修改点
  - 使用 grep_search 查找入口文件
  - 使用 read_file 读取入口方法
  
  输出: 入口方法名称、文件路径、方法职责

步骤 2 - 追踪调用链路（至少 3 层）:
  - 在入口方法中找到关键调用
  - 使用 grep_search 查找被调用方法
  - 使用 read_file 读取被调用方法
  
  🔴 特别注意: 发现统一机制时立即标注
  
  统一机制识别特征:
    ✅ runner/middleware/hook/event 等关键词
    ✅ 所有查询/操作都通过同一个函数
    ✅ 工厂模式/装饰器模式
    ✅ 生命周期钩子

步骤 3 - 绘制调用链路图:
  📊 调用链路图
  
  入口: collection.find({ status: 'active' })
    ↓
  Layer1: createCachedRunner(collection, method, args)
    ↓  ← ⚠️ 统一机制发现！
  Layer2: withSlowQueryLog(runner, options)
    ↓
  Layer3: hooks.onSlowQueryEmit(queryMeta)
    ↓
  Layer4: emit('slow-query', meta)
  
  🎯 关键发现:
  1. 所有查询都通过 createCachedRunner 统一处理
  2. 只需要在 hooks 中添加新逻辑
  3. 无需修改 11 个查询文件

步骤 4 - 验证完整性（反向查找）:
  - 查找入口方法的所有使用方
  - 确认都会经过统一机制
  - 如有例外，单独分析

步骤 5 - 向用户确认:
  ✅ 深度分析完成（层次追踪）
  
  分析范围: 15 个文件，4 层调用
  分析方法: 层次追踪
  关键发现: 存在 runner 统一机制
  
  调用链路图: [见上文]
  
  📋 完整性验证: 7/7 项完成
  
  ❓ 我的分析是否完整？是否有遗漏？
```

## 🔴 方法 C: 层次追踪 + 分批阅读

### 适用场景

```yaml
适用:
  - 复杂/超大项目（> 15 文件）
  - 需要分阶段理解
  - 避免信息过载
```

### 执行步骤

```yaml
步骤 1 - 文件分批:
  批次 1: 核心文件（5-10 个）
  批次 2: 机制文件（5-10 个）
  批次 3: 边缘文件（5-10 个）

步骤 2 - 执行层次追踪:
  对批次 1 执行方法 B

步骤 3 - 批次间确认:
  每批次完成后输出总结，询问是否继续
  
步骤 4 - 超大项目确认计划:
  > 50 文件时，必须向用户说明分析计划
```

---

## 📋 完整性验证清单（7 项）

```yaml
自检清单:
  □ 是否找到了入口点？
  □ 是否追踪了完整调用链路（≥ 3 层）？
  □ 是否发现了统一机制（runner/middleware/hook）？
  □ 是否验证了所有使用方（反向查找）？
  □ 是否理解了数据流向？
  □ 是否绘制了调用链路图？
  □ 是否向用户确认了理解？
```

## ⚠️ 常见错误

### 错误 1: 未发现统一机制

```yaml
错误:
  AI: "需要在 11 个文件中添加回调"（未发现 runner 统一机制）

正确:
  AI: "发现统一机制 createCachedRunner，只需要在 hooks 中添加逻辑"
```

### 错误 2: 凭想象设计方案

```yaml
错误:
  AI: "根据经验，应该是这样实现的..."

正确:
  AI: "让我先读取相关代码文件..."
  AI: "已读取 3 个文件，发现以下关键逻辑..."
```
