## 核心原则

```text
🔴 安全修复原则：验证 > 修复 > 加固 > 全局扫描
```

| 原则 | 说明 |
|------|------|
| 验证优先 | 修复前必须确认漏洞真实存在且可复现/可利用 |
| 最小修复 | 优先最小范围精准修复，避免大范围改动引入新风险 |
| 防御纵深 | 不仅修复当前漏洞，还要考虑多层防御（输入验证 + 输出编码 + 权限校验） |
| 全局扫描 | 同类漏洞全项目扫描（三步扫描 Step 1 强化执行） |
| 信息保密 | 漏洞细节仅在内部文档中记录，不在公开渠道暴露攻击路径 |

## 阶段叠加规则

### 阶段1 — 问题定位（叠加：漏洞分类 + 可利用性评估）

在默认步骤 1.1~1.4 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 1.5 | 确定漏洞类型（参考 OWASP Top 10 / CWE 分类） | 漏洞分类 |
| 1.6 | 关联 CVE 编号（如有已知 CVE） | CVE 信息 |
| 1.7 | 评估可利用性：攻击向量 · 攻击复杂度 · 所需权限 · 用户交互 | 可利用性评估 |
| 1.8 | 评估影响范围：机密性 · 完整性 · 可用性 | 影响评估 |
| 1.9 | 确定严重程度（参考 CVSS 评分维度） | 严重等级 |
| 1.10 | 验证漏洞可复现（通过代码分析确认攻击路径存在） | 复现确认 |

**漏洞严重程度分级：**

| 等级 | CVSS 参考 | 定义 | 响应要求 |
|:----:|:---------:|------|---------|
| 严重 | 9.0~10.0 | 远程无认证可利用 / 数据泄露 / 权限提升 | 立即修复，最高优先级 |
| 高危 | 7.0~8.9 | 需低权限或用户交互可利用 / 局部数据泄露 | 尽快修复 |
| 中危 | 4.0~6.9 | 利用条件较多 / 影响有限 | 正常排期 |
| 低危 | 0.1~3.9 | 理论风险 / 极难利用 | 按优先级排队 |

**CP1 追加输出：**

```text
🔒 安全漏洞概况:
  漏洞类型: [XSS / SQL注入 / CSRF / 权限绕过 / ...]
  OWASP/CWE: [分类编号（如有）]
  CVE 编号: [CVE-YYYY-NNNNN（如有）/ 无]
  严重程度: [严重/高危/中危/低危]
  可利用性: [攻击向量 + 复杂度 + 所需权限 + 用户交互]
  影响范围: [机密性 + 完整性 + 可用性]
  复现状态: [已确认可复现 / 理论可利用]
  攻击路径: [简述攻击路径]
```

### 阶段2 — 修复方案（叠加：防御纵深 + 加固措施）

在默认步骤 2.1~2.4 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 2.5 | 设计修复方案：直接修复漏洞点 | 漏洞修复方案 |
| 2.6 | 设计加固措施：在修复基础上增加防御纵深 | 加固方案 |
| 2.7 | 评估修复是否影响正常功能（安全修复常见的误杀风险） | 误杀评估 |
| 2.8 | 检查相关依赖是否需要升级（安全漏洞常源于过时依赖） | 依赖检查 |
| 2.9 | 制定安全验证计划（修复后如何确认漏洞已无法利用） | 验证计划 |

**CP2 追加输出：**

```text
🛡️ 安全修复策略:
  修复方案: [漏洞修复描述]
  加固措施:
    - 第1层: [输入验证/参数校验]
    - 第2层: [输出编码/转义]
    - 第3层: [权限校验/访问控制]
  依赖升级: [需要/不需要] — [依赖名@版本 → 新版本]
  误杀风险: [低/中/高] — [说明]
  验证方式: [如何确认漏洞已修复]
```

**🔴 安全修复方案规则：**

| 规则 | 说明 |
|------|------|
| 输入验证 | 所有用户输入必须校验（白名单优于黑名单） |
| 输出编码 | 所有输出到浏览器/SQL/命令行的内容必须编码/转义 |
| 最小权限 | 修复后的代码遵循最小权限原则 |
| 禁止硬编码 | 🔴 约束 #3 — 修复代码中绝对禁止硬编码密钥/密码/Token |
| 敏感信息保护 | 日志/错误信息中不暴露敏感数据（密码、Token、内部路径等） |

### 阶段3 — 执行修复 + 验证（叠加：安全验证 + 全局扫描强化）

在默认步骤 3.1~3.6 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 3.7 | 验证漏洞已无法利用（通过代码分析确认攻击路径已切断） | 修复验证 |
| 3.8 | 检查修复是否引入新的安全风险 | 新风险检查 |
| 3.9 | 全项目同类漏洞扫描（三步扫描 Step 1 强化，见下方） | 同类扫描结果 |
| 3.10 | 落地加固措施（如有额外防御层需要实现） | 加固代码 |
| 3.11 | 依赖升级（如有安全相关依赖需要更新） | 依赖更新 |
| 3.12 | 确认敏感信息无泄露（代码中、日志中、错误信息中） | 泄露检查 |

## 三步扫描强化（安全专属）

安全修复的三步扫描在默认 fix 流程基础上强化执行：

### Step 1 强化 — 同类漏洞全局扫描

| 项 | 默认 fix | 安全修复（强化） |
|----|---------|----------------|
| 搜索范围 | 同类 Bug 模式 | 同类漏洞模式 + 关联攻击向量 |
| 搜索深度 | grep 关键模式 | grep 多个变体模式（不同编码/绕过方式） |
| 结果处理 | 一并修复或记录 | 🔴 必须逐一修复，不可仅记录 |

**安全扫描模式参考：**

| 漏洞类型 | grep 搜索模式（示例） |
|---------|-------------------|
| SQL 注入 | 字符串拼接 SQL · 未参数化查询 · `${}` 直接嵌入 SQL |
| XSS | `innerHTML` · `dangerouslySetInnerHTML` · 未转义用户输入输出 |
| 路径穿越 | `../` 未过滤 · 用户输入直接拼接文件路径 |
| 命令注入 | `exec()` · `spawn()` 中拼接用户输入 |
| 硬编码凭证 | `password =` · `secret =` · `apiKey =` · `token =`（非环境变量） |
| 敏感信息日志 | `console.log(password)` · `logger.info(token)` |
| 不安全反序列化 | `JSON.parse(userInput)` 无校验 · `eval()` |

### Step 2 — 数据联动扫描（同默认 fix，无额外强化）

### Step 3 强化 — 声明前复核

在默认声明前复核清单之外，追加安全专属检查：

```text
□ 漏洞已确认无法利用（攻击路径已切断）？
□ 同类漏洞全项目扫描完成，零残留？
□ 修复未引入新的安全风险？
□ 代码中无硬编码凭证（约束 #3）？
□ 日志/错误信息中无敏感数据泄露？
□ 依赖已升级到安全版本（如适用）？
□ 加固措施已落地（防御纵深）？
```

## 常见漏洞修复参考

### OWASP Top 10 速查

| # | 漏洞类型 | 修复要点 |
|:-:|---------|---------|
| A01 | 权限控制失效 | 服务端强制权限校验 · 默认拒绝 · RBAC/ABAC |
| A02 | 加密失败 | 强加密算法 · 密钥管理 · TLS 传输 |
| A03 | 注入 | 参数化查询 · 输入验证 · ORM |
| A04 | 不安全设计 | 威胁建模 · 安全设计模式 |
| A05 | 安全配置错误 | 最小化安装 · 默认安全 · 自动化配置审查 |
| A06 | 易受攻击的组件 | 依赖扫描 · 及时更新 · SCA 工具 |
| A07 | 身份认证失败 | MFA · 密码策略 · 会话管理 |
| A08 | 数据完整性失败 | 签名验证 · 安全反序列化 · CI/CD 安全 |
| A09 | 日志与监控不足 | 安全事件日志 · 告警 · 审计追踪 |
| A10 | SSRF | URL 白名单 · 网络分段 · 禁止内网访问 |

### 各类型漏洞修复模式

#### SQL 注入

| 修复方式 | 说明 |
|---------|------|
| ✅ 参数化查询 | 使用 `?` 或 `$1` 占位符 + 参数绑定 |
| ✅ ORM 方法 | 使用 ORM 提供的查询构建器 |
| ✅ 输入验证 | 白名单校验（类型 + 长度 + 格式） |
| ❌ 字符串拼接 | 禁止将用户输入直接拼接到 SQL 语句 |
| ❌ 黑名单过滤 | 仅过滤 `'` `"` 等字符不可靠，容易绕过 |

#### XSS（跨站脚本）

| 修复方式 | 说明 |
|---------|------|
| ✅ 输出编码 | HTML 实体编码 / URL 编码 / JS 编码（按上下文） |
| ✅ CSP 头 | Content-Security-Policy 限制脚本来源 |
| ✅ 模板引擎自动转义 | 使用默认自动转义的模板引擎 |
| ❌ innerHTML | 禁止将用户输入赋值给 innerHTML |
| ❌ eval() | 禁止 eval 用户输入 |

#### CSRF（跨站请求伪造）

| 修复方式 | 说明 |
|---------|------|
| ✅ CSRF Token | 表单/请求中携带随机 Token 并服务端验证 |
| ✅ SameSite Cookie | 设置 Cookie 的 SameSite=Strict 或 Lax |
| ✅ Referer/Origin 校验 | 服务端检查请求来源 |
| ❌ 仅依赖 Cookie | Cookie 会被浏览器自动发送，不能作为身份凭证的唯一校验 |

#### 权限绕过

| 修复方式 | 说明 |
|---------|------|
| ✅ 服务端权限校验 | 每个 API 端点强制校验当前用户权限 |
| ✅ 默认拒绝 | 未明确授权 = 拒绝访问 |
| ✅ 对象级权限 | 检查用户是否有权访问**特定资源**（不仅是功能权限） |
| ❌ 前端权限控制 | 仅在前端隐藏按钮/菜单不构成权限控制 |
| ❌ 可预测 ID | 使用自增 ID 暴露资源，应使用 UUID 或加权限校验 |

## 敏感信息保护规则

| 场景 | 规则 | 违规示例 |
|------|------|---------|
| 代码中 | 🔴 约束 #3 — 禁止硬编码 | `const apiKey = "sk-xxxx"` |
| 日志中 | 禁止记录密码/Token/密钥 | `logger.info(\`User password: ${pwd}\`)` |
| 错误响应中 | 禁止暴露内部路径/堆栈 | 返回 `Error in /home/app/src/auth.js:42` |
| 文档中 | 使用 `<REDACTED>` 替代敏感值 | 漏洞报告中写明真实密钥 |
| 版本控制中 | 敏感文件加入 `.gitignore` | `.env` 文件被提交到仓库 |

## 报告追加章节

安全修复类任务的报告（使用 `templates/report-fix.md`）追加以下章节：

| 章节 | 内容 |
|------|------|
| 漏洞概况 | 漏洞类型 + OWASP/CWE + CVE + 严重程度 + 可利用性 |
| 攻击路径 | 攻击向量描述（🔴 仅内部文档记录，不在公开渠道暴露） |
| 修复方案 | 漏洞修复 + 防御纵深加固措施 |
| 修复验证 | 漏洞无法利用的确认方式和结果 |
| 全局扫描结果 | 同类漏洞扫描范围 + 发现数量 + 处理结果 |
| 依赖更新 | 已升级的安全相关依赖清单（如有） |
| 加固建议 | 可进一步加固的方向（CSP · WAF · 监控 · 审计等） |

## 与默认 fix 流程的差异

| 对比项 | 默认 fix | 安全修复（本 checklist） |
|--------|---------|----------------------|
| CP 流程 | CP1 → CP2 → 执行 | CP1 → CP2 → 执行（不变） |
| 定位深度 | 定位根因 | 定位根因 + 漏洞分类 + 可利用性评估 |

| 硬编码检查 | 约束 #3 覆盖 | 约束 #3 + 显式敏感信息全面检查 |
| 信息保护 | 无特殊要求 | 🔴 漏洞细节仅内部记录，`<REDACTED>` 替代敏感值 |
| 验证要求 | 修复有效即可 | 修复有效 + 无绕过 + 无新风险 |
