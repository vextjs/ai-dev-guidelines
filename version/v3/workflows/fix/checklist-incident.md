# 事故处理 Checklist

> `fix` 工作流的场景变体（`RULES.md §10` fix 子类型：事故处理）。
> 当 N04 意图识别将任务判定为 `fix > 事故处理` 时，由路由表自动加载本 checklist，
> 叠加到 `fix/README.md` 默认流程上执行。CP1→CP2 不变，每个阶段增加事故处理专属步骤。

## 适用场景

> 🔴 本 checklist 的加载由 `RULES.md §2` 意图识别系统驱动，**不依赖关键词匹配**。
> AI 基于语义理解判断用户需求属于"事故处理"二级分类后，自动叠加本 checklist。

| 场景 | 用户消息示例 |
|------|------------|
| 线上故障响应 | "线上出了事故，服务不可用" |
| 紧急问题处理 | "P0 故障需要紧急修复" |
| 服务宕机恢复 | "服务宕机了，需要立即处理" |
| 生产环境异常 | "生产环境出现大量报错" |
| 事故复盘分析 | "对昨天的事故做复盘" |

## 核心原则

```text
🔴 事故处理优先级：止血 > 根因 > 复盘 > 防复现
```

| 原则 | 说明 |
|------|------|
| 止血优先 | 先恢复服务可用性，再深入排查根因 |
| 时间线记录 | 从发现到修复的每个关键节点必须记录时间 |
| 无责文化 | 复盘目的是改进流程和系统，不是追究个人责任 |
| 防复现闭环 | 修复必须包含防止同类问题再次发生的措施 |

## 阶段叠加规则

### 阶段1 — 问题定位（叠加：故障记录 + 时间线）

在默认步骤 1.1~1.4 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 1.5 | 记录故障发现时间和报告来源（用户反馈 / 监控告警 / 自动发现） | 故障来源 |
| 1.6 | 收集故障基本信息：错误日志、监控数据、用户报告 | 故障数据 |
| 1.7 | 评估影响范围：受影响用户数 · 影响的功能/模块 · 影响时长（持续中 / 已恢复） | 影响评估 |
| 1.8 | 评估严重程度并标注等级（见下方分级表） | 严重等级 |
| 1.9 | 开始建立时间线（每个关键操作标注时间戳） | 时间线草稿 |

**故障严重程度分级：**

| 等级 | 定义 | 响应要求 |
|:----:|------|---------|
| P0 | 核心服务不可用 / 数据丢失 / 安全事件 | 立即响应，最高优先级 |
| P1 | 核心功能严重降级 / 大范围影响 | 尽快响应，当天解决 |
| P2 | 非核心功能异常 / 小范围影响 | 正常排期修复 |
| P3 | 体验问题 / 边缘场景 | 按优先级排队 |

**CP1 追加输出：**

```text
🚨 事故概况:
  严重等级: [P0/P1/P2/P3]
  发现时间: [YYYY-MM-DD HH:MM]
  报告来源: [用户反馈 / 监控告警 / ...]
  影响范围: [受影响用户数 · 功能/模块 · 时长]
  当前状态: [持续中 / 已临时恢复 / 已修复]
  根因方向: [初步定位结果]
```

### 阶段2 — 修复方案（叠加：止血方案 + 回滚评估）

在默认步骤 2.1~2.4 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 2.5 | 评估是否有快速止血方案（回滚 / 配置降级 / 功能开关 / 流量切换） | 止血方案 |
| 2.6 | 如有止血方案，先输出止血方案供确认；否则直接走根因修复 | 止血 or 修复 |
| 2.7 | 评估回滚可行性（是否有可回滚的版本 / 回滚是否会导致数据问题） | 回滚评估 |
| 2.8 | 制定防复现措施（监控告警 / 限流 / 兜底逻辑 / 代码防御） | 防复现清单 |

**CP2 追加输出：**

```text
🛠️ 修复策略:
  止血方案: [有/无] — [回滚/降级/开关/切换]
  回滚可行性: [可回滚(说明版本) / 不可回滚(说明原因)]
  根因修复: [修复方案概述]
  防复现措施:
    - [措施1: 描述]
    - [措施2: 描述]
```

**🔴 止血与根因修复的执行顺序：**

| 场景 | 执行策略 |
|------|---------|
| 服务仍不可用（P0 持续中） | 先执行止血方案恢复服务 → 再排查根因做永久修复 |
| 服务已恢复但有隐患 | 直接做根因修复 |
| 止血方案 = 根因修复 | 合并执行 |

### 阶段3 — 执行修复 + 验证（叠加：时间线 + 验证强化 + 防复现落地）

在默认步骤 3.1~3.6 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 3.7 | 完善时间线记录（从发现到修复完成的完整时间线） | 完整时间线 |
| 3.8 | 验证修复有效（确认故障不再复现） | 修复验证 |
| 3.9 | 落地防复现措施（新增监控 / 告警 / 限流 / 防御代码） | 防复现代码 |
| 3.10 | 执行 5-Why 根因分析（见下方） | 5-Why 结论 |
| 3.11 | 记录事故复盘（经验教训 + 改进措施） | 复盘记录 |

## 5-Why 根因分析法

在阶段3修复完成后执行，深入追溯问题根本原因：

```text
模板:
  问题: [故障现象]
  Why 1: 为什么发生？ → [直接原因]
  Why 2: 为什么会有这个直接原因？ → [间接原因]
  Why 3: 为什么没有预防？ → [预防缺失]
  Why 4: 为什么缺失预防？ → [流程/系统问题]
  Why 5: 为什么流程/系统有此问题？ → [根本原因]
```

**示例：**

```text
  问题: 用户服务不可用
  Why 1: 为什么不可用？ → 数据库连接池耗尽
  Why 2: 为什么连接池耗尽？ → 慢查询导致连接长时间占用
  Why 3: 为什么有慢查询？ → 新上线的功能缺少索引
  Why 4: 为什么没加索引？ → 代码审查未覆盖 SQL 性能
  Why 5: 为什么审查不覆盖？ → 缺少数据库变更审查流程

  根因: 缺少数据库变更审查流程
  改进: 建立数据库变更 checklist + 上线前 SQL 审查
```

**🔴 5-Why 规则：**

| 规则 | 说明 |
|------|------|
| 追到系统/流程层面 | 不能停在"某人忘了"，要追到"为什么系统没有防住" |
| 客观分析 | 不归咎个人，关注系统性改进 |
| 可执行结论 | 根因必须对应可执行的改进措施 |
| 至少 3 层 | 最少追问 3 次 Why，通常 5 次可到根因 |

## 时间线记录格式

```text
📅 事故时间线:
  [HH:MM] 🔴 故障发现 — [发现方式：监控告警/用户反馈/...]
  [HH:MM] 📋 问题确认 — [确认故障影响范围]
  [HH:MM] 🔍 开始排查 — [排查方向]
  [HH:MM] 🎯 根因定位 — [根因描述]
  [HH:MM] 🛠️ 止血执行 — [止血操作（如有）]
  [HH:MM] ✅ 服务恢复 — [恢复确认]
  [HH:MM] 🔧 根因修复 — [修复代码提交]
  [HH:MM] ✅ 修复验证 — [验证通过]
  [HH:MM] 📄 复盘完成 — [复盘报告输出]
```

> 每个关键操作都必须标注时间戳。如果是 AI 辅助修复，时间线记录 AI 的分析和执行节点。

## 影响评估维度

| 维度 | 评估内容 | 记录格式 |
|------|---------|---------|
| 用户影响 | 受影响用户数量 / 比例 | N 用户 / 占比 X% |
| 功能影响 | 哪些功能不可用 / 降级 | 功能清单 |
| 数据影响 | 是否有数据丢失 / 不一致 | 有/无 + 详情 |
| 时长影响 | 故障持续时间 | 从 HH:MM 到 HH:MM，共 N 分钟 |
| 业务影响 | 收入损失 / SLA 违约 / 品牌声誉 | 定性或定量描述 |

## 改进措施要求

事故修复完成后，必须输出改进措施清单，每条措施需包含：

| 字段 | 说明 | 必填 |
|------|------|:----:|
| 措施内容 | 具体要做什么 | ✅ |
| 类型 | 短期(1周) / 中期(1月) / 长期(3月) | ✅ |
| 优先级 | 🔴高 / 🟡中 / 🟢低 | ✅ |
| 责任方 | 谁来执行（团队/角色） | ✅ |
| 验收标准 | 怎样算做完了 | ✅ |
| 预计完成 | 目标完成日期 | 🟡建议 |

**改进措施模板：**

```text
📋 改进措施:
| # | 措施 | 类型 | 优先级 | 责任方 | 验收标准 |
|:-:|------|:----:|:------:|--------|---------|
| 1 | 添加数据库慢查询监控告警 | 短期 | 🔴 | 运维 | 慢查询>1s 自动告警 |
| 2 | 建立数据库变更审查流程 | 中期 | 🔴 | 研发 | 所有 DDL 需经审查 |
| 3 | 引入连接池自动扩容机制 | 长期 | 🟡 | 架构 | 连接池可按负载自动扩缩 |
```

## 事故处理专属验证清单

在 N13 合规自检阶段，叠加以下检查：

| # | 检查项 | 通过标准 |
|:-:|--------|---------|
| I1 | 时间线完整 | 从发现到修复完成的完整时间线，每个节点有时间戳 |
| I2 | 影响评估准确 | 影响范围和用户数量已明确记录 |
| I3 | 严重等级标注 | 已标注 P0/P1/P2/P3 等级 |

| I7 | 防复现措施 | 至少 1 条可执行的防复现措施 |

| I9 | 三步扫描完成 | fix 流程的修复后三步扫描已执行 |
| I10 | 止血 vs 根因 | 如先止血后修复，两个阶段都有记录 |

## 报告追加章节

事故类任务的报告（使用 `templates/report-fix.md`）追加以下章节：

| 章节 | 内容 |
|------|------|
| 事故概况 | 严重等级 + 发现时间 + 报告来源 + 当前状态 |
| 影响评估 | 用户/功能/数据/时长/业务五维度评估 |
| 时间线 | 完整事故时间线（每个节点有时间戳） |
| 止血记录 | 止血方案 + 执行过程 + 恢复确认（如有） |
| 5-Why 根因分析 | 至少 3 层 Why 追溯 + 根本原因结论 |
| 改进措施 | 分短/中/长期的可执行改进清单 |
| 经验教训 | 本次事故的关键教训 + 可推广的经验 |

## 与默认 fix 流程的差异

| 对比项 | 默认 fix | 事故处理（本 checklist） |
|--------|---------|----------------------|
| CP 流程 | CP1 → CP2 → 执行 | CP1 → CP2 → 执行（不变） |
| 定位深度 | 定位根因即可 | 5-Why 追到系统/流程层面 |
| 影响评估 | 简要说明 | 五维度详细评估 |
| 时间线 | 无要求 | 🔴 强制，每个节点有时间戳 |
| 止血方案 | 无 | 评估止血可行性，P0 优先止血 |
| 防复现 | 三步扫描覆盖 | 三步扫描 + 显式防复现措施清单 |
| 改进措施 | 后续建议 | 结构化改进措施（类型/优先级/责任方/验收标准） |
| 复盘 | 无 | 🔴 强制，无责文化 |
