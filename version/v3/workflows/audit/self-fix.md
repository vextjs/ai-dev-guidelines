# 规范自修复机制（self-fix）

> 反应式问题修复：检测规范问题并按分级策略执行修复。
> 本文件由 `RULES.md§11` 和 `workflows/audit/README.md` 阶段 4 引用。

**版本**: v3.0.0
**最后更新**: 2026-03-10

---

## 概述

self-fix 是一套反应式修复机制，让 AI 在发现规范问题时能够：

1. **自动检测** — 识别规范冲突、过时、缺失、冗余
2. **分级修复** — 按安全等级选择自动/半自动/辅助修复
3. **闭环验证** — 修复后回写验证，防止断层
4. **记录追溯** — 所有修复操作留有记录

**与 audit 的关系**：audit 是主动式全面体检（整体审查），self-fix 是反应式定点修复（发现即修）。audit 可触发 self-fix，self-fix 也可独立触发。

---

## 触发条件

### 1. audit 衔接触发

| 条件 | 行为 |
|------|------|
| audit 发现 🔴 必须修复问题 | 自动进入 self-fix 流程 |
| audit 发现 🟡 建议修复问题 | 输出变更计划 → 用户确认后进入 self-fix |

### 2. N13 合规检查触发

| 条件 | 行为 |
|------|------|
| N13 形式合规（FC）检查不通过 | 自动修复（补充缺失字段、修正格式） |
| N13 实质合规（SC）检查不通过 | 自动修复（安全类）或报告用户（内容类） |
| 连续 2 次同类偏差 | 升级触发：先执行 self-fix，再启动 audit 定向审查该维度 |

### 3. AI 执行中发现规范冲突（贯穿式触发）

| 条件 | 行为 |
|------|------|
| 读取规范文件时发现版本号不一致 | 立即触发版本号同步修复 |
| 读取规范文件时发现引用链接失效 | 记录问题，任务完成后修复 |
| 执行规则与规范定义矛盾 | 暂停执行 → 报告冲突 → 等待用户决策 |

### 4. 用户反馈触发

| 用户表述模式 | AI 识别与响应 |
|-------------|-------------|
| "你为什么不按规范来？" | 检查相关规范 → 是 AI 违反则道歉并修正行为 → 是规范不清则建议修复规范 |
| "规范不对""这条规范有问题" | 定位相关规范 → 分析问题 → 生成修复建议 |
| "需要新增一条规范" | 评估必要性 → 确认位置 → 生成新增方案 → 等待用户确认 |
| "规范中没有定义 XXX" | 检查是否确实缺失 → 确认后生成补充方案 |
| 用户重复同一个需求（第 2 次以上） | 检查上次是否理解偏差 → 分析规范表述是否模糊 → 建议修复 |

## 检测类型

### 冲突检测

| 检测项 | 方法 | 示例 |
|--------|------|------|
| 规范间矛盾 | 对比两个规范文件中对同一规则的定义 | RULES.md 说"CP2 后生成实施方案"，workflow 说"CP2 后开始写代码" |
| 版本号不一致 | 扫描所有规范文件的 `**版本**:` 字段 | RULES.md v3.0.0 但 document-sync.md v2.9.0 |
| 约束编号错位 | grep 搜索 `#N` 标注，与 RULES.md§4 对比 | workflow 引用 `#18 修复扫描` 但 RULES.md 中 #18 已变更含义 |

### 过时检测

| 检测项 | 方法 | 示例 |
|--------|------|------|
| 引用链接失效 | 遍历 `[text](path)` 链接，验证目标文件存在 | 链接指向已删除的 v2 文件 |
| 废弃术语残留 | grep 搜索已废弃的术语/概念 | 仍然引用"快速模式"（已被 v2.13.0 废弃） |
| 日期过时 | 检查"最后更新"字段 | 文件内容已变更但日期未更新 |

### 缺失检测

| 检测项 | 方法 | 示例 |
|--------|------|------|
| 约束未落地 | 对比 RULES.md§4 的 22 条与各 workflow 中的引用 | 约束 #18 在 fix/README.md 中无体现 |
| CP 定义缺失 | 对比 RULES.md§3 与 confirmation-points.md | RULES.md 提到 CP6 但 CP 文件中无定义 |
| 文件缺失 | 对比 RULES.md§10 路由表与实际文件 | 路由表列出 `audit/README.md` 但文件不存在 |

### 冗余检测

| 检测项 | 方法 | 示例 |
|--------|------|------|
| 规则重复定义 | 对比多个文件中对同一规则的定义 | CP 规则在 RULES.md、confirmation-points.md、build/README.md 中三处重复 |
| 废弃内容残留 | 检查标注为"已废弃"但未删除的内容 | v2 遗留的旧格式说明仍在文件中 |

> 🔴 **冗余检测的谨慎原则**：发现"疑似重复"时，必须验证是否真的是重复内容还是同概念不同详略视角。事故参考：O1(spec snippets) — 错误地将不同视角的内容判定为冗余。

## 修复分级

### 自动修复（安全类 — AI 直接执行，无需用户确认）

| 适用场景 | 修复方式 | 示例 |
|---------|---------|------|
| 版本号不一致 | 全量同步所有规范文件版本号 | 统一更新为最新版本号 |
| "最后更新"日期不一致 | 同步日期 | 更新为当前日期 |
| 约束编号引用错误 | 修正编号 | `#18` → 正确编号 |
| 格式缺失（缺少必填字段） | 补充缺失字段 | 报告头部缺 Agent 字段 → 补充 |
| 断开的内部链接 | 修正路径 | `core/xxx.md` → `version/v3/xxx.md` |

> 🔴 **自动修复约束**：仅限上述明确列出的安全类操作。执行后必须在报告中记录修复内容。

### 半自动修复（内容类 — 需用户确认）

| 适用场景 | 修复方式 | 用户交互 |
|---------|---------|---------|
| 规范表述不清 | 生成改进方案 | 输出变更计划 → 等待 [C]/[M]/[S] |
| 术语不一致 | 列出所有不一致处 + 统一建议 | 展示对比 → 等待确认统一为哪个 |
| 规则缺失 | 生成补充方案 + 建议写入位置 | 输出方案 → 等待确认 |
| 流程步骤遗漏 | 生成补充步骤 | 输出变更计划 → 等待确认 |
| 内容过时需更新 | 生成更新方案 | 输出新旧对比 → 等待确认 |

**半自动修复的确认格式：**

```text
🔧 规范修复建议

## 问题
[问题描述 — 精确到文件+位置]

## 修复方案
[具体修复内容]

## 影响范围
[列出受影响的文件]

---

❓ 是否执行修复？

[C] Confirm — 执行修复
[M] Modify — 调整修复方案
[S] Skip — 跳过（记录原因）
```

### 辅助修复（架构类 — 生成指南，用户手动执行）

| 适用场景 | AI 输出 | 用户行动 |
|---------|--------|---------|
| 规范体系重大变更 | 变更影响分析 + 迁移指南 | 用户按指南手动执行 |
| 新增工作流类型 | 工作流设计建议 + 文件结构 | 用户确认后 AI 协助创建 |
| 约束体系调整 | 约束增删方案 + 影响评估 | 用户决策后执行 |

## 修复执行流程

```text
触发（4 种触发源之一）
  ↓
识别问题类型（冲突/过时/缺失/冗余）
  ↓
确定修复分级（自动/半自动/辅助）
  ↓
├── 自动修复 → 直接执行 → 记录修复内容
├── 半自动修复 → 输出方案 → 等待用户确认 → 执行 → 记录
└── 辅助修复 → 输出指南 → 用户决策 → 协助执行 → 记录
  ↓
闭环验证（6 项检查点）
  ↓
记录修复历史
```

---

## 闭环验证（修复后必须执行）

修复完成后，必须逐项确认以下 6 个检查点，防止修复断层：

| # | 检查点 | 说明 |
|:-:|--------|------|
| V1 | 修复目标文件已更新 | 被修复的文件确实包含了修复后的内容 |
| V2 | 关联文件已同步 | 引用被修复内容的其他文件已检查并同步（约束 #17） |
| V3 | 交叉引用有效 | 修复后的文件间链接仍然有效（约束 #11） |
| V4 | 版本号已同步 | 如涉及版本变更，所有文件版本号已统一 |
| V5 | 修复记录已写入 | 修复内容已记录到报告中（见下方记录格式） |
| V6 | 反复问题检查 | 本次修复的问题是否在历史中出现过？如是，是否需要升级修复力度？ |

> **知识库断层定义**：修复只写入了执行入口（RULES.md / workflow README），但未回写到检测/触发规则中，导致未来审查无法用最新规则检测。V3 检查点专门防止此类断层。

## 反复问题升级机制

当同类问题出现 ≥2 次时，触发升级分析：

| 出现次数 | 行为 |
|:--------:|------|
| 1 次 | 正常修复 + 记录 |
| 2 次 | 🟡 升级：分析根因（为什么会再次出现？） → 加强检测规则 |
| ≥3 次 | 🔴 升级：审查是否需要改变规范结构或约束等级 → 建议用户决策 |

**升级分析输出格式：**

```text
⚠️ 反复问题检测

## 问题
[问题描述]

## 历史记录
- 第 1 次: [日期] — [修复方式]
- 第 2 次: [日期] — [修复方式]（本次）

## 根因分析
[为什么修复后问题再次出现？]

## 升级建议
[建议如何从根本上解决 — 如加强检测规则、调整约束等级、修改流程等]
```

## 修复记录格式

每次 self-fix 执行后，修复内容记录到**审查报告**（`templates/report-audit.md`）的修复记录章节中：

```markdown
## 修复记录

### 修复 #1 — [简述]

| 项 | 内容 |
|----|------|
| **触发方式** | audit 衔接 / N13 检查 / 贯穿式发现 / 用户反馈 |
| **问题类型** | 冲突 / 过时 / 缺失 / 冗余 |
| **修复分级** | 自动 / 半自动 / 辅助 |
| **问题描述** | [精确到文件+位置的问题描述] |
| **修复内容** | [具体修改了什么] |
| **影响文件** | [列出所有被修改的文件] |
| **用户确认** | 自动修复（无需确认）/ ✅ 已确认 / ⏭️ 跳过 |
| **闭环验证** | V1✅ V2✅ V3✅ V4✅ V5✅ V6✅ |
| **反复检查** | 首次 / 第 N 次（升级：[升级措施]） |
```

如果修复发生在非 audit 流程中（贯穿式触发或 N13 触发），修复记录追加到**当前任务的报告**末尾，格式相同。

## 与主线流程的集成点

| 集成点 | 位置 | 说明 |
|--------|------|------|
| 意图识别 | N04 | `audit` 意图类型 → N06 路由到 N09A |
| 审查流程 | N09A（audit） | audit 阶段 4 衔接 self-fix |
| 合规检查 | N13 | FC/SC 检查不通过 → 触发 self-fix |
| 贯穿检测 | 全流程 | AI 执行中发现规范冲突 → 随时触发 |
| 记忆记录 | N14 | 修复记录写入记忆的 `💡 关键决策` |

### 贯穿式触发的执行优先级

当 AI 在正常任务执行过程中发现规范问题时：

| 问题严重程度 | 处理策略 |
|:----------:|---------|
| 不影响当前任务 | 记录问题 → 当前任务完成后再处理 |
| 影响当前任务但可绕过 | 记录问题 → 用降级策略完成任务 → 任务后修复 |
| 阻塞当前任务 | 暂停任务 → 报告冲突 → 等待用户决策 |

> 🔴 **原则**：self-fix 不应干扰用户的主任务。非阻塞性问题延后处理，阻塞性问题立即报告。

## 安全约束

| 约束 | 说明 |
|------|------|
| 约束 #1 | 半自动/辅助修复涉及的文件修改必须等待用户确认 |
| 约束 #11 | 修复规范文件后必须执行交叉验证 |
| 约束 #17 | 修复后检查所有关联文件并同步 |
| 修复范围 | 仅修复 `version/v3/` 目录下的规范文件，不修改 `version/v2/`（v2 归档）、项目源码、用户文件 |
| 修复上限 | 单次 self-fix 最多修复 5 个文件，超出建议拆分为多次 |
| 不确定时 | 询问用户而非擅自判断 |
