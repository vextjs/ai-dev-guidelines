> `audit` 工作流阶段2 按需读取。定义每个审查维度的目标、检查项、检查方法和输出格式。
> 全面体检时扫描全部 15 维度；定向审查时按需选择子集。

**版本**: v3.0.0
**最后更新**: 2026-03-12

## 分组 A — 结构与覆盖

### D1 — 文件结构完整性 🔴

| 项 | 内容 |
|----|------|
| **审查目标** | 规划的文件是否都存在？是否有孤立文件？ |
| **审查标准** | `RULES.md§10` 路由表列出的工作流文件、`RULES.md§6` 引用的模板文件 |

**检查项：**

| # | 检查内容 | 方法 |
|:-:|---------|------|
| 1 | RULES.md 是否存在且可读 | `read_file` 验证 |
| 2 | `workflows/` 下每个路由表指向的文件是否存在 | 逐个 `read_file` 或 `list_directory` |
| 3 | `templates/` 下报告/记忆模板是否齐全 | `list_directory` 对照模板清单 |
| 4 | common/ 下 confirmation-points.md · document-sync.md · task-memory.md 是否存在 | `list_directory` |
| 5 | 是否有规范目录中存在但无任何文件引用的孤立文件 | `grep` 搜索文件名在其他文件中的引用 |

**输出格式：**

```text
文件完整性: 预期 N 个文件 / 实际存在 M 个 / 缺失 K 个
  ✅ 存在: [文件列表]
  🔴 缺失: [文件列表]
  💡 孤立: [文件列表（存在但未被引用）]
```

### D2 — 约束覆盖率 🔴

| 项 | 内容 |
|----|------|
| **审查目标** | 23 条约束是否在对应节点/文件中有落地执行规则？ |
| **审查标准** | `RULES.md§4` 的 23 条约束编号（#1~#23） |

**检查项：**

| # | 检查内容 | 方法 |
|:-:|---------|------|
| 1 | 每条约束是否在至少一个 workflow README 中被引用 | `grep` 搜索 `#N` 或约束关键词 |
| 2 | 约束的触发位置是否与 workflow 的约束触发清单一致 | 对比 RULES.md 约束定义与各 workflow 的约束触发清单 |
| 3 | P0 约束是否在所有相关工作流中都有强制执行点 | 逐条核对 |
| 4 | 是否有约束被定义但从未在任何执行文件中被引用（"死约束"） | `grep` 全项目搜索 |

**输出格式：**

```text
约束覆盖率: 23 条约束 / 已覆盖 N 条 / 未覆盖 K 条
  各约束覆盖详情:
  | 约束 | 优先级 | 引用文件数 | 状态 |
  | #1   | P0     | 4         | ✅    |
  | #X   | P1     | 0         | 🔴 未覆盖 |
```

### D5 — 交叉引用有效性 🔴

| 项 | 内容 |
|----|------|
| **审查目标** | 文件间的交叉引用链接是否指向正确位置 |
| **审查标准** | 所有 Markdown 相对链接 `[text](path)` 和锚点 `[text](path#anchor)` |

**检查项：**

| # | 检查内容 | 方法 |
|:-:|---------|------|
| 1 | Markdown 相对链接目标文件是否存在 | 遍历所有 `[text](path)` → `read_file` 或 `list_directory` 验证 |
| 2 | 锚点链接的目标章节是否存在 | 读取目标文件搜索对应标题 |
| 3 | Mermaid `click` 链接的路径是否正确 | `grep` 搜索 `click` 语句 → 验证路径 |
| 4 | "相关文档"章节中的链接是否全部有效 | 逐文件检查尾部链接列表 |

**输出格式：**

```text
交叉引用有效性: N 个链接 / 有效 M 个 / 断链 K 个
  🔴 断链清单:
  | 来源文件 | 链接文本 | 目标路径 | 问题 |
  | build/README.md | CP 定义 | ../common/confirmation-points.md | ✅ 有效 |
  | fix/README.md | report-fix.md | ../../templates/report-fix.md | 🔴 文件不存在 |
```

### D6 — 术语一致性 🟡

| 项 | 内容 |
|----|------|
| **审查目标** | 相同概念在不同文件中是否使用相同术语 |
| **审查标准** | 规范中的核心术语表 |

**检查项：**

| # | 检查内容 | 关注术语 |
|:-:|---------|---------|
| 1 | "确认点"与"CP"是否统一 | 不能交替使用"确认点"和"检查点"指代 CP |
| 2 | "工作流"与"流程"是否一致 | 同一语境下应使用统一术语 |
| 3 | "记忆"与"索引"是否区分明确 | 记忆文件 = .ai-memory 下的文件，不混淆为报告 |
| 4 | "报告"与"文档"是否区分 | 报告 = reports/ 下，文档 = 归档文档/规范文件 |
| 5 | "约束"与"规则"是否区分 | 约束 = RULES.md§4 的 23 条，规则 = 更宽泛的执行规定 |
| 6 | CP/IP 的命名是否一致 | CP = Confirmation Point · IP = Inquiry Point |
| 7 | "三步扫描"与"修复后扫描"是否统一 | 应统一为"修复后三步扫描" |

**检查方法：**

1. `grep` 搜索可能不一致的术语变体
2. 对比同一概念在 RULES.md、各 workflow README、common/ 文件中的表述
3. 特别关注中英文混用场景（如"确认点"vs"CP"应明确等价关系）

**输出格式：**

```text
术语一致性:
  | 标准术语 | 变体表述 | 出现位置 | 状态 |
  | 修复后三步扫描 | 三步扫描 | fix/README.md L145 | 🟡 建议统一 |
  | 确认点(CP) | 检查点 | build/README.md L88 | 🔴 混淆（检查点应指编码检查点） |
```
