# 确认点机制

> 关键决策必须经过用户确认的流程控制。本文件是 `RULES.md§3` 的完整展开。

**版本**: v3.0.0
**最后更新**: 2026-03-10

---

## 📋 概述

确认点（Confirmation Point, CP）是 AI 执行流程中必须暂停并等待用户确认的关键节点。

- CP1~CP3：🔴 强制（dev/fix 工作流）
- CP4~CP6：按需（满足条件时触发）
- IP1~IP2：询问点（确定产物范围）

---

## 🎯 CP 定义

### CP1: 需求确认

```yaml
触发时机: 需求分析完成后，进入方案设计前
适用工作流: build · fix

必须展示:
  - 需求复述（用 AI 的话重述核心目标）
  - 具体要求（逐条列出）
  - 关键约束
  - 改造范围（需要改 / 不需要改）

输出格式: |
  📋 需求理解确认

  我理解您的需求是：

  1. **核心目标**: [一句话描述]
  2. **具体要求**:
     - [要求1]
     - [要求2]
  3. **关键约束**:
     - [约束1]
  4. **改造范围**:
     - 需要改: [列举]
     - 不需要改: [列举]

  ❓ 我的理解是否正确？

  [✅ 确认] [✏️ 修改]

用户响应处理:
  ✅ 确认: 继续到技术方案设计（唯一合法下一步）
  ✏️ 修改: 根据反馈重新理解，再次输出 CP1
```

### CP2: 方案确认

```yaml
触发时机: 技术方案设计完成后，进入实施方案生成前
适用工作流: build · fix
前置条件: CP1 已确认

必须展示:
  - 技术方案概述
  - 修改文件清单（表格：文件 | 操作 | 说明）
  - 依赖变更
  - 🔴 P0 清单（如有高危操作）

输出格式: |
  📋 技术方案确认

  ## 方案概述
  [简要描述]

  ## 修改清单
  | 文件 | 操作 | 说明 |
  |-----|------|------|
  | file1.ts | 新增 | 添加限流中间件 |
  | file2.ts | 修改 | 注册中间件 |

  ## 依赖变更
  - 新增: package@^1.0.0

  🔴 P0 操作清单（如有）:
  [展示危险操作，见下方 P0 章节]

  ❓ 方案是否确认？

  [✅ 确认] [✏️ 修改] [❌ 取消]

用户响应处理:
  ✅ 确认: 生成实施方案 + IMPLEMENTATION-PLAN.md → 输出 CP3（⚠️ 唯一合法下一步）
  ✏️ 修改: 根据反馈调整方案，再次输出 CP2
  ❌ 取消: 终止流程，记忆记录取消原因
```

> 🔴 **CP2→CP3 过渡规则（FIX-015）**:
> - CP2 确认 = 用户认可技术方案的**设计思路**，**不等于授权执行代码**
> - CP2 确认后的唯一合法下一步：生成 `03-实施方案/` + `IMPLEMENTATION-PLAN.md` → 输出 CP3 等待确认
> - 🚫 禁止将 CP2 的"确认"解读为"可以开始写代码"
> - 事故参考：issue#15 — AI 在 CP2 确认后直接执行代码变更，被用户发现后回滚

### CP3: 实施方案确认

> 🔴 **CP3 是执行代码的唯一授权点**：只有 CP3 确认后，AI 才被授权执行代码变更。

```yaml
触发时机: 实施方案生成后，执行代码修改前
适用工作流: build（fix 不经过 CP3，CP2 确认后直接执行修复；≥5 文件建议切 dev）
前置条件: CP2 已确认 + 03-实施方案/ 和 IMPLEMENTATION-PLAN.md 已生成（🔴 FIX-015 硬阻塞）

必须展示:
  - 实施前检查清单
  - 每个文件的变更摘要（文件名 + 操作 + 行数估算）
  - 实施步骤（按依赖顺序）
  - IMPLEMENTATION-PLAN.md 内容
  - 验证清单

输出格式: |
  📋 实施方案确认

  ## 实施前检查
  - [ ] 前置条件已满足
  - [ ] 修改范围已确认
  - [ ] 回滚方案已准备

  ## 变更清单
  - src/middleware/rate-limit.ts (新增, ~85 行)
  - config/rate-limit.config.ts (新增, ~25 行)
  - src/app.ts (修改, +5 行)

  ## 实施步骤
  1. 创建配置文件
  2. 创建中间件
  3. 注册到应用

  ❓ 实施方案是否确认？确认后开始执行代码修改。

  [✅ 确认] [✏️ 修改] [❌ 取消]

用户响应处理:
  ✅ 确认: 🟢 代码执行授权生效 — 按实施方案逐文件执行
  ✏️ 修改: 根据反馈调整实施方案，再次输出 CP3
  ❌ 取消: 返回方案设计阶段（CP2）
```

### CP4: 测试确认（按需）

```yaml
触发时机: 测试执行完成后
触发条件: 任务涉及测试（用户要求测试或 IP1 选择了测试产物）

必须展示:
  - 测试统计（总数/通过/失败/跳过）
  - 失败用例详情
  - 覆盖率（如有）

输出格式: |
  📋 测试结果确认

  ## 测试统计
  - 总用例: N · 通过: N · 失败: N · 跳过: N

  ## 失败用例（如有）
  - ❌ test/xxx.test.ts — [错误描述]

  ❓ 测试结果如何处理？

  [✅ 确认继续] [🔄 重新测试] [⏭️ 跳过失败用例]

用户响应处理:
  ✅ 确认: 继续到文档生成
  🔄 重测: 修复后重新运行测试
  ⏭️ 跳过: 记录失败用例，继续流程
```

### CP5: 文档确认（按需）

```yaml
触发时机: 归档文档生成完成后
触发条件: 任务生成了归档文档（需求定义/技术方案/实施方案）

必须展示:
  - 生成的文档列表
  - 文档存放位置

输出格式: |
  📋 文档确认

  ## 生成的文档
  1. 01-需求定义.md
  2. 02-技术方案.md
  3. 03-实施方案/
  4. IMPLEMENTATION-PLAN.md（🔴 强制）

  ## 存放位置
  projects/<project>/requirements/<描述>/

  ❓ 文档是否确认？

  [✅ 确认] [✏️ 修改]

用户响应处理:
  ✅ 确认: 进入验证检查
  ✏️ 修改: 根据反馈修改文档
```

### CP6: 项目文档更新确认（按需）

```yaml
触发时机: 代码/功能实现完成后，任务结束前
触发条件（满足任一即触发）:
  - 代码文件被修改/新增/删除
  - 配置文件被修改
  - 新功能开发完成
  - Bug 修复完成

必须展示:
  - STATUS.md 更新内容预览（diff 格式）
  - CHANGELOG.md 新增条目预览
  - TASK-INDEX.md 追加记录
  - README.md 更新建议（如需要）

输出格式: |
  📋 项目文档更新确认

  ## STATUS.md 更新
  - 版本: v1.2.0 → v1.3.0
  - 最后更新: [当前日期]
  - 进度: +1 个已完成需求

  ## CHANGELOG.md 新增条目
  ### v1.3.0 ([日期])
  #### 新功能
  - [变更描述]

  ## TASK-INDEX.md
  - 追加本次任务记录

  ❓ 是否应用项目文档更新？

  [C] Continue — 确认应用更新
  [M] Modify — 手动编辑更新内容
  [S] Skip — 跳过更新（需确认原因）

用户响应处理:
  [C]: 应用所有更新
  [M]: 允许用户编辑更新内容后应用
  [S]: 记录跳过原因，继续完成任务
```

> 📎 CP6 的触发条件判断由 N12A 节点执行，详见 `document-sync.md`

## 🔴 询问点（Inquiry Points）

询问点与确认点不同：CP 是"这样做对不对？"，IP 是"要做哪些？"

### IP1: 测试产物询问

```yaml
触发时机: CP3 确认后，代码实现完成后
目的: 确定需要生成哪些测试产物

输出格式: |
  🔴 测试产物选择

  请选择需要生成的测试产物：

  [1] 单元测试 — Jest/Vitest 测试用例
  [2] HTTP 测试 — .http 文件（REST Client）
  [3] OpenAPI 文档 — Swagger/OpenAPI 规范
  [A] 全部生成
  [S] 跳过测试生成

  请选择 [1/2/3/A/S]:

响应处理:
  [1]: 仅生成单元测试
  [2]: 仅生成 HTTP 测试文件
  [3]: 仅生成 OpenAPI 文档
  [A]: 生成所有测试产物
  [S]: 跳过，继续到文档阶段
```

### IP2: 文档产物询问

```yaml
触发时机: CP4 确认后（或 IP1 选择跳过后）
目的: 确定需要更新/生成哪些文档

输出格式: |
  🔴 文档产物选择

  请选择需要生成/更新的文档：

  [1] README.md — 更新项目说明
  [2] CHANGELOG.md — 添加变更记录
  [3] 脚本文档 — scripts/README.md
  [4] API 文档 — 接口使用说明
  [A] 全部生成
  [S] 跳过文档生成

  请选择 [1/2/3/4/A/S]:

响应处理:
  [1]: 仅更新 README.md
  [2]: 仅更新 CHANGELOG.md
  [3]: 仅更新脚本文档
  [4]: 仅生成 API 文档
  [A]: 生成所有文档
  [S]: 跳过，继续到完成阶段
```

## 📊 P0 操作的确认规则

### P0 清单展示要求

P0（高危操作）在 CP2 中必须单独展示，AI 必须逐项列出：

```yaml
展示格式: |
  🔴 P0 操作清单
  ════════════════════════════════════════

  ⚠️ 以下操作需要您逐项确认：

  1. [数据库] 将添加索引 users.email
     影响: 索引创建期间可能短暂阻塞写操作
     回滚: db.users.dropIndex("email_1")
     确认: [Y/N]

  2. [配置] 将修改 Redis 连接超时
     原值: 5000ms → 新值: 10000ms
     回滚: 改回 5000ms
     确认: [Y/N]

  ════════════════════════════════════════

AI 侧（强制）:
  - 必须逐项展示每个 P0 操作（含影响说明 + 回滚方案）
  - 每个操作独立列出，不合并展示
  - 回滚方案必须是具体可执行的（有 DDL/命令，不是"回退修改"）

用户侧（灵活）:
  - 可以逐项回复 Y/N
  - 也可以明确表示"已逐项阅读，全部确认"（AI 可接受）

禁止:
  - AI 合并展示多个 P0 操作（如"以下 3 个操作需确认 [Y/N]"而不逐项列出）
  - AI 省略回滚方案（如"操作可回滚"而无具体命令）
```

## 🔄 用户响应选项速查

### 标准选项

| 选项 | 符号 | AI 行为 |
|-----|:----:|--------|
| 确认 | ✅ | 执行下一步骤 |
| 修改 | ✏️ | 等待修改指令，重新输出当前 CP |
| 取消 | ❌ | 保存进度到记忆，退出流程 |
| 跳过 | ⏭️ | 记录跳过原因，继续 |
| 重试 | 🔄 | 重新执行当前步骤 |
| Stop & Save | 📦 | 写入完整记忆后退出 |

---

## 🚨 违规处理

| 违规 | 示例 | 后果 |
|------|------|------|
| 跳过 CP | CP2 未确认就写代码 | 回滚代码，重新等待确认 |
| 假设同意 | "如果没问题我就继续了" | 必须明确等待用户响应 |
| 内容缺失 | CP2 未展示修改清单 | 重新展示完整内容 |
| CP2=CP3 | CP2 确认后直接执行代码（FIX-015） | 停止 → 回滚 → 生成实施方案 → CP3 |
| 合并确认 | 一次确认 CP1+CP2 | 拆开，逐个确认 |
