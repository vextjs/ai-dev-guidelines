# 任务记忆机制

> 跨会话上下文传递的核心基础设施。每次会话自动输出报告 + 记忆索引，实现状态持久化。
> 本文件整合 v2 `task-memory.md` + `triggers.md` + `templates.md` + `multi-agent.md` 的完整内容。

**版本**: v3.0.0
**最后更新**: 2026-03-10

---

## 核心架构：报告为主体、记忆为索引

```text
🔴 架构规则（NO EXCEPTIONS）:

  报告文件（reports/）    → 每次会话的完整分析内容载体
  记忆文件（.ai-memory/） → 索引 + 摘要（指向报告的指针）
  对话输出                → 结论摘要 + 报告路径

  关系图:
    用户提问 → AI 分析 → 写入报告文件（完整内容）
                       → 追加/更新记忆文件（摘要+报告链接+对话记录）
                       → 对话输出结论摘要 + 报告路径

    下次会话 → 读记忆（最新日期文件末尾）→ 跟链接读报告 → 完整恢复上下文
```

**设计理由：**

| # | 收益 | 说明 |
|:-:|------|------|
| 1 | 减少 token 浪费 | 记忆不重复写分析内容，只存摘要+链接 |
| 2 | 可追溯性 | 每次对话都有报告文件留存 |
| 3 | 跨会话恢复 | 新会话读记忆 → 读报告 → 完整上下文不丢失 |
| 4 | 职责清晰 | 报告=完整内容，记忆=索引+摘要 |
| 5 | 序号解耦 | 报告和记忆各自独立管理 |
| 6 | 文件数可控 | 每天一个记忆文件，月底最多 31 个 |

## 🔴 自动输出，禁止询问

```text
🔴 强制规则（NO EXCEPTIONS）:

  报告文件和记忆文件的写入是 AI 的自动行为，绝不询问用户。

  ❌ 绝对禁止:
    - "需要我把分析写入报告吗？"
    - "要不要我写入记忆？"
    - "是否需要保存本次结果？"
    - 任何形式的询问用户是否写入报告/记忆的行为

  ✅ 正确行为:
    - 预检查完成后 → 自动创建初始记忆（状态 🔄）
    - 分析完成后   → 自动写入报告文件
    - 报告写入后   → 自动更新记忆文件（追加报告链接）
    - 任务完成后   → 自动更新记忆状态为 ✅
    - 对话中       → 输出结论摘要 + 报告路径
```

### 命名规范

| 文件类型 | 命名格式 | 说明 |
|---------|---------|------|
| 记忆文件 | `YYYYMMDD.md` | 每天一个，会话内以 `## 会话 NN` 分段追加 |
| 报告文件 | `NN-<类型>-<简述>.md` | NN 仅扫描当前 `<agent>/YYYYMMDD/` 目录 |
| Agent 隔离 | 由目录 `clients/<agent>/` 保证 | 不写入文件名 |

**🔴 序号独立规则：**

| 序号类型 | 来源 | 说明 |
|---------|------|------|
| 报告 NN | 仅扫描 `reports/<子目录>/<agent>/YYYYMMDD/` | 与记忆无关 |
| 记忆 会话 NN | 文件内以 `## 会话 NN` 分段 | 当天第几个会话 |
| ❌ 禁止 | 报告和记忆共享全局序号 | v1.6 设计缺陷，已废弃 |

### 🔴 禁止创建位置

| 位置 | 状态 |
|------|:----:|
| ❌ 工作区根目录（如 `Workspace/.ai-memory/`） | 禁止 |
| ❌ `ai-dev-guidelines/` 根目录 | 禁止 |
| ❌ 业务项目源码目录（如 `schema-dsl/.ai-memory/`） | 禁止 |
| ✅ `projects/<project>/.ai-memory/` | 唯一正确 |

> 事故参考：Agent 在工作区根目录创建记忆文件，导致记忆与项目脱钩、其他 Agent 无法恢复上下文。

## 消息驱动 5+1 阶段

### 阶段总览

| 阶段 | 触发时机 | 目的 | 自动？ |
|:----:|---------|------|:------:|
| **0** | AI 收到首条消息（预检查后） | 创建/追加记忆 + 硬性阻塞 | 🔴 自动 |
| **1** | 用户发消息时 | 捕获用户输入，防追加需求丢失 | 🔴 自动 |
| **2** | AI 回复时 | 写报告 + 更新记忆 | 🔴 自动 |
| **3** | AI 执行完毕时 | 记录变更清单 + 编码检查点 | 🔴 自动 |
| **4.5** | 任务结束前 | 执行合规性回溯检查（出口门禁） | 🔴 自动 |
| **4** | 任务结束时 | 最终状态更新 ✅ | 🔴 自动 |

**执行顺序：**

```text
首轮:   0 → 1 → 2+3 →
后续轮: 1 → 2+3 → 1 → 2+3 → ...
结束:   4.5 → 4
```

**入口-出口闭环：**

```text
阶段 0（入口门禁）确保会话开始时预检查+记忆写入不被跳过
阶段 4.5（出口门禁）确保任务结束前流程合规性已验证
```

---

### 阶段 0：会话初始化（🔴 硬性阻塞）

| 项 | 说明 |
|----|------|
| **触发条件** | AI 收到本次会话的第一条用户消息，预检查第 5 项完成后 |
| **触发方式** | 🔴 自动执行，禁止询问用户 |
| **目的** | 创建/追加记忆文件 + 作为预检查第 6 行输出（硬性阻塞） |

**🔴 时序强制规则（NO EXCEPTIONS）：**

阶段 0 的记忆写入已嵌入预检查输出的第 6 行，与预检查 1-5 项合并为不可分割的原子操作。
在阶段 0 记忆写入完成之前，禁止执行任何分析性读取（如 `read_file` 用户请求相关文件）。

**执行步骤：**

1. 确定当前 Agent 标识（预检查第 4 项已获取）
2. 逐层列出 `.ai-memory/clients/<agent>/tasks/` 目录内容（🔴 禁止 glob）
3. 检查当天是否已有记忆文件 `YYYYMMDD.md`
   - 已存在 → 读取已有会话数 → 追加 `## 会话 NN+1`
   - 不存在 → 创建新文件 → 写入 `## 会话 01`
4. 写入初始内容：任务摘要（占位）· 状态 🔄 · 开始时间 · 任务类型
5. 记录用户首条消息摘要到 📨 对话记录（阶段 0+1 合并）

**写入内容（最小集）：**

```text
## 会话 NN

**状态**: 🔄 进行中
**开始时间**: YYYY-MM-DD HH:MM
**任务类型**: [待识别 / dev / fix / analyze / audit / chat]

### 🎯 任务摘要
[待填充]

### 📨 对话记录
| 轮次 | 方向 | 摘要 | 关联 |
|------|------|------|------|
| 1 | 👤→ | [用户首条消息核心意图] | |
```

### 阶段 1：用户发消息时（捕获用户输入）

| 项 | 说明 |
|----|------|
| **触发条件** | 用户发送包含新意图/新需求/方向修正的消息 |
| **触发方式** | 🔴 自动执行 |
| **目的** | 第一时间捕获用户输入，防止追加需求丢失 |

**执行步骤：**

1. 记录用户消息的核心意图到 📨 对话记录（1 行摘要 + 关联引用，不是全文复制）
2. 如用户消息含方向修正或新需求 → 追加到 ⚠️ 待跟进

**过滤规则：** 纯确认性消息（"好的""Y""继续"）不记录。

---

### 阶段 2：AI 回复时（写报告 + 更新记忆）

| 项 | 说明 |
|----|------|
| **触发条件** | AI 准备向用户输出分析结果/回复 |
| **触发方式** | 🔴 自动执行 |
| **目的** | 记录 AI 的分析结论 + 写入报告 |

**执行步骤：**

1. 确定报告类型和目标子目录（analysis / bugs / diagnostics / requirements / optimizations）
2. 扫描 `reports/<子目录>/<agent>/YYYYMMDD/` 获取当前最大 NN → NN+1
3. 写入报告文件（含报告头部必填字段 — 参见 `RULES.md§6`）
4. 更新记忆文件：追加报告链接到 📄 关联报告 + AI 回复摘要到 📨 对话记录
5. 对话中输出结论摘要 + 报告路径

---

### 阶段 3：AI 执行完毕时（记录变更清单）

| 项 | 说明 |
|----|------|
| **触发条件** | AI 在一轮消息中的所有工具调用/文件操作完成 |
| **触发方式** | 🔴 自动执行 |
| **目的** | 记录实际执行的变更 |

**执行步骤：**

1. 更新记忆文件 → 追加本轮实际修改的文件列表和关键决策
2. 更新 ⚠️ 待跟进中已完成的条目标记 `[x]`
3. 如触发编码检查点 → 写入 📦 编码检查点（见下方）

---

### 编码检查点（Coding Checkpoint）

> 解决多文件变更中途中断后进度丢失问题。适用于所有涉及 3+ 个文件变更的编码任务。

**触发条件：** 编码任务 + 计划变更文件数 ≥ 3

| 模式 | 触发粒度 | 写入内容 |
|------|---------|---------|
| 默认（有 IMPL-PLAN） | 每完成一个任务编号 | 更新 PLAN 状态 ✅ + 追加检查点 |
| 兜底（无 IMPL-PLAN） | 每完成 3 个文件 | 追加检查点 + 更新待跟进 |
| 不触发 | 变更 < 3 文件 | 沿用阶段 3 默认机制 |

**检查点写入格式：**

```text
📦 编码检查点 (T3/T5 完成):
  - ✅ T1: config/xxx.ts — 已创建
  - ✅ T2: src/xxx.ts — 已实现
  - ✅ T3: src/app.ts — 已集成
  - ⏳ T4: test/xxx.test.ts — 待写
  - ⏳ T5: docs/README.md — 待更新
```

**同步操作（检查点写入时必须一并完成）：**

1. 追加 📦 编码检查点表格行
2. 更新 ⚠️ 待跟进中对应条目标记 `[x]`
3. 更新 04-实施计划.md 中该任务状态 🔲→✅（无 PLAN 时跳过）

**恢复逻辑：**

```text
新会话恢复时:
  1. 读记忆 → 找到 📦 编码检查点 → 知道最后一个检查点位置
  2. 读 IMPL-PLAN → 找到第一个 ⏳ 任务 → 从此处继续
  3. 无需扫描全项目推断进度（节省 ~3000-4000 token）
```

**Token 成本对比：**

| 场景 | 成本 |
|------|------|
| 单个检查点写入 | ~280 token |
| 中断后恢复（有检查点） | ~500 token |
| 中断后恢复（无检查点，需扫描推断） | ~3000-5000 token |
| **ROI** | 每个检查点投入 ~280，中断恢复节省 ~3000-4000 |

---

### 阶段 4.5：执行合规性回溯检查（出口门禁）

| 项 | 说明 |
|----|------|
| **触发条件** | AI 准备进入阶段 4（任务结束）之前 |
| **触发方式** | 🔴 自动执行 |
| **目的** | 回溯检查本次会话是否按流程执行，与阶段 0（入口门禁）形成闭环 |

**回溯检查清单：**

| # | 检查项 | 说明 |
|:-:|--------|------|
| 1 | 预检查已完成？ | 阶段 0 是否执行 |
| 2 | 意图识别正确？ | 路由到了正确的工作流 |
| 3 | CP 已遵守？ | 涉及代码修改时 CP 是否按顺序执行 |
| 4 | 报告已写入？ | 报告文件是否存在 |
| 5 | 记忆已更新？ | 记忆文件是否包含本次会话记录 |
| 6 | 约束已遵守？ | 关键约束（#1 修改确认 / #3 禁硬编码 / #15 验证 / #18 三步扫描）是否遵守 |
| 7 | 工作流专属检查？ | build: B1~B6 / fix: 三步扫描 / analyze: 禁改代码 |

**发现问题时：**

- 可自修复 → 立即修复（最多 2 次尝试）
- 不可自修复 → 记录到记忆 ⚠️ 待跟进
- 连续 2 次同类偏差 → 升级触发 audit 工作流

---

### 阶段 4：任务结束时（最终状态更新）

| 项 | 说明 |
|----|------|
| **触发条件** | 用户确认任务完成 / AI 判断任务已完成（🔴 前置条件：阶段 4.5 已通过） |
| **触发方式** | 🔴 自动执行 |
| **目的** | 最终状态更新 |

**执行步骤：**

1. 执行完成验证（见下方 §任务完成验证）
2. 更新记忆文件状态 🔄 → ✅
3. 补全 🎯 任务摘要（用 2-3 句话总结）
4. 更新 📄 全天关联报告汇总
5. 更新 ⚠️ 全天待跟进汇总
6. 更新 🗺️ 下一阶段建议（如有 IMPL-PLAN）
7. 更新 SUMMARY.md（追加本次任务记录）

## 🔴 任务完成验证

在 AI 声称"任务完成"之前，必须逐项确认：

| # | 检查项 | 说明 |
|:-:|--------|------|
| 1 | 需求覆盖 | 用户的所有需求点是否都已处理？ |
| 2 | 报告存在 | 报告文件是否已写入？ |
| 3 | 记忆完整 | 记忆是否包含任务摘要 + 对话记录 + 关联报告？ |
| 4 | CP 完整 | 所有必经 CP 是否都已通过？（build: CP1~CP3 / fix: CP1~CP2） |
| 5 | 阶段 4.5 通过 | 合规性回溯检查是否已通过？ |
| 6 | 约束遵守 | 关键约束是否已遵守？ |
| 7 | 工作流验证 | 工作流专属验证清单是否通过？ |

> 任何一项未通过 → 不得声称完成，需回到对应步骤处理。

## 多 Agent 支持

### Agent 标识检测（优先级从高到低）

| 优先级 | 检测方法 | 示例 |
|:------:|---------|------|

| 规则 | 说明 |
|------|------|
| 写入隔离 | 只写入 `clients/<自己的agent>/` 目录 |
| 读取规则 | 优先读自己目录；如需跨 Agent 信息 → 读全局 SUMMARY.md |
| ❌ 禁止 | 写入其他 Agent 的记忆目录 |
| ❌ 禁止 | 删除/修改其他 Agent 的记忆文件 |
| 全局 SUMMARY | 所有 Agent 可读可追加（不删除其他 Agent 的条目） |

### 向后兼容

| 场景 | 处理方式 |
|------|---------|
| 旧格式文件（`YYYYMMDD-NN-TYPE-id.md`） | 仍可读取 |
| 新会话 | 一律使用新格式（`YYYYMMDD.md`） |
| 预检查扫描 | 同时兼容两种格式 |

## 新会话恢复流程

### 预检查时输出上次记忆（第 5 项）

```text
扫描逻辑:
  1. 确定当前 Agent 标识（预检查第 4 项已获取）
  2. 逐层列出 .ai-memory/clients/<agent>/tasks/ 目录内容
     🔴 禁止使用 glob 扫描（glob 引擎跳过隐藏目录）
  3. 按文件名日期排序（YYYYMMDD 自然排序）
  4. 取最新日期的文件
  5. 读取该文件末尾最新会话段落，提取状态字段

输出格式:
  找到记忆:   "5. 上次记忆: .../tasks/20260310.md §会话09 (🔄 进行中)"
  无记忆:     "5. 上次记忆: ⚠️ 无（首次任务）"
  目录不存在: "5. 上次记忆: ⚠️ 无（.ai-memory 目录不存在）"
```

### 完整恢复步骤

```text
AI 收到任务请求后:
  1. 输出上次记忆路径（预检查第 5 项）
  2. 读取 .ai-memory/SUMMARY.md（全局关键决策）
  3. 读取 .ai-memory/clients/<agent>/SUMMARY.md（本 Agent 任务列表）
  4. 如涉及之前任务，读取 tasks/YYYYMMDD.md:
     - 📨 对话记录 → 快速回忆对话脉络
     - 📄 关联报告 → 获取报告链接
     - 📦 编码检查点 → 恢复编码进度（如有）
  5. 按需读取关联报告文件恢复详细上下文
  6. 创建新的记忆（阶段 0）
  7. 记录用户首条消息摘要（阶段 0+1 合并）
```

### Token 耗尽恢复

```text
Token 即将耗尽时:
  1. 写入紧急检查点（📦 + 当前进度）
  2. 在记忆中标注 "⚠️ Token 耗尽中断，进度见上方检查点"
  3. 建议用户开新会话继续

新会话恢复:
  1. 预检查找到上次记忆（状态 🔄 进行中）
  2. 读取最后一个 📦 编码检查点
  3. 从检查点位置继续执行
  4. 无需重新分析已完成的步骤
```

## 归档策略

### 自动归档

| 触发条件（满足任一） | 操作 |
|---------------------|------|
| 单日记忆文件超过 8 个会话（NN ≥ 9） | 触发归档 |
| 单日记忆文件超过 400 行 | 触发归档 |

**归档操作：**

1. 创建归档文件：`YYYYMMDD-archive.md`（与当日文件同目录）
2. 将最早的会话（会话 01 ~ 会话 NN-3）整体移入归档文件
3. 当日文件只保留：
   - 头部元信息
   - `## 📦 归档说明`（指向归档文件 + 归档会话范围）
   - 最近 3 个会话（完整内容）
   - `## 📄 全天关联报告汇总`（保留完整）
   - `## ⚠️ 全天待跟进汇总`（保留完整）

**二次归档：**
- 如归档后当日文件再次超限 → 追加到已有 `YYYYMMDD-archive.md` 末尾
- 当日文件始终只保留最近 3 个会话

**恢复时的归档处理：**
- 正常恢复只读当日文件（最近 3 个会话 + 待跟进汇总，足够恢复上下文）
- 需要更早上下文时 → 读取 `YYYYMMDD-archive.md` 中对应会话段落
- 归档文件中的报告链接仍然有效

### 清理策略

| 策略 | 说明 |
|------|------|
| SUMMARY.md | 保留最近 20 条，更早的只保留关键决策 |
| tasks/ | 超过 30 天的文件可清理 |
| reports/ | 超过 30 天的文件可清理 |
| 用户触发 | 用户说"清理旧记忆"触发清理 |
| 清理粒度 | 删除 `YYYYMMDD.md` 即清除该天所有会话记忆 |
