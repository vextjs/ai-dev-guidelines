# 技术调研 Checklist

> `analyze` 工作流的场景变体（`RULES.md §10` analyze 子类型：技术调研）。
> 当 N04 意图识别将任务判定为 `analyze > 技术调研` 时，由路由表自动加载本 checklist，
> 叠加到 `analyze/README.md` 默认流程上执行。🔴 analyze 的核心规则不变：**禁止修改项目源码**。

**版本**: v3.0.0
**最后更新**: 2026-03-10

## 适用场景

> 🔴 本 checklist 的加载由 `RULES.md §2` 意图识别系统驱动，**不依赖关键词匹配**。
> AI 基于语义理解判断用户需求属于"技术调研"二级分类后，自动叠加本 checklist。
> 技术调研的最终产出是**调研报告**（含推荐结论），不是代码变更。

**典型场景示例**（仅供理解，非穷举触发词）：

| 场景 | 用户消息示例 |
|------|------------|
| 技术选型 | "帮我做数据库选型" |
| 方案对比评估 | "对比 Redis 和 Memcached" |
| 可行性验证 | "做一个 POC 验证可行性" |
| 架构方案评估 | "评估微服务改造方案" |
| 替代方案调研 | "有没有替代 Moment.js 的方案" |

## 核心原则

```text
🔴 技术调研原则：客观 > 全面 > 可落地
```

| 原则 | 说明 |
|------|------|
| 客观中立 | 基于事实和数据对比，避免主观偏见或默认推荐熟悉的方案 |
| 多方案覆盖 | 至少列出 2~3 个候选方案，单方案调研不构成"选型" |
| 维度全面 | 从功能、性能、生态、学习成本、维护成本、许可证等多维度评估 |
| 数据驱动 | 尽可能用具体数据（benchmark、star 数、发布频率等）支撑结论 |
| 结论明确 | 必须给出明确推荐方案 + 推荐理由 + 风险提示，禁止"各有优劣请自行选择" |
| 可落地 | 推荐结论应考虑项目实际情况（技术栈兼容性、团队能力、时间约束等） |

## 阶段叠加规则

### 阶段1 — 范围界定（叠加：调研背景 + 评估维度定义）

在默认步骤 1.1~1.5 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 1.6 | 明确调研背景：为什么要调研？要解决什么问题？ | 调研动机 |
| 1.7 | 明确评估约束：技术栈限制 · 团队经验 · 时间/预算约束 · 许可证要求 | 约束条件 |
| 1.8 | 确定评估维度（见下方标准维度表），根据场景裁剪 | 评估维度清单 |
| 1.9 | 初步列出候选方案清单（至少 2 个，建议 3~5 个） | 候选方案 |
| 1.10 | 确认产出预期：是否需要 POC？是否需要 benchmark？ | 产出规格 |

**标准评估维度：**

| 维度 | 说明 | 权重建议 |
|------|------|:--------:|
| 功能匹配度 | 是否满足核心需求？缺少哪些功能？ | 🔴 高 |
| 性能 | 吞吐量 · 延迟 · 资源占用 · 可扩展性 | 🔴 高 |
| 社区与生态 | GitHub Star · npm 下载量 · 发布频率 · Issue 响应速度 · 插件/中间件生态 | 🟡 中 |
| 学习曲线 | 文档质量 · API 设计友好度 · 上手难度 | 🟡 中 |
| 维护成本 | 长期维护负担 · 升级兼容性 · 技术债风险 | 🟡 中 |
| 许可证 | 开源协议兼容性（MIT / Apache / GPL / 商业） | 🟡 中 |
| 团队适配 | 团队现有经验 · 招聘难度 · 培训成本 | 💡 按需 |
| 成本 | 免费/付费 · 基础设施成本 · 运维成本 | 💡 按需 |
| 安全性 | 已知 CVE · 安全审计记录 · 安全特性 | 💡 按需 |
| 成熟度 | 版本稳定性 · 生产环境使用案例 · 大厂背书 | 💡 按需 |

> 根据具体调研场景选择 5~8 个核心维度，不必每次都覆盖全部 10 个。

---

### 阶段2 — 信息收集（叠加：多方案并行收集 + 数据采集）

在默认步骤 2.1~2.5 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 2.6 | 为每个候选方案收集基本信息（官方文档 · GitHub · npm/PyPI 等） | 方案概况 |
| 2.7 | 按评估维度逐方案采集数据（功能清单 · 性能数据 · 社区指标） | 维度数据 |
| 2.8 | 收集实际使用案例和社区评价（谁在用？效果如何？） | 使用案例 |
| 2.9 | 检查项目技术栈兼容性（依赖冲突 · 版本要求 · 运行环境） | 兼容性报告 |
| 2.10 | 如需 POC：设计 POC 范围和验证点（🔴 POC 代码建议在独立目录/文件中，不修改项目源码） | POC 计划 |

**🔴 信息收集规则（继承 analyze 默认规则 + 调研强化）：**

| 规则 | 说明 |
|------|------|
| 数据来源标注 | 每个数据点必须标注来源（官方文档 URL · GitHub · npm · 测试结果） |
| 时效性标注 | 社区指标（Star · 下载量 · 版本号）标注采集时间 |
| 禁止凭印象 | 禁止"据说性能很好""社区很活跃"等无数据支撑的描述 |
| 公平对比 | 各方案使用相同维度和标准评估，禁止选择性忽略某方案的优势 |

---

### 阶段3 — 分析与推理（叠加：横向对比 + 推荐决策）

在默认步骤 3.1~3.5 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 3.6 | 制作横向对比表（方案 × 维度矩阵） | 对比矩阵 |
| 3.7 | 逐维度评分或定性评价（✅ 优秀 / 🟡 一般 / ❌ 不足 / ⚠️ 未验证） | 维度评价 |
| 3.8 | 综合评估：结合约束条件和维度权重，形成推荐排序 | 综合排序 |
| 3.9 | 给出明确推荐方案 + 推荐理由 + 主要风险 | 推荐结论 |
| 3.10 | 给出备选方案及适用条件（"如果 X 场景，建议选 Y"） | 备选建议 |
| 3.11 | 如需 POC：输出 POC 建议（范围 · 验证点 · 预期结论 · 建议步骤） | POC 建议 |

**🔴 推荐决策规则：**

| 规则 | 说明 |
|------|------|
| 必须有推荐 | 禁止"各有优劣，请自行选择"。必须给出明确推荐 + 理由 |
| 推荐有条件 | 推荐结论应标注适用条件和前提假设 |
| 风险透明 | 推荐方案的已知风险和不足必须明确列出 |
| 备选方案 | 除推荐外，至少给出 1 个备选方案及其适用场景 |
| 三项验证 | 推荐结论同样需要通过三项验证（合理性 · 可实施性 · 收益） |

---

### 阶段4 — 报告产出（叠加：调研报告结构）

在默认步骤 4.1~4.4 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 4.5 | 按调研报告专属结构编写（见下方） | 调研报告 |
| 4.6 | 二次验证中追加：对比表数据是否准确 · 推荐结论是否有数据支撑 | 验证结果 |

## 横向对比表格式

调研报告中必须包含横向对比表，格式如下：

```text
| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 功能匹配度 | ✅ 完全满足 | 🟡 部分满足（缺 X） | ✅ 完全满足 |
| 性能 | ✅ P99<50ms | ✅ P99<30ms | 🟡 P99<200ms |
| 社区生态 | ✅ 45k Star | 🟡 8k Star | ✅ 52k Star |
| 学习曲线 | 🟡 文档较少 | ✅ 文档完善 | ❌ 上手困难 |
| 维护成本 | ✅ 低 | 🟡 中 | ❌ 高 |
| 许可证 | ✅ MIT | ✅ Apache-2.0 | ⚠️ GPL-3.0 |
| 综合评价 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
```

**评价标签说明：**

| 标签 | 含义 |
|:----:|------|
| ✅ | 优秀 / 完全满足 / 推荐 |
| 🟡 | 一般 / 部分满足 / 可接受 |
| ❌ | 不足 / 不满足 / 不推荐 |
| ⚠️ | 未验证 / 需进一步确认 |

## POC 建议规范

如果调研结论建议进行 POC 验证，输出格式如下：

| 项 | 内容 |
|----|------|
| POC 目标 | 验证什么？（如"验证 Redis 在高并发下的性能表现"） |
| 验证点 | 具体要测试的指标/功能（列表，每项一行） |
| 范围 | POC 涉及的代码范围（建议独立目录，不影响现有代码） |
| 预期结论 | 什么结果算通过？什么结果算失败？ |
| 建议步骤 | 按顺序的实施步骤（1~5 步） |
| 预估工时 | 预估 POC 所需时间 |
| 后续衔接 | POC 通过后 → 开 `dev` 任务落地；POC 失败 → 评估备选方案 |

> 🔴 **POC 代码执行需要开启新的 `dev` 任务**。analyze 流程中只输出 POC 计划和建议，不执行代码编写。
> 如用户要求在调研中直接写 POC 代码，应引导切换到 `dev` 工作流或明确告知将违反 analyze 的禁止修改代码规则。

## 调研报告专属结构

调研类任务的报告（使用 `templates/report-analysis.md`）应包含以下章节：

| 章节 | 内容 | 必须 |
|------|------|:----:|
| 调研背景 | 为什么要调研？要解决什么问题？ | ✅ |
| 约束条件 | 技术栈限制 · 团队经验 · 时间/预算 · 许可证要求 | ✅ |
| 候选方案概览 | 每个候选方案的一句话简介 | ✅ |
| 评估维度 | 本次调研使用的评估维度清单 | ✅ |
| 横向对比表 | 方案 × 维度矩阵（含评价标签） | ✅ |
| 各方案详细分析 | 每个方案的逐维度分析（含数据来源标注） | ✅ |
| 推荐结论 | 明确推荐方案 + 推荐理由 + 三项验证 | ✅ |
| 备选方案 | 备选方案及适用条件 | ✅ |
| 风险提示 | 推荐方案的已知风险和注意事项 | ✅ |
| POC 建议 | 如需 POC 验证，输出 POC 计划 | 🟡 按需 |
| 后续行动 | 推荐的下一步（🔴 必须在对话中同步输出摘要） | ✅ |

## 调研专属验证清单

在 N13 合规自检阶段，叠加以下检查（在 analyze 默认报告自检清单之后执行）：

| # | 检查项 | 通过标准 |
|:-:|--------|---------|
| R1 | 候选方案充足 | 至少列出 2 个候选方案（建议 3~5 个） |
| R2 | 维度覆盖 | 至少覆盖 5 个评估维度 |
| R3 | 横向对比表 | 报告中包含方案 × 维度的横向对比表 |
| R4 | 数据来源 | 每个维度评价有数据来源标注（URL / 版本号 / 测试结果） |
| R5 | 时效性 | 社区指标（Star · 下载量）标注采集时间 |
| R6 | 客观中立 | 各方案使用相同维度评估，无选择性忽略 |
| R7 | 推荐明确 | 有明确推荐方案 + 推荐理由（非"请自行选择"） |
| R8 | 推荐有验证 | 推荐结论附带三项验证（合理性 · 可实施性 · 收益） |
| R9 | 备选方案 | 至少给出 1 个备选方案及适用条件 |
| R10 | 风险透明 | 推荐方案的已知风险和不足已列出 |
| R11 | 项目适配 | 推荐结论考虑了项目约束条件（技术栈 · 团队 · 时间） |
| R12 | 禁止改代码 | 🔴 调研过程中未修改项目源码 |

## 与默认 analyze 流程的差异

| 对比项 | 默认 analyze | 技术调研（本 checklist） |
|--------|-------------|----------------------|
| 分析目标 | 通用分析（代码质量/架构/性能等） | 专门针对技术方案选型 |
| 候选方案 | 无要求 | 🔴 至少 2 个，建议 3~5 个 |
| 横向对比 | 无要求 | 🔴 必须有方案 × 维度矩阵 |
| 评估维度 | 按分析目标自定义 | 标准维度表（10 个可选维度） |
| 推荐结论 | 行动建议 | 🔴 明确推荐方案 + 备选方案 + 风险提示 |
| POC | 无 | 按需输出 POC 建议（代码执行需开 dev 任务） |
| 数据要求 | 基于文件验证 | 基于文件验证 + 外部数据来源标注 |
| 三项验证 | 每条问题/建议 | 每条问题/建议 + 推荐结论 |
| 禁止改代码 | ✅ | ✅（不变） |

## 常见调研场景速查

| 场景 | 重点维度 | 注意事项 |
|------|---------|---------|
| 框架选型 | 功能 · 生态 · 学习曲线 · 团队适配 | 关注长期维护成本和社区活跃度 |
| 数据库选型 | 性能 · 数据模型 · 扩展性 · 运维成本 | 关注数据量增长后的表现 |
| 中间件选型 | 性能 · 可靠性 · 运维复杂度 · 生态 | 关注集群部署和故障恢复 |
| 库/工具替换 | 功能兼容性 · 迁移成本 · 包大小 · 许可证 | 关注 API 兼容性和迁移工作量 |
| 架构方案 | 复杂度 · 可扩展性 · 团队能力 · 实施成本 | 关注渐进式迁移可行性 |
| 云服务选型 | 功能 · 成本 · SLA · 锁定风险 · 合规 | 关注厂商锁定和数据主权 |
