## 核心原则

```text
🔴 数据库变更原则：可回滚 > 可验证 > 可追踪 > 最小影响
```

| 原则 | 说明 |
|------|------|
| 可回滚 | 🔴 任何变更必须有对应的回滚脚本，且回滚脚本已验证可用 |
| 可验证 | 变更执行后有验证脚本确认结果正确 |
| 可追踪 | 所有变更通过迁移脚本管理，禁止手动执行 DDL |
| 最小影响 | 避免长时间锁表、大批量数据操作应分批执行 |
| 向后兼容 | 优先采用向后兼容的变更方式（先加后删、双写过渡） |

## 阶段叠加规则

### 阶段1 — 需求理解（叠加：变更范围分析）

在默认步骤 1.1~1.5 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 1.6 | 确定变更类型（见下方分类表） | 变更类型 |
| 1.7 | 分析受影响的表和关联关系（外键、索引、视图、触发器） | 影响清单 |
| 1.8 | 评估数据量和变更对线上服务的影响（锁表时间、性能波动） | 风险评估 |
| 1.9 | 确认数据库类型和版本（MySQL/PostgreSQL/SQLite/MongoDB 等） | 数据库信息 |
| 1.10 | 确认是否涉及数据迁移（仅结构变更 vs 结构+数据） | 迁移范围 |

**变更类型分类：**

| 类型 | 操作 | 风险等级 | 说明 |
|------|------|:--------:|------|
| 加字段 | `ALTER TABLE ADD COLUMN` | 🟢 低 | 通常安全，注意默认值和 NOT NULL |
| 改字段 | `ALTER TABLE MODIFY/ALTER COLUMN` | 🟡 中 | 可能影响现有数据和应用兼容性 |
| 删字段 | `ALTER TABLE DROP COLUMN` | 🔴 高 | 不可逆，必须确认无引用 |
| 加表 | `CREATE TABLE` | 🟢 低 | 通常安全 |
| 删表 | `DROP TABLE` | 🔴 高 | 不可逆，数据丢失 |
| 加索引 | `CREATE INDEX` | 🟡 中 | 大表加索引可能锁表 |
| 删索引 | `DROP INDEX` | 🟡 中 | 可能导致查询性能下降 |
| 重命名 | `RENAME TABLE/COLUMN` | 🔴 高 | 所有引用必须同步修改 |
| 数据迁移 | `INSERT/UPDATE/DELETE` 批量 | 🔴 高 | 数据量大时需分批，需验证数据完整性 |
| 约束变更 | `ADD/DROP CONSTRAINT` | 🟡 中 | 可能影响现有数据合法性 |

**🔴 回滚方案规则：**

| 变更类型 | 回滚方案要求 |
|---------|-------------|
| 加字段 | `ALTER TABLE DROP COLUMN` |
| 改字段 | `ALTER TABLE MODIFY` 还原为原类型/定义 |
| 删字段 | 🔴 不可逆 — 必须先备份数据，回滚 = 重建字段 + 恢复数据 |
| 加表 | `DROP TABLE`（确认无数据写入后） |
| 删表 | 🔴 不可逆 — 必须先备份完整表数据和结构，回滚 = `CREATE TABLE` + 恢复数据 |
| 加索引 | `DROP INDEX` |
| 数据迁移 | 保留原始数据快照 / 反向 DML 脚本 |

### 阶段3 — 代码实现（叠加：脚本编写 + 应用代码同步）

在默认编码执行规则之上，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 3.X+1 | 编写迁移脚本（`migration.sql` 或框架迁移文件） | 迁移脚本 |
| 3.X+2 | 编写回滚脚本（`rollback.sql`） | 回滚脚本 |
| 3.X+3 | 编写验证脚本（`verify.sql`） | 验证脚本 |
| 3.X+4 | 同步修改应用代码（Model/Entity/DTO/Repository 等） | 应用代码 |
| 3.X+5 | 同步修改类型定义（TypeScript interface/type 等） | 类型定义 |
| 3.X+6 | 检查所有 SQL 查询/ORM 调用是否兼容变更后的 Schema | 查询兼容性 |

**🔴 脚本编写规则：**

| 规则 | 说明 |
|------|------|
| 幂等性 | 迁移脚本应支持重复执行不报错（`IF NOT EXISTS` / `IF EXISTS`） |
| 事务包裹 | DDL 变更尽量在事务中执行（注意：MySQL DDL 隐式提交） |
| 注释充分 | 每条 DDL 语句附带注释说明目的 |
| 数据备份 | 🔴 删除类操作前必须先备份数据（`CREATE TABLE backup_xxx AS SELECT ...`） |
| 分批执行 | 大批量数据操作必须分批（`LIMIT` + 循环），避免长事务锁表 |
| 字符集一致 | 新表/新字段的字符集和排序规则与现有表保持一致 |

## 大表操作指南

当变更涉及的表数据量较大时（通常 > 100 万行），需要特别关注：

### 锁表风险评估

| 操作 | MySQL (InnoDB) | PostgreSQL | 说明 |
|------|:-------------:|:----------:|------|
| ADD COLUMN (有默认值) | 🔴 锁表 (< 8.0.12) / 🟢 即时 (≥ 8.0.12) | 🟢 即时 (≥ 11) | 版本敏感 |
| ADD COLUMN (无默认值) | 🟢 即时 | 🟢 即时 | 通常安全 |
| DROP COLUMN | 🟡 短暂锁 | 🟢 即时 (标记删除) | PG 仅标记，不释放空间 |
| ADD INDEX | 🟡 可能锁 | 🟢 `CONCURRENTLY` | 推荐使用在线 DDL |
| MODIFY COLUMN | 🔴 可能锁表 | 🟡 取决于类型 | 类型变更风险最高 |
| 批量 UPDATE | 🔴 长事务 | 🔴 长事务 | 必须分批执行 |

### 大表变更策略

| 策略 | 适用场景 | 工具/方法 |
|------|---------|----------|
| 在线 DDL | MySQL ≥ 5.6 的 ADD/DROP COLUMN | `ALTER TABLE ... ALGORITHM=INPLACE` |
| pt-online-schema-change | MySQL 大表结构变更 | Percona Toolkit |
| gh-ost | MySQL 大表结构变更（GitHub） | 基于 binlog 的无触发器方案 |
| CONCURRENTLY | PostgreSQL 索引创建 | `CREATE INDEX CONCURRENTLY` |
| 分批 DML | 大批量数据更新/删除 | `LIMIT` + 循环 + 间隔 |

### 分批执行模板

```sql
-- 分批更新示例（每批 1000 条，间隔 1 秒）
-- 🔴 禁止一次性 UPDATE 全表

SET @batch_size = 1000;
SET @affected = 1;

WHILE @affected > 0 DO
    UPDATE target_table
    SET column_name = new_value
    WHERE condition
      AND id > @last_id
    ORDER BY id
    LIMIT @batch_size;

    SET @affected = ROW_COUNT();
    -- 记录最后处理的 ID
    -- 等待 1 秒，释放锁，让其他查询有机会执行
    DO SLEEP(1);
END WHILE;
```

## 向后兼容策略

当数据库变更需要与应用代码同步部署时，推荐采用分阶段发布：

| 阶段 | 操作 | 说明 |
|:----:|------|------|
| 1 | 加新字段/新表（保留旧结构） | 数据库先变更，旧代码不受影响 |
| 2 | 应用代码双写（新旧字段/表都写入） | 新代码上线，同时写新旧 |
| 3 | 数据迁移（旧数据同步到新结构） | 历史数据对齐 |
| 4 | 应用代码切读（从新结构读取） | 确认新结构数据正确 |
| 5 | 清理旧字段/旧表 | 确认无引用后删除 |

> 🔴 **禁止一步到位**：同时删旧字段 + 上新代码 = 部署窗口内服务不可用。
> 除非是停机维护窗口，否则必须采用分阶段发布。
