> 当 N04 意图识别将任务判定为 `dev > 性能优化` 时，由路由表自动加载本 checklist，
> 叠加到 `build/README.md` 默认流程上执行。CP1→CP2→CP3 不变，每个阶段增加优化专属步骤。

## 适用场景

| 资源占用降低 | "服务内存占用太高，需要优化" |
| 构建/启动加速 | "项目启动太慢了，能加速吗" |

## 阶段叠加规则

### 阶段1 — 需求理解（叠加：建立性能基线）

在默认步骤 1.1~1.5 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 1.6 | 确定优化目标指标（响应时间 / QPS / 资源占用 / 构建时间） | 指标定义 |
| 1.7 | 收集当前性能基线数据（用户提供 或 AI 通过代码分析推断） | 基线数据 |
| 1.8 | 设定优化目标值（如 P99 < 200ms、QPS > 1000） | 目标值 |
| 1.9 | 确定测试环境说明（数据量、并发数、硬件配置） | 环境信息 |

**CP1 追加输出：**

```text
📊 性能基线:
  指标: [响应时间 P50/P95/P99 | QPS | CPU/内存 | ...]
  当前值: [当前性能数据]
  目标值: [优化后目标]
  测试环境: [环境说明]
```

### 阶段2 — 技术方案（叠加：瓶颈分析 + 多方案对比）

在默认步骤 2.1~2.5 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 2.6 | 定位性能瓶颈（代码分析 / 算法复杂度 / 数据库查询 / 网络请求 / 资源竞争） | 瓶颈定位 |
| 2.7 | 列出候选优化方案（至少 2 个），对比优缺点 | 方案对比表 |
| 2.8 | 选择最佳方案并说明理由 | 推荐方案 |
| 2.9 | 评估优化风险（功能回归 / 数据一致性 / 缓存穿透等） | 风险清单 |

**CP2 追加输出：**

```text
🔍 瓶颈分析:
  瓶颈位置: [文件+行号/函数名]
  瓶颈原因: [具体原因]
  预期改进: [优化后预估值]

📋 候选方案:
  方案A: [描述] — 优势: ... / 风险: ...
  方案B: [描述] — 优势: ... / 风险: ...
  推荐: 方案[X]，理由: ...
```

### 阶段3 — 代码实现（叠加：对比验证）

在默认编码执行规则之上，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 3.X+1 | 实现优化代码（按 IMPL-PLAN 执行） | 代码变更 |
| 3.X+2 | 验证功能正确性（优化未破坏原有功能） | 功能验证 |
| 3.X+3 | 收集优化后性能数据（如可在 AI 环境中测量） | 优化后数据 |
| 3.X+4 | 对比优化前后数据，确认达到目标 | 对比结果 |

**🔴 对比验证规则：**

| 规则 | 说明 |
|------|------|
| 有基线才对比 | 必须有阶段1的基线数据，禁止无基线声称"性能提升 X%" |
| 多次测试 | 性能数据应基于多次测试取平均，避免单次测试偶然性 |
| 无退化 | 确认其他性能指标未明显下降（如优化了响应时间但 CPU 翻倍，需说明） |
| 边界验证 | 极端情况下（大数据量/高并发）仍能正常工作 |

### 阶段4 — 产物交付（叠加：对比报告）

在默认交付步骤之上，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 4.X+1 | 生成性能对比报告（基线 vs 优化后） | 对比数据 |
| 4.X+2 | 总结优化经验和后续建议 | 经验记录 |

## 归档文档补充

除 build 默认归档文档（01-需求定义 / 02-技术方案 / 03-实施方案 / IMPL-PLAN）外，
优化类任务的归档文档应在相应文件中包含以下专属内容：

| 归属文件 | 追加内容 |
|---------|---------|
| `01-需求定义.md` | 性能基线数据 + 优化目标值 + 测试环境说明 |
| `02-技术方案.md` | 瓶颈分析 + 候选方案对比 + 推荐方案理由 + 风险评估 |
| 报告文件 | 优化前后对比数据 + 达标/未达标结论 + 后续优化建议 |

**文件规模划分（沿用 build 默认规则）：**

| 条件 | 处理方式 |
|------|---------|
| < 5 文件变更 | lite 模板，归档内容可精简 |
| ≥ 5 文件变更 | core 模板，完整内容 |

## 瓶颈定位参考

### 常见瓶颈类型

| 类型 | 排查方法 | 典型优化手段 |
|------|---------|-------------|
| 数据库查询 | 分析查询语句 / 执行计划 / 索引使用 | 添加索引 · 优化 SQL · 读写分离 |
| 算法复杂度 | 分析时间/空间复杂度 | 更优算法 · 数据结构替换 |
| 网络 I/O | 分析请求次数 / 响应大小 | 批量请求 · 缓存 · 压缩 |
| 内存使用 | 分析对象创建 / GC 频率 | 对象池 · 流式处理 · 减少拷贝 |
| 并发竞争 | 分析锁争用 / 线程等待 | 减少锁粒度 · 无锁数据结构 · 异步化 |
| 构建/启动 | 分析依赖加载 / 编译时间 | 懒加载 · 并行构建 · 缓存 |

### 数据库优化专项

> 当优化涉及数据库时，额外执行以下检查：

| 检查项 | 说明 |
|--------|------|
| 执行计划分析 | 添加索引前必须先分析执行计划（EXPLAIN） |
| 索引成本评估 | 评估索引大小和写入性能影响 |
| 锁表风险 | 大表加索引的锁表时间评估，考虑在线 DDL |
| 缓存失效策略 | 如引入缓存，说明失效策略和更新机制 |
| 缓存穿透/雪崩 | 评估缓存穿透和缓存雪崩风险 |

### 缓存优化专项

> 当优化涉及引入缓存时，额外执行以下检查：

| 检查项 | 说明 |
|--------|------|
| 缓存策略 | Cache-Aside / Write-Through / Write-Behind 选择理由 |
| 失效策略 | TTL / 事件驱动 / 手动清除 |
| 更新机制 | 缓存与数据源的一致性保证方案 |
| 命中率预估 | 预估缓存命中率，评估缓存收益 |
| 穿透防护 | 空值缓存 / 布隆过滤器 等防穿透措施 |
| 雪崩防护 | 过期时间随机化 / 多级缓存 等防雪崩措施 |

## 性能指标参考

| 指标 | 说明 | 常用工具/方法 |
|------|------|-------------|
| 响应时间 P50/P95/P99 | 百分位响应时间 | wrk · k6 · ab |
| QPS / TPS | 每秒请求/事务数 | wrk · k6 · jmeter |
| CPU 使用率 | 处理器占用百分比 | top · htop · perf |
| 内存使用量 | 堆/非堆内存 | heapdump · v8-profiler |
| 数据库连接数 | 活跃/空闲连接 | SHOW PROCESSLIST |
| 网络吞吐量 | 传输字节数 | iftop · nethogs |
| GC 暂停时间 | 垃圾回收耗时 | --trace-gc · GC 日志 |

## 优化专属验证清单

在 N13 合规自检阶段，叠加以下检查：

| # | 检查项 | 通过标准 |
|:-:|--------|---------|
| O1 | 性能基线存在 | 有优化前的性能数据（非推测） |
| O2 | 对比数据存在 | 有优化后的性能数据 |
| O3 | 多次测试 | 数据基于多次测试取平均（非单次） |
| O4 | 达到目标 | 优化后性能达到阶段1设定的目标值 |
| O5 | 稳定性确认 | 多次测试结果一致，无大幅波动 |
| O6 | 无退化 | 其他性能指标未明显下降 |
| O7 | 功能正常 | 优化未破坏原有功能 |
| O8 | 边界验证 | 极端条件下仍能正常工作 |
| O9 | 归档完整 | 基线数据 + 对比数据 + 优化经验已写入归档文档 |

## 报告追加章节

优化类任务的报告（使用 `templates/report-build.md`）追加以下章节：

| 章节 | 内容 |
|------|------|
| 性能基线 | 优化前的性能指标（指标名 + 数值 + 测试条件） |
| 瓶颈分析 | 瓶颈位置 + 原因 + 定位方法 |
| 优化方案对比 | 候选方案 + 选择理由 |
| 优化结果 | 优化后性能指标 + 与基线对比 + 达标/未达标 |
| 后续建议 | 可进一步优化的方向 + 监控建议 |
