## 核心原则

```text
🔴 重构铁律：行为不变 · 分步推进 · 每步可验证 · 每步可回滚
```

| 原则 | 说明 |
|------|------|
| 行为不变 | 重构不改变外部行为，只改善内部结构。如需同时改行为，必须在 CP1 中明确声明 |
| 分步推进 | 大型重构必须分阶段（Phase），每阶段独立可交付 |
| 每步可验证 | 每个阶段完成后必须验证功能不受影响（测试 / diagnostics / 人工确认） |
| 每步可回滚 | 每个阶段必须有明确的回滚方案，不能出现"改到一半无法回退"的状态 |
| 修复后三步扫描 | 🔴 约束 #18 — 重构本质上是"修复架构问题"，必须执行修复后三步扫描 |

## 阶段叠加规则

### 阶段1 — 需求理解（叠加：现状分析 + 重构范围界定）

在默认步骤 1.1~1.5 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 1.6 | 分析当前架构/代码的问题（为什么要重构？痛点是什么？） | 问题清单 |
| 1.7 | 确定重构范围：哪些模块/文件需要改 · 哪些保持不变 | 范围边界 |
| 1.8 | 确定重构目标：期望达到什么效果（可维护性 / 性能 / 扩展性 / 代码质量） | 目标定义 |
| 1.9 | 评估重构规模：影响的文件数 · 涉及的模块数 · 是否需要分阶段 | 规模评估 |
| 1.10 | 识别外部依赖方：哪些其他模块/服务依赖被重构的代码 | 依赖方清单 |

**重构规模判定：**

| 规模 | 标准 | 执行策略 |
|:----:|------|---------|
| 小型 | ≤ 5 文件，单模块内部 | 一次性执行，无需分阶段 |
| 中型 | 6~15 文件，跨 2~3 模块 | 建议分 2~3 阶段 |
| 大型 | > 15 文件，跨多模块 / 涉及接口变更 | 🔴 必须分阶段，每阶段独立 CP3 |

**CP1 追加输出：**

```text
🏗️ 重构概况:
  问题/痛点: [当前架构的具体问题]
  重构目标: [期望达到的效果]
  重构范围: [涉及的模块/文件]
  不变范围: [明确不动的部分]
  规模评估: [小型/中型/大型] — [N 个文件 · M 个模块]
  外部依赖方: [依赖被重构代码的其他模块清单]
  行为变更: [是/否] — [如有，明确列出哪些行为会改变]
```

### 阶段2 — 技术方案（叠加：目标架构 + 分阶段计划 + 兼容性分析）

在默认步骤 2.1~2.5 之后，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 2.6 | 设计目标架构（模块划分 · 依赖关系 · 接口定义） | 目标架构 |
| 2.7 | 对比现状与目标的差异（current → target gap 分析） | 差异清单 |
| 2.8 | 制定分阶段计划（中型/大型重构必须，小型可选） | 阶段划分 |
| 2.9 | 分析兼容性影响：接口变更是否影响调用方 · 数据结构变更是否影响存储 | 兼容性分析 |
| 2.10 | 制定回滚方案：每个阶段的回滚策略（🔴 P0 操作必须有具体可执行的回滚步骤） | 回滚方案 |
| 2.11 | 确定验证策略：每个阶段用什么方式验证行为不变（测试 / diagnostics / 对比） | 验证策略 |

**CP2 追加输出：**

```text
🎯 目标架构:
  [架构描述 / 模块划分 / 关键设计决策]

📋 分阶段计划（中/大型重构）:
  Phase 1: [目标] — 涉及 [N 文件] — 回滚方案: [具体步骤]
  Phase 2: [目标] — 涉及 [N 文件] — 回滚方案: [具体步骤]
  ...

⚠️ 兼容性影响:
  接口变更: [有/无] — [变更列表]
  数据结构变更: [有/无] — [变更列表]
  调用方影响: [需同步更新的调用方清单]
```

**🔴 大型重构的分阶段规则：**

| 规则 | 说明 |
|------|------|
| 阶段独立 | 每个阶段完成后系统必须可正常运行（不存在"半成品"状态） |
| 阶段验证 | 每个阶段完成后必须验证功能不受影响 |
| 阶段可回滚 | 每个阶段必须有独立的回滚方案 |
| 逐阶段 CP3 | 每个阶段生成独立的实施方案，独立走 CP3 确认 |
| 顺序依赖 | 后续阶段可依赖前序阶段的产出，但不应有循环依赖 |

### 阶段3 — 代码实现（叠加：修复后三步扫描 + 增量验证）

在默认编码执行规则之上，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 3.X+1 | 按 IMPL-PLAN 逐文件执行重构（严格按方案，不顺带改其他代码） | 代码变更 |
| 3.X+2 | 每个文件修改后运行 `diagnostics`（约束 #19） | 诊断结果 |
| 3.X+3 | 每个文件修改后检查关联文件是否需要同步更新（约束 #17） | 同步更新 |
| 3.X+4 | 执行修复后三步扫描（约束 #18 — 重构专属，见下方） | 扫描结果 |
| 3.X+5 | 阶段性验证：当前阶段的所有文件修改完成后，整体验证功能正常 | 验证结果 |

**🔴 重构专属的三步扫描要点：**

| 步骤 | 重构场景中的重点 |
|------|----------------|
| Step 1 同类全局扫描 | 搜索项目中是否还有使用旧模式/旧 API/旧结构的代码 → 确认零残留 |
| Step 2 数据联动扫描 | 被重构的函数/类/接口的所有调用方是否已同步更新 → 确认零遗漏 |
| Step 3 声明前复核 | 额外确认：旧代码是否已清理？是否有被注释而非删除的废弃代码？ |

**重构 Step 3 扩展复核清单：**

```text
□ grep 全项目确认旧模式/旧 API 零残留？
□ 所有调用方已更新到新接口/新结构？
□ 旧代码已清理（非注释保留）？
□ 导入/导出路径已全部更新？
□ 类型定义（如 TypeScript）已同步更新？
□ diagnostics 无 error？
□ 测试通过（如有）？
□ 修改范围与 CP3 方案一致（未超出范围）？
□ 回滚方案仍然有效（修改未使回滚方案失效）？
```

### 阶段4 — 产物交付（叠加：回归验证 + 重构总结）

在默认交付步骤之上，追加：

| 步骤 | 执行内容 | 产出 |
|:----:|---------|------|
| 4.X+1 | 回归测试验证：运行完整测试（如有），确认无回归 | 回归结果 |
| 4.X+2 | 架构对比总结：current vs target 的达成情况 | 达成评估 |
| 4.X+3 | 遗留清单：本次重构未覆盖的改进点（如有） | 遗留清单 |
| 4.X+4 | 如为分阶段重构的中间阶段 — 输出下一阶段的前置条件和建议 | 下阶段建议 |

## 归档文档补充

除 build 默认归档文档（01-需求定义 / 02-技术方案 / 03-实施方案 / IMPL-PLAN）外，
重构类任务的归档文档应在相应文件中包含以下专属内容：

| 归属文件 | 追加内容 |
|---------|---------|
| `01-需求定义.md` | 重构问题/痛点清单 + 重构目标 + 范围边界 + 规模评估 |
| `02-技术方案.md` | 现状架构 + 目标架构 + 差异分析 + 分阶段计划 + 兼容性分析 + 回滚方案 |
| 报告文件 | 重构达成评估 + 遗留清单 + 后续建议 |

**分阶段重构的归档结构（大型重构）：**

```text
projects/<project>/requirements/<中文描述>/
├── 01-需求定义.md              # 整体需求（含所有阶段的范围说明）
├── 02-技术方案.md              # 整体方案（含目标架构 + 阶段划分）
├── 03-实施方案/
│   ├── phase-1/               # 阶段1 实施方案
│   │   ├── README.md          # 阶段1 索引（前检查 + 步骤 + 验证）
│   │   └── <Module>.md        # 各文件变更
│   ├── phase-2/               # 阶段2 实施方案
│   │   ├── README.md
│   │   └── ...
│   └── ...
├── 04-实施计划.md     # 全局实施计划（跨阶段任务追踪）
└── docs/
    ├── architecture.md        # 架构对比文档（可选）
    └── migration-guide.md     # 迁移指南（如有接口变更，可选）
```

## 兼容性处理策略

当重构涉及接口/API 变更时，需选择适当的兼容性策略：

| 策略 | 适用场景 | 说明 |
|------|---------|------|
| 透明重构 | 内部实现变更，接口不变 | 最安全，调用方无感知 |
| 适配器模式 | 需要新接口但要兼容旧调用 | 旧接口 → 适配器 → 新实现 |
| 渐进迁移 | 调用方众多，无法一次性更新 | 新旧接口共存 → 标记旧接口 deprecated → 逐步迁移 → 移除旧接口 |
| 大爆炸式 | 调用方少、控制范围内 | 一次性切换所有调用方到新接口 |

**🔴 渐进迁移的额外规则：**

| 规则 | 说明 |
|------|------|
| deprecated 标记 | 旧接口必须添加 `@deprecated` 注释或 console.warn |
| 迁移追踪 | 记录所有待迁移的调用方，在 IMPL-PLAN 中追踪 |
| 移除时机 | 所有调用方迁移完成后，在后续任务中移除旧接口 |
| 零残留验证 | 移除旧接口前 grep 确认零调用 |

## 常见重构模式参考

| 重构模式 | 触发信号 | 操作要点 |
|---------|---------|---------|
| 提取函数/方法 | 函数过长、职责混杂 | 识别可独立的逻辑块 → 提取 → 验证调用方 |
| 提取模块/类 | 文件过大、职责不清 | 按职责拆分 → 更新导入路径 → 验证导出完整 |
| 合并重复代码 | 多处逻辑高度相似 | 🔴 逐一验证"相似"而非"相同"（事故参考 O1） → 提取公共函数 |
| 替换依赖 | 依赖过时/不维护 | 新旧 API 映射 → 逐文件替换 → 三步扫描零残留 |
| 数据结构重组 | 结构嵌套过深/冗余 | 新结构设计 → 数据迁移/适配 → 调用方同步 |
| 目录结构调整 | 目录组织不合理 | 新结构设计 → 移动文件 → 更新所有导入路径 → 三步扫描 |
| API 重新设计 | 接口不一致/难用 | 新 API 设计 → 适配器 / 渐进迁移 → 移除旧 API |
| 类型系统完善 | `any` 过多 / 类型缺失 | 逐模块补充类型 → 修复类型错误 → 启用严格模式 |

**🔴 合并重复代码的事故教训（O1）：**

> 方案声称"5 处内容重复可用 snippet 统一"，实际验证发现是同概念不同详略视角，合并反而丢失信息。
> **规则：合并前必须逐一对比每处内容的具体差异，确认确实是"相同"而非"相似"。**

## 重构专属验证清单

在 N13 合规自检阶段，叠加以下检查：

| # | 检查项 | 通过标准 |
|:-:|--------|---------|
| R1 | 行为不变 | 重构后功能行为与重构前一致（或变更已在 CP1 中声明） |
| R2 | 旧模式零残留 | grep 全项目确认旧 API / 旧结构 / 旧导入路径零残留 |
| R3 | 调用方同步 | 所有依赖被重构代码的调用方已更新 |
| R4 | 导入路径正确 | 所有导入/导出路径已更新，无断链 |
| R5 | 类型定义同步 | TypeScript 项目的类型定义已同步更新 |
| R6 | 废弃代码清理 | 旧代码已删除（非注释保留），无死代码残留 |
| R7 | diagnostics 通过 | `diagnostics` 无 error |
| R8 | 测试通过 | 测试通过（如有测试覆盖） |
| R9 | 三步扫描完成 | 修复后三步扫描已执行（约束 #18） |
| R10 | 回滚方案有效 | 回滚方案未因修改而失效 |
| R11 | 归档完整 | 需求定义 + 技术方案 + 实施方案 + IMPL-PLAN 文件均存在 |
| R12 | 分阶段独立 | 每个阶段完成后系统可正常运行（大型重构） |

## 报告追加章节

重构类任务的报告（使用 `templates/report-build.md`）追加以下章节：

| 章节 | 内容 |
|------|------|

| 三步扫描结果 | 旧模式扫描结果 + 调用方同步结果 + 零残留确认 |
| 遗留清单 | 本次未覆盖的改进点 + 后续建议 |

## 与其他工作流/变体的边界

| 场景 | 推荐 | 原因 |
|------|------|------|
| 纯内部重构（不改行为） | `build` + 本 checklist | 标准重构路径 |
| 重构 + 新功能 | 先 `build`(重构) → 再 `build`(新需求) | 拆分为两个任务，避免混杂 |
| 性能优化式重构 | `build` + `checklist-optimization.md` | 以性能为目标的改造走优化 checklist |
| 简单 Bug 修复中发现需要重构 | 先 `fix` 完成 → 再开 `build`(重构) | 避免修复范围蔓延 |
| 数据库 Schema 重构 | `build` + `checklist-database.md` | 涉及 DDL 走数据库 checklist |
| 代码审查发现需要重构 | 先 `analyze` 出报告 → 再开 `build`(重构) | 分析与执行分离 |
