# CP Checklist 模板

> 确认点执行跟踪清单。在涉及代码修改的工作流（build / fix）中使用。
> AI 在每个 CP 输出时，按本模板格式化确认内容；用户确认后记录到记忆文件。

**版本**: v3.0.0
**最后更新**: 2026-03-10

---

## CP1 — 需求/问题确认

**工作流**: build（需求确认） / fix（问题确认）

### build 模式

```text
📋 CP1 — 需求确认

🎯 需求理解:
  [用 AI 自己的话复述用户需求的核心目标]

📂 改造范围:
  需要改: [模块/文件清单]
  不需要改: [明确排除的部分]

🔒 关键约束:
  - [技术栈约束]
  - [性能约束]
  - [兼容性约束]

🏷️ 场景变体: [新需求(默认) / 重构 / 数据库 / 项目初始化 / 性能优化]

请确认以上理解是否正确？如有偏差请指出，我会修正后重新确认。
```

### fix 模式

```text
📋 CP1 — 问题确认

🔍 问题定位:
  文件: [文件路径]
  位置: [行号 / 函数名]
  根因: [根因描述]

💥 影响范围:
  - [受影响的功能/模块]
  - [关联文件]

🛠️ 修复方向:
  [修复思路概述]

🏷️ 场景变体: [常规Bug(默认) / 事故处理 / 安全修复]

请确认根因定位和修复方向是否正确？
```

### 确认记录格式（写入记忆）

```text
CP1 [✅确认 / ✏️修正] — [时间戳]
  确认内容: [一句话摘要]
  修正内容: [如有修正，记录修正点]
```

## CP2 — 方案确认

**工作流**: build（技术方案确认） / fix（修复方案确认）

### build 模式

```text
📋 CP2 — 技术方案确认

🏗️ 技术方案:
  [方案概述 · 关键设计决策]

📝 文件修改清单:
  | # | 文件 | 操作 | 说明 |
  |:-:|------|:----:|------|
  | 1 | path/file.ts | 新建 | [说明] |
  | 2 | path/file.ts | 修改 | [说明] |
  | 3 | path/file.ts | 删除 | [说明] |

📦 依赖变更:
  - [新增/升级/移除的依赖]

🔴 P0 高危操作:
  - [数据库 DDL / 配置变更 / 删除操作等（如有）]

📏 模板选择: [lite（< 5 文件）/ core（≥ 5 文件）]

⚠️ 提醒: CP2 确认 ≠ 代码执行授权。确认后我会生成实施方案，
   再由 CP3 确认后才开始编码。

请确认技术方案是否可行？可选择：✅ 确认 / ✏️ 修改 / ❌ 取消
```

### fix 模式

```text
📋 CP2 — 修复方案确认

🛠️ 修复方案:
  [修复方案描述 · 最小化变更原则]

📝 修改文件清单:
  | # | 文件 | 操作 | 变更说明 |
  |:-:|------|:----:|---------|
  | 1 | path/file.ts | 修改 | [说明] |

⚠️ 副作用评估:
  - [关联文件影响]
  - [回归风险]

🔴 P0 高危操作:
  - [（如有）]

💡 说明: fix 工作流中 CP2 确认 = 变更授权。确认后我将直接执行修复。
   如修复涉及 ≥5 个文件，建议切换到 build 工作流（含 CP3 + 实施方案）。

请确认修复方案？可选择：✅ 确认 / ✏️ 修改 / ❌ 取消
```

### 确认记录格式（写入记忆）

```text
CP2 [✅确认 / ✏️修正 / ❌取消] — [时间戳]
  确认内容: [一句话摘要]
  修正内容: [如有修正，记录修正点]
  P0 操作: [有/无 — 具体内容]
```

## CP3 — 实施方案确认（🟢 代码执行授权点）

**工作流**: 仅 build（fix 不经过 CP3）

```text
📋 CP3 — 实施方案确认 🟢

📄 04-实施计划.md 已生成:
  路径: [projects/<project>/requirements/<描述>/04-实施计划.md]

📋 实施步骤（按依赖顺序）:
  | # | 任务 | 文件 | 状态 | 说明 |
  |:-:|------|------|:----:|------|
  | T1 | [任务描述] | path/file.ts | ⏳ | [说明] |
  | T2 | [任务描述] | path/file.ts | ⏳ | [说明] |
  | T3 | [任务描述] | path/file.ts | ⏳ | [说明] |

🔗 依赖关系: T1 → T2 → T3

🔄 中断恢复: 按任务编号从最后一个 ✅ 的下一个开始

🟢 确认后将开始代码实现。
   执行过程中如发现需要额外修改（超出方案范围），我会暂停并说明。

请确认实施方案？可选择：✅ 确认（开始编码） / ✏️ 修改 / ❌ 取消（回到方案阶段）
```

### 确认记录格式（写入记忆）

```text
CP3 [✅确认 / ✏️修正 / ❌取消] — [时间戳]
  🟢 代码执行授权: 已生效
  IMPL-PLAN: [路径]
  任务数: N 个
```

**工作流**: build（阶段4，代码实现完成后）

```text

代码实现已完成。请选择测试策略:

  A. 生成单元测试（推荐）
```

## 🔴 CP 执行规则速查

| 规则 | 说明 |
|------|------|
| CP 不可跳过 | 即使用户暗示跳过，仍必须输出 CP 等待确认 |
| CP 不可合并 | CP1 和 CP2 不合并为一个输出；CP2 和 CP3 不合并 |
| CP 必须等待 | 输出 CP 后必须等待用户明确响应（✅/✏️/❌） |
| CP2 ≠ 代码授权 | build 流程中 CP2 确认后不可直接编码（需 CP3） |
| fix 无 CP3 | fix 流程 CP2 确认 = 变更授权，直接执行修复 |
| 确认写入记忆 | 每次 CP 确认/修正/取消都必须写入记忆的对话记录 |
| 用户"好的"处理 | "好的"可视为对**当前**CP 的确认，不可视为多个 CP 的确认 |
