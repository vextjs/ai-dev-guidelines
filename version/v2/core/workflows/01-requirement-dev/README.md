# 工作流 1: 需求开发（AI 执行手册）

> **任务类型**: 需求开发  
> **适用场景**: 新功能开发、功能增强、第三方集成

---

## 🚨 强制执行规则（违反=规范失效）

```
❌ 绝对禁止的行为：
  - 收到需求后直接分析代码、直接执行代码修改
  - 认为"需求很清晰"就跳过 CP1 需求理解确认
  - 输出技术方案后不等用户确认就执行代码（跳过 CP2）
  - 🆕 CP2 确认后直接执行代码变更，跳过 CP3 实施方案生成+确认（FIX-015）
  - 将用户对 CP2 的"确认"等同于 CP3 的"确认"，合并/跳过确认点（FIX-015）
  - 在同一条回复中同时输出"方案"和"执行结果"
  - 未执行 Layer 3 意图推断，将最终目的为变更的请求误判为"分析"任务而跳过确认点（FIX-010 → Layer 3）

✅ 正确的执行顺序：
  收到需求 → 输出需求理解 → 等待 CP1 确认
           → 输出技术方案 → 等待 CP2 确认
           → 输出实施方案 → 等待 CP3 确认
           → 执行代码

🔴 CP 间过渡规则（FIX-015 — 每个 CP 确认后的唯一合法下一步）：
  CP1 确认后 → 唯一合法动作 = 生成技术方案 → 输出并等待 CP2
  CP2 确认后 → 唯一合法动作 = 生成实施方案（03-实施方案/ + 04-实施计划.md）→ 输出并等待 CP3
  CP3 确认后 → 唯一合法动作 = 按实施方案逐文件执行代码变更
  ⚠️ 用户对 CPn 的"确认"仅代表该阶段产物被认可，不等于授权执行后续阶段的操作
  ⚠️ 任何 CP 之间不得合并、跳过或假设用户同时确认多个阶段

🔴 意图推断判定（v2.13.0，FIX-010 升级为 Layer 3）：
  通过 Layer 3 三问分析用户最终目的：最终目的是变更 → 按需求开发处理
  详见 core/workflows/00-task-identification/rules.md §Layer 3 意图推断
```

## 🎯 统一 7 阶段流程

> 🔴 **v2.13.0 起移除快速/完整双模式**（FIX-016 — 快速模式是多次流程跳过问题的温床 issue#11/#14/#15），所有需求开发统一为 7 阶段流程。

| 项目 | 说明 |
|-----|------|
| 流程 | 7 个阶段（理解→方案→实施方案→执行→报告→归档文档→验证检查） |
| 确认点 | CP1→CP2→CP3（强制）+ CP4（测试，按需）+ CP5（文档，按需） |
| 文档产物 | 所有需求均须生成归档文档文件（小需求内容可精简，但文件必须存在） |

### 模板选择规则

| 条件 | 模板 | 归档文档 |
|-----|------|---------|
| 小需求（< 5 个文件、单模块） | `core/templates/lite/` | 精简内容，但文件必须创建 |
| 大需求（>= 5 个文件、多模块协作） | `core/templates/core/` | 完整内容 |

## 📋 执行流程（统一 7 阶段）

```
阶段 1: 理解需求        →  CP1 确认
阶段 2: 技术方案        →  CP2 确认
阶段 3: 实施方案        →  CP3 确认（含实施前检查）
阶段 4: 执行实现        →  对照实施方案逐文件执行
阶段 5: 完成报告        →  总结
阶段 6: 生成归档文档    →  CP5 确认（01-需求定义 / 02-技术方案 / 03-实施方案 等）
阶段 7: 验证检查        →  代码和文档质量检查
```

### 阶段 1: 理解需求

**目标**: 确保正确理解用户需求

**执行步骤**:
1. 阅读用户需求描述
2. 分析涉及的模块和功能
3. 用一句话复述核心需求
4. 列出关键改动点

**输出格式**:
```markdown
📋 需求理解:
- 核心需求: [一句话描述]
- 涉及模块: [模块1, 模块2]
- 关键改动: 
  1. [改动点1]
  2. [改动点2]

请确认以上理解是否正确？[确认/修改]
```

**⏸️ CP1 确认点（绝对强制）**:
> 🔴 **无论需求描述多清晰，都必须输出需求理解并等待用户明确回复"确认"后才能继续。**
> 🔴 **禁止"需求很清晰所以跳过确认"，禁止直接进入下一阶段，禁止边分析边执行。**

---

### 阶段 2: 技术方案

**目标**: 设计怎么做（只描述方案，不写具体代码）

**执行步骤**:
1. 分析相关代码文件
2. 确定修改范围和架构设计
3. 生成技术方案（模块设计、接口契约、数据模型、流程）

**参考模板**: 小需求 [lite/technical-lite.md](../../templates/lite/technical-lite.md)，大需求 [core/technical-template.md](../../templates/core/technical-template.md)

**输出格式**:
```markdown
📝 技术方案:

## 架构设计
[简述架构/模块划分]

## 改动清单
| 文件 | 修改类型 | 说明 |
|-----|---------|------|
| xxx.ts | 新增 | xxx |
| yyy.ts | 修改 | xxx |

## 实现要点
1. [要点1]
2. [要点2]

## 影响评估
- 风险: 低/中/高
- 回滚方案: [简述]

请确认方案是否可行？[确认/修改]
```

**⏸️ CP2 确认点（绝对强制）**:
> 🔴 **技术方案必须输出完整，且等待用户明确回复"确认"后才能进入实施阶段。**
> 🔴 **禁止在输出技术方案的同时或之后立即执行代码，禁止"顺手"就把代码改了。**

---

### 阶段 3: 实施方案

**目标**: 生成可执行的实施方案，供用户确认后才执行

> 🔴 **CP2→CP3 强制过渡（硬阻塞 — FIX-015）**
> 
> CP2 确认 = 用户认可技术方案的设计思路，**不等于授权执行代码**。
> CP2 确认后的唯一合法动作：生成 `03-实施方案/` + `04-实施计划.md` → 输出 CP3 等待用户确认。
> 
> ❌ 禁止：CP2 确认后直接创建/修改任何业务源码文件
> ❌ 禁止：将 CP2 的"确认"解读为"可以开始实施"
> ✅ 唯一正确路径：CP2 ✅ → 生成实施方案文档 → 输出 CP3 → 等待用户确认 → 执行代码
> 
> 事故参考：issue#15 — AI 在 CP2 确认后直接执行 Phase 0 DSL 迁移 + Phase 1 Model 变更，被用户发现后回滚

**核心原则**:
- 技术方案说"怎么做"（架构、设计思路）
- 实施方案说"具体改什么"（每个文件的完整变更内容）
- **用户确认所有变更内容后，才开始执行代码**

**执行步骤**:
1. 基于技术方案，为每个需要新增/修改的文件生成变更文档
2. 生成实施前检查清单（前置条件验证）
3. 生成实施步骤（按依赖顺序排列）
4. 生成验证清单

**实施方案结构**:

```yaml
小需求（< 5 个文件）:
  直接在聊天中输出:
    1. 实施前检查清单
    2. 每个文件的完整变更内容
    3. 实施步骤（按依赖顺序）
    4. 验证清单

大需求（>= 5 个文件）:
  生成 03-实施方案/ 目录:
    projects/<project>/requirements/<中文描述>/
    ├── 03-实施方案/
    │   ├── README.md              # 实施总索引
    │   ├── <Model>.md             # 数据模型变更（完整代码）
    │   ├── <Service>.md           # 服务层变更（完整代码）
    │   ├── <Controller>.md        # 控制层变更（完整代码）
    │   ├── <config>.md            # 配置文件变更
    │   └── <test>.md              # 测试文件
    └── scripts/
```

**03-implementation/README.md 必须包含**:

```markdown
## 实施前检查
- [ ] 确认 xxx 服务/模块已存在
- [ ] 确认 xxx 依赖已安装
- [ ] 确认数据库连接正常

## 实施文件清单
| 文件 | 路径 | 操作 | 文档 |
|------|------|------|------|
| 数据模型 | app/model/xxx.ts | 新增 | [查看](./Xxx.model.md) |
| 服务层 | app/service/XxxService.ts | 新增 | [查看](./XxxService.md) |
| 控制器 | app/controller/xxx.ts | 修改 | [查看](./XxxController.md) |

## 实施步骤（按依赖顺序）
步骤1: 配置 → 步骤2: 模型 → 步骤3: Service → 步骤4: Controller → 步骤5: 测试

## 验证清单
- [ ] 功能验证项...
- [ ] 性能验证项...
```

**每个变更文件文档格式**:

```markdown
# [模块名称]

**文件路径**: `app/service/XxxService.ts`
**操作**: 新增 / 修改

[完整的代码内容]
```

**🔴 04-实施计划.md — 实施计划（强制生成）**:

> 所有涉及代码变更的需求，无论大小，都必须在阶段 3 生成 04-实施计划.md。
> 它与 03-实施方案/ 互补：PLAN 管理项目级进度（任务编号、依赖顺序、状态追踪），03-实施方案/ 管理代码级变更（每个文件的完整代码）。

```yaml
04-实施计划.md — 实施计划:

  定义: 项目级实施计划文件，用于任务编号追踪、进度管理和中断恢复
  生成时机: 阶段 3（实施方案）— 与 03-实施方案/ 同步生成
  存储位置: projects/<project>/requirements/<中文描述>/04-实施计划.md

  与 03-实施方案/ 的关系:
    03-实施方案/: 代码级 — 每个文件具体改什么（变更内容、完整代码）
    04-实施计划.md: 项目级 — 任务编号、依赖顺序、进度状态
    两者互补: PLAN 引用 03-实施方案/ 中的模块文档

  大需求（>= 5 个文件 或 多模块 或 预估 > 1 天）:
    必须包含:
      1. 总体进度看板（Phase/百分比/状态）
      2. 任务编号表（编号/任务名/估时/依赖/状态 🔲/✅）
      3. 依赖拓扑（Level 分层或文字描述依赖关系）
      4. Phase 划分（如有多阶段）
      5. 完成标准
      6. 风险事项（可选）

  小需求（< 5 个文件）:
    简化版，必须包含:
      1. 任务编号表（编号/任务名/状态 🔲/✅）
      2. 实施顺序（按依赖排列）
    示例（~10 行）:
      ## 实施计划
      | # | 任务 | 状态 |
      |---|------|:----:|
      | 1 | 创建配置文件 config/rate-limit.ts | 🔲 |
      | 2 | 创建中间件 middleware/rate-limit.ts | 🔲 |
      | 3 | 注册到 app.ts | 🔲 |
      实施顺序: 1 → 2 → 3（按依赖）

  状态追踪:
    🔲 待开始 → ✅ 已完成
    编码检查点（约束 #18）在每完成一个任务编号时更新 🔲→✅
    中断恢复: 新会话读 PLAN 找到第一个 🔲 任务 → 从该任务继续
```

**⏸️ CP3 确认点**: 用户确认 实施计划 + 实施方案（变更内容）后才执行

### 阶段 4: 执行实现

**目标**: 对照实施方案，逐个文件执行变更

**前置条件**: CP3 已确认 + 实施前检查全部通过

> 🔴 **方案-实施一致性**（约束）: 执行过程中若发现实施方案存在问题，必须暂停执行、修订方案文档并重新走 CP3 确认，禁止"默默修正"。
> 详见 [AI-CODE-STANDARDS.md §0.3](../../best-practices/AI-CODE-STANDARDS.md)

**执行步骤**:
1. 先执行实施前检查清单，逐项验证
2. 按实施方案中的步骤顺序，逐文件执行
3. 每完成一个文件/批次，报告进度
4. 遇到问题暂停，向用户报告

**实施前检查格式**:
```markdown
🔍 实施前检查:
- [x] xxx 服务已存在 ✅
- [x] xxx 依赖版本正确 ✅
- [ ] xxx 接口未就绪 ❌ → 需要先处理

检查结果: ✅ 全部通过 / ❌ 有阻塞项
```

**进度报告格式**:
```markdown
🔄 实施进度: [2/5]
✅ 已完成: model/xxx.ts, service/XxxService.ts
🔄 正在处理: controller/xxx.ts
⏳ 待处理: config/xxx.ts, test/xxx.test.ts
```

**多文件改动分批策略（涉及 5+ 个文件时）**:

```yaml
分批原则:
  依赖排序（底层优先）:
    批次1: 基础层 - config、types/interfaces、model
    批次2: 逻辑层 - service、middleware、utils
    批次3: 接口层 - controller、route
    批次4: 验证层 - test、scripts

  每批完成后:
    1. 报告本批完成进度
    2. 🔴 触发编码检查点 — 同步更新记忆文件（🆕 v2.11.0）:
       - 追加 📦 编码检查点记录（本批完成的文件列表）
       - 更新 ⚠️ 待跟进中已完成条目标记 [x]
       - 有 实施计划 时: 更新对应任务状态 🔲→✅
       - 详见 task-memory/triggers.md §编码检查点
    3. 运行相关验证（编译检查/测试）
    4. 确认无报错后进入下一批

  Token 不够时:
    - 因编码检查点已在每批完成时同步更新记忆，中断后新会话可精确恢复
    - 下次会话: 用户说"继续"时，读记忆 📦 编码检查点 → 从断点续接
```

### 阶段 5: 完成报告

**目标**: 总结实施结果

**输出格式**:
```markdown
✅ 需求实现完成

## 修改汇总
| 文件 | 操作 | 说明 |
|-----|------|------|
| xxx.ts | 新增 | xxx |

## 验证结果
- [x] 功能验证通过
- [x] 编译/类型检查通过

## 注意事项
- [如有特别说明]
```

**📝 接口文档（按需生成）**:
如果本次需求涉及 API 接口新增或变更，在完成报告后补充 `04-接口文档.md`：

```yaml
🔴 接口文档触发条件（判断是否需要生成）:
  必须生成:
    - 新增 API 接口/端点
    - 修改已有接口的入参/出参/错误码
    - 第三方系统对接
    - 接口行为变更（限流策略、鉴权方式等）
  建议生成（AI 主动建议，用户确认）:
    - 中间件变更影响所有接口行为
    - WebSocket/gRPC 等非 REST 协议接口新增
  不需要生成:
    - 纯内部逻辑修改（不影响接口契约）
    - 纯前端修改
    - 仅数据库层面变更（不影响接口响应）
```

- **模板**: `core/templates/core/api-doc-template.md`（API 接口文档）或 `core/templates/core/integration-template.md`（涉及第三方系统对接时）
- **存储路径**: `projects/<project>/requirements/<中文描述>/04-接口文档.md`
- 使用模板生成文档文件，存放到任务目录下

**🔗 `.http` 文件（自动跟随接口文档生成）**:
当生成 `04-接口文档.md` 时，**自动同时生成** `api.http` 文件，作为接口文档的可执行调试附件：

```yaml
🔴 .http 文件生成规则:
  触发条件: 与 04-接口文档.md / 04-对接文档.md 同步——生成接口文档时自动生成
  文件名: api.http（固定命名，不使用中文）
  存储路径: projects/<project>/requirements/<中文描述>/api.http
  内容结构:
    - 顶部: 环境变量定义（@baseUrl / @token 占位符）
    - 每个接口: ### 分隔的请求块（方法 + URL + Headers + Body）
    - 可选: 预期响应注释（关键字段说明）
  生成方式: 生成文件到任务目录下

🟡 OpenAPI 规范文件（按需，不自动生成）:
  触发条件: 仅在用户明确要求时才生成（如"需要 OpenAPI 文档"/"生成 Swagger"）
  文件名: openapi.yaml
  存储路径: projects/<project>/requirements/<中文描述>/openapi.yaml
  说明: OpenAPI 适合对外暴露的 API 服务，内部接口用 .http 即可满足调试需求
```

### 阶段 6: 生成归档文档

**目标**: 生成归档文档套件（所有需求均须执行此阶段，小需求内容可精简但文件必须创建）

**输出文档**:
1. `01-需求定义.md` - 需求文档（做什么、为什么做）— 小需求可精简为核心需求+验收标准
2. `02-技术方案.md` - 技术方案（怎么做，只方案描述）— 小需求可精简为改动清单+实现要点
3. `03-实施方案/` - 实施方案目录（阶段 3 已生成）
4. `04-接口文档.md` - 接口文档（涉及 API 接口新增/变更时，可选）或 `04-对接文档.md`（涉及第三方系统对接时，可选）

> 🔴 **04 编号文档触发规则**：不仅限于第三方系统对接。任何涉及 API 接口新增或变更的需求都应生成接口文档。
> 第三方系统对接场景使用 `integration-template.md`（含 API + 认证 + 流程），
> 纯接口新增/变更场景使用 `api-doc-template.md`（纯接口契约）。
> 详见阶段 5 §接口文档触发条件。

**参考模板**: `core/templates/core/` 目录

**执行步骤**:
1. 基于阶段 1-5 的内容整理
2. 使用完整版模板生成文档
3. 保存到 `projects/<project>/requirements/<task-id>/`

### 📎 文档职责划分与强关联

```yaml
文档关系:

  01-需求定义.md（需求层 - 做什么）:
    职责: 定义业务需求、功能需求、验收标准
    不包含: 技术实现细节、代码
    下游: 02-技术方案.md 引用本文档的需求点

  02-技术方案.md（方案层 - 怎么做）:
    职责: 架构设计、模块设计、接口契约、数据库设计、流程图
    不包含: 具体代码实现（只写接口签名、伪代码、配置结构）
    上游: 引用 01-需求定义.md 的需求编号
    下游: 03-实施方案/ 引用本文档的模块名和改动清单

  03-实施方案/（实施层 - 具体改什么）:
    职责: 实施前检查、每个变更文件的完整代码、实施步骤、验证清单
    上游: 引用 02-技术方案.md 的模块设计
    结构:
      README.md - 实施总索引（检查+步骤+验证）
      <模块名>.md - 每个变更文件的完整代码（一个文件对应一个文档）
    时机: 在执行代码之前生成，用户确认后才执行

  04-接口文档.md / 04-对接文档.md（接口层 - 可选）:
    职责: API 接口契约文档（接口清单、请求/响应、错误码）或第三方系统对接说明
    上游: 引用 01-需求定义 和 02-技术方案
    触发条件: 涉及 API 接口新增/变更/第三方对接时生成（详见阶段 5 §接口文档触发条件）
    模板选择:
      纯接口新增/变更 → api-doc-template.md
      第三方系统对接 → integration-template.md（含认证+流程+接口）
      前端对接指南 → frontend-integration-template.md（面向前端开发者）

🔴 文件命名语言规则（FIX-011 2026-02-28）:
  任务产物文件名使用中文（与 copilot-instructions.md "所有输出使用中文" 一致）
  序号前缀保留数字: 01-、02-、03-、04-

文档目录结构:

  ├── api.http                     # 接口调试文件（跟随 04-接口文档自动生成，可选）
  └── scripts/                     # 脚本目录

文档间引用规范:
  02-技术方案.md 开头必须写:
    > 关联需求: [01-需求定义.md](./01-需求定义.md)
```
