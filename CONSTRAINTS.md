# 开发执行约束

> 本规范定义 AI 在 ai-dev-guidelines 框架下执行开发任务时的约束条件

---

## 🎯 核心约束（15 条）

### 1. 删除操作需确认

删除现有代码/文件/配置前必须用户确认：

```yaml
需确认删除:
  - 文件删除
  - 函数/方法删除
  - 配置项删除
  - 路由删除

操作方式:
  AI: "需要删除 userService.js，是否确认？[Y/N]"
  用户: "Y"
  AI: "已删除"
```

### 2. Git 操作需确认

执行 git commit/push 前必须用户确认：

```yaml
需确认操作:
  - git commit
  - git push
  - git merge

操作方式:
  AI: "代码完成，是否提交？
        分支: feature/xxx
        变更: 3 个文件
        [Y/N]"
```

### 3. 方案选择需确认

多方案时必须用户决策：

```yaml
需确认场景:
  - 技术方案选择
  - 依赖版本选择
  - 架构设计决策

操作方式:
  AI: "[A] 方案 A - 优点/缺点
        [B] 方案 B - 优点/缺点
        建议 A，请选择 [A/B]"
```

### 4. 多任务需拆分

同时收到多个任务时建议分开执行：

```yaml
检测到多任务:
  AI: "检测到 2 个任务：
        1. 添加限流功能（需求开发）
        2. 优化查询性能（性能优化）
        建议分开执行，先做哪个？[1/2]"
```

## 📝 代码输出约束（3 条）

### 5. 必须包含错误处理

```typescript
// ✅ 正确
async function getUser(id: string) {
  try {
    if (!id) throw new Error('ID required');
    return await db.users.findById(id);
  } catch (error) {
    logger.error('getUser failed', { id, error });
    throw error;
  }
}

// ❌ 错误 - 无错误处理
async function getUser(id: string) {
  return await db.users.findById(id);
}
```

### 6. 禁止硬编码敏感信息

```typescript
// ✅ 正确
const API_KEY = process.env.API_KEY;

// ❌ 错误
const API_KEY = 'sk-abc123...';
```

### 7. 使用结构化输出

输出使用表格/列表/代码块，避免大段叙述：

```markdown
✅ 正确:
| 步骤 | 操作 | 文件 |
|-----|------|------|
| 1 | 添加方法 | userService.ts |

❌ 错误:
"这个功能需要修改用户服务，首先我们要添加一个新的方法..."
```

## ⚠️ 失败处理

任何失败必须明确报告，不能静默跳过：

```yaml
报告格式:
  AI: "❌ [失败类型]
        错误: [错误信息]
        建议:
        1. [建议1]
        2. [建议2]"
```

## 🔴 报告验证约束（v2.0.1 增强）

### 8. 报告需验证

生成分析报告或问题清单时，**必须逐项验证**：

```yaml
禁止:
  - 基于推断列出问题（未实际查看文件）
  - 复制之前报告内容而不重新验证
  - 假设某内容应该存在但未确认
  - 只搜索 1-2 处就断言问题范围（必须全局搜索）
  - 把"路线图待办"或"优化建议"当作"Bug"报告

必须:
  - 每个问题都读取实际文件验证
  - 路径/链接类问题必须 grep_search 全局搜索所有引用位置
  - 严格按 4 类分类:
      🔴 Bug（实际错误，需修复）
      🟡 待办（已在路线图中，非Bug）
      💡 建议（可选优化）
      ❌ 误报（验证后不存在）
  - 只有 🔴Bug 列入"需要修复"清单

详见: best-practices/AI-CODE-STANDARDS.md#报告必须验证
```

## 🔴 修复完整性约束（v2.0.2 新增）

### 9. 修复需扫描

执行任何修复后，声称"修复完毕"之前**必须完成 3 步扫描**：

```yaml
Step 1 - 同类问题全局扫描:
  修了什么 → grep_search 全项目搜同类 → 确认零残留

Step 2 - 数据联动扫描:
  改了哪个文件的数据 → 找引用该数据的文件 → 逐一同步

Step 3 - 声明前复核:
  □ grep 全项目确认零残留？
  □ 引用修改数据的文件都检查了？
  □ 对照实际目录验证了数量/结构？
  全部 ✅ 才能声称修复完毕

详见: best-practices/AI-CODE-STANDARDS.md#修复后完整性扫描
```

## 🧠 主动分析约束（v2.1.0 新增）

### 10. 主动合理性分析

收到用户消息后，在执行之前**必须先评估合理性**：

```yaml
触发条件: 每次收到用户的需求/指令
执行时机: 预检查之后、正式执行之前

检查项:
  1. 需求合理性:
     - 用户的方案是否有更优选择？
     - 命名/结构/流程是否符合项目现有规范？
     - 是否有潜在的副作用或遗漏？
  
  2. 处理方式:
     合理且无更好建议: 直接执行，无需额外输出
     有更好建议: 先提出建议，等用户确认后再执行
     明显不合理: 说明原因，给出替代方案

输出格式（仅在有建议时）:
  AI: "💡 建议:
        用户方案: [用户说的]
        我的建议: [更好的方案]
        原因: [为什么更好]
        
        是否采纳？[采纳/按原方案执行]"

⚠️ 不要:
  - 每次都输出"分析合理"之类的废话
  - 在明显合理的请求上浪费确认时间
  - 只提问不给建议（必须附带具体方案）
```

### 11. 自动关联文件检查

修改任何文件时，**必须自动扫描关联文件**：

```yaml
触发条件: 每次修改文件内容
执行时机: 完成单个文件修改后、声称完成之前

执行步骤:
  Step 1 - 识别关联:
    改了什么 → 这个内容被哪些文件引用？
    方法: grep_search 搜索被修改的关键字/路径/名称
  
  Step 2 - 分类处理:
    无影响的关联文件: 自动一并修改（不需确认）
    有影响的关联文件: 列出影响，请用户确认是否一起修改
    
  Step 3 - 报告:
    自动修改的: 在修改列表中标注"（自动同步）"
    需确认的: 暂停执行，等待用户决策

判断标准（无影响 vs 有影响）:
  无影响（自动修改）:
    - 同一数据/路径/名称在多处出现的简单同步
    - 引用链接更新
    - 版本号/日期同步
    - 注释中的引用更新
    
  有影响（需确认）:
    - 修改会改变其他文件的功能逻辑
    - 修改涉及其他工作流的流程定义
    - 不确定修改是否正确

输出格式:
  AI: "📎 关联文件检查:
        ✅ 已自动同步: fileA.md, fileB.md（路径引用更新）
        ⚠️ 需要确认: fileC.md - [说明影响]
        
        是否一并修改 fileC.md？[Y/N]"
```
