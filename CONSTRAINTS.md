# AI 禁止项清单

> AI 在执行开发任务时必须遵守的约束

---

## 🚨 核心原则

```text
================================================================================
⛔ 禁止项是强制约束 - 无任何例外 ⛔
================================================================================

禁止项的优先级高于所有其他规范。
违反禁止项将导致：
  - 任务立即终止
  - 需要用户确认才能继续
  - 可能需要回滚已执行的操作

================================================================================
```

---

## 🔴 操作禁止项（6 条）

### 1. 禁止跳过预检查

```yaml
描述: 每次任务必须先执行预检查流程

违规示例:
  用户: "在 user 项目添加限流功能"
  AI: "好的，我来分析项目结构..."  ❌ 直接开始分析

正确示例:
  用户: "在 user 项目添加限流功能"
  AI: "🔴 预检查清单:
        ✅ 1. 工作区: <项目工作区路径>
        ..."                              ✅ 先执行预检查

违规后果: 重新开始任务
```

### 2. 禁止未确认删除

```yaml
描述: 删除任何现有代码/文件/配置前必须用户确认

禁止删除的内容:
  - 现有 import 语句
  - 现有路由定义
  - 现有函数/方法
  - 现有配置项
  - 现有文件

违规示例:
  AI: "我删除了旧的 userService.js 文件"  ❌

正确示例:
  AI: "检测到需要删除 userService.js，是否确认？[Y/N]"
  用户: "Y"
  AI: "已删除 userService.js"              ✅

违规后果: 立即停止，尝试恢复
```

### 3. 禁止猜测技术栈

```yaml
描述: 必须读取项目规范确认技术栈，不能凭记忆猜测

禁止猜测的内容:
  - 框架版本
  - 数据库类型
  - 依赖版本
  - 目录结构
  - 配置格式

违规示例:
  AI: "这个项目应该是用 Express..."  ❌ 猜测

正确示例:
  AI: 调用 read_file("projects/user/TECH-STACK.md")
  AI: "根据项目规范，技术栈是 Express 4.18 + TypeScript"  ✅

违规后果: 重新分析
```

### 4. 禁止跳步执行

```yaml
描述: 必须按照工作流步骤顺序执行，不能跳过

禁止跳过的步骤:
  - 预检查
  - 需求确认
  - 方案确认（确认点）
  - 代码确认（确认点）
  - 测试确认（确认点）
  - 三轮验证

违规示例:
  AI: "方案看起来简单，我直接写代码了"  ❌

正确示例:
  AI: "方案设计完成，请确认后继续 [确认/修改]"
  用户: "确认"
  AI: "收到确认，开始代码实现..."        ✅

违规后果: 退回上一步
```

### 5. 禁止假设配置

```yaml
描述: 不确定的配置项必须询问用户

禁止假设的内容:
  - 环境变量值
  - 数据库连接
  - API 密钥
  - 端口号
  - 路径配置

违规示例:
  AI: "我假设数据库端口是 27017..."  ❌

正确示例:
  AI: "请确认 MongoDB 连接配置：
        端口: [默认 27017]
        数据库: [请输入]"              ✅

违规后果: 重新确认
```

### 6. 禁止自动提交 Git

```yaml
描述: 执行 git commit/push 前必须用户确认

禁止自动执行:
  - git add
  - git commit
  - git push
  - git merge
  - git rebase

违规示例:
  AI: "代码写完了，我提交一下..."  ❌

正确示例:
  AI: "代码实现完成，是否提交到 Git？
        分支: feature/rate-limit
        变更: 3 个文件
        [提交/暂不提交]"              ✅

违规后果: 立即停止
```

## 🟡 输出禁止项（4 条）

### 7. 禁止大段自然语言

```yaml
描述: 输出必须使用结构化格式

禁止的格式:
  - 长篇叙述
  - 没有段落的大块文字
  - 缺少列表/表格的说明

正确的格式:
  - 表格
  - 列表
  - 代码块
  - 章节标题

违规示例:
  AI: "这个功能需要修改用户服务，首先我们要添加一个新的方法，
        这个方法会调用数据库查询用户信息，然后验证用户的权限，
        如果权限通过就返回数据，否则抛出错误..."  ❌

正确示例:
  AI: "## 修改计划
        
        | 步骤 | 操作 | 文件 |
        |-----|------|------|
        | 1 | 添加方法 | userService.ts |
        | 2 | 数据库查询 | userService.ts |
        | 3 | 权限验证 | authMiddleware.ts |"  ✅
```

### 8. 禁止遗漏错误处理

```yaml
描述: 生成的代码必须包含错误处理

必须包含:
  - async 函数的 try-catch
  - Promise 的 .catch()
  - 边界条件检查
  - 日志记录

违规示例:
  async function getUser(id) {
    const user = await db.users.findById(id);  ❌ 无错误处理
    return user;
  }

正确示例:
  async function getUser(id) {
    try {
      if (!id) throw new Error('ID is required');
      const user = await db.users.findById(id);
      return user;
    } catch (error) {
      logger.error('getUser failed', { id, error });
      throw error;
    }
  }  ✅
```

### 9. 禁止硬编码敏感信息

```yaml
描述: 敏感信息必须使用环境变量

敏感信息包括:
  - API 密钥
  - 数据库密码
  - 加密密钥
  - Token
  - 私钥

违规示例:
  const API_KEY = 'sk-abc123...';  ❌

正确示例:
  const API_KEY = process.env.API_KEY;  ✅
```

### 10. 禁止跳过验证

```yaml
描述: 代码/方案必须经过三轮验证

禁止跳过:
  - 逻辑验证（6 项）
  - 技术验证（7 项）
  - 完整性验证（10 项）

违规示例:
  AI: "代码写完了，任务完成"  ❌ 未验证

正确示例:
  AI: "🔍 三轮验证中...
        [1/3] 逻辑验证: ✅
        [2/3] 技术验证: ✅
        [3/3] 完整性验证: ✅
        
        代码实现完成，请确认..."  ✅
```

## 🟢 流程禁止项（3 条）

### 11. 禁止自作主张

```yaml
描述: 方案选择必须用户确认，不能自动决定

需要确认的决策:
  - 技术方案选择
  - 架构设计
  - 依赖选择
  - 实现方式

违规示例:
  AI: "我觉得方案 A 更好，我按方案 A 做了"  ❌

正确示例:
  AI: "有两个可选方案：
        [A] 方案 A - 优点...缺点...
        [B] 方案 B - 优点...缺点...
        
        建议选择方案 A，请确认 [A/B]"
  用户: "A"
  AI: "收到确认，按方案 A 执行"           ✅
```

### 12. 禁止一次执行多任务

```yaml
描述: 多个任务应分开执行

违规示例:
  用户: "添加限流功能并优化查询性能"
  AI: "好的，我同时做这两件事..."  ❌

正确示例:
  用户: "添加限流功能并优化查询性能"
  AI: "检测到 2 个任务：
        1. 需求开发 - 添加限流功能
        2. 性能优化 - 优化查询性能
        
        建议分开执行，请选择先执行哪个 [1/2]"  ✅
```

### 13. 禁止静默失败

```yaml
描述: 任何失败必须明确报告

需要报告的失败:
  - 工具调用失败
  - 验证失败
  - 依赖安装失败
  - 测试失败

违规示例:
  AI: [工具调用失败]
  AI: "我换个方法试试..."  ❌ 未报告失败

正确示例:
  AI: [工具调用失败]
  AI: "❌ 依赖安装失败
        错误: npm ERR! network timeout
        
        建议操作：
        1. 检查网络连接
        2. 配置 npm 镜像
        3. 重试安装
        
        请选择 [1/2/3]"  ✅
```
