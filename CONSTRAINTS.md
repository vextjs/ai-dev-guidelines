# 开发执行约束

> 本规范定义 AI 在 ai-dev-guidelines 框架下执行开发任务时的约束条件

**版本**: v2.11.0  
**更新日期**: 2026-02-27

---

## 🎯 核心约束（19 条）

### 1. 删除操作需确认

删除现有代码/文件/配置前必须用户确认：

```yaml
需确认删除:
  - 文件删除
  - 函数/方法删除
  - 配置项删除
  - 路由删除

操作方式:
  AI: "需要删除 userService.js，是否确认？[Y/N]"
  用户: "Y"
  AI: "已删除"
```

### 2. Git 操作需确认

执行 git commit/push 前必须用户确认：

```yaml
需确认操作:
  - git commit
  - git push
  - git merge

操作方式:
  AI: "代码完成，是否提交？
        分支: feature/xxx
        变更: 3 个文件
        [Y/N]"
```

### 3. 方案选择需确认

多方案时必须用户决策：

```yaml
需确认场景:
  - 技术方案选择
  - 依赖版本选择
  - 架构设计决策

操作方式:
  AI: "[A] 方案 A - 优点/缺点
        [B] 方案 B - 优点/缺点
        建议 A，请选择 [A/B]"
```

### 4. 多任务需拆分

同时收到多个任务时建议分开执行：

```yaml
检测到多任务:
  AI: "检测到 2 个任务：
        1. 添加限流功能（需求开发）
        2. 优化查询性能（性能优化）
        建议分开执行，先做哪个？[1/2]"
```

### 5. 必须包含错误处理

```typescript
// ✅ 正确
async function getUser(id: string) {
  try {
    if (!id) throw new Error('ID required');
    return await db.users.findById(id);
  } catch (error) {
    logger.error('getUser failed', { id, error });
    throw error;
  }
}

// ❌ 错误 - 无错误处理
async function getUser(id: string) {
  return await db.users.findById(id);
}
```

### 6. 禁止硬编码敏感信息

```typescript
// ✅ 正确
const API_KEY = process.env.API_KEY;

// ❌ 错误
const API_KEY = 'sk-abc123...';
```

### 7. 使用结构化输出

输出使用表格/列表/代码块，避免大段叙述：

```markdown
✅ 正确:
| 步骤 | 操作 | 文件 |
|-----|------|------|
| 1 | 添加方法 | userService.ts |

❌ 错误:
"这个功能需要修改用户服务，首先我们要添加一个新的方法..."
```

### 8. 报告需验证

生成分析报告或问题清单时，**必须逐项验证**：

```yaml
禁止:
  - 基于推断列出问题（未实际查看文件）
  - 复制之前报告内容而不重新验证
  - 假设某内容应该存在但未确认
  - 只搜索 1-2 处就断言问题范围（必须全局搜索）
  - 把"路线图待办"或"优化建议"当作"Bug"报告

必须:
  - 每个问题都读取实际文件验证
  - 路径/链接类问题必须 grep_search 全局搜索所有引用位置
  - 严格按 4 类分类:
      🔴 Bug（实际错误，需修复）
      🟡 待办（已在路线图中，非Bug）
      💡 建议（可选优化）
      ❌ 误报（验证后不存在）
  - 只有 🔴Bug 列入"需要修复"清单

详见: best-practices/AI-CODE-STANDARDS.md#报告必须验证
```

### 9. 文件名日期规则

报告文件和记忆文件使用**不同的命名规则**（🆕 v2.10.0 / task-memory v1.7）：

```yaml
报告文件（reports/）:
  格式: YYYYMMDD-NN-<类型>-<简述>.md
  NN: 2 位当日序号（01、02、03...），仅扫描 reports/<子目录>/ 目录计算
  ✅ 正确: 20260227-01-analysis-deep-review.md
  ✅ 正确: 20260227-02-bug-login-timeout.md

记忆文件（.ai-memory/clients/<agent>/tasks/）:
  格式: YYYYMMDD.md（每天一个文件，会话内以 ## 会话 NN 分段）
  不使用 NN 序号，直接以纯日期命名
  ✅ 正确: 20260226.md
  ✅ 正确: 20260227.md
  ❌ 错误: 20260226-01-ANALYSIS-v3-deep-review.md  # v1.6 旧格式（已废弃）

❌ 通用错误:
  v3-architecture-deep-analysis.md       # 缺少日期前缀
  2026-02-27-analysis-xxx.md             # 日期格式错误（有分隔符）

🔴 报告 NN 和记忆序号完全独立:
  - 报告 NN 仅扫描 reports/ 目录
  - 记忆文件无 NN，用日期定位
  - ❌ 禁止报告和记忆共享全局序号池（v1.6 及之前的设计缺陷，已废弃）

获取日期: 优先使用 now() 工具获取真实日期，禁止留 YYYY-MM-DD 占位符
```

### 10. 多 Agent 目录隔离

记忆文件按 Agent 目录隔离存储，**不跨目录读写**：

```yaml
存储结构:
  .ai-memory/
  ├── SUMMARY.md                    # 全局只读摘要（关键决策汇总）
  └── clients/
      ├── zed-copilot/              # Zed Copilot 专属
      │   ├── SUMMARY.md
      │   └── tasks/
      ├── webstorm-copilot/         # WebStorm Copilot 专属
      │   ├── SUMMARY.md
      │   └── tasks/
      └── unknown-agent/            # 无法确定编辑器时的默认目录

规则:
  写入: 只写入 clients/<当前agent>/ 目录
  读取: 先读全局 SUMMARY.md，再读 clients/<当前agent>/SUMMARY.md
  禁止: 绝不写入其他 Agent 的 clients/ 目录
  恢复: 只从本 Agent 目录查找 🔄 状态的任务

跨 Agent 恢复:
  - 全局 SUMMARY.md 发现其他 Agent 未完成任务时，提示用户确认
  - 用户确认后在本 Agent 目录创建新记忆，标注"继承自 <原 Agent>"
  - 不修改原 Agent 目录下的文件

详见: workflows/common/task-memory.md §多 Agent 支持
```

### 11. 修复需扫描

执行任何修复后，声称"修复完毕"之前**必须完成 3 步扫描**：

```yaml
Step 1 - 同类问题全局扫描:
  修了什么 → grep_search 全项目搜同类 → 确认零残留

Step 2 - 数据联动扫描:
  改了哪个文件的数据 → 找引用该数据的文件 → 逐一同步

Step 3 - 声明前复核:
  □ grep 全项目确认零残留？
  □ 引用修改数据的文件都检查了？
  □ 对照实际目录验证了数量/结构？
  全部 ✅ 才能声称修复完毕

详见: best-practices/AI-CODE-STANDARDS.md#修复后完整性扫描
```

### 12. 主动合理性分析

收到用户消息后，在执行之前**必须先评估合理性**：

```yaml
触发条件: 每次收到用户的需求/指令
执行时机: 预检查之后、正式执行之前

检查项:
  1. 需求合理性:
     - 用户的方案是否有更优选择？
     - 命名/结构/流程是否符合项目现有规范？
     - 是否有潜在的副作用或遗漏？
  
  2. 处理方式:
     合理且无更好建议: 直接执行，无需额外输出
     有更好建议: 先提出建议，等用户确认后再执行
     明显不合理: 说明原因，给出替代方案

输出格式（仅在有建议时）:
  AI: "💡 建议:
        用户方案: [用户说的]
        我的建议: [更好的方案]
        原因: [为什么更好]
        
        是否采纳？[采纳/按原方案执行]"

⚠️ 不要:
  - 每次都输出"分析合理"之类的废话
  - 在明显合理的请求上浪费确认时间
  - 只提问不给建议（必须附带具体方案）
```

### 13. 自动关联文件检查

修改任何文件时，**必须自动扫描关联文件**：

```yaml
触发条件: 每次修改文件内容
执行时机: 完成单个文件修改后、声称完成之前

执行步骤:
  Step 1 - 识别关联:
    改了什么 → 这个内容被哪些文件引用？
    方法: grep_search 搜索被修改的关键字/路径/名称
  
  Step 2 - 分类处理:
    无影响的关联文件: 自动一并修改（不需确认）
    有影响的关联文件: 列出影响，请用户确认是否一起修改
    
  Step 3 - 报告:
    自动修改的: 在修改列表中标注"（自动同步）"
    需确认的: 暂停执行，等待用户决策

判断标准（无影响 vs 有影响）:
  无影响（自动修改）:
    - 同一数据/路径/名称在多处出现的简单同步
    - 引用链接更新
    - 版本号/日期同步
    - 注释中的引用更新
    
  有影响（需确认）:
    - 修改会改变其他文件的功能逻辑
    - 修改涉及其他工作流的流程定义
    - 不确定修改是否正确

输出格式:
  AI: "📎 关联文件检查:
        ✅ 已自动同步: fileA.md, fileB.md（路径引用更新）
        ⚠️ 需要确认: fileC.md - [说明影响]
        
        是否一并修改 fileC.md？[Y/N]"
```

### 14. 规范修改需交叉验证

修改任何规范文件后，**必须逐个检查所有引用该规范的文件是否需要同步更新**：

```yaml
触发条件: 修改规范文件后（copilot-instructions.md、workflows/、standards/、QUICK-REFERENCE.md 等）
执行时机: 修改完成后、声称完成之前

交叉检查对照表:
  copilot-instructions.md → 00-pre-check/README.md、QUICK-REFERENCE.md
  00-pre-check/README.md  → copilot-instructions.md、task-memory.md
  task-memory.md           → copilot-instructions.md、00-pre-check/README.md
  QUICK-REFERENCE.md       → copilot-instructions.md
  CONSTRAINTS.md           → QUICK-REFERENCE.md、decision-tree.yaml
  🔴 版本号变更时          → 8 个文件全量同步（见 QUICK-REFERENCE.md §版本号文件清单）:
                              copilot-instructions.md、README.md、QUICK-REFERENCE.md、
                              CONSTRAINTS.md、STATUS.md、CHANGELOG.md、
                              decision-tree.yaml、00-pre-check/README.md

检查清单:
  □ 预检查项数各处一致？
  □ 文件命名格式各处一致？
  □ 多 Agent 方案各处一致？
  □ 约束条数与实际条目数一致？
  □ 版本号（被动）: 修改的文件已更新版本号和最后更新日期？
  □ 🔴 版本号（主动全量）: 8 个版本号文件是否全部一致？（逐个 read_file 确认，禁止推断）
  □ 所有引用链接指向的文件确实存在？

详见: QUICK-REFERENCE.md §规范修改交叉验证清单
```

### 15. 任务完成验证

声称任务完成前，**必须执行完成验证**：

```yaml
触发条件: AI 准备声称"任务完成"之前
执行时机: 所有修改/生成操作完成后

验证清单:
  1. 输出文件存在性:
     - [ ] 所有声称创建的文件确实存在？（用工具验证）
     - [ ] 文件路径正确？（在预期目录下）
     - [ ] 文件名符合命名规范？

  2. 内容结构完整性:
     - [ ] 文档头部包含所有必填字段？（日期、Agent、状态）
     - [ ] 日期字段是真实日期，非占位符？
     - [ ] 没有中途截断的内容？

  3. 记忆文件完整性:
     - [ ] .ai-memory 中的任务记忆已更新？
     - [ ] SUMMARY.md 已追加/更新对应条目？

  4. 关联文件一致性（如修改了规范文件）:
     - [ ] 已执行交叉验证清单？
     - [ ] 所有引用处已同步更新？

🔴 未通过验证项必须修复后才能声称完成。

详见: workflows/common/task-memory.md §任务完成验证
```

### 16. 文件修改需确认（🆕 v2.7.0）

涉及文件修改/删除操作时，**必须先输出变更计划，等待用户确认后再执行**：

```yaml
触发条件: AI 准备执行文件修改或删除操作
执行时机: 分析完成后、实际修改前

适用范围:
  需要确认:
    - 修改现有文件内容
    - 删除文件或目录
    - 批量重命名/移动文件
    - 修改规范/配置文件

  无需确认（纯只读操作 + AI 自动输出）:
    - 读取文件进行分析
    - 搜索/grep 操作
    - 写入会话报告（输出到 reports/）← AI 自动行为
    - 创建/更新记忆文件（.ai-memory/ 下的写入）← AI 自动行为

操作方式:
  AI: "📋 变更计划:
        | 序号 | 文件 | 操作 | 说明 |
        |------|------|------|------|
        | 1 | CONSTRAINTS.md | 修改 | 新增约束 #16 |
        | 2 | README.md | 修改 | 更新约束条数注释 |
        | 3 | outputs/ | 删除 | 已弃用的目录 |
        
        总计: 修改 2 个文件，删除 1 个目录
        
        确认后开始执行。"
  
  用户: "确认"
  AI: 开始执行修改

⚠️ 例外情况:
  - 用户在请求中明确说"直接修复"/"直接改" → 视为已授权，可跳过确认
  - 自动同步的无影响关联文件（约束 #13 中定义的"无影响"类） → 可自动修改
  - 报告文件的创建（reports/ 下）→ 属于 AI 自动输出机制，无需确认（约束 #17）
  - 记忆文件的创建和更新（.ai-memory/ 下）→ 属于 AI 自动输出机制，无需确认（约束 #17）

🔴 禁止行为:
  - 分析完问题后直接修复，不等用户确认
  - 用"建议修复"的措辞实际已执行了修改
  - 变更计划描述模糊（如"修改若干文件"），必须列出具体文件和操作
```

### 17. 报告+记忆自动输出（🆕 v2.8.0）

报告文件和记忆文件的写入是 AI 的**自动行为**，每次会话必须执行，**绝不询问用户**：

```yaml
🔴 强制规则（NO EXCEPTIONS）:

  每次用户提问 + AI 完成分析/执行后，必须自动:
    1. 写入报告文件（reports/ 下） → 完整分析内容
    2. 创建/更新记忆文件（.ai-memory/ 下） → 摘要 + 报告链接
    3. 对话中只输出结论摘要 + 报告文件路径

  架构:
    报告文件（reports/）  = 完整分析内容（主体）
    记忆文件（.ai-memory/）= 摘要 + 报告链接（索引）
    对话输出             = 结论摘要 + 报告路径

  ❌ 绝对禁止:
    - "需要我写入报告吗？"
    - "要不要我保存记忆？"
    - "是否需要把分析结果存下来？"
    - 任何形式的询问用户是否写入报告/记忆的行为
    - 在对话中堆砌大段分析内容而不写报告文件
    - 会话结束时没有任何报告文件产出

  为什么:
    - 报告和记忆是 AI 内部工作机制，不需要用户决策
    - 询问会打断用户思路，增加不必要的交互轮次
    - 遗忘写入 = 下次会话丢失上下文 = 重复劳动

详见: workflows/common/task-memory.md §核心架构 + §自动输出
详见: workflows/common/temp-reports.md §核心规则
```

### 18. 消息驱动记忆触发（🆕 v2.9.0）

记忆写入以**对话消息事件**为锚点，而非 AI 内部工作流阶段：

```yaml
🔴 消息驱动 5 阶段（task-memory.md v1.7）:

  阶段 0: 会话初始化（首条消息时 — 预检查 + 创建/追加记忆 YYYYMMDD.md）
  阶段 1: 用户发消息时（捕获用户输入 — 新意图/需求/修正）
  阶段 2: AI 回复时（写报告 + 更新记忆 + 追加对话记录含关联引用）
  阶段 3: AI 执行完毕时（记录变更清单 — 与阶段 2 通常合并）
  阶段 4: 任务结束时（最终状态更新 ✅）

  典型顺序: 0 → 1 → 2+3 → 1 → 2+3 → ... → 4

  🔴 阶段 1 强制规则:
    - 用户发送包含新意图/需求/修正的消息时，必须立即记录到 📨 对话记录
    - 记录发生在 AI 开始分析之前（极轻量，~100 token）
    - 确保 token 耗尽时用户说了什么至少有记录

  阶段 1 过滤规则（以下消息不触发）:
    - 纯确认: "好的" / "Y" / "确认"
    - 流程控制: "继续" / "接着" / "下一步"
    - 情绪表达: "好的谢谢" / "明白了"

  📨 对话记录（🆕 v1.7 — 4 列表格，新增"关联引用"列）:
    | 轮次 | 方向 | 摘要 | 关联引用 |
    |:----:|:----:|------|----------|
    | 1 | 👤→ | [用户消息摘要] | 🧠 YYYYMMDD.md, 📋 profile/README.md |
    | 1 | 🤖← | [AI 回复摘要] | 📄 YYYYMMDD-NN-类型-简述 |

  关联引用标记: 📋 项目规范  📄 报告文件  🧠 记忆文件  📑 规范文件
  对话记录关联引用列 = 时间线视角；§全天关联报告汇总 = 汇总视角，互补不重复

  为什么需要阶段 1:
    - 用户消息是最有价值的原始输入，不应等 AI 消化后才记录转述版
    - 用户追加需求/方向修正必须被第一时间捕获
    - token 耗尽时，用户说了什么不应丢失

详见: workflows/common/task-memory.md §触发时机（v1.7 消息驱动 5 阶段）
```

### 19. 报告/记忆序号独立（🆕 v2.10.0）

报告 NN 序号和记忆文件**完全解耦**，禁止共享全局序号池：

```yaml
🔴 强制规则（NO EXCEPTIONS）:

  报告文件:
    - 命名: YYYYMMDD-NN-<类型>-<简述>.md
    - NN 来源: 仅扫描 reports/<子目录>/ 目录，取 max+1
    - 与记忆文件无关

  记忆文件:
    - 命名: YYYYMMDD.md（每天一个文件，无 NN 序号）
    - 会话内以 ## 会话 NN 分段追加（NN 为当天第几个会话）
    - 与报告 NN 无关

  ❌ 绝对禁止:
    - 报告和记忆共享同一个全局 NN 序号池
    - 记忆文件使用旧格式 YYYYMMDD-NN-TYPE-id.md（新会话）
    - 创建报告时扫描 .ai-memory/ 目录获取序号

  为什么:
    - v1.6 及之前的共享序号导致断号困惑（记忆 07 跳到 09）
    - 共享序号增加了序号分配复杂度和跨目录耦合
    - 解耦后阶段 0 大幅简化（不再需要扫描计算记忆序号）

详见: workflows/common/task-memory.md §序号独立规则
详见: workflows/common/temp-reports.md §报告 NN 序号独立规则
```
