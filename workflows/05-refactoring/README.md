# 架构重构流程

> **任务类型**: 架构重构  
> **使用场景**: 大型代码重构、架构升级  
> **输出目录**: `projects/<project-name>/refactoring/<ref-id>/`  
> **版本**: v2.1.0

---

## 📋 流程概览

```
现状分析 → 目标设计 → 拆分计划 → 实施方案 → 分步执行 → 灰度上线 → 完成总结
    CP1         CP2                  CP3（每阶段执行前确认）
```

---

## 🎯 执行步骤

### Step 1: 现状分析
- 分析当前架构问题
- 评估重构必要性
- 确定重构范围

**输出**: `01-current-state.md`

**⏸️ CP1 确认点**: 等待用户确认分析结果和重构范围

---

### Step 2: 目标设计
- 设计目标架构
- 制定重构原则
- 评估技术风险

**输出**: `02-target-design.md`

**⏸️ CP2 确认点**: 等待用户确认目标架构可行

---

### Step 3: 拆分计划
- 将重构拆分为多个阶段
- 确定每个阶段的目标和范围
- 制定回滚方案

**输出**: `03-plan.md`

**内容要点**:
- 阶段划分（Phase 1/2/3...）
- 每阶段的目标和涉及模块
- 阶段间依赖关系
- 回滚方案

---

### Step 4: 实施方案

**目标**: 为每个阶段生成具体的实施方案

**核心原则**:
- 拆分计划说"分几步走"（宏观规划）
- 实施方案说"每步具体改什么"（每个文件的完整变更）
- **每个阶段执行前，生成该阶段的实施方案并确认**

**实施方案结构**:

```yaml
按阶段生成:
  projects/<project>/refactoring/<ref-id>/
  ├── 04-implementation/
  │   ├── phase-1/
  │   │   ├── README.md            # 阶段1实施索引（前检查+步骤+验证）
  │   │   ├── <Module>.md          # 各文件变更
  │   │   └── ...
  │   ├── phase-2/
  │   │   ├── README.md
  │   │   └── ...
  │   └── ...
  └── scripts/
```

**⏸️ CP3 确认点**: 每个阶段执行前确认该阶段的实施方案

---

### Step 5: 分步执行
- 按阶段执行重构
- 每阶段执行前先确认实施方案
- 每阶段完成后验证并报告
- 记录每个阶段的变更

**执行流程（每个阶段）**:
1. 生成该阶段实施方案 → 用户确认
2. 执行实施前检查
3. 对照方案逐文件执行
4. 验证该阶段无问题
5. 进入下一阶段

---

### Step 6: 灰度上线
- 制定灰度策略
- 监控关键指标
- 逐步扩大灰度范围

**输出**: 更新 `04-implementation/` 最后阶段

---

### Step 7: 完成总结
- 总结重构效果
- 记录经验教训
- 整理技术文档
- **更新 `projects/<project>/TASK-INDEX.md`**
- **生成任务记忆** `dev-docs/projects/<project>/.ai-memory/`（详见 [common/task-memory.md](../common/task-memory.md)）

**输出**: `05-summary.md`

---

## 📦 输出示例

```
projects/user-service/refactoring/REF-user-auth-20260211/
├── 01-current-state.md        # 当前架构
├── 02-target-design.md        # 目标架构
├── 03-plan.md                 # 实施计划（阶段划分）
├── 04-implementation/         # 实施方案
│   ├── phase-1/               # 阶段1：基础拆分
│   │   ├── README.md
│   │   ├── AuthService.md
│   │   └── UserModel.md

│       ├── AuthController.md
│       └── AuthMiddleware.md
├── 05-summary.md              # 完成总结

    └── migration-guide.md     # 迁移指南
```

---

## 🚨 注意事项

- 大型重构必须分阶段进行
- 每个阶段都要有实施方案和用户确认
- 每个阶段执行后都要验证
- 制定详细的回滚方案
- 重要变更需要灰度上线

---

## ✅ 完成检查清单

- [ ] 重构目标明确
- [ ] 目标架构已确认
- [ ] 实施计划详细（阶段划分清晰）
- [ ] 每阶段实施方案已确认
- [ ] 每阶段已验证
- [ ] 测试覆盖充分
- [ ] 文档已更新

---

## ✅ 架构重构专属验证

完成重构后，执行以下验证：

```yaml
架构重构验证清单:
  计划验证:
    ✅ 目标明确: 重构目标和收益清晰
    ✅ 分阶段: 有明确的阶段划分和里程碑
    ✅ 风险可控: 每阶段有回滚方案
  
  实施验证:
    ✅ 测试覆盖: 回归测试覆盖核心功能
    ✅ 无破坏: 现有功能未受影响
    ✅ 性能稳定: 性能无明显下降
  
  文档验证:
    ✅ 架构文档: 新架构已文档化
    ✅ 迁移指南: 其他开发者可按指南迁移
    ✅ 变更记录: CHANGELOG 已更新
```
