# 架构重构流程

> **任务类型**: 架构重构  
> **使用场景**: 大型代码重构、架构升级  
> **输出目录**: `projects/<project-name>/refactoring/<ref-id>/`  

---

## 📋 流程概览

```
现状分析 → 目标设计 → 拆分计划 → 分步实施 → 灰度上线 → 完成总结
```

---

## 🎯 执行步骤

### Step 1: 现状分析
- 分析当前架构问题
- 评估重构必要性
- 确定重构范围

**输出**: `01-current-state.md`

### Step 2: 目标设计
- 设计目标架构
- 制定重构原则
- 评估技术风险

**输出**: `02-target-design.md`

### Step 3: 拆分计划
- 将重构拆分为多个阶段
- 确定每个阶段的目标
- 制定回滚方案

**输出**: `03-plan.md`

### Step 4: 分步实施
- 按阶段实施重构
- 记录每个阶段的变更
- 持续测试验证

**输出**: `04-implementation.md`

### Step 5: 灰度上线
- 制定灰度策略
- 监控关键指标
- 逐步扩大灰度范围

**输出**: 更新 `04-implementation.md`

### Step 6: 完成总结
- 总结重构效果
- 记录经验教训
- 整理技术文档

**输出**: `05-summary.md`

---

## 📦 输出示例

```
projects/user-service/refactoring/REF-user-auth-20260211/
├── 01-current-state.md        # 当前架构
├── 02-target-design.md        # 目标架构
├── 03-plan.md                 # 实施计划
├── 04-implementation.md       # 实施记录
├── 05-summary.md              # 完成总结
└── docs/
    ├── architecture.png       # 架构图
    └── migration-guide.md     # 迁移指南
```

---

## 🚨 注意事项

- 大型重构必须分阶段进行
- 每个阶段都要有测试验证
- 制定详细的回滚方案
- 重要变更需要灰度上线

---

## ✅ 完成检查清单

- [ ] 重构目标明确
- [ ] 实施计划详细
- [ ] 每阶段已验证
- [ ] 测试覆盖充分
- [ ] 文档已更新

---

## ✅ 架构重构专属验证

完成重构后，执行以下验证：

```yaml
架构重构验证清单:
  计划验证:
    ✅ 目标明确: 重构目标和收益清晰
    ✅ 分阶段: 有明确的阶段划分和里程碑
    ✅ 风险可控: 每阶段有回滚方案
  
  实施验证:
    ✅ 测试覆盖: 回归测试覆盖核心功能
    ✅ 无破坏: 现有功能未受影响
    ✅ 性能稳定: 性能无明显下降
  
  文档验证:
    ✅ 架构文档: 新架构已文档化
    ✅ 迁移指南: 其他开发者可按指南迁移
    ✅ 变更记录: CHANGELOG 已更新
```
