# 任务记忆机制

**版本**: v1.6
**最后更新**: 2026-02-27

> AI 每次会话自动输出报告文件 + 记忆索引，实现跨会话上下文传递

---

## 🎯 解决的问题

- AI 会话断开后，新会话无法知道之前做了什么
- TASK-INDEX.md 只记录了任务编号和路径，缺少上下文
- 长期积累后 AI 无法快速回忆项目历史决策
- **多编辑器/多 Agent 场景下记忆文件共存混乱，无法区分来源**
- **对话中直接输出的分析结果，会话关闭后无法追溯**（🆕 v1.5）
- **记忆文件为了保留上下文被迫塞入大量细节，偏离"摘要索引"定位**（🆕 v1.5）
- **触发时机以 AI 内部阶段为中心，忽略了"用户发消息"这个关键输入时刻**（🆕 v1.6）
- **用户追加需求/方向修正没有记忆写入点，可能在 token 耗尽时丢失**（🆕 v1.6）

---

## 🏗️ 核心架构：报告为主体、记忆为索引（🆕 v1.5）

> **核心原则**: AI 每次会话的分析结果必须写入报告文件，记忆文件只存摘要+报告链接。
> 对话中不再直接输出大段分析内容 —— 完整内容写报告，对话中只输出结论摘要+报告路径。

```yaml
🔴 架构规则（NO EXCEPTIONS）:

  报告文件（reports/）:
    - 定位: 每次会话的完整分析内容载体
    - 内容: 完整的分析过程、数据、结论、建议
    - 存放: projects/<project>/reports/<子目录>/
    - 命名: YYYYMMDD-NN-<类型>-<简述>.md
    - 详见: workflows/common/temp-reports.md

  记忆文件（.ai-memory/）:
    - 定位: 索引 + 摘要（指向报告的指针）
    - 内容: 任务目标、状态、关键决策、报告链接、待跟进
    - 存放: projects/<project>/.ai-memory/clients/<agent>/tasks/
    - 命名: YYYYMMDD-NN-TYPE-id.md

  对话输出:
    - 只输出结论摘要 + 报告文件路径
    - 不在对话中堆砌大段分析内容

  关系图:
    用户提问 → AI 分析 → 写入报告文件（完整内容）
                       → 写入/更新记忆文件（摘要+报告链接）
                       → 对话输出结论摘要 + 报告路径

    下次会话 → 读记忆 → 跟链接读报告 → 完整恢复上下文

为什么这样设计:
  1. 减少 token 浪费: 记忆文件不再重复写分析内容，只存摘要+链接
  2. 可追溯性: 每次对话都有报告文件留存，不随会话丢失
  3. 跨会话恢复: 新会话读记忆→读报告，完整上下文不丢失
  4. 职责清晰: 报告=完整内容，记忆=索引+摘要，分工明确
```

## 🔴 自动输出，禁止询问（🆕 v1.5）

```yaml
🔴 强制规则（NO EXCEPTIONS）:

  报告文件和记忆文件的写入是 AI 的自动行为，绝不询问用户。
  每次用户提问、AI 回复，都必须自动写入报告 + 更新记忆。

  ❌ 绝对禁止:
    - "需要我把分析写入报告吗？"
    - "要不要我写入记忆？"
    - "是否需要保存本次结果？"
    - 任何形式的询问用户是否写入报告/记忆的行为

  ✅ 正确行为:
    - 预检查完成后，自动创建初始记忆（状态 🔄）
    - 分析完成后，自动写入报告文件
    - 报告写入后，自动更新记忆文件（追加报告链接）
    - 任务完成后，自动更新记忆状态为 ✅
    - 对话中输出结论摘要 + 报告路径

  为什么:
    - 报告和记忆是 AI 内部工作机制，不是用户需要决策的事项
    - 询问会打断用户思路，增加不必要的交互轮次
    - 遗忘写入 = 下次会话丢失上下文 = 重复劳动
    - 约束 #16（文件修改需确认）已明确豁免记忆文件和报告文件的创建
```

## 🔄 触发时机（v1.6 消息驱动 5 阶段）

### ⚡ 核心原则

> **原则 1 — 消息驱动（🆕 v1.6）**: 触发器以**对话中的消息事件**为锚点
> （用户发消息 / AI 回复 / AI 执行完毕），而非 AI 内部工作流阶段。
>
> **原则 2 — 自动输出（v1.5）**: 报告和记忆的写入是 AI 的自动行为，绝不询问用户。
> 每次会话都必须输出报告文件+更新记忆，无一例外。
>
> **原则 3 — 先写后干**: AI 无法感知自身剩余 token，输出中途截断时没有机会执行任何收尾动作。
> 开始执行前先创建初始记忆，过程中在每个消息事件增量更新+写入报告，
> 确保 token 耗尽时至少有最近一次快照可用。

### 5 阶段总览（🆕 v1.6）

```yaml
阶段 0: 会话初始化（AI 收到首条消息时 — 预检查 + 创建记忆）
阶段 1: 用户发消息时（每条有实质内容的用户消息 — 捕获用户输入）  ← 🆕
阶段 2: AI 回复时（每次 AI 输出结果 — 写报告 + 更新记忆）
阶段 3: AI 执行完毕时（单轮工具调用全部完成 — 记录变更清单）
阶段 4: 任务结束时（整个任务完成 — 最终状态更新 ✅）

执行顺序:
  首轮:   0 → 1 → 2+3 →
  后续轮: 1 → 2+3 → 1 → 2+3 → ... →
  结束:   4

典型会话时间线:
  用户消息1 → [阶段0+1] → AI分析执行 → AI回复1 → [阶段2+3]
  用户消息2 → [阶段1]   → AI分析执行 → AI回复2 → [阶段2+3]
  用户确认  →                                      → [阶段4]

v1.5 → v1.6 的核心变化:
  - 🆕 新增阶段 0（会话初始化）和阶段 1（用户发消息）
  - 阶段编号与实际执行顺序一致（不再是 1→3→2→4 的混乱顺序）
  - 阶段 3 触发条件从主观的"关键节点"改为客观的"执行完毕"
  - 记忆模板新增 §📨 对话记录（用户消息+AI回复摘要表格）
```

### 阶段 0：会话初始化（首条消息时）

```yaml
触发条件: AI 收到本次会话的第一条用户消息
触发方式: 🔴 自动执行，禁止询问用户
目的: 创建记忆文件 + 执行预检查

AI 执行:
  1. 执行预检查 5 项必做（见 00-pre-check/README.md）
  2. 确定当前 Agent 标识（预检查第 4 项）
  3. 获取当前真实日期（使用 now() 工具）
  4. 确保 .ai-memory/clients/<agent>/tasks/ 目录存在（不存在则创建）
  5. 扫描 tasks/ 目录，统计当天已有文件数量，确定序号 NN（max+1 或 01）
  6. 创建 .ai-memory/clients/<agent>/tasks/<YYYYMMDD>-<NN>-<TYPE>-<id>.md（状态 🔄）
  7. 写入已知信息:
     - 任务目标（来自用户首条消息）
     - 涉及的项目/模块、预计变更范围
     - 📨 对话记录表格 → 记录首条用户消息摘要（轮次 1, 👤→）
  8. 更新 .ai-memory/clients/<agent>/SUMMARY.md → 追加到最近任务（状态 🔄）

记忆文件命名格式:
  <YYYYMMDD>-<NN>-<TYPE>-<id>.md
    YYYYMMDD : 日期
    NN       : 2 位当日序号（01、02、03...），每天从 01 开始
    TYPE     : 任务类型（大写：ANALYSIS / FIX / REQ / OPT / BUG / VERIFY）
    id       : 简短描述

  示例:
    20260226-01-ANALYSIS-v3-deep-review.md   # 当天第 1 个会话
    20260226-02-FIX-arch-verification.md     # 当天第 2 个会话
    20260226-03-OPT-hot-reload.md            # 当天第 3 个会话

输出: "📝 初始记忆已创建 → .ai-memory/clients/<agent>/tasks/<filename>.md"

本质: 合并了 v1.5 的"预检查"和"阶段 1（任务开始时）"
与 v1.5 区别: 明确触发器是"首条消息"，并在初始记忆中记录用户消息摘要
```

### 阶段 1：用户发消息时（捕获用户输入）🆕 v1.6

```yaml
触发条件: 用户发送包含新意图/新需求/方向修正的消息
触发方式: 🔴 自动执行，禁止询问用户
目的: 第一时间捕获用户输入，防止追加需求丢失

AI 执行:
  1. 记录用户消息的核心意图到 📨 对话记录（1 行摘要，不是全文复制）
  2. 如果用户修正了方向 → 更新记忆中的 🎯 任务摘要
  3. 如果用户补充了新需求 → 追加到记忆的 ⚠️ 待跟进

写入内容:
  - 📨 对话记录追加: | N | 👤→ | [用户消息摘要] |
  - 如有方向修正 → 更新 🎯 任务摘要
  - 如有新需求 → 追加到 ⚠️ 待跟进

⚠️ 重要: 这一步发生在 AI 开始分析之前
  - 极轻量 — 只记录摘要，不做分析，预估消耗 ~100 token
  - 确保即使 AI 在后续分析中 token 耗尽，用户说了什么至少有记录
  - 首条消息的记录已在阶段 0 完成，阶段 1 主要覆盖后续追加消息

🔴 过滤规则 — 以下消息不触发阶段 1:
  - 纯确认性消息: "好的" / "Y" / "确认" / "可以"
  - 流程控制: "继续" / "接着" / "下一步"
  - 情绪表达: "好的谢谢" / "明白了"
  判断标准: 用户消息包含新的意图/需求/约束/方向修正时才写入

为什么需要这个阶段（v1.5 缺失的问题）:
  - v1.5 中用户消息不是触发条件，追加需求没有记忆写入点
  - 用户消息是最有价值的原始输入，不应等 AI 理解后才记录转述版
  - 多轮对话中用户可能多次修正方向，每次修正都应被捕获
```

### 阶段 2：AI 回复时（写报告 + 更新记忆）

```yaml
触发条件: AI 准备向用户输出分析结果/回复
触发方式: 🔴 自动执行，禁止询问用户
目的: 记录 AI 的分析结论 + 写入报告

AI 执行:
  1. 确定报告类型和目标子目录（analysis/ / bugs/ / diagnostics/ 等）
  2. 扫描目标目录，确定当天序号 NN
  3. 写入报告文件到 reports/<子目录>/YYYYMMDD-NN-<类型>-<简述>.md
     报告包含完整分析内容（规范详见 temp-reports.md）
  4. 更新记忆文件:
     - 📄 关联报告 → 追加报告链接
     - 📨 对话记录 → 追加: | N | 🤖← | [AI 回复摘要 + 报告路径] |
  5. 对话中只输出结论摘要 + 报告文件路径

对话输出格式:
  AI: "[结论摘要 — 2-5 句话]

       📄 完整报告: reports/<子目录>/YYYYMMDD-NN-<类型>-<简述>.md"

与 v1.5 区别:
  - 触发器从"分析完成"改为"AI 回复"（可观测事件，更清晰）
  - 新增 📨 对话记录追加
```

### 阶段 3：AI 执行完毕时（记录变更清单）

```yaml
触发条件: AI 在一轮消息中的所有工具调用/文件操作完成
触发方式: 🔴 自动执行，禁止询问用户
目的: 记录实际执行的变更

AI 执行:
  1. 更新记忆文件 → 追加本轮实际修改的文件列表和关键决策
  2. 状态保持 🔄 进行中

写入内容:
  - 💡 关键决策 → 追加本轮决策
  - 如修改了文件 → 在 ⚠️ 待跟进 中更新已完成/新增项

注意: 增量更新应简洁高效，不要花大量 token 写冗长记录

与 v1.5 的区别:
  - 触发器从主观的"关键节点"改为客观的"执行完毕"
  - 不再需要判断"是否重要" — 每次有工具调用的轮次结束后都更新
  - 如果 AI 纯文字回复（无工具调用），阶段 3 不触发，只触发阶段 2

⚠️ 实操简化: 阶段 2 和阶段 3 通常在同一时刻写入
  （AI 准备回复用户时，同时完成写报告 + 记录变更）
  逻辑上保持两个概念的区分（报告/结论 vs 变更清单/决策），
  但实际可以合并为一次记忆文件更新操作。
```

```yaml
AI 执行:
  1. 执行完成验证（见下方 §任务完成验证）
  2. 最终更新记忆文件 → 状态改为 ✅ 完成
  3. 更新 clients/<agent>/SUMMARY.md → 更新状态 + 补充"关键决策"
  4. 同步关键决策到全局 .ai-memory/SUMMARY.md
  5. 更新 TASK-INDEX.md → 追加任务记录行（已有机制）
```

### 🔴 任务完成验证（声称完成前必须执行）

```yaml
触发条件: AI 准备声称"任务完成"之前

验证清单:
  1. 输出文件存在性:
     - [ ] 所有声称创建的文件确实存在？（用工具验证，不要假设）
     - [ ] 文件路径正确？（在预期的目录下）
     - [ ] 文件名符合命名规范？（YYYYMMDD-NN-<TYPE>-<id>.md）

     - [ ] 报告有完整的结构（标题→正文→结论）？
     - [ ] 没有中途截断的内容？

  3. 记忆文件完整性:
     - [ ] .ai-memory 中的任务记忆已更新？
     - [ ] 记忆文件包含报告链接？
     - [ ] SUMMARY.md 已追加/更新对应条目？

  4. 关联文件一致性（如修改了规范文件）:
     - [ ] 已执行交叉验证清单？
```

### ⚠️ 新会话继续任务（含 Token 耗尽恢复）

> 当用户在新会话中表达"继续"意图时，预检查流程自动识别并进入恢复模式。
> 完整恢复流程定义在 `workflows/00-pre-check/README.md §继续任务`。

```yaml
触发语句（任一匹配即触发）:
  - "继续" / "接着" / "继续上次任务"
  - "恢复记忆" / "恢复上下文"
  - "上次卡住了" / "上次断了"
  - "token 不够了继续" / "刚才没完成"

恢复流程摘要:
  1. 扫描 projects/*/.ai-memory/clients/<agent>/SUMMARY.md → 找状态 🔄 的任务
  2. 0个=没有未完成任务；1个=直接恢复；多个=让用户选
  3. 读取 clients/<agent>/tasks/<id>.md → 提取目标、报告链接、待办
  4. 读取 📨 对话记录 → 快速回忆对话脉络（🆕 v1.6）
  5. 读取关联的报告文件 → 恢复完整分析上下文
  6. 向用户输出摘要并确认
  7. 从断点继续执行，同时更新记忆文件

因为采用了"报告为主体+消息驱动+先写后干"策略，即使 token 耗尽，也有:
  - 📨 对话记录中的用户消息摘要（阶段 0+1 写入的）  ← 🆕 v1.6
  - 记忆文件中的任务目标和报告链接（阶段 0 写入的）
  - 报告文件中的完整分析内容（阶段 2 写入的）
  - 已更新的变更和决策（阶段 3 写入的）
```

## 🤖 多 Agent / 多编辑器支持（v1.3）

> **问题背景**: 用户可能同时使用多个编辑器（如 Zed + WebStorm）安装不同的 AI 助手，
> 它们共享同一个文件系统上的 `.ai-memory/` 目录。

### 🔴 方案：目录隔离 + 全局摘要

```
.ai-memory/
├── SUMMARY.md                          # 全局只读摘要（所有 Agent 的关键决策）
└── clients/
    ├── webstorm-copilot/               # WebStorm Copilot 专属
    │   ├── SUMMARY.md                  # 该 Agent 的任务摘要
    │   └── tasks/
    │       └── 20260227-01-FIX-xxx.md
    ├── zed-copilot/                    # Zed Copilot 专属
    │   ├── SUMMARY.md
    │   └── tasks/
    │       └── 20260226-01-ANALYSIS-xxx.md
    └── unknown-agent/                  # 无法确定编辑器时的默认目录
        ├── SUMMARY.md
        └── tasks/
```

### 为什么目录隔离？

```yaml
❌ 共享目录+文件名前缀的问题（v1.2 已废弃）:
  - SUMMARY.md 被多个 Agent 并发写入，容易互相覆盖
  - 恢复任务需要解析文件名中的 Agent 前缀来过滤，复杂且脆弱
  - 文件名过长（日期+Agent+类型+描述）可读性差

✅ 目录隔离的优点:
  - 每个 Agent 完全独立读写自己的 SUMMARY.md 和 tasks/，零冲突
  - 恢复任务时只扫描自己的目录，无需过滤
  - 用户在文件管理器中一眼看出哪个编辑器产生了哪些记忆
  - 全局 SUMMARY.md 仅追加关键决策摘要，不存在并发问题
```

### 读写规则

```yaml
写入:
  1. 报告写入 projects/<project>/reports/<子目录>/（所有 Agent 共享 reports/ 目录）
  2. 记忆写入 .ai-memory/clients/<agent>/tasks/ 和 clients/<agent>/SUMMARY.md
  3. 关键决策同步追加到全局 .ai-memory/SUMMARY.md（标注 Agent 来源）
  4. AI 绝不写入其他 Agent 的 clients/ 目录

读取:
  1. 先读取全局 .ai-memory/SUMMARY.md（了解跨编辑器的关键决策）
  2. 再读取 .ai-memory/clients/<agent>/SUMMARY.md（本 Agent 的任务列表）
  3. 读取记忆中的报告链接 → 按需读取报告文件恢复详细上下文
  4. 恢复任务时只从本 Agent 目录查找 🔄 状态的任务

跨 Agent 恢复:
  - 读取全局 SUMMARY.md 发现其他 Agent 有未完成任务时，提示用户确认
  - 用户确认后，在本 Agent 目录创建新记忆文件，标注"继承自 <原 Agent>"
  - 不修改原 Agent 目录下的文件
```

### Agent 检测方法

```yaml
优先级:
  1. 系统信息/环境线索（如本次对话的 "JetBrains-WS editor" → webstorm-copilot）
  2. 用户在对话中提及的编辑器名称
  3. 无法确定时使用 unknown-agent，提示用户确认

常见映射:
  JetBrains IDE (WebStorm/IDEA) → webstorm-copilot
  Zed Agent 面板               → zed-copilot
  VS Code Copilot Chat         → vscode-copilot
  Cursor                       → cursor
  Windsurf                     → windsurf
```
