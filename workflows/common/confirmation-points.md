# 确认点机制

> 确保关键决策经过用户确认的流程控制

---

## 📋 概述

确认点（Confirmation Point, CP）是 AI 执行流程中必须暂停并等待用户确认的关键节点。这确保了：
- 用户对方案有控制权
- 重要决策不被自动执行
- 可以在任何阶段调整方向

---

## 🎯 确认点定义

### 核心确认点（5 个）

| 确认点 | 位置 | 展示内容 | 用户选项 |
|-------|------|---------|---------|
| CP1 | 需求理解后 | 需求复述、改造清单 | 确认/修改 |
| CP2 | 方案设计后 | 技术方案、P0 清单 | 确认/修改/取消 |
| CP3 | 代码实现后 | 代码变更清单 | 确认/修改/取消 |
| CP4 | 测试完成后 | 测试结果 | 确认/重测/跳过 |
| CP5 | 文档生成后 | 文档内容 | 确认/修改 |

---

## 📊 各确认点详情

### CP1: 需求确认

```yaml
触发时机: 需求分析完成后，进入方案设计前
必须展示:
  - 需求复述（用 AI 的话重述）
  - 核心目标
  - 具体要求
  - 关键约束
  - 改造范围（如涉及代码改造）

输出格式:
  📋 需求理解确认
  
  我理解您的需求是：
  
  1. **核心目标**: [一句话描述]
  2. **具体要求**:
     - [要求1]
     - [要求2]
  3. **关键约束**:
     - [约束1]
     - [约束2]
  4. **改造范围**:
     - 需要改: [列举]
     - 不需要改: [列举]
  
  ❓ 我的理解是否正确？
  
  [✅ 确认] [✏️ 修改]

用户响应处理:
  ✅ 确认: 继续到方案设计
  ✏️ 修改: 根据反馈重新理解
```

### CP2: 方案确认

```yaml
触发时机: 技术方案设计完成后，进入代码实现前
必须展示:
  - 技术方案概述
  - 架构设计
  - 修改文件清单
  - 依赖变更
  - 🔴 P0 清单（如果是 P0 操作）

输出格式:
  📋 技术方案确认
  
  ## 方案概述
  [简要描述]
  
  ## 修改清单
  | 文件 | 操作 | 说明 |
  |-----|------|------|
  | file1.ts | 新增 | 添加限流中间件 |
  | file2.ts | 修改 | 注册中间件 |
  
  ## 依赖变更
  - 新增: flex-rate-limit@^1.0.0
  
  ---
  
  🔴 P0 操作清单（如有）:
  [展示危险操作]
  
  ---
  
  ❓ 方案是否确认？
  
  [✅ 确认] [✏️ 修改] [❌ 取消]

用户响应处理:
  ✅ 确认: 继续到代码实现
  ✏️ 修改: 根据反馈调整方案
  ❌ 取消: 终止流程
```

### CP3: 代码确认

```yaml
触发时机: 代码实现完成后，进入测试前
必须展示:
  - 代码变更清单
  - 新增文件
  - 修改文件（diff 摘要）
  - 删除文件（如有）

输出格式:
  📋 代码实现确认
  
  ## 变更统计
  - 新增: 2 个文件
  - 修改: 3 个文件
  - 删除: 0 个文件
  
  ## 变更详情
  
  ### 新增文件
  - src/middleware/rate-limit.ts (85 行)
  - config/rate-limit.config.ts (25 行)
  
  ### 修改文件
  - src/app.ts (+5 行, -0 行)
    - 添加中间件导入
    - 注册中间件
  
  ## 三轮验证结果
  ✅ 逻辑验证: 6/6 通过
  ✅ 技术验证: 7/7 通过
  ✅ 完整性验证: 10/10 通过
  
  ---
  
  ❓ 代码是否确认？
  
  [✅ 确认] [✏️ 修改] [❌ 取消]

用户响应处理:
  ✅ 确认: 继续到测试
  ✏️ 修改: 根据反馈修改代码
  ❌ 取消: 回滚代码变更
```

### CP4: 测试确认

```yaml
触发时机: 测试执行完成后，进入文档生成前
必须展示:
  - 测试结果汇总
  - 通过/失败用例
  - 覆盖率（如有）

输出格式:
  📋 测试结果确认
  
  ## 测试统计
  - 总用例: 15
  - 通过: 14
  - 失败: 1
  - 跳过: 0
  
  ## 失败用例
  - test/rate-limit.test.ts
    - ❌ should handle concurrent requests
      Error: timeout exceeded
  
  ## 覆盖率
  - 行覆盖: 85%
  - 分支覆盖: 78%
  
  ---
  
  ❓ 测试结果如何处理？
  
  [✅ 确认继续] [🔄 重新测试] [⏭️ 跳过失败用例]

用户响应处理:
  ✅ 确认继续: 继续到文档生成
  🔄 重新测试: 修复后重新运行测试
  ⏭️ 跳过: 记录失败用例，继续流程
```

### CP5: 文档确认

```yaml
触发时机: 文档生成完成后，进入最终完成前
必须展示:
  - 生成的文档列表
  - 文档内容摘要

输出格式:
  📋 文档确认
  
  ## 生成的文档
  1. 01-requirement.md - 需求文档
  2. 02-technical.md - 技术方案
  3. 03-implementation.md - 实施记录
  
  ## 文档存放位置
  projects/user-service/requirements/20260212-rate-limit/
  
  ---
  
  ❓ 文档是否确认？
  
  [✅ 确认] [✏️ 修改]

用户响应处理:
  ✅ 确认: 完成任务
  ✏️ 修改: 根据反馈修改文档
```

### CP6: 项目文档更新确认 🆕

```yaml
触发时机: 代码/功能实现完成后，任务结束前
触发条件（满足任一即触发）:
  - 代码文件被修改
  - 配置文件被修改
  - 新功能开发完成
  - Bug 修复完成

必须展示:
  - STATUS.md 更新内容预览
  - CHANGELOG.md 新增条目预览
  - plans/ 文档状态

输出格式:
  📋 项目文档更新确认
  
  ## STATUS.md 更新
  - 版本: v1.2.0 → v1.3.0
  - 最后更新: 2026-02-12
  - 进度: +1 个已完成需求
  
  ## CHANGELOG.md 新增条目
  ```
  ### v1.3.0 (2026-02-12)
  
  #### 新功能
  - 添加 API 限流中间件
  
  #### 变更
  - 更新依赖 flex-rate-limit@1.0.0
  ```
  
  ## plans/ 文档状态
  - requirements/20260212-rate-limit/ ✅ 已完成
  
  ---
  
  ❓ 是否应用项目文档更新？
  
  [C] Continue - 确认应用更新
  [M] Modify - 手动编辑更新内容
  [S] Skip - 跳过更新（需确认原因）

用户响应处理:
  [C] Continue: 应用更新，完成任务
  [M] Modify: 允许用户编辑更新内容
  [S] Skip: 记录跳过原因，完成任务
```

## 🔴 询问点（Inquiry Points）🆕

询问点与确认点不同，用于在关键阶段询问用户需要生成哪些产物。

### IP1: 测试产物询问

```yaml
触发时机: 代码实现确认后（CP3 之后）
目的: 确定需要生成哪些测试相关产物

输出格式:
  🔴 测试产物选择
  
  请选择需要生成的测试产物：
  
  [1] 单元测试 - Jest/Vitest 测试用例
  [2] HTTP 测试 - .http 文件（用于 REST Client）
  [3] OpenAPI 文档 - Swagger/OpenAPI 规范
  [A] 全部生成
  [S] 跳过测试生成
  
  请选择 [1/2/3/A/S]:

用户响应处理:
  [1]: 仅生成单元测试
  [2]: 仅生成 HTTP 测试文件
  [3]: 仅生成 OpenAPI 文档
  [A]: 生成所有测试产物
  [S]: 跳过，继续到文档阶段
```

### IP2: 文档产物询问

```yaml
触发时机: 测试确认后（CP4 之后）
目的: 确定需要更新/生成哪些文档

输出格式:
  🔴 文档产物选择
  
  请选择需要生成/更新的文档：
  
  [1] README.md - 更新项目说明
  [2] CHANGELOG.md - 添加变更记录
  [3] 脚本文档 - scripts/README.md
  [4] API 文档 - 接口使用说明
  [A] 全部生成
  [S] 跳过文档生成
  
  请选择 [1/2/3/4/A/S]:

用户响应处理:
  [1]: 仅更新 README.md
  [2]: 仅更新 CHANGELOG.md
  [3]: 仅更新脚本文档
  [4]: 仅生成 API 文档
  [A]: 生成所有文档
  [S]: 跳过，继续到完成阶段
```

## 🔄 用户响应选项

### 标准选项

| 选项 | 符号 | 说明 | AI 行为 |
|-----|------|------|--------|
| 确认 | ✅ | 同意当前内容，继续下一步 | 执行下一步骤 |
| 修改 | ✏️ | 需要调整当前内容 | 等待修改指令 |
| 取消 | ❌ | 终止当前流程 | 保存进度，退出 |
| 跳过 | ⏭️ | 跳过当前步骤 | 记录跳过，继续 |
| 重试 | 🔄 | 重新执行当前步骤 | 重新执行 |

### 特殊选项

| 选项 | 符号 | 说明 | AI 行为 |
|-----|------|------|--------|
| Stop & Save | 📦 | 保存当前进度后退出 | 生成进度文件 |
| 全部确认 | ✅✅ | 确认所有 P0 操作 | 继续执行 |

## 📊 确认点与风险等级

### P0 操作的确认点

```yaml
特殊要求:
  - CP2 必须展示 P0 清单
  - 用户必须逐项确认
  - 必须确认回滚方案

P0 清单格式:
  🔴 P0 操作清单
  ════════════════════════════════════════
  
  ⚠️ 以下操作需要您逐项确认：
  
  1. [数据库] 将添加索引 users.email
     影响: 索引创建期间可能短暂阻塞写操作
     回滚: db.users.dropIndex("email_1")
     确认: [Y/N]
  
  2. [配置] 将修改 Redis 连接超时
     原值: 5000ms
     新值: 10000ms
     回滚: 改回 5000ms
     确认: [Y/N]
  
  ════════════════════════════════════════
```

### P1 操作的确认点

```yaml
标准要求:
  - 4 个确认点
  - 每个确认点都可以修改或取消
```

### P2 操作的确认点

```yaml
简化要求:
  - 0-2 个确认点
  - 可以合并确认
```

## 🚨 确认点违规处理

### 违规情况

```yaml
违规1: 跳过确认点
  示例: AI 在 CP2 未获得确认就开始写代码
  后果: 回滚代码，重新等待确认

违规2: 假设用户同意
  示例: AI 说"如果没问题我就继续了"
  后果: 必须明确等待用户响应

违规3: 不展示必要内容
  示例: CP2 没有展示修改清单
  后果: 重新展示完整内容
```

### 正确做法

```yaml
正确做法:
  - 明确输出"请确认"
  - 提供清晰的选项
  - 等待用户明确响应
  - 不做任何假设
```
