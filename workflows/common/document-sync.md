# 文档同步机制

> 确保项目文档（STATUS.md / CHANGELOG.md）与代码变更保持同步

---

## 📋 概述

文档同步机制确保每次代码变更后，项目级别的文档能够自动更新，保持文档与代码的一致性。

---

## 🎯 触发条件

### 强制触发（满足任一即触发）

| 条件 | 说明 | 优先级 |
|------|------|--------|
| 代码文件修改 | `src/**`, `lib/**`, `app/**` 等 | P0 |
| 配置文件修改 | `package.json`, `tsconfig.json`, `.env.example` | P0 |
| 新功能开发完成 | 需求开发任务完成 | P0 |
| Bug 修复完成 | Bug 修复任务完成 | P1 |

### 豁免条件

| 条件 | 说明 | 行为 |
|------|------|------|
| 仅测试文件修改 | `test/**`, `__tests__/**` | 跳过同步 |
| 仅文档文件修改 | `docs/**`, `*.md` | 跳过同步 |
| 用户明确跳过 | 在 CP6 选择 [S] Skip | 记录原因，跳过 |

---

## 📁 必须同步的文件

### 1. STATUS.md（P0 必须更新）

```yaml
更新内容:
  - 版本号（如有版本变更）
  - 最后更新日期（当前日期）
  - 进度统计（已完成/进行中/计划中）
  - 需求状态更新

更新位置: 项目根目录 STATUS.md

检测逻辑:
  1. 读取当前 STATUS.md
  2. 检测版本号是否需要更新
  3. 更新最后修改日期
  4. 更新进度统计数据
  5. 生成 diff 展示给用户
```

### 2. CHANGELOG.md（P0 必须更新）

```yaml
更新内容:
  - 版本概览表新增条目
  - 重要更新内联摘要（可选）
  - changelogs/vX.Y.Z.md 详细变更文档

存储结构（参考 monSQLize）:
  项目根目录/
  ├── CHANGELOG.md              # 版本概览（摘要表）
  └── changelogs/               # 详细变更目录
      ├── README.md             # 目录说明
      ├── TEMPLATE.md           # 变更文档模板
      ├── v1.0.0.md            # v1.0.0 详细变更
      └── vX.Y.Z.md            # 其他版本详细变更

CHANGELOG.md 格式:
  ## 版本概览
  
  | 版本 | 日期 | 变更摘要 | 详细 |
  |------|------|---------|------|
  | [v1.1.0](./changelogs/v1.1.0.md) | 2026-02-12 | 新功能描述 | [查看](./changelogs/v1.1.0.md) |
  | [v1.0.0](./changelogs/v1.0.0.md) | 2026-02-01 | 初始版本 | [查看](./changelogs/v1.0.0.md) |
  
  ## 🚨 重要更新
  
  ### vX.Y.Z - 版本标题
  
  **发布日期**: YYYY-MM-DD
  **版本类型**: Release / Feature / Bugfix
  
  #### 核心变更
  - 变更1
  - 变更2

changelogs/vX.Y.Z.md 必需章节:
  1. 版本信息（版本号/日期/类型/风险/兼容性）
  2. 变更摘要（一句话总结）
  3. 背景说明（为什么变更）
  4. 变更内容（改了什么）
  5. 影响范围（影响哪些文件）
  6. 详细变更清单（Added/Changed/Fixed/Removed）
  7. 验证方法（如何验证）

检测逻辑:
  1. 读取当前 CHANGELOG.md
  2. 确定新版本号
  3. 收集本次任务的变更
  4. 更新版本概览表
  5. 生成 changelogs/vX.Y.Z.md 详细文档
  6. 插入重要更新摘要（如是重大版本）
```

### 3. README.md（P1 视情况更新）

```yaml
更新条件:
  - 新功能涉及用户可见的功能变更
  - API 接口有变更
  - 配置项有变更
  - 安装/使用方式有变更

更新内容:
  - 功能列表
  - 使用说明
  - 配置说明
  - API 文档链接
```

## 🔄 同步流程

### 标准流程

```
代码实现完成
    ↓
CP3: 代码确认
    ↓
CP4: 测试确认
    ↓
CP5: 文档确认
    ↓
检测是否触发同步
    ↓
[触发] → 生成更新预览
    ↓
CP6: 项目文档更新确认
    ↓
[C] → 应用更新 → 完成
[M] → 用户编辑 → 应用 → 完成
[S] → 记录跳过 → 完成
```

### AI 执行步骤

```yaml
步骤 1: 检测触发条件
  - 读取本次任务的文件变更列表
  - 判断是否满足触发条件
  - 判断是否满足豁免条件

步骤 2: 收集变更信息
  - 任务类型（需求/Bug/优化等）
  - 功能描述
  - 影响范围
  - 版本号建议

步骤 3: 生成更新内容
  - STATUS.md 更新内容
  - CHANGELOG.md 新条目
  - README.md 更新建议（如需要）

步骤 4: 展示更新预览
  - 显示 diff 格式的变更
  - 展示更新前后对比

步骤 5: 等待用户确认
  - 在 CP6 等待用户选择
  - 处理用户响应

步骤 6: 应用更新
  - [C] 应用所有更新
  - [M] 等待用户编辑后应用
  - [S] 记录跳过原因，不应用
```

## 📝 版本号规则

### 语义化版本 (SemVer)

```yaml
格式: MAJOR.MINOR.PATCH

MAJOR: 不兼容的 API 变更
MINOR: 新功能（向后兼容）
PATCH: Bug 修复（向后兼容）

示例:
  1.0.0 → 1.0.1  # Bug 修复
  1.0.0 → 1.1.0  # 新功能
  1.0.0 → 2.0.0  # 破坏性变更
```

### 版本号更新建议

| 任务类型 | 版本变更 | 示例 |
|---------|---------|------|
| Bug 修复 | PATCH +1 | 1.2.3 → 1.2.4 |
| 新功能 | MINOR +1 | 1.2.3 → 1.3.0 |
| 性能优化 | PATCH +1 | 1.2.3 → 1.2.4 |
| 重构 | MINOR +1 | 1.2.3 → 1.3.0 |
| 破坏性变更 | MAJOR +1 | 1.2.3 → 2.0.0 |

```diff
# 项目状态

- **版本**: v1.2.0

+ **最后更新**: 2026-02-12

## 进度统计

- **已完成**: 15 → 16
- **进行中**: 3 → 2
- **计划中**: 5 → 5
```

```diff
# 更新日志

+ ### 变更

+

## [1.2.0] - 2026-02-01

```

## 🚨 同步失败处理

### 常见失败情况

| 情况 | 原因 | 处理方式 |
|------|------|---------|
| 文件不存在 | 项目缺少 STATUS.md | 提示创建模板 |
| 格式不匹配 | 文档格式非标准 | 尝试智能解析 |
| 版本冲突 | 版本号已存在 | 询问用户处理 |
| 权限问题 | 文件只读 | 报告错误 |

### 失败后的行为

```yaml
失败处理:
  1. 报告失败原因
  2. 提供手动更新建议
  3. 允许用户选择跳过
  4. 记录到审计日志
```
