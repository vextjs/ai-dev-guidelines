## 工作流概览

当前系统中有 8 个核心工作流：

```
WF-01 ══════ 新需求规范化
      │
      ↓
WF-02 ══════ 技术方案设计
      │
      ↓
WF-03 ══════ 核心功能开发
      │
      ↓
WF-04 ══════ 集成和测试
      │
      ↓
WF-05 ══════ 文档补充
      │
      ├─ WF-06 ─ 性能优化 (可选平行)
      │
      ├─ WF-07 ─ Bug 修复 (可以中断)
      │
      ├─ WF-08 ─ 架构重构 (可选，大任务)
      │
      ↓
WF-09 ══════ 完成验证和交付
```

## 转移规则矩阵

### 工作流间转移规则表

```
FROM \ TO  | WF-01 | WF-02 | WF-03 | WF-04 | WF-05 | WF-06 | WF-07 | WF-08 | WF-09
-----------|-------|-------|-------|-------|-------|-------|-------|-------|-------
WF-01      |   -   |  ✅   |  ❌   |  ❌   |  ❌   |  ❌   |  ✅*  |  ❌   |  ❌
WF-02      |  ✅** |   -   |  ✅   |  ❌   |  ❌   |  ❌   |  ✅*  |  ✅   |  ❌
WF-03      |  ✅** |  ✅** |   -   |  ✅   |  ❌   |  ✅   |  ✅*  |  ❌   |  ❌
WF-04      |  ✅** |  ✅** |  ✅** |   -   |  ✅   |  ❌   |  ✅*  |  ❌   |  ❌
WF-05      |  ✅** |  ✅** |  ✅** |  ✅** |   -   |  ❌   |  ✅*  |  ❌   |  ✅
WF-06      |  ❌   |  ❌   |  ❌   |  ❌   |  ✅   |   -   |  ✅*  |  ❌   |  ✅
WF-07      |  ❌   |  ❌   |  ❌   |  ❌   |  ❌   |  ❌   |   -   |  ❌   |  ✅
WF-08      |  ✅** |  ❌   |  ❌   |  ❌   |  ❌   |  ❌   |  ✅*  |   -   |  ❌
WF-09      |  ❌   |  ❌   |  ❌   |  ❌   |  ❌   |  ❌   |  ❌   |  ❌   |   -

图例:
  ✅  = 合法转移
  ❌  = 不合法转移
  *   = 紧急插队 (bug 中断当前工作流)
  **  = 需要审批 (发现需求或设计问题)
```

### 转移规则说明

```yaml
✅ 合法直接转移:
  # 正常流程
  WF-01 → WF-02 (需求规范完成，开始设计)
  WF-02 → WF-03 (设计完成，开始开发)
  WF-03 → WF-04 (开发完成，开始测试)
  WF-04 → WF-05 (测试通过，补充文档)
  WF-05 → WF-09 (文档完成，准备交付)
  
  # 优化平行工作
  WF-03 → WF-06 (开发同时进行性能优化)
  WF-04 → WF-05 (集成测试同时补充文档)
  WF-06 → WF-05 (性能优化完成，补充性能文档)
  WF-07 → WF-09 (Bug 修复完成，交付)

❌ 不合法转移 (需要先完成中间步骤):
  WF-01 → WF-03 (跳过设计)
  WF-02 → WF-04 (设计完成但开发未完成)
  WF-03 → WF-05 (开发完成但测试未完成)
  
  原因: 这些步骤之间有逻辑依赖，不能跳过

⚠️  需要审批的转移:
  WF-02 → WF-02 (设计返工，发现设计问题)
  WF-03 → WF-02 (开发返工，发现设计缺陷)
  WF-04 → WF-03 (测试失败，需要返工)
  
  说明: 有跨越主要阶段的回溯，需要评估影响
  处理: 创建 GitHub issue 记录问题，讨论是否需要改动需求

🚨 紧急插队:
  任何状态 → WF-07 (发现 bug，立即修复)
  
  说明: Bug 修复可以中断当前工作
  处理: 
    1. 创建新的 bug 分支
    2. 修复完成后回到原工作流
    3. 完成剩余工作
    4. 进入 WF-09 交付
```

## 特定转移场景

### 场景 1: 正常流程完成

```
初始状态: WF-01 (新需求)
│
├─ 需求规范完成
│  检查: 需求是否清晰? 是否有验收标准?
│  结果: ✅ 是
│
↓ 转移到 WF-02

├─ 技术方案完成
│  检查: 方案是否完整? 有无遗漏的复杂性?
│  结果: ✅ 是
│
↓ 转移到 WF-03

├─ 核心功能开发完成
│  检查: 代码是否通过 linting? 主要测试是否通过?
│  结果: ✅ 是
│
├─ 可选: 转移到 WF-06 (同步进行性能优化)
│  或继续到 WF-04
│
↓ 转移到 WF-04

├─ 集成和测试完成
│  检查: 所有测试通过? 代码覆盖率 >= 80%?
│  结果: ✅ 是
│
↓ 转移到 WF-05

├─ 文档补充完成
│  检查: API 文档完整? 设计文档完整?
│  结果: ✅ 是
│
↓ 转移到 WF-09

└─ 完成验证和交付
   生成 commit 描述，等待人工提交
```

### 场景 2: 发现设计问题，需要返工

```
当前状态: WF-03 (开发中)
│
│ 发现问题: 数据库字段不足，需要修改 schema
│
├─ 评估影响:
│  问题: 需要改 DB schema
│  影响范围: 中等 (影响 3 个数据模型)
│  开发工作: 估计 4 小时
│  
│ 决策: ⚠️ 需要审批
│
↓ 回溯到 WF-02 (设计返工)
│
├─ 修改技术方案
│  - 更新 schema 设计
│  - 更新相关 API 设计
│  - 更新类图和流程图
│
│ 审批: ✅ 方案已更正
│
↓ 重新进入 WF-03 (续)
│
├─ 继续开发
│  - 应用新的 schema
│  - 更新相关模型和服务
│  - 重新测试
│
│ 检查: ✅ 功能完整，测试通过
│
↓ 转移到 WF-04 (集成和测试)
```

### 场景 3: 发现严重 Bug，紧急插队

```
当前状态: WF-04 (集成测试)
│
│ 📞 生产环境报告: 支付流程崩溃
│ 影响: 订单无法完成，直接损失
│
├─ 决策: 🚨 立即停止当前工作，进行 Bug 修复
│
↓ 创建新分支: hotfix/payment-crash
│
├─ 快速分析: 支付 API 调用失败
│ 修复: 更新 API 调用超时时间
│ 验证: 测试支付流程恢复
│
↓ 完成 Bug 修复 (WF-07)
│
├─ 生成 Commit:
│  Message: "fix: increase payment API timeout to 30s"
│
↓ 转移到 WF-09 (交付)
│
├─ 部署修复到生产
│
└─ 完成后:
   - 通知所有受影响方
   - 返回 WF-04 继续原工作
   - 根据时间决定是否需要架构改进 (WF-08)
```

### 场景 4: 性能优化平行工作

```
当前状态: WF-03 (开发完成)
            WF-04 (集成测试进行中)
│
├─ 团队分工:
│  - 开发人员 A: 继续 WF-04
│  - 开发人员 B: 转移到 WF-06 (性能优化)
│
↓ 开发人员 B → WF-06
│
├─ 性能检查:
│  - 分析瓶颈
│  - 优化数据库查询
│  - 添加缓存
│
│ 完成: ✅ 性能提升 40%
│
↓ 转移到 WF-05 (文档)
│
├─ 编写性能优化文档:
│  - 优化前后的基准测试
│  - 优化方案解释
│  - 建议的部署步骤
│
│ 同步: 等待 开发人员 A 完成 WF-05
│
└─ 都完成后 → WF-09 (交付)
```

### 场景 5: 架构重构 (大型任务)

```
当前状态: WF-02 (设计阶段)
│
│ 识别问题: 当前架构无法支持未来的扩展需求
│ 范围: 整个微服务架构重构
│ 工作量: 200+ 小时
│
├─ 决策:
│  问题: 这个需求是否足够紧急?
│
│  ✅ 如果是: 开始 WF-08 (架构重构)
│  ❌ 如果不是: 先完成当前任务
│
↓ 开始 WF-08 (假设决定进行)
│
├─ WF-08 需要:
│  1. 创建详细的架构设计文档
│  2. 进行架构 ADR (Architecture Decision Record)
│  3. 获得团队和管理层批准
│  4. 制定实施路线图 (分阶段)
│  5. 评估风险和回滚计划
│
│ 完成: ✅ 架构设计和批准
│
↓ 根据结果:
│
├─ 如果批准: 按阶段重构
│  - Phase 1: 微服务拆分
│  - Phase 2: 数据一致性保障
│  - Phase 3: 性能优化
│  - Phase 4: 监控和告警
│
└─ 每个阶段都走完整的 WF-03 → WF-04 → WF-05 → WF-09
```

## 决策树

### 工作流选择决策树

```
开始一个新任务
│
├─ 这是什么类型的任务?
│  │
│  ├─ 📝 新需求
│  │  └─ 进入 WF-01 (新需求规范化)
│  │
│  ├─ 🔍 需求发现问题
│  │  └─ 停止 WF-01 并修正
│  │
│  ├─ 📐 技术方案相关
│  │  ├─ 问题: 这个需求是否已经有方案?
│  │  │  ├─ 是: 继续开发 → WF-03
│  │  │  └─ 否: 制定方案 → WF-02
│  │  │
│  │  └─ 问题: 这是需求层的架构改进吗?
│  │     ├─ 是: 可能需要 WF-08 (架构重构)
│  │     └─ 否: WF-02 (设计)
│  │
│  ├─ 💻 代码开发
│  │  └─ 问题: 是否有完整的设计和需求?
│  │     ├─ 是: 进入 WF-03 (开发)
│  │     └─ 否: 返回 WF-01/WF-02 完成前置
│  │
│  ├─ ✅ 测试和集成
│  │  └─ 进入 WF-04 (集成和测试)
│  │
│  ├─ 📚 文档
│  │  └─ 问题: 功能代码是否完成?
│  │     ├─ 是: 进入 WF-05 (文档)
│  │     └─ 否: 返回 WF-03 先完成代码
│  │
│  ├─ ⚡ 性能优化
│  │  └─ 问题: 是否与当前任务平行?
│  │     ├─ 是: 进入 WF-06 (可选)
│  │     └─ 否: 推迟，先完成主要功能
│  │
│  ├─ 🐛 Bug 修复
│  │  └─ 问题: 影响程度?
│  │     ├─ 🚨 严重 (生产崩溃):
│  │     │  └─ 立即进入 WF-07 (紧急)
│  │     │
│  │     ├─ ⚠️  中等 (功能不正常):
│  │     │  └─ 创建 issue，等待当前任务完成后处理
│  │     │
│  │     └─ ℹ️  低 (轻微问题):
│  │        └─ 列入 backlog，后续统一处理
│  │
│  └─ 🏗️ 架构重构
│     └─ 问题: 范围有多大?
│        ├─ 小 (单个模块): 作为 WF-03 的一部分
│        ├─ 中 (多个模块): 需要单独的 WF-08
│        └─ 大 (整体架构): 需要 WF-08 + 分阶段执行
│
└─ 确定工作流后，开始执行
```

### 返工决策树

```
发现需要返工
│
├─ 返工的原因是什么?
│  │
│  ├─ 需求不清晰
│  │  └─ 返回 WF-01
│  │     原因: 需求规范不完整
│  │     影响: 重新评估，可能改变工作量
│  │
│  ├─ 设计有缺陷
│  │  └─ 返回 WF-02
│  │     原因: 设计遗漏或错误
│  │     影响: 修改设计，返工开发代码
│  │
│  ├─ 代码有问题
│  │  └─ 停留 WF-03
│  │     原因: 代码实现不符合设计
│  │     影响: 修复代码，不影响设计
│  │
│  ├─ 测试失败
│  │  ├─ 如果是代码问题:
│  │  │  └─ 返回 WF-03 修复
│  │  │
│  │  └─ 如果是设计问题:
│  │     └─ 返回 WF-02 修改设计
│  │
│  ├─ 文档不完整
│  │  └─ 停留 WF-05
│  │     原因: 补充文档
│  │     影响: 仅影响文档，不影响代码
│  │
│  └─ 其他问题
│     └─ 评估影响范围后确定返工点
│
├─ 评估返工成本
│  ├─ 有多少代码需要改动? (%)
│  ├─ 需要多少时间? (小时)
│  └─ 是否会影响其他相关工作?
│
├─ 决定是否返工
│  ├─ 成本低 (< 20%): 立即返工
│  ├─ 成本中 (20-50%): 评估和讨论后决定
│  └─ 成本高 (> 50%): 需要管理层批准
│
└─ 执行返工
   └─ 返工完成后，使用 ✅ 检查新的质量，再继续
```

## 常见模式

### 模式 1: 快速迭代循环

适用于: 内部工具、临时脚本、原型验证

```
WF-01 (快速规范, 30 分钟)
  ↓
WF-02 (轻量设计, 1 小时)
  ↓
WF-03 (快速开发, 4 小时)
  ↓
WF-04 (基础测试, 2 小时)
  ↓
WF-05 (轻量文档, 1 小时)
  ↓
WF-09 (交付)

总时间: 9 小时
特点: 减少设计和文档的形式化要求
```

### 模式 2: 标准开发循环

适用于: 新功能开发、中等复杂度任务

```
WF-01 (需求规范, 2 小时)
  ↓
WF-02 (技术设计, 4 小时)
  ↓
WF-03 (核心开发, 20 小时)
  ↓
WF-04 (集成测试, 8 小时)
  ↓
WF-05 (文档补充, 4 小时)
  ↓
WF-09 (交付)

总时间: 38 小时 (标准 1 周工作)
特点: 完整的规范化流程，适合生产环境
```

### 模式 3: 复杂任务 + 平行工作

适用于: 大型功能、多个相关任务

```
WF-01 (2小时)
  ↓
WF-02 (6小时)
  ↓
┌─ WF-03A (开发功能 A)
│
├─ WF-03B (开发功能 B, 平行)
│
└─ WF-06 (性能优化, 平行)
  ↓
WF-04 (集成所有功能)
  ↓
WF-05 (完整文档)
  ↓
WF-09 (交付)

总时间: 40 小时 (3 人, 2 周)
特点: 通过平行工作加快进度
```

### 模式 4: Bug 修复 + 快速交付

适用于: 生产环保的 bug 修复

```
发现 Bug (生产环境)
  ↓
WF-07 (Bug 分析和修复, 2 小时)
  ↓
快速验证 (测试修复, 1 小时)
  ↓
WF-09 (交付，可能跳过文档)

总时间: 3 小时
特点: 加速流程，优先恢复服务
```

### 模式 5: 架构重构 + 分阶段

适用于: 系统级改进，大范围重构

```
WF-08 第 1 阶段 (架构设计)
  ↓
Phase 1: 数据库迁移
  ├─ WF-03 (迁移脚本)
  ├─ WF-04 (验证数据一致性)
  └─ WF-09 (部署)
  ↓
Phase 2: 微服务拆分
  ├─ WF-02 (设计新的服务边界)
  ├─ WF-03 (实现新服务)
  ├─ WF-04 (集成测试)
  └─ WF-09 (灰度部署)
  ↓
Phase 3: 性能优化
  ├─ WF-06 (性能优化)
  └─ WF-09 (部署)

总时间: 200+ 小时 (分阶段在 3+ 个月完成)
特点: 降低单次部署的风险，允许中途调整
```

## 工具和流程

### Git 分支命名对应工作流

```
wf-01/requirement-xxx       # WF-01: 需求规范
wf-02/design-xxx            # WF-02: 技术设计
wf-03/feature-xxx           # WF-03: 核心开发
wf-04/test-xxx              # WF-04: 集成测试 (通常在 wf-03 中完成)
wf-05/docs-xxx              # WF-05: 文档补充 (通常在 wf-03 中完成)
wf-06/perf-xxx              # WF-06: 性能优化
wf-07/bugfix-xxx            # WF-07: Bug 修复
wf-08/refactor-xxx          # WF-08: 架构重构
hotfix/urgent-xxx           # 生产紧急修复

示例:
git checkout -b wf-03/user-authentication
git checkout -b hotfix/payment-crash
```

### GitHub Issue 中标记工作流状态

```markdown
# 用户认证功能

## Metadata
- WF: WF-01 → WF-02 → WF-03 (当前) → WF-04 → WF-05
- 状态: 🟡 开发中
- 优先级: 高
- 工作量: 40 小时 (已用 15 小时)

## 进度
- [x] WF-01: 需求规范 (2/2)
- [x] WF-02: 技术设计 (4/4)
- [ ] WF-03: 核心开发 (15/20)
  - [x] 用户模型设计
  - [x] 认证 API 实现
  - [ ] 授权逻辑
  - [ ] 刷新令牌
- [ ] WF-04: 集成测试 (0/8)
- [ ] WF-05: 文档补充 (0/4)

## 转移检查清单 (WF-03 → WF-04)
```
