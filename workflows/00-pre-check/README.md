# 工作流 00 - 前置检查流程

> **流程代码**: WF-00  
> **驱动方式**: AI 自动执行 + 用户确认  
> **执行时机**: 所有工作流开始前（第一步）  
> **预计耗时**: 2-5 分钟  
> **目标**: 确保环境、规范和资源就绪

---

## 📋 流程概述

在执行任何 AI 开发任务前，必须先通过前置检查。这个流程验证：
- ✅ 项目规范是否完整
- ✅ 环境是否满足要求
- ✅ Git 仓库状态是否正常
- ✅ 输出目录是否可用
- ✅ 关键工具是否已安装

**设计理念**: AI 自动检查，用户确认关键点

---

## 🎯 执行步骤

### Step 1: 项目规范检查

**目标**: 验证项目的规范文件完整性

**AI 自动执行**:
```bash
检查项目目录: projects/<project-name>/

必需文件检查:
  ✅ PROJECT-PROFILE.md      - 项目描述
  ✅ TECH-STACK.md           - 技术栈
  ✅ CODE-STANDARDS.md       - 代码规范
  ✅ DEPLOYMENT.md           - 部署规范
  ✅ ARCHITECTURE.md         - 架构文档

可选但推荐:
  ⚠️ TESTING.md              - 测试规范
  ⚠️ CI-CD.md                - 持续集成规范
  ⚠️ MONITORING.md           - 监控规范
  ⚠️ SECURITY.md             - 安全规范
```

**检查结果示例**:
```

═════════════════════════════════════════
项目         ✅ 存在 (projects/user/)
必需文件:
  PROJECT-PROFILE.md        ✅ 存在 (831 行)
  TECH-STACK.md            ✅ 存在 (245 行)
  CODE-STANDARDS.md        ✅ 存在 (420 行)
  DEPLOYMENT.md            ✅ 存在 (180 行)
  ARCHITECTURE.md          ✅ 存在 (590 行)

可选文件:
  TESTING.md              ⚠️  缺失（建议补充）
  CI-CD.md                ⚠️  缺失（建议补充）
═════════════════════════════════════════

状态: ✅ 基础规范完整，可继续
```

**人工决策点**:
```
⚠️ 如果缺失必需文件:
   提示: "项目规范不完整，建议先补充 [缺失文件]"
   选择:
     [A] 继续执行（跳过缺失规范）
     [B] 中止，补充规范后重试
     [C] 使用默认规范（给出警告）
```

### Step 2: 环境检查

**目标**: 验证 AI 所需的工具和环境

**AI 自动执行**:
```bash
检查项:
  1. Node.js 版本
     要求: >= 16.0.0
     命令: node --version
  
  2. npm/yarn 包管理器
     要求: 已安装
     命令: npm --version 或 yarn --version
  
  3. Git 工具
     要求: >= 2.20.0
     命令: git --version
  
  4. 磁盘空间
     要求: >= 1GB 可用空间
     命令: 检查 /dev/disk
  
  5. 网络连接
     要求: 可访问 npm registry
     命令: ping registry.npmjs.org
```

**检查结果示例**:
```
环境检查
═════════════════════════════════════════
Node.js                  ✅ v18.16.0 (满足要求)
npm                      ✅ v9.6.7 (满足要求)
Git                      ✅ v2.40.1 (满足要求)
磁盘空间                 ✅ 可用 150GB
网络连接                 ✅ 可访问 npm registry
═════════════════════════════════════════

状态: ✅ 环境就绪
```

**故障处理**:
```
❌ 如果版本不匹配或工具缺失:
   示例: "Node.js 版本过低（需要 16.0+，当前 14.2.0）"
   建议: "请更新 Node.js 后重试"
   选择:
     [A] 继续（忽略警告，可能出错）
     [B] 中止，修复环境后重试
```

---

### Step 3: Git 仓库检查

**目标**: 确保 Git 仓库状态正常

**AI 自动执行**:
```bash
检查项:
  1. 仓库状态
     命令: git status
     期望: 仓库存在，可访问
  
  2. 当前分支
     命令: git branch --show-current
     期望: 显示当前分支名
  
  3. 工作树状态
     命令: git status --porcelain
     期望: 检查是否有未提交的更改
  
  4. 远程配置
     命令: git remote -v
     期望: 远程仓库已配置
```

**检查结果示例**:
```
Git 仓库检查
═════════════════════════════════════════
仓库状态                 ✅ 有效
当前分支                 ⚠️  feature/user-auth (非 main)
工作树状态               ⚠️  有 3 个未提交的文件:
                         - src/user.ts (已修改)
                         - test/user.test.ts (新文件)
                         - docs/API.md (新文件)
远程仓库                 ✅ origin (https://...)
═════════════════════════════════════════

状态: ⚠️ 警告：在非 main 分支上工作
```

**人工决策点**:
```
⚠️ 如果在非 main 分支:
   提示: "您在 feature/user-auth 分支上工作"
   选择:
     [A] 继续（在当前分支上工作）
     [B] 切换到 main 分支后重试
     [C] 中止

⚠️ 如果有未提交的更改:
   提示: "工作树有 3 个未提交的文件"
   建议: "建议先提交或保存这些更改"
   选择:
     [A] 继续（可能覆盖这些更改）
     [B] 先提交或 stash 后重试
     [C] 中止
```

### Step 4: 输出目录初始化

**目标**: 准备任务输出目录

**AI 自动执行**:
```bash
步骤:
  1. 生成任务 ID
     格式: <project>-<task-type>-<timestamp>
     示例: user-req-20260211-143052
  
  2. 创建输出目录结构
     inputs/
       ├── <project>/
       │   └── <task-id>/
       │       ├── REQUIREMENT.md (若干)
       │       ├── TECH-PLAN.md (若干)
       │       ├── IMPLEMENTATION.md (若干)
       │       ├── test-scripts/
       │       ├── reports/
       │       └── analysis/
  
  3. 初始化 INDEX 文件
     创建 outputs/<project>/INDEX.md
     用于追踪所有生成的任务
```

**目标**: 验证项目依赖的完整性

## ✅ 验收清单

前置检查通过的标准：

```yaml
必需项 (所有必须通过):
  □ 项目规范文件完整
  □ Node.js/npm 版本符合要求
  □ Git 仓库可访问
  □ 输出目录已初始化

警告项 (可选继续):
  □ 确认是否在非 main 分支工作
  □ 确认是否覆盖已有输出文件
  □ 确认缺失规范是否可以跳过

故障项 (必须中止):
  □ 环境工具不可用
  □ Git 仓库损坏或无权限
  □ 磁盘空间不足
  □ 网络连接失败
```

## 🔄 重新执行

如果前置检查失败或需要重新执行：

```bash
# 完整重新执行前置检查
ai-exec workflow 00-pre-check --project user --reset

# 跳过某些检查（谨慎使用）
ai-exec workflow 00-pre-check --project user --skip-network

# 显示详细日志
ai-exec workflow 00-pre-check --project user --verbose
```

## 💡 最佳实践

### 1. 定期运行
```
建议在以下时间运行:
  - 每个开发会话开始前
  - 切换分支后
  - 从远程拉取代码后
  - 更新了项目规范后
```

### 2. 保持规范最新
```
建议:
  - 定期审查项目规范
  - 新技术采用时更新规范
  - 团队决策体现在规范中
  - 不要让规范成为负担
```

### 3. 处理失败
```
当前置检查失败时:
  ✅ 理解失败的原因
  ✅ 修复根本问题（不是规避问题）
  ✅ 重新运行验证
  ✅ 文档化常见问题
```

## 🚫 常见问题

### Q: 前置检查需要多长时间？
**A**: 正常情况下 2-5 分钟。如果网络慢可能需要 10 分钟。

### Q: 如果我在非 main 分支可以继续吗？
**A**: 可以，但要确认。系统会提醒您当前分支。

### Q: 如果缺少项目规范可以跳过吗？
**A**: 可以，但会失去项目特定的约束。建议先补充规范。

### Q: 环境检查失败怎么办？
**A**: 按照建议修复环境，然后重新运行前置检查。

### Q: 如何在 CI/CD 中使用？
**A**: 前置检查设计用于本地开发。CI/CD 应该有自己的环境验证流程。
