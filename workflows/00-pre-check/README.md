# 工作流 00 - 简化预检查 v2

> **版本**: v2.0.0  
> **核心改进**: 精简为 3 项必做检查，移除形式化检查

---

## 🎯 设计理念

**之前的问题**:
- 检查 Node.js 版本、磁盘空间等对文档生成没有实际价值
- 增加不必要的执行开销
- 用户已在使用项目，环境通常没问题

**改进后**:
- 聚焦有价值的检查
- 快速完成，减少摩擦
- 按需加载项目规范

---

## 📋 简化预检查（3 项必做）

### ✅ 检查清单

```text
🔍 预检查:
1. 工作区: [当前路径]
2. 任务类型: [需求/Bug/优化/...]
3. 输出位置: [projects/<project>/...]
```

### 检查项说明

| # | 检查项 | 获取方式 | 必要性 |
|---|-------|---------|-------|
| 1 | 工作区确认 | 自动识别当前目录 | ✅ 必须 |
| 2 | 任务类型识别 | 分析用户请求关键词 | ✅ 必须 |
| 3 | 输出位置确认 | 基于项目名构建路径 | ✅ 必须 |

---

## 📋 按需检查（涉及代码时）

当任务涉及代码修改时，额外检查:

```text
🔍 扩展检查:
4. 项目规范: ✅ 已加载 / ⚠️ 未找到（使用默认规范）
```

### 项目规范加载逻辑

```yaml
查找顺序:
  1. projects/<project>/PROJECT.md      # 新版统一规范
  2. projects/<project>/PROJECT-PROFILE.md  # 旧版兼容
  3. 使用默认规范继续
```

---

## 🚀 执行流程

### Step 1: 工作区确认

```yaml
执行: 识别当前工作目录
输出: "1. 工作区: E:\Worker\user-service"
```

### Step 2: 任务类型识别

```yaml
执行: 分析用户请求，匹配任务类型

关键词映射:
  需求开发: ["实现", "开发", "添加", "新增", "集成"]
  Bug修复: ["修复", "解决", "Bug", "报错", "异常"]
  性能优化: ["优化", "慢", "性能", "加速"]
  技术调研: ["调研", "选型", "对比"]
  架构重构: ["重构", "重写", "改造"]
  
输出: "2. 任务类型: 需求开发"
```

### Step 3: 输出位置确认

```yaml
执行: 基于项目名和任务类型构建路径

路径模式:
  需求: projects/<project>/requirements/<feature>/
  Bug: projects/<project>/bugs/<bug-id>/
  优化: projects/<project>/optimizations/<opt-id>/
  
输出: "3. 输出位置: projects/user/requirements/20260212-xxx/"
```

### Step 4: 项目规范加载（按需）

```yaml
条件: 任务涉及代码修改

执行:
  1. 尝试读取 projects/<project>/PROJECT.md
  2. 如失败，尝试 projects/<project>/PROJECT-PROFILE.md
  3. 如都失败，使用默认规范

输出:
  成功: "4. 项目规范: ✅ 已加载 (Node.js + TypeScript)"
  失败: "4. 项目规范: ⚠️ 未找到，使用默认规范"
```

## ❌ 移除的检查项

以下检查项已移除（理由：对 AI 文档生成没有实际价值）:

| 移除项 | 原因 |
|-------|------|
| Node.js 版本检查 | 用户环境已可用 |
| npm/yarn 版本检查 | 用户环境已可用 |
| Git 版本检查 | 用户环境已可用 |
| 磁盘空间检查 | 极少遇到空间不足 |
| 网络连接检查 | 与文档生成无关 |
