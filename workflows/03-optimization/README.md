# 性能优化流程

> **任务类型**: 性能优化  
> **使用场景**: 提升系统性能、解决性能瓶颈  
> **输出目录**: `projects/<project-name>/optimizations/<中文描述>/`  
> **版本**: v2.2.0

---

## 📋 流程概览

```
性能基线 → 瓶颈识别 → 优化方案 → 实施方案 → 执行优化 → 效果验证 → 对比报告
              CP1          CP2        CP3（含实施前检查）
```

---

## 🎯 执行步骤

### Step 1: 建立性能基线

**目标**: 记录优化前的性能指标

**AI 执行**:
1. 询问用户当前性能问题
2. 确定需要优化的指标（响应时间、吞吐量、资源使用等）
3. 收集当前性能数据
4. 建立性能基线

**输出**: `01-性能基线.md`

**内容要点**:
- 优化目标
- 当前性能指标
- 性能问题描述
- 测试环境说明

---

### Step 2: 瓶颈识别

**目标**: 找出性能瓶颈

**AI 执行**:
1. 分析代码
2. 检查数据库查询
3. 分析网络请求
4. 识别性能瓶颈点

**输出**: `02-瓶颈分析.md`

**内容要点**:
- 瓶颈位置
- 瓶颈原因
- 优化优先级
- 预期改进空间

**⏸️ CP1 确认点**: 等待用户确认瓶颈分析结果

---

### Step 3: 优化方案设计

**目标**: 设计优化方案（只描述方案，不写具体代码）

**AI 执行**:
1. 提出多个优化方案
2. 对比方案优劣
3. 选择最佳方案
4. 评估实施风险

**输出**: `03-优化方案.md`

**模板**: `templates/core/optimization-template.md`

**内容要点**:
- 优化方案设计
- 技术实现要点（描述逻辑，不写代码）
- 改动文件清单
- 风险评估
- 预期效果

**⏸️ CP2 确认点**: 等待用户确认方案可行

---

### Step 4: 实施方案

**目标**: 生成可执行的实施方案，供用户确认后才执行

**核心原则**:
- 优化方案说"怎么优化"（思路、策略）
- 实施方案说"具体改什么"（每个文件的完整变更内容）
- **用户确认所有变更内容后，才开始修改代码**

**AI 执行**:
1. 基于优化方案，为每个需要修改的文件生成变更内容
2. 生成实施前检查清单
3. 生成实施步骤（按依赖顺序）
4. 生成验证清单

**实施方案结构**:

```yaml
小优化（< 5 个文件）:
  直接在聊天中输出:
    1. 实施前检查清单
    2. 每个文件的完整变更内容
    3. 实施步骤
    4. 验证清单

大优化（>= 5 个文件）:
  生成 04-实施方案 目录:
    projects/<project>/optimizations/<中文描述>/
    ├── 04-实施方案/
    │   ├── README.md              # 实施总索引
    │   ├── <Module>.md            # 各文件变更
    │   └── ...
    └── scripts/
```

**⏸️ CP3 确认点**: 用户确认所有变更内容后才执行

---

### Step 5: 执行优化

**目标**: 对照实施方案，逐个文件执行优化

**前置条件**: CP3 已确认 + 实施前检查全部通过

**AI 执行**:
1. 先执行实施前检查清单，逐项验证
2. 按实施方案中的步骤顺序，逐文件修改
3. 每完成一个文件/批次，报告进度
4. 遇到问题暂停，向用户报告

---

### Step 6: 效果验证

**目标**: 验证优化效果

**AI 执行**:
1. 运行性能测试
2. 收集优化后的性能指标
3. 对比优化前后数据
4. 验证功能正确性

**输出**: `05-验证报告.md`

**内容要点**:
- 优化后性能指标
- 性能提升百分比
- 对比图表
- 功能验证结果

---

### Step 7: 生成对比报告

**目标**: 总结优化效果

**AI 执行**:
1. 汇总优化前后数据
2. 生成对比报告
3. 总结经验教训
4. 提出后续优化建议
5. **更新 `projects/<project>/TASK-INDEX.md`**
6. **生成任务记忆** `ai-dev-guidelines/projects/<project>/.ai-memory/`（详见 [common/task-memory.md](../common/task-memory.md)）

**输出**: 更新所有文档，确保完整性

## 📎 文档职责划分

```yaml
文档关系:
  01-性能基线.md: 优化前的性能基线数据
  02-瓶颈分析.md: 瓶颈识别和原因分析
  03-优化方案.md: 优化方案设计（只描述，不写代码）
  04-实施方案/: 实施方案（具体变更内容，确认后才执行）
  05-验证报告.md: 优化后效果验证和对比
```

## 📝 命名规范

```yaml
🔴 目录命名:
  格式: <中文描述>/
  示例: 创建项目AI标题异步化/、数据库查询优化/
  ❌ 不使用 ID 前缀（已在 optimizations/ 下，无需冗余标记）
  ❌ 不使用英文描述

🔴 文件命名（按优化规模区分）:
  小优化（< 5 个文件改动）:
    单文件: README.md（合并基线+分析+方案+实施+验证）
    原因: 小改动拆 5 个文件过重，增加维护负担
  
  大优化（>= 5 个文件改动）:
    多文件: NN-<中文名>.md
    示例: 01-性能基线.md、02-瓶颈分析.md、03-优化方案.md
    NN: 固定步骤序号（01~05），与工作流步骤对应
  
  ❌ 不使用英文名（如 01-baseline.md）
  原因: copilot-instructions.md 入口声明"所有输出使用中文"

🔴 状态同步:
  代码修改完成后，必须同步更新所有关联文件的状态:
    - 任务产物文件（README.md 或各 NN-*.md）
    - TASK-INDEX.md
    - 记忆文件
  ❌ 禁止只更新部分文件的状态
```

## 🚨 注意事项

### 性能测试
```yaml
必须包含:
  1. 测试环境说明
  2. 测试数据量
  3. 并发数
  4. 多次测试取平均值
```

### 数据库优化
```yaml
如果优化数据库:
  1. 添加索引必须先分析执行计划
  2. 记录索引大小和维护成本
  3. 验证是否影响写入性能
```

### 缓存策略
```yaml
如果添加缓存:
  1. 说明缓存失效策略
  2. 说明缓存更新机制
  3. 评估缓存命中率
  4. 考虑缓存穿透/雪崩问题
```

### 性能指标
```yaml
常见优化指标:
  - 响应时间（P50, P95, P99）
  - QPS/TPS
  - CPU 使用率
  - 内存使用量
  - 数据库连接数
```

## ✅ 完成检查清单

在向用户报告完成前，确认：

- [ ] 所有文档已生成（小优化: README.md / 大优化: 01-05）
- [ ] 性能基线已建立
- [ ] 瓶颈已明确识别
- [ ] 优化方案已用户确认
- [ ] 实施方案已用户确认
- [ ] 实施前检查已通过
- [ ] 代码变更已执行
- [ ] 性能测试已完成
- [ ] 有对比数据证明优化效果
- [ ] 脚本文件已生成（如需要）

---

## ✅ 性能优化专属验证

完成优化后，执行以下验证：

```yaml
性能优化验证清单:
  数据验证:
    ✅ 基线数据: 有优化前的性能数据
    ✅ 对比数据: 有优化后的性能数据
    ✅ 多次测试: 数据来自多次测试取平均
  
  效果验证:
    ✅ 达到目标: 性能提升达到预期
    ✅ 稳定性: 多次测试结果一致
    ✅ 无退化: 其他指标未明显下降
  
  功能验证:
    ✅ 功能正常: 优化未破坏原有功能
    ✅ 边界情况: 极端情况下仍能正常工作
```
