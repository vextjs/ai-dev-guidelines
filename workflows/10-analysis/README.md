# 深度分析工作流

> 定义 AI 执行"深度分析/审查/评估"类任务的标准化步骤

**版本**: v1.0
**最后更新**: 2026-02-27
**适用**: 用户说"分析"、"审查"、"评估"、"深度分析"、"全面检查"等

---

## 🎯 适用场景

| 关键词 | 示例 |
|--------|------|
| 分析、审查、评估 | "分析这个项目的架构" |
| 深度分析、全面检查 | "全面审查当前规范" |
| 对比、评审 | "对比 A 和 B 方案" |
| 健康检查、质量评估 | "检查代码质量" |

## 📋 执行流程（5 阶段）

### 阶段 1：范围确认

```yaml
输入: 用户的分析请求
输出: 明确的分析范围 + 评估维度

步骤:
  1. 理解用户意图 — 要分析什么？分析到什么程度？
  2. 确定分析对象 — 具体文件/目录/模块/项目
  3. 确定评估维度 — 一致性/完整性/性能/安全/架构等
  4. 确定输出期望 — 报告格式/是否需要修复建议

🔴 如果范围模糊，先确认再执行（CP1）
```

### 阶段 2：数据采集

```yaml
输入: 明确的分析范围
输出: 所有相关文件的内容已加载到上下文

步骤:
  1. 列出需要检查的文件清单
  2. 逐个 read_file 读取（不要推断内容）
  3. 对数值类检查（版本号/条数/数量），使用 grep_search 或终端命令批量验证
  4. 记录每个文件的关键数据点

原则:
  - 🔴 逐文件读取，禁止推断
  - 🔴 数值类必须实际比对，不可假设
  - 优先使用终端命令批量扫描，再用 read_file 确认细节
  - 记录"已检查"和"未检查"的文件，确保无遗漏
```

### 阶段 3：交叉验证

```yaml
输入: 所有文件的数据点
输出: 一致性/完整性检查结果

步骤:
  1. 同一数据在多文件中的一致性 — 如版本号、条数、数量
  2. 定义与示例的一致性 — 如命名格式定义 vs 示例中的实际命名
  3. 引用链接有效性 — 引用的文件是否真的存在
  4. 历史标记合理性 — 旧版本引用是否属于历史记录（变更日志、错误示例）
  5. 标注与实际的一致性 — 如"工具脚本 N 个" vs 实际目录文件数

检查类型:
  🔢 数值一致性: 同一数值在所有引用处是否相同
  📝 格式一致性: 定义的格式与示例/模板中的格式是否匹配
  🔗 引用有效性: 所有内部链接是否指向存在的文件
  📊 标注准确性: 数量/计数类标注是否与实际文件系统一致
```

### 阶段 4：输出报告

```yaml
输入: 交叉验证结果
输出: 结构化分析报告

报告结构:
  1. 评审摘要（表格）
  2. 通过的检查项（按维度分组）
  3. 发现的问题（按严重度排序: 🔴 P0 → 🟡 P1 → 💡 建议）
  4. 综合评分（10 分制，各维度分别评分）
  5. 修复建议（如有问题）

模板: templates/lite/analysis-lite.md

🔴 报告自动写入 reports/<子目录>/<agent>/YYYYMMDD/ 目录
🔴 禁止询问用户是否写报告
```

### 阶段 5：更新记忆

```yaml
输入: 报告内容
输出: 记忆文件已更新

步骤:
  1. 追加会话记录到 .ai-memory/clients/<agent>/tasks/YYYYMMDD.md
  2. 关键发现 + 报告链接 + 对话记录
  3. 更新待跟进汇总

🔴 记忆自动写入，禁止询问用户
```

## 🔄 多轮审查模式

当用户要求"N 轮审查"时：

```yaml
每轮审查聚焦不同维度:
  第 1 轮: 数值一致性（版本号/条数/数量 — 全量对比）
  第 2 轮: 内容一致性（格式/示例/引用 — 深度检查）
  第 3 轮: 架构完整性（交叉引用/历史标记/链接有效性）

每轮输出:
  - 通过项 + 发现项
  - 如有发现，立即标记并累积到最终报告

最终报告:
  - 汇总所有轮次结果
  - 综合评分
  - 修复建议（按优先级排序）
```

---

## ⚡ 快速 vs 完整

| 模式 | 适用 | 流程 |
|------|------|------|
| **快速模式** | 小范围/单文件/特定问题 | 阶段 2→4（跳过范围确认和多轮） |
| **完整模式** | 全项目/多文件/全面审查 | 完整 5 阶段 + 多轮审查 |

用户说"快速分析" → 快速模式
用户说"全面审查"/"深度分析" → 完整模式 + 多轮
