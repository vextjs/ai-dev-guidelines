# 数据库变更流程

> **任务类型**: 数据库变更  
> **使用场景**: Schema 变更、数据迁移、索引优化  
> **输出目录**: `projects/<project-name>/database/<db-change-id>/`  
> **版本**: v2.1.0

---

## 📋 流程概览

```
变更设计 → 脚本编写 → 测试验证 → 生产执行 → 执行后验证
   CP1        CP2（脚本确认）  CP3（执行前确认）
```

---

## 🎯 执行步骤

### Step 1: 变更设计
- 分析变更需求
- 设计 Schema 变更
- 评估影响范围
- 制定回滚方案

**输出**: `01-design.md`

**内容要点**:
- 变更目标
- Schema 变更设计
- 影响范围（涉及的表、索引、视图）
- 数据迁移计划（如涉及）
- 回滚方案

**⏸️ CP1 确认点**: 等待用户确认变更设计

---

### Step 2: 脚本编写
- 编写迁移脚本
- 编写回滚脚本
- 编写验证脚本

**输出**: `scripts/*.sql`

**脚本要求**:
```markdown
必须生成的脚本:
- migration.sql   - 迁移脚本（正向变更）
- rollback.sql    - 回滚脚本（逆向变更）
- verify.sql      - 验证脚本（检查变更结果）
```

**⏸️ CP2 确认点**: 等待用户确认所有脚本内容

---

### Step 3: 测试验证
- 在测试环境执行迁移脚本
- 验证数据正确性
- 测试回滚流程
- 验证应用兼容性

**输出**: `02-validation.md`

**内容要点**:
- 测试环境信息
- 迁移执行结果
- 回滚测试结果
- 数据完整性验证
- 应用兼容性验证

---

### Step 4: 生产执行

**前置条件**: CP2 已确认 + 测试验证通过

**⏸️ CP3 确认点**: 执行前最终确认
```markdown
🔍 生产执行前检查:
- [ ] 测试环境验证通过
- [ ] 回滚脚本已测试
- [ ] 已选择低峰期执行
- [ ] 监控告警已就绪
- [ ] DBA/运维已知晓

确认执行？[Y/N]
```

**AI 执行**:
- 制定执行计划（执行时间、步骤）
- 灰度执行（如可能）
- 监控数据库性能

**输出**: `03-execution.md`

---

### Step 5: 执行后验证
- 验证数据完整性
- 检查应用功能
- 确认性能指标
- **更新 `projects/<project>/TASK-INDEX.md`**

**输出**: 更新 `03-execution.md`

---

projects/user-service/database/DB-add-user-index-20260211/

---

## 🚨 高危操作警告

### 必须有回滚脚本
```sql
-- 任何数据库变更都必须有对应的回滚脚本
-- 示例: 添加字段的回滚
ALTER TABLE users DROP COLUMN new_field;
```

### 必须在测试环境验证
- 执行迁移脚本
- 执行回滚脚本
- 验证数据完整性

### 大表操作注意事项
- 避免锁表时间过长
- 考虑分批执行
- 准备降级方案

## ✅ 数据库变更专属验证

```yaml
数据库变更验证清单:
  脚本验证:
    ✅ 迁移脚本: 正确执行，无报错
    ✅ 回滚脚本: 可完整回滚变更
    ✅ 验证脚本: 能检测变更是否成功
  
  数据验证:
    ✅ 数据完整: 无数据丢失或损坏
    ✅ 数据一致: 新旧数据关系正确
    ✅ 性能稳定: 查询性能无明显下降
  
  应用验证:
    ✅ 功能正常: 应用功能未受影响
    ✅ 兼容性: 应用代码兼容新 Schema
```
