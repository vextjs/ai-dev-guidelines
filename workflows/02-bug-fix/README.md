# Bug 修复流程

> **任务类型**: Bug 修复  
> **使用场景**: 修复生产环境或测试环境中发现的问题  
> **输出目录**: `projects/<project-name>/bugs/<bug-id>/`  
> **版本**: v2.1.0

---

## 📋 流程概览

```
问题复现 → 原因分析 → 解决方案 → 实施方案 → 执行修复 → 验证测试 → 文档归档
              CP1         CP2        CP3（含实施前检查）
```

---

## 🎯 执行步骤

### Step 1: 问题复现

**目标**: 确认问题真实存在，获取复现步骤

**AI 执行**:
1. 询问用户获取问题详情:
   - 问题描述
   - 发生时间
   - 影响范围
   - 错误日志
2. 尝试复现问题
3. 记录复现步骤

**输出**: 无（信息收集阶段）

---

### Step 2: 原因分析

**目标**: 定位问题根本原因

**AI 执行**:
1. 分析错误日志
2. 检查相关代码
3. 确定影响范围
4. 分析根本原因

**输出**: `01-analysis.md`

**模板**: `templates/core/bug-analysis-template.md`

**内容要点**:
- 问题描述
- 复现步骤
- 影响范围和严重程度
- 根因分析
- 相关代码定位

**⏸️ CP1 确认点**: 等待用户确认分析结果，确保根因正确

---

### Step 3: 解决方案设计

**目标**: 设计修复方案（只描述方案，不写具体代码）

**AI 执行**:
1. 设计修复方案
2. 列出需要修改的文件清单
3. 评估风险
4. 制定回滚方案

**输出**: `02-solution.md`

**内容要点**:
- 修复方案设计
- 改动文件清单（哪些文件、改什么）
- 技术实现要点（描述逻辑，不写代码）
- 风险评估
- 回滚方案

**⏸️ CP2 确认点**: 等待用户确认方案可行

---

### Step 4: 实施方案

**目标**: 生成可执行的实施方案，供用户确认后才执行

**核心原则**:
- 解决方案说"怎么修"（思路、方向）
- 实施方案说"具体改什么"（每个文件的完整变更内容）
- **用户确认所有变更内容后，才开始修改代码**

**AI 执行**:
1. 基于解决方案，为每个需要修改的文件生成变更内容
2. 生成实施前检查清单
3. 生成实施步骤（按依赖顺序）
4. 生成验证清单

**实施方案结构**:

```yaml
小 Bug（< 5 个文件）:
  直接在聊天中输出:
    1. 实施前检查清单
    2. 每个文件的完整变更内容
    3. 实施步骤
    4. 验证清单

大 Bug（>= 5 个文件）:
  生成 03-implementation 目录:
    projects/<project>/bugs/<bug-id>/
    ├── 03-implementation/
    │   ├── README.md              # 实施总索引
    │   ├── <Module>.md            # 各文件变更（完整代码）
    │   └── ...
    └── scripts/
```

**实施前检查格式**:
```markdown
## 实施前检查
- [ ] 确认问题已复现
- [ ] 确认修改范围不影响其他功能
- [ ] 确认回滚方案可行
- [ ] 确认测试环境可用
```

**⏸️ CP3 确认点**: 用户确认所有变更内容后才执行

---

### Step 5: 执行修复

**目标**: 对照实施方案，逐个文件执行修复

**前置条件**: CP3 已确认 + 实施前检查全部通过

**AI 执行**:
1. 先执行实施前检查清单，逐项验证
2. 按实施方案中的步骤顺序，逐文件修改
3. 每完成一个文件报告进度
4. 遇到问题暂停，向用户报告

**进度报告格式**:
```markdown
🔄 修复进度: [2/4]
✅ 已完成: service/UserService.ts, model/User.ts
🔄 正在处理: controller/UserController.ts
⏳ 待处理: test/user.test.ts
```

---

### Step 6: 验证测试

**目标**: 确认问题已解决

**AI 执行**:
1. 执行单元测试
2. 手动验证修复效果（复现步骤不再触发问题）
3. 回归测试（相关功能不受影响）
4. 性能影响评估

**输出格式**:
```markdown
🔍 验证结果:
- 问题复现: ❌ 已不可复现 ✅
- 单元测试: ✅ 全部通过
- 回归测试: ✅ 无影响
- 性能影响: ✅ 无下降
```

---

### Step 7: 文档归档

**目标**: 完成文档并归档

**AI 执行**:
1. 检查所有文档完整性
2. 生成脚本文件（如需要）
3. **更新任务索引** `projects/<project>/TASK-INDEX.md`
4. **生成任务记忆** `ai-dev-guidelines/projects/<project>/.ai-memory/`（详见 [common/task-memory.md](../common/task-memory.md)）
5. 向用户报告完成

**索引更新**:
```markdown
| BUG-xxx | YYYY-MM-DD | Bug 标题 | P1 | ✅ 修复 | `bugs/BUG-xxx/` |
```

**输出**: 完整的 Bug 修复文档包

---

## 📎 文档职责划分

```yaml
文档关系:

  01-analysis.md（分析层 - 什么问题）:
    职责: 问题描述、复现步骤、影响范围、根因分析
    不包含: 修复方案、代码

  02-solution.md（方案层 - 怎么修）:
    职责: 修复方案设计、改动文件清单、风险评估、回滚方案
    不包含: 具体代码实现
    上游: 引用 01-analysis.md 的根因

  03-implementation/（实施层 - 具体改什么）:
    职责: 实施前检查、每个变更文件的完整代码、实施步骤、验证清单
    上游: 引用 02-solution.md 的方案设计
    结构:
      小 Bug: 单文件 03-implementation.md
      大 Bug: 目录 03-implementation/README.md + 各模块.md
    时机: 在执行修复之前生成，用户确认后才执行
```

## 📦 输出示例

### 小 Bug（< 5 个文件）
```
projects/chat-service/bugs/BUG-chat-001-message-loss/
├── 01-analysis.md             # 问题分析
├── 02-solution.md             # 解决方案
├── 03-implementation.md       # 实施方案（单文件）
└── scripts/
    ├── fix-data.sql           # 数据修复脚本（如需要）
    └── rollback.sql           # 回滚脚本（如需要）
```

### 大 Bug（>= 5 个文件）
```
projects/payment-service/bugs/BUG-pay-003-order-duplicate/
├── 01-analysis.md             # 问题分析
├── 02-solution.md             # 解决方案
├── 03-implementation/         # 实施方案目录
│   ├── README.md              # 实施总索引（前检查+步骤+验证）
│   ├── OrderService.md        # 订单服务变更
│   ├── PaymentService.md      # 支付服务变更
│   └── OrderModel.md          # 订单模型变更
└── scripts/
    ├── fix-duplicate-orders.sql
    └── rollback.sql
```

## 🚨 注意事项

### 严重 Bug 处理
```yaml
如果 Bug 严重程度为 P0/P1:
  1. 优先修复，暂停其他任务
  2. 详细记录修复过程
  3. 制定详细回滚方案
  4. 修复后进行充分测试
```

### 数据修复
```yaml
如果需要修复数据:
  1. 必须生成 SQL 脚本
  2. 必须先在测试环境验证
  3. 必须制定回滚脚本
  4. 记录影响的数据量
```

### 配置变更
```yaml
如果涉及配置变更:
  1. 记录变更前后的配置
  2. 说明变更原因
  3. 标注需要重启的服务
```

## ✅ 完成检查清单

在向用户报告完成前，确认：

- [ ] 所有文档已生成（01-03）
- [ ] 问题根因已明确定位
- [ ] 修复方案已详细记录
- [ ] 实施方案已用户确认
- [ ] 实施前检查已通过
- [ ] 代码变更已记录
- [ ] 测试已通过
- [ ] 脚本文件已生成（如需要）
- [ ] 回滚方案已制定

---

## ✅ Bug 修复专属验证

完成修复后，执行以下验证：

```yaml
Bug 修复验证清单:
  问题验证:
    ✅ 复现确认: 修复前能复现问题
    ✅ 修复有效: 修复后问题不再出现
    ✅ 根因明确: 问题原因已清晰记录
  
  副作用检查:
    ✅ 无新问题: 修复未引入新 Bug
    ✅ 功能完整: 相关功能未受影响
    ✅ 性能稳定: 无明显性能下降
  
  回滚准备:
    ✅ 回滚方案: 有明确的回滚步骤
    ✅ 数据安全: 数据修复有回滚脚本
```
