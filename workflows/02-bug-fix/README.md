# Bug 修复流程

> **任务类型**: Bug 修复  
> **使用场景**: 修复生产环境或测试环境中发现的问题  
> **输出目录**: `projects/<project-name>/bugs/<bug-id>/`  

---

## 📋 流程概览

```
问题复现 → 原因分析 → 解决方案 → 代码修复 → 验证测试 → 文档归档
```

---

## 🎯 执行步骤

### Step 1: 问题复现

**目标**: 确认问题真实存在，获取复现步骤

**AI 执行**:
1. 询问用户获取问题详情:
   - 问题描述
   - 发生时间
   - 影响范围
   - 错误日志

2. 尝试复现问题
3. 记录复现步骤

**输出**: 无（信息收集阶段）

---

### Step 2: 原因分析

**目标**: 定位问题根本原因

**AI 执行**:
1. 分析错误日志
2. 检查相关代码
3. 确定影响范围
4. 分析根本原因

**输出**: `01-analysis.md`

**模板**: `templates/core/bug-analysis-template.md`

**内容要点**:
- 问题描述
- 复现步骤
- 影响范围和严重程度
- 根因分析
- 相关代码定位

---

### Step 3: 解决方案设计

**目标**: 设计修复方案

**AI 执行**:
1. 设计修复方案
2. 评估风险
3. 制定回滚方案
4. 确认技术实现要点

**输出**: `02-solution.md`

**内容要点**:
- 修复方案设计
- 技术实现要点
- 风险评估
- 回滚方案

---

### Step 4: 代码修复

**目标**: 实施修复

**AI 执行**:
1. 修改代码
2. 编写单元测试
3. 记录所有变更

**输出**: `03-implementation.md`

**模板**: `templates/core/implementation-template.md`

**内容要点**:
- 代码修改清单
- 配置变更记录
- 测试用例
- 部署说明

---

### Step 5: 验证测试

**目标**: 确认问题已解决

**AI 执行**:
1. 执行单元测试
2. 手动验证修复效果
3. 回归测试
4. 性能影响评估

**输出**: 更新 `03-implementation.md` 的测试结果部分

---

### Step 6: 文档归档

**目标**: 完成文档并归档

**AI 执行**:
1. 检查所有文档完整性
2. 生成脚本文件（如需要）
3. 向用户报告完成

**输出**: 完整的 Bug 修复文档包

---

## 📦 输出示例

```
projects/chat-service/bugs/BUG-chat-001-message-loss/
├── 01-analysis.md             # 问题分析
├── 02-solution.md             # 解决方案
├── 03-implementation.md       # 实施记录
└── scripts/
    ├── fix-data.sql           # 数据修复脚本（如需要）
    └── rollback.sql           # 回滚脚本（如需要）
```

---

## 🚨 注意事项

### 严重 Bug 处理
```yaml
如果 Bug 严重程度为 P0/P1:
  1. 优先修复，暂停其他任务
  2. 详细记录修复过程
  3. 制定详细回滚方案
  4. 修复后进行充分测试
```

### 数据修复
```yaml
如果需要修复数据:
  1. 必须生成 SQL 脚本
  2. 必须先在测试环境验证
  3. 必须制定回滚脚本
  4. 记录影响的数据量
```

### 配置变更
```yaml
如果涉及配置变更:
  1. 记录变更前后的配置
  2. 说明变更原因
  3. 标注需要重启的服务
```

## ✅ 完成检查清单

在向用户报告完成前，确认：

- [ ] 所有文档已生成（01-03）
- [ ] 问题根因已明确定位
- [ ] 修复方案已详细记录
- [ ] 代码变更已记录
- [ ] 测试已通过
- [ ] 脚本文件已生成（如需要）
- [ ] 回滚方案已制定
