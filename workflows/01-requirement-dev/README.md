# 工作流 1: 需求开发（AI 执行手册）v2

> **任务类型**: 需求开发  
> **适用场景**: 新功能开发、功能增强、第三方集成  
> **版本**: v2.0.0 (支持快速/完整模式)

---

## 🎯 执行模式

### 快速模式 (默认)

**适用条件**:
- 简单需求、小改动
- 需求描述 < 100 字
- 涉及文件 < 5 个
- 用户说"快速"、"简单"

**流程**: 4 个阶段  
**输出**: 精简文档 + 代码实现  
**预计时间**: 15-30 分钟

### 完整模式

**适用条件**:
- 复杂需求、核心功能
- 涉及多模块、多文件
- 用户说"完整"、"详细"、"需要文档"

**流程**: 6 个阶段  
**输出**: 完整文档套件 + 代码实现  
**预计时间**: 45-90 分钟

---

## 📋 快速模式流程（默认）

```
阶段 1: 理解需求        →  确认点
阶段 2: 生成方案        →  CP2 确认
阶段 3: 执行实现        →  CP3 确认
阶段 4: 完成报告        →  总结
```

### 阶段 1: 理解需求

**目标**: 确保正确理解用户需求

**执行步骤**:
1. 阅读用户需求描述
2. 分析涉及的模块和功能
3. 用一句话复述核心需求
4. 列出关键改动点

**输出格式**:
```markdown
📋 需求理解:
- 核心需求: [一句话描述]
- 涉及模块: [模块1, 模块2]
- 关键改动: 
  1. [改动点1]
  2. [改动点2]

请确认以上理解是否正确？[确认/修改]
```

**⏸️ CP1 确认点**: 等待用户确认后继续

---

### 阶段 2: 生成方案

**目标**: 快速生成技术方案

**执行步骤**:
1. 分析相关代码文件
2. 确定修改范围
3. 生成精简技术方案

**参考模板**: [templates/lite/technical-lite.md](../../templates/lite/technical-lite.md)

**输出格式**:
```markdown
📝 技术方案:

## 改动清单
| 文件 | 修改类型 | 说明 |
|-----|---------|------|
| xxx.ts | 修改 | xxx |

## 实现要点
1. [要点1]
2. [要点2]

## 影响评估
- 风险: 低/中/高
- 回滚方案: [简述]

请确认方案是否可行？[确认/修改]
```

**⏸️ CP2 确认点**: 等待用户确认后继续

**等待用户确认后继续**

---

### 阶段 3: 执行实现

**目标**: 按方案逐步实现代码

**执行步骤**:
1. 按修改清单顺序执行
2. 每完成一个文件报告进度
3. 遇到问题及时沟通

**进度报告格式**:
```markdown
🔄 实现进度: [2/5]
✅ 已完成: file1.ts, file2.ts
🔄 正在处理: file3.ts
⏳ 待处理: file4.ts, file5.ts
```

**注意事项**:
- 大文件: 参见 [best-practices/token-optimization.md](../../best-practices/token-optimization.md)
- 复杂项目: 参见 [best-practices/large-projects.md](../../best-practices/large-projects.md)

---

### 阶段 4: 完成报告

**目标**: 总结实施结果

**输出格式**:
```markdown
✅ 需求实现完成

## 修改汇总
| 文件 | 操作 |
|-----|------|
| xxx.ts | 新增/修改/删除 |

## 测试建议
- [ ] [测试点1]
- [ ] [测试点2]

## 注意事项
- [如有特别说明]
```

**⭐ 索引更新（必须）**:
完成后更新 `projects/<project>/TASK-INDEX.md`，添加本次任务记录：
```markdown
| REQ-xxx | YYYY-MM-DD | 任务标题 | ✅ 完成 | `requirements/YYYYMMDD-xxx/` |
```

## 📋 完整模式流程

```
阶段 1-4: 同快速模式
阶段 5: 生成完整文档
阶段 6: 验证检查
```

### 阶段 5: 生成完整文档

**目标**: 生成完整的文档套件

**输出文档**:
1. `01-requirement.md` - 需求文档
2. `02-technical.md` - 技术方案
3. `03-implementation.md` - 实施记录

**参考模板**: `templates/core/` 目录

**执行步骤**:
1. 基于阶段 1-4 的内容整理
2. 使用完整版模板生成文档
3. 保存到 `projects/<project>/requirements/<task-id>/`

---

### 阶段 6: 验证检查

**目标**: 确保实现质量

**检查清单**:
- [ ] 代码符合项目规范
- [ ] 关键逻辑有注释
- [ ] 无明显的错误处理遗漏
- [ ] 文档内容完整

**输出格式**:
```markdown
🔍 验证结果:
- 代码规范: ✅ 通过
- 注释完整: ✅ 通过
- 错误处理: ✅ 通过
- 文档完整: ✅ 通过

总体评估: ✅ 可以交付
```

## 🔄 模式切换

### 自动判断

AI 根据以下条件自动选择模式:

| 条件 | 快速模式 | 完整模式 |
|-----|---------|---------|
| 需求描述 | < 100 字 | >= 100 字 |
| 涉及文件 | < 5 个 | >= 5 个 |
| 复杂度 | 简单改动 | 多模块协作 |
| 用户关键词 | "快速"、"简单" | "完整"、"详细" |

### 手动切换

用户可以随时说:
- "切换到完整模式" - 使用完整流程
- "只要代码，不要文档" - 跳过文档生成
- "需要完整文档" - 补充生成文档

## 📝 示例

### 快速模式示例

```
用户: "在 user 项目添加一个获取用户积分的接口"

AI 执行:
1. 理解需求 → "新增 GET /api/user/credits 接口" → 用户确认
2. 生成方案 → 修改 UserController + UserService → 用户确认
3. 执行实现 → 逐文件修改，报告进度
4. 完成报告 → 汇总修改，测试建议
```

### 完整模式示例

```
用户: "需要详细文档，在 payment 项目实现完整的退款功能"

AI 执行:
1-4. 同快速模式
5. 生成完整文档 → requirement.md + technical.md + implementation.md
6. 验证检查 → 代码和文档质量检查
```

## ✅ 需求开发专属验证

完成代码后，执行以下验证：

```yaml
需求开发验证清单:
  功能验证:
    ✅ 需求点覆盖: 每个需求点有对应实现
    ✅ 接口契约: 入参/出参/错误码定义完整
    ✅ 边界处理: null/空值/异常输入已处理
  
  代码质量:
    ✅ 错误处理: async 函数有 try-catch
    ✅ 日志记录: 关键操作有日志
    ✅ 配置外置: 无硬编码配置值
  
  文档同步:
    ✅ README: 新功能说明已添加
    ✅ 配置说明: 新配置项有文档
    ✅ CHANGELOG: 变更记录已添加
```
