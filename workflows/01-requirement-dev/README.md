# 工作流 1: 需求开发（AI 执行手册）v2

> **任务类型**: 需求开发  
> **适用场景**: 新功能开发、功能增强、第三方集成  
> **版本**: v2.1.0 (支持快速/完整模式)

---

## 🚨 强制执行规则（违反=规范失效）

```
❌ 绝对禁止的行为：
  - 收到需求后直接分析代码、直接执行代码修改
  - 认为"需求很清晰"就跳过 CP1 需求理解确认
  - 输出技术方案后不等用户确认就执行代码（跳过 CP2）
  - 在同一条回复中同时输出"方案"和"执行结果"

✅ 正确的执行顺序：
  收到需求 → 输出需求理解 → 等待 CP1 确认
           → 输出技术方案 → 等待 CP2 确认
           → 输出实施方案 → 等待 CP3 确认
           → 执行代码
```

---

## 🎯 执行模式

### 快速模式 (默认)

**适用条件**:
- 简单需求、小改动
- 需求描述 < 100 字
- 涉及文件 < 5 个
- 用户说"快速"、"简单"

**流程**: 5 个阶段  
**输出**: 代码实现 + 实施方案（大需求时生成文件）  
**预计时间**: 15-30 分钟

### 完整模式

**适用条件**:
- 复杂需求、核心功能
- 涉及多模块、多文件
- 用户说"完整"、"详细"、"需要文档"

**流程**: 7 个阶段  
**输出**: 完整文档套件 + 代码实现  
**预计时间**: 45-90 分钟

---

## 📋 快速模式流程（默认）

```
阶段 1: 理解需求        →  CP1 确认
阶段 2: 技术方案        →  CP2 确认
阶段 3: 实施方案        →  CP3 确认（含实施前检查）
阶段 4: 执行实现        →  对照实施方案逐文件执行
阶段 5: 完成报告        →  总结
```

### 阶段 1: 理解需求

**目标**: 确保正确理解用户需求

**执行步骤**:
1. 阅读用户需求描述
2. 分析涉及的模块和功能
3. 用一句话复述核心需求
4. 列出关键改动点

**输出格式**:
```markdown
📋 需求理解:
- 核心需求: [一句话描述]
- 涉及模块: [模块1, 模块2]
- 关键改动: 
  1. [改动点1]
  2. [改动点2]

请确认以上理解是否正确？[确认/修改]
```

**⏸️ CP1 确认点（绝对强制）**:
> 🔴 **无论需求描述多清晰，都必须输出需求理解并等待用户明确回复"确认"后才能继续。**
> 🔴 **禁止"需求很清晰所以跳过确认"，禁止直接进入下一阶段，禁止边分析边执行。**

---

### 阶段 2: 技术方案

**目标**: 设计怎么做（只描述方案，不写具体代码）

**执行步骤**:
1. 分析相关代码文件
2. 确定修改范围和架构设计
3. 生成技术方案（模块设计、接口契约、数据模型、流程）

**参考模板**: [templates/lite/technical-lite.md](../../templates/lite/technical-lite.md)

**输出格式**:
```markdown
📝 技术方案:

## 架构设计
[简述架构/模块划分]

## 改动清单
| 文件 | 修改类型 | 说明 |
|-----|---------|------|
| xxx.ts | 新增 | xxx |
| yyy.ts | 修改 | xxx |

## 实现要点
1. [要点1]
2. [要点2]

## 影响评估
- 风险: 低/中/高
- 回滚方案: [简述]

请确认方案是否可行？[确认/修改]
```

**⏸️ CP2 确认点（绝对强制）**:
> 🔴 **技术方案必须输出完整，且等待用户明确回复"确认"后才能进入实施阶段。**
> 🔴 **禁止在输出技术方案的同时或之后立即执行代码，禁止"顺手"就把代码改了。**

---

### 阶段 3: 实施方案

**目标**: 生成可执行的实施方案，供用户确认后才执行

**核心原则**:
- 技术方案说"怎么做"（架构、设计思路）
- 实施方案说"具体改什么"（每个文件的完整变更内容）
- **用户确认所有变更内容后，才开始执行代码**

**执行步骤**:
1. 基于技术方案，为每个需要新增/修改的文件生成变更文档
2. 生成实施前检查清单（前置条件验证）
3. 生成实施步骤（按依赖顺序排列）
4. 生成验证清单

**实施方案结构**:

```yaml
小需求（< 5 个文件）:
  直接在聊天中输出:
    1. 实施前检查清单
    2. 每个文件的完整变更内容
    3. 实施步骤（按依赖顺序）
    4. 验证清单

大需求（>= 5 个文件）:
  生成 03-implementation 目录:
    projects/<project>/requirements/<task-id>/
    ├── 03-implementation/
    │   ├── README.md              # 实施总索引
    │   ├── <Model>.md             # 数据模型变更（完整代码）
    │   ├── <Service>.md           # 服务层变更（完整代码）
    │   ├── <Controller>.md        # 控制层变更（完整代码）
    │   ├── <config>.md            # 配置文件变更
    │   └── <test>.md              # 测试文件
    └── scripts/
```

**03-implementation/README.md 必须包含**:

```markdown
## 实施前检查
- [ ] 确认 xxx 服务/模块已存在
- [ ] 确认 xxx 依赖已安装
- [ ] 确认数据库连接正常

## 实施文件清单
| 文件 | 路径 | 操作 | 文档 |
|------|------|------|------|
| 数据模型 | app/model/xxx.ts | 新增 | [查看](./Xxx.model.md) |
| 服务层 | app/service/XxxService.ts | 新增 | [查看](./XxxService.md) |
| 控制器 | app/controller/xxx.ts | 修改 | [查看](./XxxController.md) |

## 实施步骤（按依赖顺序）
步骤1: 配置 → 步骤2: 模型 → 步骤3: Service → 步骤4: Controller → 步骤5: 测试

## 验证清单
- [ ] 功能验证项...
- [ ] 性能验证项...
```

**每个变更文件文档格式**:

```markdown
# [模块名称]

**文件路径**: `app/service/XxxService.ts`
**操作**: 新增 / 修改

[完整的代码内容]
```

**⏸️ CP3 确认点**: 用户确认所有变更内容后才执行

---

### 阶段 4: 执行实现

**目标**: 对照实施方案，逐个文件执行变更

**前置条件**: CP3 已确认 + 实施前检查全部通过

**执行步骤**:
1. 先执行实施前检查清单，逐项验证
2. 按实施方案中的步骤顺序，逐文件执行
3. 每完成一个文件/批次，报告进度
4. 遇到问题暂停，向用户报告

**实施前检查格式**:
```markdown
🔍 实施前检查:
- [x] xxx 服务已存在 ✅
- [x] xxx 依赖版本正确 ✅
- [ ] xxx 接口未就绪 ❌ → 需要先处理

检查结果: ✅ 全部通过 / ❌ 有阻塞项
```

**进度报告格式**:
```markdown
🔄 实施进度: [2/5]
✅ 已完成: model/xxx.ts, service/XxxService.ts
🔄 正在处理: controller/xxx.ts
⏳ 待处理: config/xxx.ts, test/xxx.test.ts
```

**多文件改动分批策略（涉及 5+ 个文件时）**:

```yaml
分批原则:
  依赖排序（底层优先）:
    批次1: 基础层 - config、types/interfaces、model
    批次2: 逻辑层 - service、middleware、utils
    批次3: 接口层 - controller、route
    批次4: 验证层 - test、scripts

  每批完成后:
    1. 报告本批完成进度
    2. 运行相关验证（编译检查/测试）
    3. 确认无报错后进入下一批

  Token 不够时:
    - 保存进度: 列出已完成/未完成文件清单
    - 下次会话: 用户说"继续"时续传
```

---

### 阶段 5: 完成报告

**目标**: 总结实施结果

**输出格式**:
```markdown
✅ 需求实现完成

## 修改汇总
| 文件 | 操作 | 说明 |
|-----|------|------|
| xxx.ts | 新增 | xxx |

## 验证结果
- [x] 功能验证通过
- [x] 编译/类型检查通过

## 注意事项
- [如有特别说明]
```

**⭐ 索引更新（必须）**:
完成后更新 `projects/<project>/TASK-INDEX.md`，添加本次任务记录：
```markdown
| REQ-xxx | YYYY-MM-DD | 任务标题 | ✅ 完成 | `requirements/YYYYMMDD-xxx/` |
```

**🧠 任务记忆（必须）**:
完成后生成任务记忆，保存到 `dev-docs/projects/<project>/.ai-memory/`：
1. 生成 `tasks/<date>-REQ-<id>.md`（做了什么、关键决策、变更文件）
2. 更新 `SUMMARY.md`（追加最近任务 + 关键决策）

> 详见 [workflows/common/task-memory.md](../common/task-memory.md)

## 📋 完整模式流程

```
阶段 1-5: 同快速模式（含实施方案+实施前检查）
阶段 6: 生成归档文档
阶段 7: 验证检查
```

### 阶段 6: 生成归档文档

**目标**: 生成完整的文档套件用于归档

**输出文档**:
1. `01-requirement.md` - 需求文档（做什么、为什么做）
2. `02-technical.md` - 技术方案（怎么做，只方案描述）
3. `03-implementation/` - 实施方案目录（阶段 3 已生成）
4. `04-integration.md` - 对接文档（涉及第三方系统时，可选）

**参考模板**: `templates/core/` 目录

**执行步骤**:
1. 基于阶段 1-5 的内容整理
2. 使用完整版模板生成文档
3. 保存到 `projects/<project>/requirements/<task-id>/`

---

### 📎 文档职责划分与强关联

```yaml
文档关系:

  01-requirement.md（需求层 - 做什么）:
    职责: 定义业务需求、功能需求、验收标准
    不包含: 技术实现细节、代码
    下游: 02-technical.md 引用本文档的需求点

  02-technical.md（方案层 - 怎么做）:
    职责: 架构设计、模块设计、接口契约、数据库设计、流程图
    不包含: 具体代码实现（只写接口签名、伪代码、配置结构）
    上游: 引用 01-requirement.md 的需求编号
    下游: 03-implementation/ 引用本文档的模块名和改动清单

  03-implementation/（实施层 - 具体改什么）:
    职责: 实施前检查、每个变更文件的完整代码、实施步骤、验证清单
    上游: 引用 02-technical.md 的模块设计
    结构:
      README.md - 实施总索引（检查+步骤+验证）
      <Module>.md - 每个变更文件的完整代码（一个文件对应一个文档）
    时机: 在执行代码之前生成，用户确认后才执行

  04-integration.md（对接层 - 可选）:
    职责: 第三方系统对接说明
    上游: 引用 01-requirement 和 02-technical

文档目录结构:
  projects/<project>/requirements/<task-id>/
  ├── 01-requirement.md            # 需求文档
  ├── 02-technical.md              # 技术方案（只描述，不写代码）
  ├── 03-implementation/           # 实施方案目录
  │   ├── README.md                # 实施总索引（前检查+文件清单+步骤+验证）
  │   ├── XxxModel.md              # 数据模型变更（完整代码）
  │   ├── XxxService.md            # 服务层变更（完整代码）
  │   ├── XxxController.md         # 控制层变更（完整代码）
  │   └── config-xxx.md            # 配置变更
  ├── 04-integration.md            # 对接文档（可选）
  └── scripts/                     # 脚本目录

文档间引用规范:
  02-technical.md 开头必须写:
    > 关联需求: [01-requirement.md](./01-requirement.md)

  03-implementation/README.md 开头必须写:
    > 技术方案: [02-technical.md](../02-technical.md)
    > 需求文档: [01-requirement.md](../01-requirement.md)

  03-implementation/ 各变更文件开头必须写:
    > 实施索引: [README.md](./README.md)
    > 文件路径: `app/service/XxxService.ts`
```

- [ ] 关键逻辑有注释
- [ ] 无明显的错误处理遗漏
- [ ] 文档内容完整

**输出格式**:
```markdown
🔍 验证结果:
- 代码规范: ✅ 通过

```

## 🔄 模式切换

### 自动判断

AI 根据以下条件自动选择模式:

| 条件 | 快速模式 | 完整模式 |
|-----|---------|---------|
| 需求描述 | < 100 字 | >= 100 字 |
| 涉及文件 | < 5 个 | >= 5 个 |
| 复杂度 | 简单改动 | 多模块协作 |
| 用户关键词 | "快速"、"简单" | "完整"、"详细" |

### 手动切换

用户可以随时说:
- "切换到完整模式" - 使用完整流程
- "只要代码，不要文档" - 跳过文档生成
- "需要完整文档" - 补充生成文档

## 📝 示例

### 快速模式示例

```
用户: "在 user 项目添加一个获取用户积分的接口"

AI 执行:
1. 理解需求 → "新增 GET /api/user/credits 接口" → 用户确认
2. 技术方案 → 修改 UserController + UserService → 用户确认
3. 实施方案 → 生成每个文件的具体变更内容 + 实施前检查 → 用户确认
4. 执行实现 → 实施前检查通过 → 对照实施方案逐文件执行
5. 完成报告 → 汇总修改 + 验证结果
```

### 完整模式示例

```
用户: "需要详细文档，在 payment 项目实现完整的退款功能"

AI 执行:
1-5. 同快速模式
6. 生成归档文档 → 01-requirement.md + 02-technical.md + 03-implementation/
7. 验证检查 → 代码和文档质量检查
```

## ✅ 需求开发专属验证

完成代码后，执行以下验证：

```yaml
需求开发验证清单:
  功能验证:
    ✅ 需求点覆盖: 每个需求点有对应实现
    ✅ 接口契约: 入参/出参/错误码定义完整
    ✅ 边界处理: null/空值/异常输入已处理
  
  代码质量:
    ✅ 错误处理: async 函数有 try-catch
    ✅ 日志记录: 关键操作有日志
    ✅ 配置外置: 无硬编码配置值
  
  文档同步:
    ✅ README: 新功能说明已添加
    ✅ 配置说明: 新配置项有文档
    ✅ CHANGELOG: 变更记录已添加
```
