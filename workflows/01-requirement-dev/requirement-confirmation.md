# 需求理解确认流程

> 确保 AI 正确理解用户需求后再执行

---

## 📋 触发条件

```yaml
触发确认流程的条件（满足任一）:
  - 用户要求"统一XXX"、"所有XXX"
  - 涉及配置化改造
  - 涉及工具类替换
  - 用户使用限定性关键词（"只"、"不要"、"包括"等）
  - 涉及代码修改/配置变更

例外:
  - 简单问答
  - 文档查询
```

---

## 🔄 确认流程（3 步）

### Step 1: 用自己的话复述需求

```yaml
必须输出格式:

📋 需求理解确认

我理解您的需求是：

1. **核心目标**: [用一句话描述]
2. **具体要求**:
   - [要求1]
   - [要求2]
   - [要求3]
3. **关键约束**:
   - [约束1: 如"只用分钟单位"]
   - [约束2: 如"不要改定时任务"]
4. **改造范围**:
   - 需要改: [列举文件/模块]
   - 不需要改: [列举文件/模块]

❓ 我的理解是否正确？如有误解请指出。
```

### Step 2: 全局搜索（如涉及代码改造）

```yaml
必须执行:
  1. 根据改造内容确定搜索模式
  2. 执行全局搜索（grep/file_search）
  3. 生成完整清单
  4. 输出统计

必须输出格式:

## 🔍 全局搜索结果

### 搜索命令
grep -rn "pattern1" app/
grep -rn "pattern2" app/

### 搜索结果汇总

| # | 文件 | 行号 | 原代码 | 是否改造 | 改造方案 | 不改原因 |
|---|------|------|--------|---------|---------|---------|
| 1 | file1.ts | L100 | code | ✅ 是 | 新方案 | - |
| 2 | file2.ts | L200 | code | ❌ 否 | - | [原因] |

### 统计
- **总计**: 10 处匹配
- **需要改造**: 5 处
- **保持不变**: 5 处

❓ 改造范围是否完整？如有遗漏请指出。
```

### Step 3: 等待用户明确确认

```yaml
等待用户响应:
  ✅ "确认" / "对" / "是的" → 继续执行
  ❌ "不对" / "有问题" → 重新理解需求
  🔄 "修改" → 调整理解和清单

🔴 禁止行为:
  - 未等确认就继续
  - 说"如果没问题我就继续了"
  - 假设用户默认同意
```

---

## 📝 关键词理解规则

### 数量/范围关键词

| 用户说 | 正确理解 | ❌ 错误理解 |
|-------|---------|-----------|
| "只XXX" | 仅限XXX，排除其他 | XXX + 其他相关的 |
| "所有XXX" | 全局搜索，不能遗漏 | 只改当前看到的 |
| "包括XXX" | 除了前面的，还要包括 | XXX可选 |
| "不要XXX" | XXX一行都不动 | 微调没关系 |

### 示例

```yaml
用户说: "只用分钟单位"
正确: 只用分钟，不用天/小时/秒
错误: 分钟为主，但保留其他单位

用户说: "所有过期时间配置"
正确: 全局搜索，找出所有相关代码
错误: 只改几个明显的地方

用户说: "不要改定时任务"
正确: 定时任务代码一行不动
错误: 顺便优化一下定时任务
```

## ⚠️ 常见违规

### 违规 1: 边做边想

```yaml
错误:
  AI: "我先改造 file1.ts..."
  AI: "哦，我发现 file2.ts 也需要改..."
  AI: "不好意思，还有 file3.ts..."

正确:
  AI: "让我先全局搜索所有需要改造的地方..."
  AI: "搜索结果：共10处，需要改造5处，详见清单..."
  用户: "确认"
  AI: "收到确认，开始改造..."
```

### 违规 2: 假设确认

```yaml
错误:
  AI: "如果没问题我就继续了"

正确:
  AI: "请确认理解是否正确 [确认/修改]"
  用户: "确认"
  AI: "收到确认，继续执行..."
```
