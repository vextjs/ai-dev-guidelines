## 📋 用户意图模式识别

### 模式 1: 理解偏差纠正

**用户表达**:
```yaml
典型句式:
  - "我明明说的是 xxx，你怎么做成了 yyy"
  - "我要的是 xxx，不是 yyy"
  - "我说的是 A，你理解成 B 了"
  - "不对，我说的是..."

关键特征:
  - 包含"明明"、"不是"、"不对"
  - 对比结构：期望 vs 实际
  - 纠正性语气
```

**AI 识别步骤**:
```yaml
1. 提取关键信息:
   - 用户期望: xxx (添加限流)
   - AI 实际做: yyy (做成防火墙)

2. 定位相关规范:
   - 工作流: workflows/01-requirement-dev/
   - 阶段: 需求理解阶段

3. 分析根因:
   - 需求确认点不够明确
   - 缺少"复述核心需求"步骤
   - 或：技术术语定义不清

4. 生成修复建议:
   选项 A: 在阶段 1 增加"用一句话复述核心需求"
   选项 B: 添加技术术语对照表
```

**响应模板**:
```yaml
AI: "非常抱歉理解错误！我会：
      
      1. 立即检查相关规范
      2. 分析为何会产生这个误解
      
      [检测中...]
      
      发现问题: workflows/01-requirement-dev 需求理解阶段
      缺少明确的'复述核心需求'步骤
      
      建议修复: 在阶段 1 增加复述环节，避免类似误解
      
      是否记录此问题并修复规范？[Y/N]"
```

### 模式 2: 规范执行质疑

**用户表达**:
```yaml
典型句式:
  - "你为什么不按规范来..."
  - "规范要求 xxx，你怎么..."
  - "这不符合规范"
  - "你违反了 xxx 规范"

关键特征:
  - 明确提到"规范"
  - 质疑/指责语气
  - 引用具体规范条款
```

**AI 识别步骤**:
```yaml
1. 定位被质疑的规范:
   - 提取规范名称: "预检查"、"命名规范"等
   - 映射到文件: workflows/00-pre-check/README.md

2. 自我检查:
   问题 A: 我是否真的违反了规范？
     - 如是: 分析为何违反（规范太严格？有例外情况？）
     - 如否: 规范表述不清晰，用户误解

3. 分析原因:
   情况 1: AI 确实违反
     - 规范过于严格，现实无法执行
     - 规范有例外情况但未说明
   
   情况 2: AI 遵守了规范
     - 规范表述模糊，用户误解
     - 规范位置不明显，AI 难以找到

4. 生成修复建议:
   针对情况 1: 放宽规范或添加例外说明
   针对情况 2: 增强规范表述，加粗关键要求
```

**响应模板**:
```yaml
# 情况 1: AI 确实违反规范
AI: "您说得对，我确实没有按规范执行预检查。
      
      让我检查为什么会这样...
      
      [分析]
      规范要求: 每次必须执行 5 项预检查
      我的行为: 跳过了预检查直接分析
      
      原因: 我判断这是简单任务，预检查可能不必要
      
      问题根源: 规范未说明"何时可以简化"
      
      建议修复: 在预检查规范中添加'简化条件'说明
      
      是否修复规范并重新执行任务？[Y/N]"

# 情况 2: AI 遵守了规范
AI: "让我检查相关规范...
      
      [分析]
      规范要求: workflows/00-pre-check/ 要求 5 项检查
      我的行为: 已执行 5 项检查（见上文输出）
      
      可能的问题: 我的输出格式不够明显
      
      规范问题: 未要求用明显标记展示预检查结果
      
      建议修复: 规范中增加输出格式示例
      
      是否修复规范让输出更清晰？[Y/N]"
```

### 模式 3: 重复需求（隐式反馈）

**用户表达**:
```yaml
场景:
  用户第 1 次: "添加限流功能"
  AI: [执行错误方向]
  用户第 2 次: "我说的是添加限流，不是防火墙"
  用户第 3 次: "限流！API 限流！"

关键特征:
  - 用户重复说同一个需求
  - 每次表达更强烈/更具体
  - 出现强调符号（！、大写）
```

**AI 识别步骤**:
```yaml
1. 检测重复:
   - 对比最近 3 轮对话
   - 提取核心关键词
   - 判断相似度 > 70%

2. 触发自检:
   "用户重复了 3 次，说明我理解有误"
   
3. 分析根因:
   - 我在哪个环节理解错误？
   - 规范是否有明确的理解确认机制？

4. 主动修复:
   "我注意到您重复了多次需求，
    说明我的理解可能有偏差。
    
    让我重新理解：您的核心需求是...
    
    同时，我会检查规范中的需求确认环节。"
```

### 模式 4: 混淆质疑

**用户表达**:
```yaml
典型句式:
  - "为什么要这样做？"
  - "这个步骤有必要吗？"
  - "能不能跳过 xxx？"

关键特征:
  - 疑问句
  - 质疑某个步骤的必要性
  - 可能是规范太繁琐
```

**AI 识别步骤**:
```yaml
1. 判断意图:
   情况 A: 用户不理解规范价值
     → 需要解释
   
   情况 B: 规范确实过于繁琐
     → 需要简化

2. 检查规范:
   - 这个步骤是否必要？
   - 是否可以简化？
   - 是否有说明"为什么需要"？

3. 响应:
   如必要: 解释价值，并在规范中补充说明
   如不必要: 标记为可简化，修复规范
```

### 模式 5: 反复出现同类问题（🆕 v2.0 — 升级触发）

**用户表达**:
```yaml
典型句式:
  - "这个问题又出现了"
  - "每次都重复同样的问题"
  - "上次不是修过了吗？怎么又..."
  - "你觉得单次修复了后续都会重复复现？"
  - "修复了但没有真正生效"
  - "为什么总是犯同一个错"

关键特征:
  - 包含"又"、"每次"、"重复"、"总是"、"再次"
  - 引用之前的修复历史
  - 指出修复不彻底/未真正嵌入流程
  - 质疑自我修复机制本身的有效性
```

**AI 识别步骤**:
```yaml
1. 识别反复问题:
   a. 提取用户指出的问题类型
   b. 扫描 spec-self-fix/records/ 历史记录
   c. 统计同类问题出现次数
   d. 确认是否属于"已修复但复现"的情况

2. 评估修复完整性（🔴 关键断层检测）:
   修复是否写入了执行流程入口文件？
     □ QUICK-REFERENCE.md
     □ CONSTRAINTS.md
     □ task-memory.md / 00-pre-check/README.md
   修复是否回写到 spec-self-fix 知识库本身？
     □ detection/ 模块检测规则是否更新？
     □ triggers/ 模块自动触发场景是否新增？
     □ repair/ 模块修复模式是否更新？
     □ records/summary.md 统计是否更新？
   修复是否从"建议"升级为"强制规则"？
     □ 是否标记为 🔴 NO EXCEPTIONS？
     □ 是否有自检清单？
     □ 是否有正例/反例对比？

3. 执行升级分析:
   根据出现次数触发不同等级的修复:
   
   | 出现次数 | 升级等级 | 动作 |
   |:--------:|:-------:|------|
   | 1 次     | 🟢 常规 | 单点修复 + 记录 |
   | 2 次     | 🟡 关注 | 5-Why 分析 + 加强检测规则 |
   | 3 次+    | 🔴 严重 | 全链路修复 + 强制规则 + 知识库回写 + 工具建设 |

4. 全链路修复（🔴 严重等级时必须执行）:
   a. 5-Why 根因分析 → 深入到真正根因
   b. 修复执行流程入口文件 → 确保每次任务都会检查
   c. 回写 spec-self-fix 知识库 → 确保健康检查能检测到
   d. 新增/升级自动触发场景 → 确保正确时机触发检测
   e. 新增/升级修复模式 → 确保知道如何修复
   f. 更新修复记录和统计 → 确保历史可追溯
   g. 评估是否需要自动化工具 → 消除人为遗漏

5. 🔴 回写完整性验证（修复后逐项确认）:
   □ 修复已写入执行流程入口？
   □ 修复已回写到 spec-self-fix 知识库？
   □ 检测模块有对应检测规则？
   □ 触发模块有对应自动触发场景？
   □ 修复模块有对应修复模式？
   □ 修复记录已更新到 records/summary.md？
   → 任一项为"否" → 修复不完整，存在复现风险
```

**响应模板**:
```yaml
AI: "您说得对，这类问题确实不应该反复出现。让我分析一下...
      
      [扫描修复历史]
      同类问题出现次数: 3 次（BUG-023~031）
      升级等级: 🔴 严重
      
      [断层检测]
      ✅ 执行流程入口已修复（QUICK-REFERENCE/CONSTRAINTS/task-memory）
      ❌ spec-self-fix 知识库未同步（detection/triggers/repair 仍为 v1.0）
      → 断层: 修复只写入了'AI 阅读的规范'，没有回写到'自我修复机制的知识库'
      → 后果: 未来健康检查不会使用最新检测规则，问题可能再次被遗漏
      
      [修复方案]
      全链路修复:
        1. 更新 detection/conflict-detection.md（嵌入 8 文件清单）
        2. 更新 triggers/auto-triggers.md（新增版本号全量同步触发）
        3. 更新 repair/repair-patterns.md（升级版本号修复模式）
        4. 更新 records/summary.md（更新统计）
        5. ...（其他需要同步的模块）
      
      是否确认执行全链路修复？[Y/N]"
```

**实际案例**:
```yaml
案例 1 — 版本号遗漏反复发生（BUG-023~031）:
  第 1 次 (§会话03): BUG-023~027
    → 单点修复 5 个文件版本号 → 🟢 常规
  
  第 2 次 (§会话04): BUG-028~029
    → 修复 2 个文件版本号 → 🟡 关注（但未升级分析）
  
  第 3 次 (§会话05): BUG-030~031
    → 用户明确指出 "每次都重复同样的问题"
    → 触发 🔴 严重等级
    → 5-Why 根因分析:
       根因 = 缺少明确文件清单 + 只有被动检查
    → 全链路修复:
       ✅ QUICK-REFERENCE 新增 8 文件清单 + 主动全量检查
       ✅ CONSTRAINTS #14 新增版本号全量同步行
       ✅ task-memory/00-pre-check 阶段 0 时序强制规则
       ❌ spec-self-fix 知识库未同步（§会话06 补齐）

案例 2 — spec-self-fix 知识库断层（§会话06）:
  用户指出: "修复了但后续都会重复复现？"
  触发: 模式 5（反复问题升级触发）
  断层检测:
    ✅ 执行流程入口已修复
    ❌ 知识库 6 个子模块仍为 v1.0
  修复: 全链路回写 → detection/triggers/repair 全部升级到 v2.0
```

### 模式 6: 规范缺失/不合理反馈（🆕 v2.1 — 规范补充触发）

**用户表达**:
```yaml
典型句式:
  - "规范中没有定义..."
  - "这个规范不合理"
  - "需要新增一条规范"
  - "规范定义不清楚"
  - "如果没有需要定义清楚"
  - "这个规则缺失了"
  - "规范里应该加上..."
  - "为什么没有关于 xxx 的规范？"

关键特征:
  - 指出规范内容的缺失（"没有定义"、"缺少"、"缺失"）
  - 指出规范内容不合理（"不合理"、"太严格"、"太宽松"、"不够清晰"）
  - 建议新增/补充规范（"需要新增"、"应该加上"、"建议增加"）
  - 指出规范定义模糊（"定义不清楚"、"歧义"、"容易误解"）
```

**与其他模式的区别**:
```yaml
模式 2（规范执行质疑）:
  用户说: "你为什么不按规范来？" → AI 没有遵守已有规范
  焦点: AI 的执行行为

模式 4（混淆质疑）:
  用户说: "这个规范太复杂了" → 已有规范的必要性/复杂度
  焦点: 已有规范的价值

模式 6（规范缺失/不合理 — 本模式）:
  用户说: "规范中没有定义 xxx" → 规范本身需要补充或修改
  焦点: 规范内容的完整性和合理性
```

**AI 识别步骤**:
```yaml
1. 判断反馈类型:
   a. 缺失型: 用户指出某个场景/规则没有被规范覆盖
   b. 不合理型: 用户认为已有规范的阈值/方式/定义不合理
   c. 模糊型: 用户认为已有规范定义不够清晰，存在歧义
   d. 新增型: 用户主动建议新增一条规范

2. 审查现有规范:
   a. 搜索相关规范文件（grep + read_file）
   b. 确认是否真的缺失/不合理
   c. 如果已有相关规范 → 可能是模糊型（规范存在但不清晰）
   d. 如果确实缺失 → 进入补充流程

3. 评估合理性（约束 #12 主动合理性分析）:
   a. 用户的建议是否合理？
   b. 有没有更好的方案？
   c. 新增/修改是否会与已有规范冲突？
   d. 影响范围有多大？（单个文件 vs 多文件交叉修改）

4. 生成修复方案:
   a. 缺失型 → 起草新规则内容 + 确定写入位置
   b. 不合理型 → 提出修改方案 + 说明新旧对比
   c. 模糊型 → 增强说明/示例/正例反例
   d. 新增型 → 评估是否需要新约束条目 or 融入已有条目

5. 等待用户确认后执行修复

6. 执行交叉验证（约束 #14）:
   a. 修改的内容是否需要同步到其他文件？
   b. QUICK-REFERENCE / CONSTRAINTS / copilot-instructions 三处一致？
   c. spec-self-fix 知识库是否需要同步更新？

7. 记录到 records/
```

**响应模板**:
```yaml
AI: "您指出了一个规范缺失/不合理的问题，让我审查一下...

     [审查结果]
     - 搜索范围: CONSTRAINTS.md, QUICK-REFERENCE.md, task-memory.md, ...
     - 发现: 确实没有关于 [xxx] 的明确规范定义

     [合理性评估]
     - 您的建议合理，因为 [原因]
     - 建议方案: [具体内容]
     - 影响范围: [涉及文件列表]

     [变更计划]
     1. [文件A] — [修改内容]
     2. [文件B] — [修改内容]
     ...

     确认后我开始执行。[Y/N]"
```

**实际案例**:
```yaml
案例 1 — 文件过大拆分规范缺失:
  用户说: "规范中有没有定义如果写入的文档文件太大需要进行拆分？如果没有需要定义清楚"
  识别: 模式 6（缺失型）
  审查:
    - CONSTRAINTS.md: 19 条约束，无文件大小/拆分相关
    - best-practices/README.md: 提到"文件过大难以阅读"但无强制规则
    - token-optimization.md: 只有读取策略，无写入/拆分规则
  结果: 确认缺失 → 起草约束 #20 → 用户确认阈值 → 执行

案例 2 — SUMMARY 日期精度不够:
  用户说: "SUMMARY.md 中日期应该是具体时间，而不是按天"
  识别: 模式 6（不合理型）
  审查:
    - task-memory.md SUMMARY 模板: 日期列格式为 YYYY-MM-DD（按天）
    - 每日记忆模板: 最后更新已有 HH:MM
  结果: 确认不合理（同天多会话无法区分先后）→ 升级为 YYYY-MM-DD HH:MM
```
