# 用户意图检测机制

> 基于用户反馈触发规范自我修复的核心机制

**优先级**: ⭐⭐⭐⭐⭐ 最高（最实用）  
**版本**: v1.0  
**创建日期**: 2026-02-12

---

## 🎯 机制定位

这是规范自我修复中**最重要**的触发机制，因为：

1. **实时反馈**: 基于真实用户使用场景
2. **精准定位**: 用户直接指出问题所在
3. **持续改进**: 每次用户抱怨都是改进机会
4. **自然触发**: 不需要用户额外操作

---

## 📋 用户意图模式识别

### 模式 1: 理解偏差纠正

**用户表达**:
```yaml
典型句式:
  - "我明明说的是 xxx，你怎么做成了 yyy"
  - "我要的是 xxx，不是 yyy"
  - "我说的是 A，你理解成 B 了"
  - "不对，我说的是..."

关键特征:
  - 包含"明明"、"不是"、"不对"
  - 对比结构：期望 vs 实际
  - 纠正性语气
```

**AI 识别步骤**:
```yaml
1. 提取关键信息:
   - 用户期望: xxx (添加限流)
   - AI 实际做: yyy (做成防火墙)

2. 定位相关规范:
   - 工作流: workflows/01-requirement-dev/
   - 阶段: 需求理解阶段

3. 分析根因:
   - 需求确认点不够明确
   - 缺少"复述核心需求"步骤
   - 或：技术术语定义不清

4. 生成修复建议:
   选项 A: 在阶段 1 增加"用一句话复述核心需求"
   选项 B: 添加技术术语对照表
```

**响应模板**:
```yaml
AI: "非常抱歉理解错误！我会：
      
      1. 立即检查相关规范
      2. 分析为何会产生这个误解
      
      [检测中...]
      
      发现问题: workflows/01-requirement-dev 需求理解阶段
      缺少明确的'复述核心需求'步骤
      
      建议修复: 在阶段 1 增加复述环节，避免类似误解
      
      是否记录此问题并修复规范？[Y/N]"
```

---

### 模式 2: 规范执行质疑

**用户表达**:
```yaml
典型句式:
  - "你为什么不按规范来..."
  - "规范要求 xxx，你怎么..."
  - "这不符合规范"
  - "你违反了 xxx 规范"

关键特征:
  - 明确提到"规范"
  - 质疑/指责语气
  - 引用具体规范条款
```

**AI 识别步骤**:
```yaml
1. 定位被质疑的规范:
   - 提取规范名称: "预检查"、"命名规范"等
   - 映射到文件: workflows/00-pre-check/README.md

2. 自我检查:
   问题 A: 我是否真的违反了规范？
     - 如是: 分析为何违反（规范太严格？有例外情况？）
     - 如否: 规范表述不清晰，用户误解

3. 分析原因:
   情况 1: AI 确实违反
     - 规范过于严格，现实无法执行
     - 规范有例外情况但未说明
   
   情况 2: AI 遵守了规范
     - 规范表述模糊，用户误解
     - 规范位置不明显，AI 难以找到

4. 生成修复建议:
   针对情况 1: 放宽规范或添加例外说明
   针对情况 2: 增强规范表述，加粗关键要求
```

**响应模板**:
```yaml
# 情况 1: AI 确实违反规范
AI: "您说得对，我确实没有按规范执行预检查。
      
      让我检查为什么会这样...
      
      [分析]
      规范要求: 每次必须执行 3 项预检查
      我的行为: 跳过了预检查直接分析
      
      原因: 我判断这是简单任务，预检查可能不必要
      
      问题根源: 规范未说明"何时可以简化"
      
      建议修复: 在预检查规范中添加'简化条件'说明
      
      是否修复规范并重新执行任务？[Y/N]"

# 情况 2: AI 遵守了规范
AI: "让我检查相关规范...
      
      [分析]
      规范要求: workflows/00-pre-check/ 要求 3 项检查
      我的行为: 已执行 3 项检查（见上文输出）
      
      可能的问题: 我的输出格式不够明显
      
      规范问题: 未要求用明显标记展示预检查结果
      
      建议修复: 规范中增加输出格式示例
      
      是否修复规范让输出更清晰？[Y/N]"
```

### 模式 3: 重复需求（隐式反馈）

**用户表达**:
```yaml
场景:
  用户第 1 次: "添加限流功能"
  AI: [执行错误方向]
  用户第 2 次: "我说的是添加限流，不是防火墙"
  用户第 3 次: "限流！API 限流！"

关键特征:
  - 用户重复说同一个需求
  - 每次表达更强烈/更具体
  - 出现强调符号（！、大写）
```

**AI 识别步骤**:
```yaml
1. 检测重复:
   - 对比最近 3 轮对话
   - 提取核心关键词
   - 判断相似度 > 70%

2. 触发自检:
   "用户重复了 3 次，说明我理解有误"
   
3. 分析根因:
   - 我在哪个环节理解错误？
   - 规范是否有明确的理解确认机制？

4. 主动修复:
   "我注意到您重复了多次需求，
    说明我的理解可能有偏差。
    
    让我重新理解：您的核心需求是...
    
    同时，我会检查规范中的需求确认环节。"
```

---

### 模式 4: 混淆质疑

**用户表达**:
```yaml
典型句式:
  - "为什么要这样做？"
  - "这个步骤有必要吗？"
  - "能不能跳过 xxx？"

关键特征:
  - 疑问句
  - 质疑某个步骤的必要性
  - 可能是规范太繁琐
```

**AI 识别步骤**:
```yaml
1. 判断意图:
   情况 A: 用户不理解规范价值
     → 需要解释
   
   情况 B: 规范确实过于繁琐
     → 需要简化

2. 检查规范:
   - 这个步骤是否必要？
   - 是否可以简化？
   - 是否有说明"为什么需要"？

3. 响应:
   如必要: 解释价值，并在规范中补充说明
   如不必要: 标记为可简化，修复规范
```

## 🔧 检测算法实现

### 算法 1: 关键词匹配

```yaml
高优先级关键词（直接触发）:
  - "明明"、"不是"、"不对"、"错了"
  - "规范"、"要求"、"应该"
  - "为什么不"、"怎么不"

中优先级关键词（综合判断）:
  - "理解"、"意思"、"意图"
  - "重复"、"再说一次"

触发逻辑:
  if 包含高优先级关键词:
    立即触发检测
  elif 包含中优先级关键词 + 负面语气:
    触发检测
```

### 算法 2: 语气分析

```yaml
负面语气特征:
  - 问号多（？？？）
  - 感叹号（！）
  - 重复字符（我说的是xxx，xxx！）
  - 短句（语气急促）

综合判断:
  负面语气 + 关键词 → 触发检测
```

### 算法 3: 对话轮次分析

```yaml
检测重复:
  1. 提取最近 5 轮对话的用户输入
  2. 提取核心关键词（去除停用词）
  3. 计算关键词重叠度
  4. 如果连续 3 轮重叠度 > 60% → 触发检测

示例:
  轮次 1: "添加限流功能"  关键词: [添加, 限流, 功能]
  轮次 2: "我说的是限流"  关键词: [说, 限流]
  轮次 3: "限流！"        关键词: [限流]
  
  重叠: 3/3 轮都有"限流" → 触发
```

## 📝 响应流程

### 标准响应流程

```yaml
Step 1: 识别用户意图
  - 提取关键信息
  - 分类意图类型（纠正/质疑/重复）

Step 2: 致歉并说明
  "非常抱歉！我会立即检查..."

Step 3: 定位相关规范
  - 根据对话内容定位规范文件
  - 如不确定，向用户确认

Step 4: 执行检测
  - 自我检查是否遵守规范
  - 检查规范是否清晰/完整

Step 5: 分析根因
  - 为什么产生偏差？
  - 规范问题 or AI 问题？

Step 6: 生成修复建议
  - 具体的修复方案
  - 预期效果

Step 7: 询问用户
  "是否记录此问题并修复规范？[Y/N]"

Step 8: 执行修复（如用户同意）
  - 应用修复
  - 生成修复记录
  - 更新 records/summary.md
```

每次触发都应记录到 `../records/YYYYMMDD-user-feedback-xxx.md`：

```markdown

**用户**: "我明明说的是添加限流，你怎么做成了防火墙？"

**AI 识别**:

- 期望: 添加限流

## 规范检查

**相关规范**: `workflows/01-requirement-dev/README.md`

**问题定位**: 阶段 1 需求理解步骤
```

2. 分析涉及的模块和功能
