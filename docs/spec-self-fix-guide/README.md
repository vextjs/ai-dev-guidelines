## 一、机制总览

### 为什么需要自我修复？

ai-dev-guidelines 规范体系由 **40+ 个文件**组成，同一规则可能在多处引用。当修改了一处而忘记同步其他地方，就会产生不一致 — AI 执行时会混乱或出错。自我修复机制让 AI 具备自动检测和修正这类问题的能力。

### 核心能力

| 能力 | 说明 | 版本 |
|------|------|:----:|
| **自动检测** | 检测规范问题（冲突、过时、冗余、缺失、时序违规） | v1.0 |
| **智能修复** | 安全修复自动执行，重要修复需用户确认 | v1.0 |
| **记录追溯** | 所有修复历史可查（`core/self-fix/records/`） | v1.0 |
| **响应用户** | 用户抱怨/纠正/质疑/建议触发规范检查 | v2.0 |
| **升级驱动** | 反复出现的同类问题自动升级修复力度 | v2.3 |
| **闭环验证** | 修复后验证知识库回写完整性，防止断层 | v2.3 |
| **规范补充** | 响应用户指出的规范缺失/不合理，主动审查并补充 | v2.4 |

### 自修复与日常任务的关系

> **重要澄清**：自修复**不会**在每次任务完成后全量运行。

| 触发频率 | 执行内容 | 说明 |
|---------|---------|------|
| **每次任务**（实时） | 场景 1（工作流完整性）+ 场景 9（第 6 行存在性） | 轻量检测，几乎无感知 |
| **任务完成后** | 检测点 7（反复问题检查）+ 检测点 8（汇总问题） | 轻量汇总，只在发现相似历史问题时触发升级 |
| **特定事件后** | 版本号变更 / 约束变更 / 时序规则修改 等 | 事件触发，精确检测 |
| **用户主动触发** | "运行规范健康检查" | 全量检测，遍历所有规范文件 |

详见 [01-detection-and-triggers.md](./01-detection-and-triggers.md) §检测频率分类。

## 三、关键数据

| 指标 | 数值 | 说明 |
|------|:----:|------|
| 检测规则 | 21 条 | 4 类：冲突 6 + 过时 6 + 冗余 4 + 完整性 5 |
| 触发场景 | 9 个自动 + 6 种用户意图 + 手动 | 详见 [01-detection-and-triggers.md](./01-detection-and-triggers.md) |
| 修复模式 | 11 种 | 模式 1~11，详见 [02-repair-and-records.md](./02-repair-and-records.md) |
| 修复验证 | 7 步 | 每次修复后执行 |
| 防复现机制 | 4 类 | 主动全量 / 硬性阻塞 / 知识库闭环 / 反复升级 |
| 回写验证 | 6 项检查点 | 防止知识库断层 |

> **数字对齐说明**：`core/self-fix/README.md` 中标注"10 种修复模式"，本指南按照 `repair-patterns.md` 中实际列出的模式 1~10 加上 `auto-repair.md` 中的模式 11（预检查-记忆原子操作修复）合计为 **11 种**。触发场景数量以 `auto-triggers.md` 中实际的场景 1~9 为准，共 **9 个**自动触发场景。
