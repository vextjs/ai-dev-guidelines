## 一、修复策略分类

> 源文件: `core/self-fix/repair/auto-repair.md`

| 策略 | 安全等级 | 适用场景 | 是否需要你确认 |
|------|:-------:|---------|:-------------:|
| **自动修复** | ✅ 安全 | 补充缺失章节、修复死链接、版本号同步、时序规则同步、格式标准化 | 否 |
| **半自动修复** | ⚠️ 需确认 | 删除冗余内容、内容合并、知识库断层修复、依赖链修复、术语统一 | 是 |
| **辅助修复** | 🔴 人工 | 解决规范冲突（需理解业务）、架构决策 | 是（AI 提供方案，你决策） |

**自动修复的安全边界：**

```yaml
✅ 可以自动:
  - 修改不影响业务逻辑的元数据（版本号、日期、条数）
  - 补充缺失的标准化章节（使用模板）
  - 修复明显的死链接（路径拼写错误）
  - 同步时序规则（以 task-memory.md 为权威来源）
  - 同步清单内容（以 CROSS-VALIDATION.md 为权威来源）

❌ 不能自动:
  - 删除任何内容
  - 修改规范的语义或逻辑
  - 解决两个规范之间的业务冲突
  - 修改用户的代码
```

### 模式 1~4：基础修复

#### 模式 1 — 补充缺失章节

```yaml
触发: 完整性检测发现 core/standards/ 下的文件缺少必要章节（标题/规则/示例/检查清单）
执行:
  1. 识别缺失的章节类型
  2. 使用对应模板生成章节骨架
  3. 填入占位内容（标注 TODO）
  4. 写入文件
安全等级: ✅ 自动（只新增内容，不修改已有内容）
```

#### 模式 2 — 修复死链接

```yaml
触发: 过时检测发现 .md 文件中的相对链接指向不存在的文件
执行:
  1. 收集所有失效链接
  2. 对每个链接:
     a. 尝试匹配正确路径（文件名相似度 > 80%）
     b. 如找到唯一匹配 → 自动修复
     c. 如找到多个匹配 → 列出候选，让用户选择
     d. 如无匹配 → 标记为待处理
  3. 批量更新
安全等级: ✅ 自动（唯一匹配时）/ ⚠️ 半自动（多匹配时）
```

#### 模式 3 — 添加可覆盖标记

```yaml
触发: core/standards/ 下的通用规范未标注"项目规范可覆盖"
执行:
  1. 在规范文件头部添加可覆盖标记
  2. 标注哪些规则可以被项目规范覆盖
安全等级: ✅ 自动
```

#### 模式 4 — 内容合并

```yaml
触发: 冗余检测发现两个文件/章节内容相似度 > 70%
执行:
  1. 对比两处内容，识别差异点
  2. 生成合并方案（保留更完整的版本）
  3. 提交合并方案供用户确认
  4. 用户确认后执行合并 + 在被合并处添加引用链接
安全等级: ⚠️ 半自动（涉及删除内容，必须用户确认）
```

### 模式 5：版本号批量更新

这是最常用的修复模式之一，已经过 3 次迭代优化。

```yaml
执行步骤:
  1. 确定目标版本号（从 copilot-instructions.md 或用户指令获取）
  2. 逐个 read_file 8 个文件，提取当前版本号
  3. 找出不一致的文件
  4. 逐个更新版本号 + "最后更新"日期
  5. 验证：再次逐个 read_file 确认全部一致
  6. 记录修复到 records/

v2.0 vs v1.0 的核心变化:
  v1.0（已废弃）: 只检查"改了的文件" → 被动检查 → 已导致 3 次 Bug
  v2.0（当前）: 对照清单逐个 read_file → 主动全量 → 不依赖记忆
```

**版本号 8 文件清单：**

| # | 文件 | 版本号位置 |
|:-:|------|-----------|
| 1 | `.github/copilot-instructions.md` | L1 `**版本**: vX.Y.Z` |
| 2 | `README.md` | L3 `> **版本**: vX.Y.Z` |
| 3 | `QUICK-REFERENCE.md` | L5 `**版本**: vX.Y.Z` |
| 4 | `CONSTRAINTS.md` | L6 `**版本**: vX.Y.Z` |
| 5 | `STATUS.md` | L5 `**当前版本**: vX.Y.Z` |
| 6 | `CHANGELOG.md` | 版本概览表新增行 |
| 7 | `core/workflows/decision-tree.yaml` | L1 注释 + L4 `version` 字段 + L82 注释 + L87 `mandatory_precheck.version` |
| 8 | `core/workflows/00-pre-check/README.md` | L3 + 文件末尾（两处） |

安全等级: ✅ 自动

### 模式 6~8：依赖与格式修复

#### 模式 6 — 依赖链修复

```yaml
触发: 文件 A 引用 B，B 引用 C，C 不存在（断链深度 ≥ 2）
执行:
  1. 追溯完整的引用链
  2. 定位断裂点
  3. 生成修复方案（创建缺失文件 / 修复链接 / 移除无效引用）
  4. 提交方案供用户确认
安全等级: ⚠️ 半自动
```

#### 模式 7 — 术语统一

```yaml
触发: 同一概念在不同文件中有不同表述（如"预检查"/"pre-check"/"前置检查"）
执行:
  1. 收集所有相关表述
  2. 确定标准术语（通常以 QUICK-REFERENCE.md 中的用法为准）
  3. 生成统一方案（列出所有需要修改的位置）
  4. 提交方案供用户确认
安全等级: ⚠️ 半自动
```

#### 模式 8 — 格式标准化

```yaml
触发: 格式不符合规范（代码块缺少语言标记、表格格式不统一等）
执行:
  1. 扫描发现的格式问题
  2. 按规范自动修复
安全等级: ✅ 自动（不影响语义）
```

### 模式 9：时序违规修复

```yaml
触发: 阶段 0 时序规则在 task-memory.md 和 00-pre-check/README.md 之间不一致
执行:
  1. 以 task-memory.md 为权威来源（时序规则的 single source of truth）
  2. 读取 task-memory.md 中的时序规则定义
  3. 对比 00-pre-check/README.md 中的时序规则
  4. 如不一致 → 将 00-pre-check 同步为 task-memory 的内容
  5. 验证两处内容一致
安全等级: ✅ 自动

根因背景:
  此模式源于 §会话05 的阶段 0 时序违规事故 —— AI 先分析问题再补写记忆，
  导致记忆写入可能被遗忘或时间线混乱。
  ✅ 正确: 预检查 → 写入记忆(§会话NN) → 输出"📝 记忆已更新" → 开始分析
  ❌ 错误: 预检查 → 分析问题 → 输出结论 → 补写记忆
```

### 模式 10：知识库断层修复

当修复只写入了执行流程入口文件（QUICK-REFERENCE / CONSTRAINTS / task-memory / 00-pre-check），但未回写到 spec-self-fix 知识库（detection / triggers / repair）时，就产生了"断层"。

```yaml
断层检测清单（6 项）:
  □ 修复是否写入了执行流程入口文件？
  □ 修复是否写入了 core/self-fix/detection/ 相关检测规则？
  □ 修复是否写入了 core/self-fix/triggers/ 相关触发规则？
  □ 修复是否写入了 core/self-fix/repair/ 相关修复模式？
  □ 修复是否在 core/self-fix/records/ 有记录？
  □ records/summary.md 是否更新？

任一项为"否" → 触发断层修复

执行步骤:
  1. 识别缺失的回写目标
  2. 从执行流程入口文件中提取新规则内容
  3. 生成回写方案（需要在知识库哪些文件中添加什么内容）
  4. 提交方案供用户确认
  5. 用户确认后执行回写

安全等级: ⚠️ 半自动（涉及多文件新增内容，需用户确认）
```

**为什么会产生断层？**

修复规范问题时，AI 通常会优先修改 QUICK-REFERENCE / CONSTRAINTS / task-memory 等 AI 日常执行时会读取的文件。但如果忘记同步更新 spec-self-fix 的检测/触发/修复规则，未来的健康检查和自动检测将无法使用最新规则，导致问题可能复现。

安全等级: ✅ 自动

## 三、修复验证（7 步）

> 源文件: `core/self-fix/repair/repair-validation.md`

每次修复完成后，执行 **7 步验证**确保修复质量：

| 步骤 | 验证内容 | 适用场景 |
|:----:|---------|---------|
| 1 | **文件完整性** — 修复后文件结构完整，无损坏 | 所有修复 |
| 2 | **链接有效性** — 修复中涉及的链接全部有效 | 所有修复 |
| 3 | **内容一致性** — 修复后跨文件内容一致 | 所有修复 |
| 4 | **功能验证** — 修复不影响规范的正常使用 | 所有修复 |
| 5 | **版本号全量一致性** — 8 个文件版本号全部相同 | 版本号相关修复后 |
| 6 | **时序规则同步一致性** — 两处时序规则内容相同 | 时序相关修复后 |
| 7 | **回写完整性** — 修复同时写入了执行流程和知识库（6 项断层检测清单） | 知识库断层修复后 |

**验证输出格式：**

```
✅ 修复验证:
  1. 文件完整性: ✅ 通过
  2. 链接有效性: ✅ 通过（已验证 3 个链接）
  3. 内容一致性: ✅ 通过（跨 2 个文件一致）
  4. 功能验证:   ✅ 通过
  5. 版本号全量: ✅ 通过（8/8 文件一致）
  6. 时序规则:   ⏭️ 不适用（本次修复不涉及时序）
  7. 回写完整性: ✅ 通过（6/6 检查点通过）
```

## 五、防复现机制

自我修复机制经过多次迭代，建立了以下 4 类防复现体系：

### 5.1 主动全量检查（替代被动检查）

| 检查类型 | 旧方式（已废弃） | 新方式（当前） | 根因 |
|---------|-----------------|---------------|------|
| 版本号 | 只检查"改了的文件" | 逐个 read_file 8 个文件，全部一致才通过 | 被动检查导致 3 次 Bug |
| 约束条数 | 只改 3 个主文件 | 逐个 read_file 11 个引用文件，全部一致才通过 | 遗漏了 8+ 处卫星文件 |
| 时序规则 | 只在 task-memory 定义 | task-memory + 00-pre-check 两处必须一致 | 定义不一致导致执行混乱 |

**为什么"主动全量"比"被动检查"好？**

被动检查依赖 AI 的记忆 — "我改了哪些文件"。但 AI 的记忆不可靠（上下文窗口有限、会话间断），导致反复遗漏。主动全量不依赖记忆，而是对照一份固定清单逐个验证，从根本上消除遗漏风险。

---

### 5.2 硬性阻塞（替代软性规则）

| 机制 | 旧方式（soft gate） | 新方式（hard gate） | 效果 |
|------|-------------------|-------------------|------|
| 阶段 0 记忆写入 | 文字规则"必须先写记忆" | 嵌入预检查第 6 行输出，不写 = 预检查未完成 | 2 次事故 → 0 次 |
| 版本号检查 | "修改后记得同步" | 版本号文件清单 + 自动触发场景 5 | 3 次 Bug → 0 次 |
| 约束条数检查 | "新增约束记得改条数" | 约束条数引用清单 + 自动触发场景 8 | 多次遗漏 → 0 次 |

**soft gate vs hard gate 的本质区别：**

- **Soft gate**：依赖 AI 自觉遵守文字规则 → AI 可能选择跳过（已发生过 2 次事故）
- **Hard gate**：将规则嵌入不可跳过的流程步骤 → AI 无法绕过（不写 = 预检查本身不完整 = 无法进入下一步）

---

### 5.3 知识库闭环（修复 = 执行流程 + 知识库双写）

```
修复一个问题时，必须同时写入两个地方:

  1. 执行流程入口（AI 日常执行时会读取的文件）:
     - QUICK-REFERENCE.md
     - CONSTRAINTS.md
     - task-memory.md
     - 00-pre-check/README.md

  2. spec-self-fix 知识库（健康检查和自动检测使用的规则）:
     - detection/*.md  — 检测规则
     - triggers/*.md   — 触发场景
     - repair/*.md     — 修复模式
     - records/*.md    — 修复记录

  只写一处不写另一处 = 知识库断层 = 模式 10 触发修复
```

**为什么需要双写？**

如果只写入执行流程（QUICK-REFERENCE 等），AI 日常执行时能遵守新规则，但健康检查时无法检测新规则是否被违反。反过来，如果只写入知识库（spec-self-fix），健康检查能发现问题，但 AI 日常执行时不知道新规则。

双写确保了"日常执行"和"健康检查"两条路径都能使用最新规则。

### 5.4 反复问题自动升级

```
同类问题第 1 次出现 → 常规修复
  → 标准修复模式（模式 1~11 之一）
  → 记录到 records/

同类问题第 2 次出现 → 触发升级分析（自动触发场景 7）
  → 5-Why 根因分析（深入到根因，不停留在表面）
  → 全链路修复（不只修表面，还修根因和防复现机制）
  → 新增防复现机制（进入反复问题追踪表）
  → 回写到 records/summary.md 反复问题追踪表
  → 回写到 spec-self-fix 知识库（防止知识库断层）

同类问题第 3 次出现 → 根治性修复
  → 审查为什么升级分析后仍然复现
  → 通常需要引入 hard gate 或自动化工具
  → 例: 版本号遗漏 3 次 → 引入 8 文件清单 + 主动全量检查
```

**升级分析的 5-Why 示例：**

```
问题: 版本号又没同步（第 3 次）

Why 1: 为什么版本号没同步？
  → 修改了 copilot-instructions.md 但忘了改其他文件

Why 2: 为什么会"忘了"？
  → 没有明确的"哪些文件含版本号"清单

Why 3: 为什么没有清单？
  → 之前用的是"扫描所有文件找版本号"的方式

Why 4: 为什么扫描方式不行？
  → 因为被动检查依赖 AI 记忆，AI 可能遗漏文件

Why 5: 根因
  → 缺少一份固定的、不依赖记忆的版本号文件清单

解决方案:
  1. 建立 8 文件清单（固定在 CROSS-VALIDATION.md 中）
  2. 检查方式从"被动扫描"升级为"主动全量"
  3. 新增自动触发场景 5（版本号变更时自动检测）
```
