## 场景 1：需求开发

> **触发关键词：** 实现、开发、添加、新增、集成
> **工作流：** `core/workflows/01-requirement-dev/`

### 快速模式（默认，< 5 个文件的简单需求）

```
┌──────────────────────────────────────────────────────────────────┐
│ 阶段 1: 理解需求                                                 │
│   → AI 用一句话复述核心需求 + 列出关键改动点                       │
│   → ⏸️ CP1: 等你确认理解正确                                     │
├──────────────────────────────────────────────────────────────────┤
│ 阶段 2: 技术方案                                                 │
│   → AI 分析相关代码，输出架构设计 + 改动文件清单 + 影响评估         │
│   → 只描述方案思路，不写具体代码                                   │
│   → ⏸️ CP2: 等你确认方案可行                                     │
├──────────────────────────────────────────────────────────────────┤
│ 阶段 3: 实施方案                                                 │
│   → AI 为每个文件生成具体变更内容 + 实施前检查清单 + 验证清单       │
│   → 小需求在聊天中直接输出，大需求生成 03-implementation/ 目录      │
│   → ⏸️ CP3: 等你确认所有变更内容                                  │
├──────────────────────────────────────────────────────────────────┤
│ 阶段 4: 执行实现                                                 │
│   → 先跑实施前检查清单，全部通过后按依赖顺序逐文件执行              │
│   → 每完成一个文件/批次报告进度                                    │
│   → 涉及 5+ 文件时按层分批：基础层 → 逻辑层 → 接口层 → 验证层      │
├──────────────────────────────────────────────────────────────────┤
│ 阶段 5: 完成报告                                                 │
│   → 修改汇总 + 验证结果 + 注意事项                                │
│   → 自动更新 TASK-INDEX.md + 写入记忆                             │
└──────────────────────────────────────────────────────────────────┘
```

### 完整模式（复杂需求、核心功能）

在快速模式基础上增加两个阶段：

```
阶段 1~5: 同快速模式
阶段 6: 生成归档文档
  → 01-requirement.md（做什么、为什么做）
  → 02-technical.md（怎么做、架构设计）
  → 03-implementation/（具体改什么、完整代码）
  → 04-接口文档.md（涉及 API 接口新增/变更时，可选）
     或 04-对接文档.md（涉及第三方系统对接时，可选）
  → api.http（跟随 04-接口文档自动生成，可选）
阶段 7: 验证检查
  → 代码规范 + 注释完整 + 错误处理 + 文档完整
```

### 文档间关系

```
01-requirement.md    做什么      ← 用户需求
      ↓ 引用
02-technical.md      怎么做      ← 架构设计（不写代码）
      ↓ 引用
03-implementation/   具体改什么   ← 每个文件的完整变更（确认后才执行）
      ↓ 对照执行
实际代码修改
```

### 产物输出目录

```
ai-dev-guidelines/projects/<项目>/requirements/<中文描述>/
├── 01-requirement.md
├── 02-technical.md
├── 03-implementation/
│   ├── README.md          # 实施总索引
│   ├── XxxService.md      # 服务层变更
│   └── XxxController.md   # 控制层变更
├── 04-接口文档.md          # 接口文档（涉及 API 新增/变更时，可选）
│   # 或 04-对接文档.md    # 对接文档（涉及第三方系统对接时，可选）
├── api.http               # 接口调试文件（跟随 04-接口文档自动生成，可选）
└── scripts/               # 自动化脚本（如需要）
```

### 需求开发专属验证

```yaml
功能验证:
  ✅ 需求点覆盖: 每个需求点有对应实现
  ✅ 接口契约: 入参/出参/错误码定义完整
  ✅ 边界处理: null/空值/异常输入已处理

代码质量:
  ✅ 错误处理: async 函数有 try-catch
  ✅ 日志记录: 关键操作有日志
  ✅ 配置外置: 无硬编码配置值

文档同步:
  ✅ README: 新功能说明已添加
  ✅ 配置说明: 新配置项有文档
  ✅ CHANGELOG: 变更记录已添加
```

## 场景 2：Bug 修复

> **触发关键词：** 修复、解决、Bug、报错、异常
> **工作流：** `core/workflows/02-bug-fix/`

```
┌──────────────────────────────────────────────────────────────────┐
│ Step 1: 问题复现                                                 │
│   → AI 询问问题详情（描述、时间、影响范围、错误日志）               │
│   → 尝试复现问题，记录复现步骤                                    │
├──────────────────────────────────────────────────────────────────┤
│ Step 2: 原因分析                                                 │
│   → 分析错误日志 + 检查相关代码 + 确定影响范围 + 根因分析          │
│   → 输出 01-analysis.md                                         │
│   → ⏸️ CP1: 等你确认根因分析正确                                 │
├──────────────────────────────────────────────────────────────────┤
│ Step 3: 解决方案设计                                              │
│   → 设计修复方案（只描述思路，不写代码）+ 风险评估 + 回滚方案      │
│   → 输出 02-solution.md                                          │
│   → ⏸️ CP2: 等你确认方案可行                                     │
├──────────────────────────────────────────────────────────────────┤
│ Step 4: 实施方案                                                 │
│   → 为每个文件生成具体变更 + 实施前检查 + 实施步骤 + 验证清单      │
│   → ⏸️ CP3: 等你确认所有变更内容                                  │
├──────────────────────────────────────────────────────────────────┤
│ Step 5: 执行修复                                                 │
│   → 实施前检查通过后，按顺序逐文件修改，每完成一个报告进度          │
├──────────────────────────────────────────────────────────────────┤
│ Step 6: 验证测试                                                 │
│   → 单元测试 + 复现验证 + 回归测试 + 性能影响评估                  │
├──────────────────────────────────────────────────────────────────┤
│ Step 7: 文档归档                                                 │
│   → 检查文档完整性 + 更新 TASK-INDEX.md + 写入记忆                │
└──────────────────────────────────────────────────────────────────┘
```

### 产物输出目录

```
# 小 Bug（< 5 个文件）
ai-dev-guidelines/projects/<项目>/bugs/<中文描述>/
├── 01-analysis.md              # 问题分析
├── 02-solution.md              # 解决方案
├── 03-implementation.md        # 实施方案（单文件）
├── 04-接口变更.md              # 接口变更说明（修复涉及接口契约变化时，可选）
├── api.http                    # 接口调试文件（跟随 04-接口变更自动生成，可选）
└── scripts/
    ├── fix-data.sql            # 数据修复脚本（如需要）
    └── rollback.sql            # 回滚脚本

# 大 Bug（≥ 5 个文件）
ai-dev-guidelines/projects/<项目>/bugs/<中文描述>/
├── 01-analysis.md
├── 02-solution.md
├── 03-implementation/          # 实施方案（目录）
│   ├── README.md               # 实施总索引
│   ├── OrderService.md
│   └── PaymentService.md
├── 04-接口变更.md              # 接口变更说明（如修复涉及接口契约变化，可选）
├── api.http                    # 接口调试文件（跟随 04-接口变更自动生成，可选）
└── scripts/
```

### 严重 Bug 特殊处理

| 严重程度 | 特殊处理 |
|---------|---------|
| P0/P1 | 优先修复，暂停其他任务；详细记录过程；制定详细回滚方案；充分测试 |
| 涉及数据修复 | 必须生成 SQL 脚本；先在测试环境验证；必须有回滚脚本 |
| 涉及配置变更 | 记录变更前后配置；说明变更原因；标注需重启的服务 |

### Bug 修复专属验证

```yaml
问题验证:
  ✅ 修复前能复现 → 修复后不再出现
  ✅ 根因明确记录

副作用检查:
  ✅ 未引入新 Bug
  ✅ 相关功能未受影响
  ✅ 无明显性能下降

回滚准备:
  ✅ 有回滚步骤
  ✅ 数据修复有回滚脚本
```

## 场景 4：深度分析

> **触发关键词：** 分析、审查、评估、深度分析、全面检查
> **工作流：** `core/workflows/10-analysis/`

```
┌──────────────────────────────────────────────────────────────────┐
│ 阶段 1: 范围确认                                                 │
│   → 理解你要分析什么、分析到什么程度、期望什么输出                  │
│   → 范围模糊时先确认再执行                                        │
├──────────────────────────────────────────────────────────────────┤
│ 阶段 2: 数据采集                                                 │
│   → 列出需要检查的文件清单 + 逐个读取（不推断内容）               │
│   → 数值类检查使用终端命令批量验证                                 │
├──────────────────────────────────────────────────────────────────┤
│ 阶段 3: 交叉验证                                                 │
│   → 🔢 数值一致性（同一数值在所有引用处是否相同）                  │
│   → 📝 格式一致性（定义 vs 示例中的实际格式）                     │
│   → 🔗 引用有效性（所有链接是否指向存在的文件）                    │
│   → 📊 标注准确性（数量/计数与实际文件系统是否匹配）               │
├──────────────────────────────────────────────────────────────────┤
│ 阶段 4: 输出报告                                                 │
│   → 评审摘要（表格）+ 通过项 + 问题项（🔴P0 → 🟡P1 → 💡建议）   │
│   → 综合评分（10 分制）+ 修复建议                                 │
│   → 自动写入 reports/<子目录>/<agent>/YYYYMMDD/                   │
├──────────────────────────────────────────────────────────────────┤
│ 阶段 5: 更新记忆                                                 │
│   → 追加会话记录 + 关键发现 + 报告链接                            │
└──────────────────────────────────────────────────────────────────┘
```

### 产物输出目录

```
ai-dev-guidelines/projects/<项目>/reports/analysis/<agent>/YYYYMMDD/
└── NN-analysis-<中文描述>.md    ← 分析报告（自动写入 reports/）
```

> 深度分析类任务通常不产生代码产物，主要产出为分析报告。报告自动写入 `reports/analysis/` 目录。

### 多轮审查模式

当你要求"全面审查"或"深度分析"时，AI 会按维度分轮：

| 轮次 | 聚焦维度 | 检查内容 |
|:----:|---------|---------|
| 第 1 轮 | 数值一致性 | 版本号、条数、数量 — 全量对比 |
| 第 2 轮 | 内容一致性 | 格式、示例、引用 — 深度检查 |
| 第 3 轮 | 架构完整性 | 交叉引用、历史标记、链接有效性 |

### 深度分析专属验证

```yaml
分析质量:
  ✅ 数据采集: 所有相关文件已逐个读取（非推断）
  ✅ 交叉验证: 数值/格式/引用/标注 4 个维度已检查
  ✅ 问题分类: 按 🔴P0 → 🟡P1 → 💡建议 分级

报告完整性:
  ✅ 评审摘要表格已生成
  ✅ 通过项与问题项均已列出
  ✅ 综合评分（10 分制）已给出
  ✅ 修复建议具体可执行
```

```
ai-dev-guidelines/projects/<项目>/research/<中文描述>/
```

### 技术调研专属验证

```yaml
调研质量:
  ✅ 候选方案: 覆盖主流方案（≥ 3 个），无明显遗漏
  ✅ 对比维度: 包含性能/成本/维护性/社区/兼容性等关键维度
  ✅ 数据来源: 基于官方文档和实际测试，非主观臆断

选型验证:
  ✅ POC 验证: 首选方案已通过最小化验证（如适用）
  ✅ 风险评估: 已列出首选方案的已知风险和缓解措施
  ✅ 迁移路径: 已给出从当前方案到推荐方案的迁移步骤
```

## 场景 6：架构重构

> **触发关键词：** 重构、重写、改造、升级
> **工作流：** `core/workflows/05-refactoring/`

```
┌──────────────────────────────────────────────────────────────────┐
│ Step 1: 现状分析                                                 │
│   → 当前架构梳理、问题识别、技术债务评估                           │
│   → 输出现状分析报告                                              │
│   → ⏸️ CP1: 确认问题识别正确                                     │
├──────────────────────────────────────────────────────────────────┤
│ Step 2: 重构方案                                                 │
│   → 目标架构设计 + 重构策略（渐进/一步到位）+ 风险评估             │
│   → ⏸️ CP2: 确认重构方案                                        │
├──────────────────────────────────────────────────────────────────┤
│ Step 3: 迁移计划                                                 │
│   → 分阶段迁移步骤 + 每阶段的回滚方案 + 兼容性策略                │
│   → ⏸️ CP3: 确认迁移计划                                        │
├──────────────────────────────────────────────────────────────────┤
│ Step 4: 分阶段执行                                               │
│   → 按迁移计划逐阶段执行，每阶段完成后验证                        │
│   → 每阶段报告进度                                                │
├──────────────────────────────────────────────────────────────────┤
```

## 场景 11：规范自我审查

> **触发关键词**：自我审查、规范审查、spec audit、全面体检、规范健康检查
> **工作流路径**：`core/workflows/11-self-audit/`

规范自我审查是对规范体系自身的 **15 维度全量深度体检**。与通用的深度分析（场景 4）不同，它有固定的审查维度和标准，专门面向 ai-dev-guidelines 规范体系本身。

### 触发条件

1. **事件触发**：修改 `core/` 下规范文件后，约束 #14 交叉验证完成后自动追加
2. **消息触发**：用户发送"自我审查"等关键词
3. 每次触发都执行全部 15 维度，不分级不省略

### 15 维度审查体系

| 分组 | 维度 | 审查内容 |
|:---:|------|---------|
| A 定位与哲学 | Ⅰ 项目定位符合性 | 规范/机制/约束是否服务于"AI 工作操作系统"定位 |
| A 定位与哲学 | Ⅱ 设计原则遵守度 | 5 条设计原则是否被所有规范文件贯穿 |
| A 定位与哲学 | Ⅲ 架构层次完整性 | 6 层架构各层数据是否与实际一致 |
| B 数据与引用 | Ⅳ 数值一致性 | 版本号/约束条数/日期等跨文件数值 |
| B 数据与引用 | Ⅴ 引用完整性 | 内部链接/锚点/路径有效性 |
| B 数据与引用 | Ⅵ 语义一致性 | 同一概念在不同文件中的描述一致性 |
| C 规则与约束 | Ⅶ 约束体系合理性 | 20 条约束的逻辑矛盾/优先级/重叠/缺失 |
| C 规则与约束 | Ⅷ 工作流内部逻辑完整性 | 转入/转出规则/边界条件/互斥关系 |
| C 规则与约束 | Ⅸ 环境假设有效性 | 对工具/平台/模型能力的隐含假设是否成立 |
| D 运行与实效 | Ⅹ 机制有效性 | 核心机制在实际执行中是否被有效遵守 |
| D 运行与实效 | Ⅺ 防复现完备性 | 每个已知问题是否建立了防复现机制 |
| D 运行与实效 | Ⅻ 层间同步一致性 | 执行层与自维护层的规则双向同步 |
| E 演进与生态 | ⅩⅢ 复杂度健康度 | 文件行数/约束数/工作流数是否在健康范围 |
| E 演进与生态 | ⅩⅣ 演进路线一致性 | 版本历史/路线图与实际状态一致 |
| E 演进与生态 | ⅩⅤ 工具链与文档覆盖度 | 工具脚本和用户文档对规范体系的覆盖 |

### 约束 #8 合规

审查报告中每个发现/建议必须附带三项验证标注：

- **合理性**：基于实际文件验证，列出读取的文件和行号
- **可实施性**：修复方案已验证可落地
- **收益/必要性**：修复后的具体收益

### 规范自我审查专属验证

```yaml
审查质量:
  ✅ 数据采集: 所有 15 维度相关文件已逐个 read_file（非推断）
  ✅ 约束 #8 合规: 每条发现附带合理性+可实施性+收益三项验证标注
  ✅ 问题分级: 按 🔴 必须修复 → 🟡 建议修复 → 💡 观察项 → ❌ 不合理 分级

报告完整性:
  ✅ 总览表: 15 维度 × 状态 + 问题数 + 验证状态
  ✅ 详细分析: 各维度通过项 + 发现项 + 结论 + 三项验证
  ✅ 综合评分: 百分制 + 各维度分项
```
