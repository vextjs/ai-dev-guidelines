# AI 执行流程完整指南

> 当你向 AI 发送任务请求时，AI 会按照本文档描述的流程执行。
> 本指南整合了所有工作流的公共流程和各场景的专属流程，帮助你理解 AI 的每一步行为。

**最后更新**: 2026-03-02
**适用范围**: 所有任务类型（需求开发、Bug 修复、性能优化、深度分析等）

## 一、全局流程总览

无论你发送什么类型的任务请求，AI 的执行流程都遵循以下顺序：

```
┌─────────────────────────────────────────────────────────┐
│  你发送消息                                              │
│  ↓                                                      │
│  ① 预检查（5 项 + 记忆写入）                             │
│  ↓                                                      │
│  ② 任务类型识别 → 匹配对应工作流                          │
│  ↓                                                      │
│  ③ 合理性评估（有更好方案会先提出）                        │
│  ↓                                                      │
│  ④ 按工作流执行（含多个确认点，等你确认才继续）             │
│  ↓                                                      │
│  ⑤ 任务完成后轻量检测（反复问题检查 + 汇总问题）           │
│  ↓                                                      │
│  ⑤.5 执行合规性回溯检查（出口门禁 — 验证流程合规性）       │
│  ↓                                                      │
│  ⑥ 自动写入报告 + 更新记忆（不会问你）                    │
│  ↓                                                      │
│  ⑦ 输出结论摘要 + 报告路径                               │
└─────────────────────────────────────────────────────────┘
```

> **与实际流程对照说明**：步骤 ⑤ 对应 `core/self-fix/triggers/auto-triggers.md` 中的检测点 7（反复问题检查）和检测点 8（汇总问题），在"完成任务"节点注入。这不是全量自修复，而是轻量检测 — 只在发现与历史修复相似的问题时才触发升级分析。步骤 ⑤.5 对应检测点 11（执行合规性回溯检查 — 场景 10），在"任务结束前"节点注入，作为出口门禁与阶段 0（入口门禁）形成闭环。

**核心原则：**

| 原则 | 说明 |
|------|------|
| 先检查后执行 | 每次会话必须先完成预检查，才开始分析你的问题 |
| 先确认后修改 | 涉及文件修改/删除，AI 必须先输出变更计划等你确认 |
| 自动记录 | 报告和记忆的写入是自动行为，AI 不会询问你 |
| 中文输出 | 所有对话、报告、记忆使用中文（技术术语可保留英文） |

## 二、公共流程：每次会话都会执行

### 阶段 A：预检查（5 项必做 + 第 6 行硬性阻塞）

AI 收到你的第一条消息后，**必须先输出以下内容**，然后才能开始分析：

```
📋 预检查:
1. 工作区: E:\MySelf（当前工作目录）
2. 任务类型: 需求开发 / Bug 修复 / 性能优化 / 深度分析 / ...
3. 输出位置:
   报告: ai-dev-guidelines/projects/<项目>/reports/<子目录>/<agent>/YYYYMMDD/
   产物: ai-dev-guidelines/projects/<项目>/<任务类型>/<中文描述>/
4. Agent: zed-copilot / webstorm-copilot / vscode-copilot / cursor / ...
5. 上次记忆: .ai-memory/clients/<agent>/tasks/YYYYMMDD.md §会话NN 或 ⚠️ 无
6. 📝 记忆已创建: .ai-memory/clients/<agent>/tasks/YYYYMMDD.md §会话NN (🔄)
```

**你需要知道的：**

- **第 1~5 项**是信息采集（工作目录、任务类型、输出路径、编辑器标识、历史记忆）
- **第 6 行**是记忆写入确认 — AI 必须在这一步创建/追加记忆文件，然后才开始分析你的问题
- 如果你看不到第 6 行，说明 AI 的预检查没有完成，可以提醒它

**任务类型是怎么识别的？**

AI 通过 **4 层递进流程** 自动识别（v2.12.0）：

1. **Layer 1 精确匹配**：你明确说了任务类型（如"这是个 Bug"）
2. **Layer 2 关键词信号**：从你的消息中提取关键词作为候选信号
3. **Layer 3 意图推断（🔴 核心）**：通过三问判断你的真实目的：
   - Q1: 最终目的是**变更**还是**获得结论**？
   - Q2: "分析/评估/审查"是**手段**还是**目的**？
   - Q3: 是否有**具体要执行的事项**？
   - **任一问题指向变更 → 走需求开发（CP1→CP2→CP3），不会走深度分析**
4. **Layer 4 上下文校验**：结合项目上下文最终确认

关键词参考表（Layer 2 候选信号，最终由 Layer 3 意图推断决定）：

| 你说的关键词 | AI 识别为 | 对应工作流 |
|-------------|---------|-----------|
| 实现、开发、添加、新增、集成 | 需求开发 | `core/workflows/01-requirement-dev/` |
| 修复、解决、Bug、报错、异常 | Bug 修复 | `core/workflows/02-bug-fix/` |
| 优化、慢、性能、加速 | 性能优化 | `core/workflows/03-optimization/` |
| 调研、选型、对比、POC | 技术调研 | `core/workflows/04-research/` |
| 重构、重写、改造、升级 | 架构重构 | `core/workflows/05-refactoring/` |
| 数据库、表结构、迁移、Schema | 数据库变更 | `core/workflows/06-database/` |
| 安全、漏洞、加固、渗透 | 安全修复 | `core/workflows/07-security/` |
| 事故、故障、复盘、P0 | 事故复盘 | `core/workflows/08-incident/` |
| 开源、初始化、脚手架 | 开源项目初始化 | `core/workflows/09-opensource-init/` |
| 分析、审查、评估、深度分析 | 深度分析 | `core/workflows/10-analysis/` |
| 自查、自审、规范检查、self-audit | 规范自审 | `core/workflows/11-self-audit/` |
| 规范走查、场景验证、walkthrough、模拟用户 | 场景化走查 | `core/workflows/12-scenario-walkthrough/` |
| 继续、接着、上次、恢复 | 继续任务 | 恢复流程（见 [02-session-and-modes.md](./02-session-and-modes.md)） |

### 阶段 B：消息驱动记忆（5 阶段循环）

AI 的记忆机制以**消息事件**为锚点，贯穿整个会话：

```
你的第一条消息  →  [阶段 0] 预检查 + 创建记忆
                   [阶段 1] 捕获你的输入（记录到对话记录表）
                                 ↓
                   AI 分析、执行
                                 ↓
AI 回复你       ←  [阶段 2] 写入报告文件（完整内容）
                   [阶段 3] 更新记忆（变更清单 + 报告状态实时更新）
                                 ↓
你的第二条消息  →  [阶段 1] 捕获新输入
                                 ↓
                   AI 分析、执行
                                 ↓
AI 回复你       ←  [阶段 2+3] 写报告 + 更新记忆
                                 ↓
                   ...重复...
                                 ↓
任务即将结束    →  [阶段 4.5] 执行合规性回溯检查（出口门禁）  ← 🆕
                                 ↓
任务结束        →  [阶段 4] 最终状态更新（📝 进行中 → ✅ 已完成）
```

**典型顺序：** `0 → 1 → 2+3 → 1 → 2+3 → ... → 4.5 → 4`

**入口-出口闭环：** 阶段 0（入口门禁）确保会话开始时预检查+记忆写入不被跳过；阶段 4.5（出口门禁）确保会话结束前流程合规性不被遗忘。二者形成对称闭环。

**什么消息会被记录？**

| 消息类型 | 是否记录 |
|---------|---------|
| 包含新意图/新需求/方向修正的消息 | ✅ 记录 |
| 纯确认（"好的" / "Y" / "确认"） | ❌ 不记录 |
| 流程控制（"继续" / "下一步"） | ❌ 不记录 |

**报告与记忆的关系：**

```
报告文件（reports/）  = 完整分析内容（主体）
记忆文件（.ai-memory/）= 摘要 + 报告链接 + 对话记录（索引）
```

这意味着记忆文件很轻量，详细内容都在报告文件里。下次恢复时，AI 读记忆 → 跟链接读报告 → 完整上下文恢复。

### 阶段 C：确认点机制（CP1 / CP2 / CP3）

AI 在执行涉及代码修改的任务时，会在关键节点**暂停等待你确认**：

| 确认点 | 时机 | AI 输出什么 | 你需要做什么 |
|:-----:|------|-----------|------------|
| **CP1** | 理解需求后 / 分析原因后 | 需求理解摘要 / 根因分析 | 确认理解正确 → 说"确认" |
| **CP2** | 技术方案后 / 解决方案后 | 改动清单 + 架构设计 + 风险评估 | 确认方案可行 → 说"确认" |
| **CP3** | 实施方案后 | 每个文件的具体变更内容 + 实施前检查清单 | 确认所有变更 → 说"确认" |

**强制规则：**

- AI **不会**在同一条回复中同时输出"方案"和"执行结果"
- 即使需求很清晰，AI **也必须**输出需求理解并等待你确认
- 你说"确认"后 AI 才会进入下一阶段

**你可以在任何确认点说：**

- "确认" → 进入下一阶段
- "修改：xxx" → AI 调整后重新输出该阶段
- "取消" → 终止流程

---

### 阶段 D：报告自动输出

每次会话结束时，AI 会自动完成以下操作（**不会询问你**）：

1. **写入报告文件** → `projects/<项目>/reports/<子目录>/<agent>/YYYYMMDD/NN-<类型>-<简述>.md`
2. **更新记忆文件** → `projects/<项目>/.ai-memory/clients/<agent>/tasks/YYYYMMDD.md`
3. **对话中输出摘要** → 结论要点 + 报告文件路径

**报告文件存放位置示例：**

```
ai-dev-guidelines/projects/chat/
├── reports/
│   ├── analysis/
│   │   └── zed-copilot/
│   │       └── 20260228/
│   │           ├── 01-analysis-架构审查.md
│   │           └── 02-bug-登录超时.md
│   ├── bugs/
│   │   └── webstorm-copilot/
│   │       └── 20260228/
│   │           └── 01-bug-数据丢失.md
│   └── optimizations/
│       └── zed-copilot/
│           └── 20260228/
│               └── 01-opt-查询优化.md
└── .ai-memory/
    └── clients/
        └── zed-copilot/
            └── tasks/
                └── 20260228.md    ← 今天的记忆文件（内含 ## 会话 01、## 会话 02 ...）
```

### 阶段 E：任务完成后轻量检测 + 合规回溯检查

> 对应 `core/self-fix/triggers/auto-triggers.md` 中的检测点 7、检测点 8 和检测点 11

任务完成后（阶段 4 之前），AI 会自动执行以下检测：

| 检测点 | 检测内容 | 触发条件 |
|:------:|---------|---------|
| **检测点 7** | 反复问题检查 — 对比本次问题与 `core/self-fix/records/` 中的历史修复，同类 ≥ 2 次触发升级分析 | 每次任务完成 |
| **检测点 8** | 汇总问题 — 延迟汇报任务过程中发现的链接失效、格式问题、小优化建议 | 每次任务完成 |
| **检测点 11** | 🆕 执行合规性回溯检查（阶段 4.5 出口门禁）— 验证预检查完整性、确认点遵守、记忆/报告写入 | 每次任务结束前 |

**检测点 7 和 8 是轻量级的问题汇总**，不是全量健康检查。全量健康检查需要你主动说"运行规范健康检查"。

**检测点 11（阶段 4.5）是出口门禁**，与阶段 0（入口门禁）形成闭环。它检查 3 项必做内容：
1. 预检查完整性（第 6 行是否输出、记忆是否创建）
2. 确认点遵守（CP1→CP2→CP3 是否按序执行，纯分析任务可跳过）
3. 记忆/报告写入（是否写入报告、是否包含链接和对话记录）

如果检查未通过：可立即修复的项目（如报告未写入）会自动修复；无法修复的项目（如确认点已跳过）会记录到记忆待跟进，并在同类违规 ≥ 2 次时触发升级分析。

详见 [规范自我修复机制指南](../spec-self-fix-guide/README.md)。
