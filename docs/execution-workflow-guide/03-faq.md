## Q1：AI 为什么每次都先输出一大段预检查？

预检查是规范体系的核心保障。它确保 AI 知道当前工作区、任务类型、输出位置和历史记忆，避免走错方向或重复工作。第 6 行的记忆写入还确保了即使会话中途中断，至少有一份快照记录。

预检查通常只有 6 行，耗时很短。如果你觉得信息冗余，可以快速扫一眼确认没问题即可。

---

## Q2：AI 可以跳过确认点直接执行吗？

不可以。确认点（CP1/CP2/CP3）是强制的，即使你觉得需求很简单。这是为了防止 AI 理解偏差导致改错代码。你可以在确认点快速说"确认"来加速流程。

**绝对禁止的行为：**

- AI 认为"需求很清晰"就跳过 CP1
- AI 输出技术方案后不等确认就执行代码
- AI 在同一条回复中同时输出"方案"和"执行结果"

## Q3：报告和记忆有什么区别？

| 维度 | 报告文件（reports/） | 记忆文件（.ai-memory/） |
|------|--------------------|-----------------------|
| **定位** | 完整的分析内容（主体） | 索引 + 摘要（指针） |
| **内容** | 分析过程、数据、结论、建议 | 任务目标、状态、关键决策、报告链接、对话记录 |
| **存放** | `reports/<子目录>/<agent>/YYYYMMDD/` | `.ai-memory/clients/<agent>/tasks/` |
| **命名** | `NN-<类型>-<简述>.md` | `YYYYMMDD.md` |
| **体积** | 较大（完整内容） | 轻量（摘要+链接） |

下次恢复时，AI 先读记忆（快速回忆）→ 跟链接读报告（恢复详细上下文）。

## Q4：我可以让 AI 不写报告/记忆吗？

不可以。报告和记忆的写入是自动行为，规范明确禁止 AI 询问用户。这是为了确保每次会话都有可追溯的记录。

> 对应规范：约束 #17「报告+记忆自动输出」— 每次用户提问 + AI 完成分析/执行后，必须自动写入报告文件和更新记忆文件。

## Q5：多个编辑器的 AI 会冲突吗？

不会。每个编辑器（Zed、WebStorm、VS Code 等）有独立的 Agent 标识和独立的 `clients/<agent>/` 目录。它们的报告序号也是独立编号的。

| Agent 标识 | 编辑器 |
|-----------|--------|
| `zed-copilot` | Zed + Copilot/Claude |
| `webstorm-copilot` | WebStorm + Copilot |
| `vscode-copilot` | VS Code + Copilot |
| `cursor` | Cursor |

> 对应规范：约束 #10「多 Agent 目录隔离」+ 约束 #19「报告/记忆序号独立」。

## Q6：Token 耗尽了怎么办？

AI 遵循"先写后干"原则 — 开始执行前先创建初始记忆，过程中在每个消息事件增量更新。所以即使 Token 中途耗尽，你下次说"继续"时 AI 可以从最近的快照恢复。

**恢复流程：**

1. 新会话中说"继续"
2. AI 自动扫描当前 Agent 的 `.ai-memory`，找到 🔄 状态的未完成任务
3. 读取记忆 + 关联报告，恢复完整上下文
4. 输出恢复摘要，等你确认后从断点继续

详见 [02-session-and-modes.md](./02-session-and-modes.md) §跨会话恢复。

## Q7：文件修改需要我确认，报告/记忆写入不需要？

是的。涉及你项目源码的文件修改/删除必须先输出变更计划等你确认。但报告和记忆是 AI 的工作记录，写入 `ai-dev-guidelines/projects/` 目录下（不是你的源码目录），属于自动行为。

**需要确认的操作：**

- 创建/修改/删除你的源码文件
- Git 操作（commit、push、merge 等）
- 安装/卸载依赖

**不需要确认的操作：**

- 写入报告文件（`reports/` 目录）
- 更新记忆文件（`.ai-memory/` 目录）
- 读取任何文件（只读操作）

> 对应规范：约束 #16「文件修改需确认」+ 约束 #17「报告+记忆自动输出」。

## Q8：什么是"合理性评估"？

AI 收到你的请求后，会先评估请求的合理性。如果 AI 认为有更好的方案或你的请求可能有问题，它会先提出建议让你确认，而不是盲目执行。

**三种处理方式：**

| 情况 | AI 行为 |
|------|--------|
| 合理且无更好建议 | 直接执行，无需额外输出 |
| 有更好建议 | 先提出建议，等你确认后再执行 |
| 明显不合理 | 说明原因，给出替代方案 |

> 对应规范：约束 #12「主动合理性分析」。

## Q9：任务完成后 AI 会自动运行自我修复吗？

**不会运行全量自我修复。** 任务完成后 AI 只会执行两个轻量检测点：

| 检测点 | 内容 | 说明 |
|:------:|------|------|
| 检测点 7 | 反复问题检查 | 对比本次问题与修复历史，同类 ≥ 2 次才触发升级分析 |
| 检测点 8 | 汇总问题 | 延迟汇报任务过程中发现的链接失效、格式问题等 |

全量健康检查（遍历所有规范文件、执行 4 类检测、生成问题报告）需要你主动触发：

- 说"运行规范健康检查"
- 说"检查版本号是否一致"
- 说"验证规范一致性"

> 详见 [规范自我修复机制指南](../spec-self-fix-guide/README.md)。
> 对应规范：`core/self-fix/triggers/auto-triggers.md` §检测点注入 + §检测频率。

## Q10：工作流有多少种？我怎么知道 AI 用的是哪个？

目前有 **11 个工作流**（编号 00-10）。AI 在预检查第 2 项"任务类型"中会明确告诉你它识别为什么类型，以及对应哪个工作流。

| 编号 | 工作流 | 触发关键词 |
|:----:|--------|-----------|
| 00 | 预检查 + 任务识别 | （自动执行，不需要你触发） |
| 01 | 需求开发 | 实现、开发、添加、新增、集成 |
| 02 | Bug 修复 | 修复、解决、Bug、报错、异常 |
| 03 | 性能优化 | 优化、慢、性能、加速 |
| 04 | 技术调研 | 调研、选型、对比、POC |
| 05 | 架构重构 | 重构、重写、改造、升级 |
| 06 | 数据库变更 | 数据库、表结构、迁移、Schema |
| 07 | 安全修复 | 安全、漏洞、加固、渗透 |
| 08 | 事故复盘 | 事故、故障、复盘、P0 |
| 09 | 开源项目初始化 | 开源、初始化、脚手架 |
| 10 | 深度分析 | 分析、审查、评估、深度分析 |

如果 AI 识别错了任务类型，你可以直接告诉它正确的类型，AI 会重新匹配工作流。

详见 [01-task-workflows.md](./01-task-workflows.md)。
