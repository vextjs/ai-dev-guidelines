## ⚙️ CI/CD 流程概述

### 核心概念

```
CI (持续集成):
  - 代码提交 → 自动运行测试 → 检查质量门禁 → 反馈结果
  - 频率: 每次 commit 或 PR
  - 目标: 快速发现问题

CD (持续部署):
  - 代码通过测试 → 自动部署到测试环境 → 自动部署到生产
  - 需要人工批准 (建议不要自动部署到生产)
  - 目标: 快速交付，降低风险
```

### 流程示意图

```
代码提交
   ↓
[CI] 自动运行测试和检查
   ├─ Linting (代码风格)
   ├─ TypeScript (类型检查)
   ├─ 单元测试
   ├─ 代码质量 (SonarQube)
   └─ 安全扫描
   ↓
[质量门禁] 是否通过所有检查?
   ├─ ✅ 通过 → 合并代码
   └─ ❌ 失败 → 反馈给开发者
   ↓
[CD] 部署到测试环境
   ├─ 拉取代码
   ├─ 构建镜像
   └─ 部署到测试环境
   ↓
[CD] 部署到生产环境
   ├─ 人工审批 (推荐)
   ├─ 构建镜像
   ├─ 灰度/蓝绿部署
   └─ 监控和验证
   ↓
[监控] 持续监控运行状态
   ├─ 错误率
   ├─ 性能指标
   └─ 基础设施健康度
```

## 🤔 关键决策点

### 决策 1: 自动部署政策

选择一个最适合项目的部署策略：

```yaml
选项 A: 手动部署 (推荐，保守)
  ✅ 适用: 生产环境、金融/支付系统
  说明: 测试通过后，人工手动触发部署
  优点:
    - 控制完全，风险最低
    - 可选择部署时间
    - 有充分的审查机会
  缺点:
    - 流程较慢，交付周期长
    - 依赖人力，容易遗漏

选项 B: 混合策略 (推荐)
  ✅ 适用: 大多数项目
  说明: 测试/Staging 自动部署，生产手动
  优点:
    - 测试快速，生产保守
    - 平衡速度和风险
    - 循序渐进验证
  缺点:
    - 两套流程，配置稍复杂

选项 C: 完全自动化 (激进，不推荐)
  ⚠️ 谨慎使用
  说明: 所有环境自动部署
  优点:
    - 最快交付，流程最简洁
  缺点:
    - 风险高，出问题影响大
    - 难以跟踪问题根源
    - 不适合关键系统

【建议】: 大多数项目选择 选项 B (混合策略)
```

### 决策 2: 自动化检查范围

```yaml
【必须项】(所有项目):
  ☑ 代码风格检查 (Linting)
  ☑ 单元测试
  ☑ 代码质量扫描
  ☑ 依赖安全扫描

【强烈推荐】:
  ☑ 类型检查 (如使用 TypeScript)
  ☑ 集成测试 (如有)
  ☑ 覆盖率检查 (>= 80%)

【可选项】:
  ☐ 性能测试 (基准对比)
  ☐ 端到端测试
  ☐ 安全深度扫描 (SAST/DAST)
  ☐ 静态分析 (SonarQube 深度)
```

### 决策 3: 部署策略

```yaml
选择生产环境部署方式:

蓝绿部署 (Blue-Green)
  ✅ 原理: 两套完整环境轮流切换
  优点:
    - 快速切换，回滚简单
    - 零停机
  缺点:
    - 资源消耗大 (维护两套环境)
    - 数据库迁移复杂
  适用: 有充足资源的项目

灰度发布 (Canary)
  ✅ 原理: 逐步放量，先 5% → 25% → 100%
  优点:
    - 风险最小，发现问题快
    - 资源消耗少
  缺点:
    - 需要完整的灰度框架
    - 监控要求高
  适用: 大型项目，用户众多

金丝雀部署 (Canary 变体)
  ✅ 原理: 同时运行新旧版本，逐步切换流量
  优点:
    - 可实时对比，快速发现问题
    - 支持自动回滚
  缺点:
    - 配置复杂
    - 需要 Istio/Envoy 等工具
  适用: 微服务架构

直接部署 (Rolling)
  ❌ 原理: 逐个替换实例，一次性全量
  优点:
    - 简单，资源消耗少
  缺点:
    - 风险高，不可快速回滚
    - 可能有短暂停机
  适用: 小型项目

【建议】: 大多数项目选择 灰度发布 或 金丝雀部署
```

### 决策 4: 回滚策略

```yaml
问题检测:
  □ 如何判断部署失败?
    - 监控告警触发
    - 错误率超过阈值
    - 关键服务不可用
  
  □ 检测延迟:
    - 即时检测: 自动告警
    - 延迟检测: 部署后 5-15 分钟检查
  
  □ 回滚时机:
    - 自动: 检测失败立即回滚
    - 手动: 人工审核后回滚

回滚方式:
  □ 自动回滚
    需要配置健康检查和告警规则
  
  □ 手动回滚
    保留回滚脚本，快速操作
  
  □ 混合
    金丝雀失败 → 自动回滚
    全量发布失败 → 人工回滚

【建议】: 
  - 灰度阶段 (0-25%) 自动回滚
  - 全量阶段 (25-100%) 手动回滚，严格审批
```

## 📋 快速决策表

```yaml
问题 1: 项目规模?
  ├─ 小型 (< 50 人) → 选择 选项 B (混合部署)
  ├─ 中型 (50-200 人) → 选择 选项 B + 灰度发布
  └─ 大型 (> 200 人) → 选择 选项 B + 金丝雀部署

问题 2: 风险容忍度?
  ├─ 低 (金融/医疗) → 选择 蓝绿部署 + 手动回滚
  ├─ 中 (普通系统) → 选择 灰度发布 + 混合回滚
  └─ 高 (实验项目) → 选择 直接部署 + 自动回滚

问题 3: 是否有 24/7 值班?
  ├─ 有 → 可用自动回滚
  └─ 无 → 必须手动回滚，严格审批

问题 4: 使用容器化吗?
  ├─ 是 (Docker/K8s) → 推荐灰度部署
  └─ 否 (虚拟机) → 推荐蓝绿部署
```
