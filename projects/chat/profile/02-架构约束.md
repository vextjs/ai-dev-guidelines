# 02 - 架构约束

> **模块**: 架构强制规范（8 条禁止项）  
> **重要性**: 🔴 P0 - Profile 的核心功能，优先级高于代码风格  
> **适用对象**: AI 助手必须严格遵守，禁止按原有项目代码模式执行  
> **版本**: v1.0  
> **最后更新**: 2026-02-24  
> **视角**: 本文档使用"你"指代 AI 助手（规范执行者）

---

## 🚫 架构约束（强制）

### 1. 禁止在 Controller 层直接操作数据库

**原因**: Controller 应仅处理请求和响应，业务逻辑和数据访问应在 Service 层

```typescript
// ❌ 错误
const trip = await ctx.model.Trip.findOne({ _id: tripId });

// ✅ 正确
const trip = await ctx.service.trip.tripService.getTripById(tripId);
```

---

### 2. 禁止在 Service 层使用 ctx.request/ctx.response

**原因**: Service 应独立于 HTTP 层，便于测试和复用

**⚠️ 例外情况**: SSE/文件上传/流式响应场景需注释 `// EXCEPTION: [原因]`

---

### 3. 禁止绕过中间件直接返回响应

**原因**: 统一响应格式由 `responseHelper` 中间件处理

```typescript
// ❌ 错误
ctx.body = { code: 0, data: result };

// ✅ 正确
ctx.success(result);
```

---

### 4. 禁止在 Model 层编写业务逻辑

**原因**: Model 仅定义数据结构和验证规则，业务逻辑应在 Service 层

---

### 5. 禁止硬编码数据库查询字段

**原因**: 字段名可能变化，必须基于实际数据库 Schema 编写代码

---

### 6. 禁止跳过 Service 层直接操作 Model

**原因**: Controller 必须通过 Service 层调用

```typescript
// ❌ 错误（Controller 直接操作 Model）
const project = await ctx.model.TripProject.create(data);

// ✅ 正确
const project = await ctx.service.trip.tripProjectService.createTripProject(data);
```

---

### 7. 禁止使用 throw new Error() 抛出业务错误

**原因**: 必须使用 schema-dsl 的错误处理，确保错误码和多语言支持

```typescript
// ❌ 错误
throw new Error('User not found');
throw new ExError('FAILED');
throw ctx.fail('ERROR_CODE', 404);

// ✅ 正确（业务错误，在 try 块中）
ctx.dsl.error.throw('USER_NOT_FOUND');

// ✅ 正确（系统错误，在 catch 块中）
throw ctx.dsl.error.create('QUERY_FAILED');
```

**详见**: [04-错误处理与多语言.md](./04-错误处理与多语言.md)

---

### 8. 禁止硬编码数据库连接字符串

**原因**: 通过 Nacos 配置中心管理

---

## ⚠️ 例外情况

以下场景允许例外：

1. **快速原型开发**（明确标注 `// TODO: 重构到 Service 层`）
2. **性能优化**（经过分析确认性能瓶颈在层级调用）
3. **紧急修复**（必须在后续版本中重构）

所有例外必须：
- 在代码中添加 `// EXCEPTION: [原因]` 注释
- 在下个迭代中优先重构
