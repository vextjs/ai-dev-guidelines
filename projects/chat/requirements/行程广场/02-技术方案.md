# 行程广场技术方案

## 零、需求概览

> 详细需求见 [01-需求定义.md](./01-需求定义.md)，本节仅提炼技术方案所需的核心背景。

### 业务目标

为行程售卖功能新增一个公开的「行程广场」入口，让任何人（含未登录访客）都能发现和浏览已上架的行程；已登录用户可点赞，卖家可灵活控制行程是否上架。

### 需求清单

| # | 需求 | 一句话描述 | 优先级 |
|:-:|------|---------|:------:|
| REQ-1 | 上架控制 | 售卖配置新增 `listed_in_square` 字段，卖家自主决定是否展示到广场 | P0 |
| REQ-2 | 免费行程 | 售卖价格支持设为 0；price=0 时公开预览展示全部内容；payment 侧拦截无意义支付 | P0 |
| REQ-3 | 广场列表 | 单一公开接口，支持关键词搜索（标题+城市）和分页；携带 Token 时自动返回点赞状态 | P0 |
| REQ-4 | 点赞功能 | 登录用户可点赞/取消，支持查询状态和查看自己的点赞列表；接口有速率限制 | P1 |
| REQ-5 | 搜索排序 | 按命中情况打分排序：同时命中 > 只命中城市 > 只命中标题；同分按点赞量→时间降序 | P1 |

### 关键约束

- 广场列表和点赞接口均放 `/home/*` 路由组**无鉴权分组**，与项目现有访客模式保持一致
- 鉴权机制依赖全局 `visitorLimit` 中间件：解析 Authorization header → 设置 `ctx.state.user`，广场列表 Controller 根据 `ctx.state.user` 是否存在决定是否注入 `liked` 字段
- 前端统一携带 token（访客 token 或注册用户 token），不存在"无 token"访问场景
- `trip_likes` 采用**事件日志模式**：每次点赞/取消各插入一条记录（`type: 'like'｜'unlike'`），不删除记录
- 点赞/取消操作使用 **MongoDB 事务**：保证 `trip_likes` 写入与 `like_count` 更新的原子性
- 所有查询使用 `ctx.msq`（monSQLize），写入使用 `ctx.model`（mongoose），聚合用 `ctx.model.aggregate()`
- `like_count` 为冗余计数字段，接受最终一致性，不要求强一致
- 卖家信息实时从 user 服务获取，不读行程快照字段

### 1.3 数据流

**广场列表请求（visitorLimit 全局解析 token）**

```
[客户端请求（前端统一携带 token：visitor_token 或 user_token）]
    │
    ▼
visitorLimit（全局中间件）
    ├─ 解析 Authorization header → JWT verify
    ├─ [visitor token]    → ctx.state.user = { _id, user_type: 'visitor' }
    ├─ [registered token] → ctx.state.user = { _id, user_type: 'registered' }
    └─ [无 token / 无效]   → ctx.state.user = null，放行
    │
    ▼
GET /home/trip_square/list（home 无鉴权分组，无额外中间件）
    │
    ▼
TripSquareController.list()
    │  参数验证：keyword?, page, page_size
    ▼
TripSquareService.list()（ctx.msq 查询）
    │  1. 构建过滤条件
    │  2. 有 keyword → 计算匹配评分
    │  3. 复合排序（匹配评分 → 点赞量 → 时间）
    │  4. 分页查询
    │  5. 批量获取卖家信息（调 user 服务）
    ▼
[ctx.state.user 有值] → TripLikeService.batchGetStatus()（聚合取最新 type）→ 注入 liked 字段
[ctx.state.user 为空] → 跳过点赞状态查询
    │
    ▼
返回：列表 + total + seller_info [+ liked（有 token 时）]
```

**点赞请求**

```
[客户端请求（携带 visitor_token 或 user_token）]
    │
    ▼
visitorLimit（全局中间件）
    → ctx.state.user = { _id, user_type } 已设置
    │
    ▼
POST /home/trip_like/:tripId（home 无鉴权分组）
    │
    ▼
flexRateLimit — Key: userId + tripId + path，60s 内最多 10 次
    │
    ▼
TripLikeController.like()
    │  手动校验 ctx.state.user 存在（无 token 请求返回 401）
    ▼
TripLikeService.like(tripId, userId)
    │  1. 校验行程存在且已上架
    │  2. 开启 MongoDB 事务
    │  3. 查最新记录 type，判断幂等
    │  4. 事务内：写入 trip_likes（type:'like'）+ 原子 +1 like_count
    │  5. 提交事务
    ▼
返回：{ liked: true }
```

## 二、数据模型设计

### 2.1 `trips` 集合变更（app/model/trip.ts）

#### 2.1.1 sell_config 新增字段

| 字段 | 类型 | 默认值 | 说明 |
|------|------|:------:|------|
| `sell_config.listed_in_square` | Boolean | `false` | 是否上架到广场 |

#### 2.1.2 sell_config 现有字段修改

| 字段 | 原约束 | 新约束 | 原因 |
|------|:------:|:------:|------|
| `sell_config.price` | min: 0.99 | min: 0 | 支持免费行程 |
| `sell_config.credit_price` | min: 1 | min: 0 | 支持免费行程 |

#### 2.1.3 顶层新增字段

| 字段 | 类型 | 默认值 | 说明 |
|------|------|:------:|------|
| `like_count` | Number | `0` | 点赞数冗余字段，用于排序加速，由点赞/取消点赞操作原子维护 |

#### 2.1.4 新增索引

| 索引字段 | 类型 | 用途 |
|---------|------|------|
| `sell_config.enabled` + `sell_config.listed_in_square` + `del_flag` + `like_count` | 复合索引（降序） | 广场列表基础过滤 + 点赞量排序 |

### 2.2 `trip_likes` 集合（新建 app/model/trip_like.ts）

> **事件日志模式**：每次点赞/取消各插入一条新记录，不删除旧记录。判断当前是否已点赞，取该用户对该行程的**最新一条记录**的 `type` 值。

| 字段 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `trip_id` | ObjectId | ✅ | 关联行程 |
| `user_id` | ObjectId | ✅ | 操作用户（支持 visitor 和 registered） |
| `type` | String | ✅ | 操作类型：`like`（点赞）/ `unlike`（取消点赞） |
| `del_flag` | Number | ✅ | 软删除标记，默认 0 |
| `created_at` | Date | — | 操作时间（自动，作为事件时间戳） |
| `updated_at` | Date | — | 更新时间（自动） |

**索引设计**

| 索引 | 类型 | 说明 |
|------|------|------|
| `{ user_id: 1, trip_id: 1, created_at: -1 }` | 普通复合索引 | 核心索引：支持"取某用户对某行程的最新记录"查询 |
| `{ user_id: 1, created_at: -1 }` | 普通复合索引 | 支持用户点赞列表聚合查询 |

> ⚠️ **不再使用唯一联合索引**：事件日志模式下，同一对 `{user_id, trip_id}` 会存在多条记录（每次操作一条），唯一性由事务内的幂等判断保证。

### 2.3 类型声明（新建 typings/entity/trip_like.d.ts）

新增 `ITripLike` 接口，包含以上所有字段的 TypeScript 类型定义，供 Service/Controller 层引用。`type` 字段对应枚举 `TripLikeTypeEnum { Like = 'like', Unlike = 'unlike' }`。

```yaml
请求参数 (Query):
  keyword:
    type: string
    required: false
    description: 搜索关键词，同时模糊匹配行程标题和目的地城市，不区分大小写

    required: false
    default: 1
    min: 1
  page_size:
    type: integer
    required: false
    default: 20
    min: 1
    max: 50

响应 (200):
  code: 200
  data:
    total: number          # 符合条件的总数
    list:
      - _id: string
        trip_name: string
        trip_cover: string
        trip_brief: string
        destination: string
        start_date: string
        end_date: string
        like_count: number
        sell_config:

          nick_name: string
```

## 四、模块设计

### 4.1 TripSquareService（新建）

**文件路径**: `app/service/trip/TripSquareService.ts`

**职责**: 行程广场列表查询、复合排序、批量获取卖家信息

| 方法 | 参数 | 返回 | 说明 |
|------|------|------|------|
| `list` | `keyword?, page, pageSize` | `{ total, list }` | 广场列表查询，含复合排序和卖家信息 |

**核心流程**:
1. 构建基础过滤：`del_flag=0` + `sell_config.enabled=true` + `sell_config.listed_in_square=true`
2. 有 `keyword` 时，对标题和城市分别计算命中，生成匹配评分字段用于排序（评分规则见第五章）
3. 按排序规则排序后分页取数据
4. 提取所有行程的 `owner_id`，批量调用 user 服务获取卖家信息
5. 将卖家信息合并到对应行程记录返回

**查询规范**: 分页查询使用 `this.ctx.msq.collection('trips').find(...)`；评分排序涉及 Aggregation Pipeline 时使用 `this.ctx.model.Trip.aggregate(...)`

**跨服务依赖**: `POST /internal/users/batch`（user 服务），降级方案为逐个查询

### 4.2 TripLikeService（新建）

**文件路径**: `app/service/trip/TripLikeService.ts`

**职责**: 点赞 CRUD 操作、点赞状态批量查询、用户点赞列表

| 方法 | 参数 | 返回 | 说明 |
|------|------|------|------|
| `like` | `tripId, userId` | `{ liked: true }` | 点赞，事务内幂等写入 + 原子 +1 |
| `unlike` | `tripId, userId` | `{ liked: false }` | 取消点赞，事务内幂等写入 + 原子 -1（不低于 0） |
| `getStatus` | `tripId, userId` | `{ liked: boolean }` | 查询单行程点赞状态（取最新记录 type） |
| `batchGetStatus` | `tripIds[], userId` | `Set<tripId>` | 批量查询已点赞行程集合，供广场列表注入 liked 字段 |
| `listByUser` | `userId, page, pageSize` | `{ total, list }` | 查询用户点赞列表，含行程详情和卖家信息 |

**关键设计约束（事件日志模式）**:

| 操作 | 实现方式 |
|------|---------|
| `like` 幂等判断 | 事务内先查 `{ user_id, trip_id }` 最新记录：type='like' → 已点赞，直接返回；type='unlike' 或无记录 → 插入新记录 type='like' + $inc like_count +1 |
| `unlike` 幂等判断 | 事务内先查最新记录：type='unlike' 或无记录 → 已取消，直接返回；type='like' → 插入新记录 type='unlike' + $inc like_count -1（$gt: 0 保护） |
| `getStatus` 查询 | `ctx.msq` 查 `{ user_id, trip_id }` 最新一条记录，判断 type |
| `batchGetStatus` 批量查询 | `ctx.model.TripLike.aggregate()`：按 `{ user_id, trip_id }` 分组，取每组 `created_at` 最大的记录，过滤 type='like'，返回命中的 trip_id 集合 |
| `listByUser` 聚合查询 | 同上分组逻辑，过滤 type='like' 后 join trips 集合获取行程详情和卖家信息 |

**查询规范**: `getStatus` 用 `ctx.msq`；`batchGetStatus` 和 `listByUser` 涉及分组聚合，用 `ctx.model.TripLike.aggregate()`；写入用 `ctx.model.TripLike.create()`

## 五、搜索与排序规则

### 5.1 搜索行为

广场列表只有一个搜索框（对应 `keyword` 参数），输入的关键词会**同时**在行程标题和目的地城市中搜索，两者都不区分大小写，也不要求精确匹配（包含即算命中）。

例如，搜索「Tokyo」，标题含「Tokyo」的行程和目的地含「Tokyo」的行程都会出现在结果中。

> **安全注意**：keyword 参数在传入数据库查询前需做正则特殊字符转义，防止正则注入攻击。

### 5.2 无关键词时的排序

不输入搜索词时，广场展示**所有已上架行程**，排序规则为：

1. **点赞量多的在前**（点赞量相同则继续比较下一条规则）
2. **发布时间晚的在前**

### 5.3 有关键词时的排序

输入关键词后，系统会根据每条行程"命中了什么"来打分，分数越高排越前：

| 命中情况 | 得分 | 举例 |
|---------|:----:|------|
| 标题和城市都命中 | 3 分 | 搜索「Tokyo」，标题含「Tokyo Tour」且目的地也是「Tokyo」|
| 只有城市命中 | 2 分 | 搜索「Tokyo」，目的地是「Tokyo」，标题不含 |
| 只有标题命中 | 1 分 | 搜索「Tokyo」，标题含「My Tokyo Trip」，目的地不是 Tokyo |
| 两者都未命中 | 0 分 | 不出现在结果中 |

**同分情况下的次级排序**：得分相同的行程之间，仍按「点赞量多的在前 → 发布时间晚的在前」排列。

### 5.4 排序规则汇总表

| 场景 | 第一排序 | 第二排序 | 第三排序 |
|------|---------|---------|---------|
| 无关键词 | 点赞量（多→少） | 发布时间（新→旧） | — |
| 有关键词 | 匹配得分（高→低） | 点赞量（多→少） | 发布时间（新→旧） |

> **实现说明**：使用 MongoDB Aggregation Pipeline 实现评分计算和复合排序，具体代码在实施方案中提供。

## 六、速率限制设计

### 6.1 接入方案

chat 服务当前未集成 `flex-rate-limit`，需要：

1. 安装依赖：`flex-rate-limit`（参考 user 服务版本）
2. 新建中间件文件 `app/middleware/flexRateLimit.ts`（从 user 服务复制适配）
3. 配置 Redis 存储（复用 chat 服务现有 Redis 配置）
4. 在 `app/config/` 中新增 `flexRateLimit` 配置项

### 6.2 限流规则

**适用接口**: 点赞 + 取消点赞（高频写操作）

| 参数 | 值 | 说明 |
|------|:--:|------|
| 时间窗口 | 60 秒 | 固定窗口算法 |
| 最大次数 | 10 次 | 每用户每行程 60s 内不超过 10 次 |
| 算法 | fixed-window | 适合突发防护场景 |

**限流 Key 构成**：

点赞操作的 Key 由以下三部分拼接，确保限流粒度精确到「某用户对某行程的操作」：

| 维度 | 来源 | 说明 |
|------|------|------|
| 用户 ID | `ctx.state.user._id` | 区分不同用户 |
| 行程 ID | 路由参数 `tripId` | 区分对不同行程的操作 |
| 请求路径 | `ctx.path` | 区分点赞和取消点赞（同一行程两类操作分别计数） |

> **为什么需要三个维度**：仅用 IP 会误伤 NAT 后的多用户；仅用用户 ID 会让用户无法同时对多个行程点赞；加入 tripId 后，限流粒度精确到「用户 + 行程」，更合理。

### 6.3 错误码

需在 DSL 多语言配置中新增：

| 错误码 | 中文提示 | 英文提示 |
|--------|---------|---------|
| `rate_limit_exceeded` | 操作过于频繁，请稍后再试 | Too many requests, please try again later |

| 广场列表（无 keyword） | < 100ms | 复合索引支持基础过滤 + 点赞量排序 |

| 点赞/取消（含事务） | < 100ms | `{ user_id, trip_id, created_at: -1 }` 复合索引支持最新记录查询 |
| 用户点赞列表 | < 150ms | `{ user_id, created_at: -1 }` 复合索引 + 聚合分组 |
| batchGetStatus（聚合） | < 50ms | `{ user_id, trip_id, created_at: -1 }` 复合索引；page_size ≤ 50 控制数量 |

### 10.3 回滚方案

1. Model 字段均为可选且有默认值，新增字段不影响现有数据和查询
2. `trip_likes` 集合独立存在，删除不影响其他功能
3. 新增路由不影响现有接口；广场和点赞路由放无鉴权分组，不影响现有有鉴权路由
4. `visitorLimit` 无需修改，新路径不在其限制规则内
5. `flex-rate-limit` 仅在路由级别挂载，移除中间件即可回滚限流
6. 如需回滚 price 验证规则，恢复 DSL min 值即可

```
chat → user（已有模式，广场列表复用）:
  ├── GET  /internal/user/:userId          — 单个卖家信息（降级方案）

      返回字段: nick_name, portrait, email, phone, link, introduction, social_media

  └── GET  /internal/trip/sell-config/:tripId — 创建订单时验证售卖状态

  ├── GET  /credit/tradable                — 查询买家可用积分
```
