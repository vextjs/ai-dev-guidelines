## 四、模块设计

### 4.1 TripSquareService（新建）

**跨服务调用**：
- 通过 `POST /internal/users/batch` 批量获取卖家信息（复用 `open/TripProjectController.fetchBatchSellerInfo` 已有模式）
- 返回字段：`nick_name`, `portrait`, `introduction`
- 降级方案：批量接口失败时逐个查询（`fetchSellerInfoFallback`）

**文件路径**: `app/service/trip/TripSquareService.ts`

**职责**: 广场列表查询 + 复合排序逻辑（REQ-3 + REQ-5）

**方法设计**:

| 方法 | 签名 | 说明 |
|------|------|------|
| `list` | `list(params: { keyword?: string; city?: string; page: number; pageSize: number }): Promise<{ total: number; list: any[] }>` | 广场列表查询，含复合排序 |

**核心逻辑**:

1. 构建基础 `$match` 过滤条件：`del_flag=0`, `sell_config.enabled=true`, `sell_config.listed_in_square=true`
2. `city` 参数存在时追加 `destination` 正则匹配到基础过滤
3. `keyword` 参数存在时：
   - `$addFields` 计算 `_title_match`（`$regexMatch` 匹配 `trip_name`）
   - `$addFields` 计算 `_city_match`（`$regexMatch` 匹配 `destination`）
   - `$addFields` 计算 `_match_score`（`$switch`：双命中=3, 城市=2, 标题=1, 无=0）
   - `$match` 过滤 `_match_score > 0`
4. `$sort`：有 keyword 时 `_match_score DESC` → `like_count DESC` → `created_at DESC`；无 keyword 时 `like_count DESC` → `created_at DESC`
5. 使用 `$facet` 同时获取 `total`（`$count`）和 `list`（`$skip` + `$limit`）
6. `$project` 仅返回需要的字段

### 4.2 TripLikeService（新建）

**文件路径**: `app/service/trip/TripLikeService.ts`

**职责**: 点赞 CRUD 操作（REQ-4）

**方法设计**:

| 方法 | 签名 | 说明 |
|------|------|------|
| `like` | `like(tripId: string, userId: string): Promise<{ liked: boolean }>` | 点赞 |
| `unlike` | `unlike(tripId: string, userId: string): Promise<{ liked: boolean }>` | 取消点赞 |
| `getStatus` | `getStatus(tripId: string, userId: string): Promise<{ liked: boolean }>` | 查询点赞状态 |

**核心逻辑**:

#### like(tripId, userId)

1. 查询 Trip 是否存在：`Trip.findOne({ _id: tripId, del_flag: 0, 'sell_config.enabled': true, 'sell_config.listed_in_square': true })`
2. 不存在 → 抛出业务错误（`trip_not_found` 或 `trip_not_listed`）
3. `TripLike.findOneAndUpdate({ trip_id: tripId, user_id: userId }, { $setOnInsert: { trip_id: tripId, user_id: userId } }, { upsert: true, new: true, rawResult: true })`
4. 通过 `rawResult.lastErrorObject.updatedExisting` 判断：
   - `false`（新插入）→ `Trip.updateOne({ _id: tripId }, { $inc: { like_count: 1 } })` 
   - `true`（已存在）→ 幂等，不操作
5. 返回 `{ liked: true }`

#### unlike(tripId, userId)

1. `TripLike.findOneAndDelete({ trip_id: tripId, user_id: userId })`
2. 删除成功（返回非 null）→ `Trip.updateOne({ _id: tripId, like_count: { $gt: 0 } }, { $inc: { like_count: -1 } })`
3. 删除失败（返回 null，说明本来就没点赞）→ 幂等，不操作
4. 返回 `{ liked: false }`

> `like_count: { $gt: 0 }` 保护条件确保 like_count 不会变为负数。

#### getStatus(tripId, userId)

1. `TripLike.findOne({ trip_id: tripId, user_id: userId, del_flag: 0 })`
2. 存在 → `{ liked: true }`；不存在 → `{ liked: false }`

## 五、复合排序算法设计（REQ-5）

### 5.1 算法概述

使用 MongoDB Aggregation Pipeline 的 `$addFields` + `$switch` 实现评分排序。

### 5.2 Pipeline 设计

```typescript
const pipeline: any[] = [];

// Step 1: 基础过滤
const baseMatch: Record<string, any> = {
    del_flag: DelFlagEnum.Normal,
    'sell_config.enabled': true,
    'sell_config.listed_in_square': true,
};
if (city) {
    baseMatch.destination = { $regex: city, $options: 'i' };
}
pipeline.push({ $match: baseMatch });

// Step 2: 有 keyword 时计算匹配评分
if (keyword) {
    const escapedKeyword = keyword.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');

    pipeline.push({
        $addFields: {
            _title_match: {
                $regexMatch: { input: '$trip_name', regex: escapedKeyword, options: 'i' }
            },
            _city_match: {
                $regexMatch: { input: '$destination', regex: escapedKeyword, options: 'i' }
            },
        }
    });

    pipeline.push({
        $addFields: {
            _match_score: {
                $switch: {
                    branches: [
                        // 城市+标题都命中 → 3 分
                        {
                            case: { $and: ['$_city_match', '$_title_match'] },
                            then: 3
                        },
                        // 只命中城市 → 2 分（城市优先）
                        {
                            case: '$_city_match',
                            then: 2
                        },
                        // 只命中标题 → 1 分
                        {
                            case: '$_title_match',
                            then: 1
                        },
                    ],
                    default: 0,
                }
            }
        }
    });

    // 只保留有匹配的结果
    pipeline.push({ $match: { _match_score: { $gt: 0 } } });
}

// Step 3: 排序
const sortFields: Record<string, 1 | -1> = {};
if (keyword) {
    sortFields._match_score = -1;   // 匹配优先级降序
}
sortFields.like_count = -1;         // 点赞量降序
sortFields.created_at = -1;         // 时间降序
pipeline.push({ $sort: sortFields });

// Step 4: 使用 $facet 同时获取 total 和分页数据
pipeline.push({
    $facet: {
        metadata: [{ $count: 'total' }],
        data: [
            { $skip: (page - 1) * pageSize },
            { $limit: pageSize },
            {
                $project: {
                    _id: 1,
                    trip_name: 1,
                    trip_cover: 1,
                    trip_brief: 1,
                    destination: 1,
                    start_date: 1,
                    end_date: 1,
                    like_count: 1,
                    'sell_config.price': 1,
                    'sell_config.payment_type': 1,
                    'sell_config.credit_price': 1,
                    created_at: 1,
                }
            }
        ],
    }
});
```

### 5.3 排序行为验证矩阵

| 场景 | keyword | city | 排序规则 |
|------|:-------:|:----:|---------|
| 无搜索 | — | — | `like_count DESC → created_at DESC` |
| 仅 city 参数 | — | "Tokyo" | destination 模糊过滤 → `like_count DESC → created_at DESC` |
| 仅 keyword | "Tokyo" | — | `_match_score DESC → like_count DESC → created_at DESC` |
| keyword + city | "Beach" | "Bali" | city 过滤 + score 排序 → `_match_score DESC → like_count DESC → created_at DESC` |

### 5.4 正则注入防护

用户输入的 `keyword` 和 `city` 参数直接用于 `$regex`，必须转义特殊字符：

```typescript
function escapeRegex(str: string): string {
    return str.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
}
```

## 八、影响评估

### 8.1 风险评估

| # | 风险 | 级别 | 缓解措施 |
|:-:|------|:----:|---------|
| 1 | 广场列表每次批量调 user 服务获取卖家信息 | **中** | page_size ≤ 50，user 服务批量接口限制 100 条，单次请求 < 50ms；后续可加 Redis 缓存 |
| 2 | price=0 修改涉及 2 个服务需同步上线 | **中** | payment 侧仅增加拦截（不破坏现有逻辑），可先上 chat 再上 payment |
| 3 | `like_count` 并发写入偶有不精确 | 低 | `$inc` 原子操作 + 唯一索引天然防重；接受最终一致 |
| 4 | 正则注入（keyword/city 直接用于 `$regex`） | 中 | 使用 `escapeRegex()` 转义特殊字符 |
| 5 | 广场列表无分页上限导致大量数据查询 | 低 | `page_size` DSL 限制 max=50 |
| 6 | home 版广场列表需额外查 trip_likes | 低 | 使用 `$in` 批量查询，有 `{trip_id, user_id}` 联合索引 |
| 7 | `seller_userinfo` 快照字段（trip 上）不可用于广场 | ⚠️ 注意 | 广场始终实时获取 user 服务，不读快照；快照仅用于购买后的 project 展示 |
| 8 | DSL 多语言迁移可能影响现有错误消息 | 低 | schema-dsl v1.2.3 index.js 自动跳过，向后兼容；error_catch.ts 改为统一 index 导出 |

### 8.2 性能评估

| 操作 | 预期性能 | 索引支持 |
|------|---------|---------|
| 广场列表（无 keyword） | < 100ms | 复合索引 `{ enabled, listed_in_square, del_flag, like_count }` |
| 广场列表（有 keyword） | < 300ms | 基础索引过滤后 `$regexMatch` 在结果集上计算 |
| 点赞/取消 | < 50ms | 唯一联合索引 `{ trip_id, user_id }` |

### 8.3 回滚方案

1. Model 字段新增为可选且有默认值，不影响现有数据
2. 新建集合 `trip_likes` 独立存在，删除不影响其他功能
3. `like_count` 字段新增默认 0，不影响现有查询
4. 路由新增不影响现有接口

## 十一、跨服务调用链

```
chat → user（已有模式，广场列表复用）:
  ├── GET  /internal/user/:userId          — 单个卖家信息
  └── POST /internal/users/batch           — 批量卖家信息（广场列表使用此接口）
      返回字段: nick_name, portrait, email, phone, link, introduction, social_media
      降级方案: 批量失败 → fetchSellerInfoFallback 逐个查询（并发限制 5）

payment → chat（已有模式，无需修改）:
  └── GET  /internal/trip/sell-config/:tripId — 创建订单时验证售卖状态

payment → user（已有模式，无需修改）:
  ├── GET  /credit/tradable                — 查询买家可用积分
  ├── POST /internal/credit/freeze         — 冻结积分
  └── POST /internal/credit/deduct         — 扣除积分
```

**广场列表卖家信息获取流程**:
1. 广场列表查询返回 N 条行程（含 `owner_id`）
2. 提取唯一 `owner_id` 集合 → `POST /internal/users/batch` 批量获取
3. 按 `owner_id` 映射回每条行程的 `seller_info` 字段
4. 参考实现: `open/TripProjectController.ts` 的 `fetchBatchSellerInfo()` + `fetchSellerInfoFallback()`
