# 行程广场功能需求定义

> **任务类型**: 需求开发（01-requirement-dev）— 完整模式
> **需求来源**: 用户提出行程广场 5 项子需求
> **涉及项目**: chat 服务（Egg.js + TypeScript + MongoDB）

---

## 一、业务背景

当前行程售卖功能已支持用户配置售卖参数（价格、预览天数、权限等），买家可通过分享链接预览并购买行程。但缺少一个**公开的行程展示入口**——即「行程广场」，让所有用户（含未登录访客）能浏览、搜索、发现优质行程。

本次需求围绕「行程广场」核心场景，新增 5 项子功能。

## 三、逐项需求定义

### REQ-1: 售卖配置新增「是否上架到广场」字段

**业务目标**: 让卖家控制行程是否对外展示在广场中，实现"开启售卖 ≠ 上架广场"的灵活配置。

**功能需求**:

1. 在行程售卖配置（`sell_config`）中新增 `listed_in_square` 布尔字段，默认为 `false`
2. 该字段仅在 `sell_config.enabled = true`（开启售卖）时有意义
3. 当 `enabled` 设为 `false` 时，`listed_in_square` 应强制设为 `false`
4. 保存售卖配置接口需新增该字段的 DSL 验证和写入逻辑
5. 国际化 label 需同步新增（中文/英文/繁体中文）

**验收标准**:

- [ ] `PUT /home/trip_project/trip/:tripId/sell-config` 接口支持 `listed_in_square` 参数
- [ ] `listed_in_square` 默认值为 `false`
- [ ] `enabled=false` 时保存，`listed_in_square` 被强制设为 `false`
- [ ] `GET /home/trip_project/trip/:tripId/sell-config` 返回值中包含 `listed_in_square`
- [ ] 三语言包均包含 `label.listed_in_square`

### REQ-2: 售卖价格支持配置为 0（免费行程）

**业务目标**: 允许卖家将行程价格设为 0，支持免费行程场景。

**功能需求**:

1. 现金价格（`price`）最小值从 100（即 $1.00）改为 0
2. 积分价格（`credit_price`）最小值从 1 改为 0
3. Model 层 `sell_config.price.min` 从 `0.99` 改为 `0`
4. Model 层 `sell_config.credit_price.min` 从 `1` 改为 `0`
5. Controller 层 DSL 验证范围从 `number:100-999999!` 改为 `number:0-999999!`
6. **修复已知 Bug**: `updateData` 中 `form.price &&` 使用 truthy 判断，导致 `price=0` 时不写入数据库，必须改为 `form.price !== undefined`

**前提说明**:

> `price=0` 的业务语义（免费购买 vs 跳过支付）由 payment 微服务侧决定。本次仅修改 chat 服务侧的验证和存储逻辑，payment 侧另行处理。

**验收标准**:

- [ ] `price=0` 时 DSL 验证通过
- [ ] `credit_price=0` 时 DSL 验证通过
- [ ] `price=0` 成功写入数据库（不被 truthy 判断跳过）
- [ ] `credit_price=0` 成功写入数据库
- [ ] MongoDB Schema 的 `min` 约束允许 0

### REQ-3: 行程广场列表（可选鉴权 + 关键词搜索）

**业务目标**: 提供一个公开的行程广场列表接口，让任何人（含未登录访客）都能浏览和搜索已上架的行程；已登录用户额外获取每条行程的点赞状态。

**功能需求**:

1. 新增 `GET /open/trip_square/list` 接口，无需登录即可访问
2. 接口放在 `/open/*` 路由组下，使用 `openRequestValid` + `userAuth(optional)` 中间件
3. **可选鉴权行为**：
   - 请求携带有效 JWT Token → 自动解析用户身份，响应中包含每条行程的 `liked` 点赞状态
   - 请求无 Token 或 Token 无效 → 静默放行，响应中不含 `liked` 字段（不返回 401 错误）
4. 查询条件：
   - 固定过滤：`del_flag=0` + `sell_config.enabled=true` + `sell_config.listed_in_square=true`
   - 可选参数 `keyword`：同时模糊匹配 `trip_name`（标题）和 `destination`（目的地城市），不区分大小写
5. 支持分页：`page`（默认 1，最小 1）、`page_size`（默认 20，最小 1，最大 50）
6. 返回字段：`_id`, `trip_name`, `trip_cover`, `trip_brief`, `destination`, `start_date`, `end_date`, `like_count`, `sell_config.price`, `sell_config.payment_type`, `sell_config.credit_price`, `seller_info`（`nick_name`, `portrait`, `introduction`）, `created_at`
7. 已登录时额外返回：`liked: boolean`（当前用户是否已点赞该行程）

**数据源说明**:

- 查询 `trips` 集合（`app/model/trip.ts`）
- 卖家信息实时从 user 服务批量获取（`POST /internal/users/batch`），不使用快照字段
- `destination` 字段为字符串类型（如 "Tokyo, Japan"），使用正则模糊匹配

**验收标准**:

- [ ] `GET /open/trip_square/list` 无需登录即可访问，返回完整列表和卖家信息
- [ ] 只返回 `enabled=true` 且 `listed_in_square=true` 且 `del_flag=0` 的行程
- [ ] `keyword` 参数能同时匹配标题和目的地城市（不区分大小写）
- [ ] 无 `keyword` 时返回全部已上架行程
- [ ] 未登录时响应中不含 `liked` 字段，不返回错误
- [ ] 已登录时响应中每条行程包含 `liked: true/false`
- [ ] 分页正常工作，返回 `total` 总数
- [ ] 空数据时返回空数组而非报错

### REQ-4: 行程点赞功能

**业务目标**: 允许已登录用户对广场中的行程点赞/取消点赞，为后续排序提供数据支撑，同时提升用户参与感；并支持查看自己点赞过的行程列表。

**功能需求**:

1. 新建 `trip_likes` 集合，记录点赞关系（`trip_id` + `user_id`）
2. 索引设计：
   - 联合唯一索引 `{ trip_id: 1, user_id: 1 }` — 防止重复点赞
   - 普通复合索引 `{ user_id: 1, created_at: -1 }` — 支持用户点赞列表查询
3. `trips` 集合顶层新增 `like_count` 冗余字段（默认 0），用于排序加速
4. 点赞时 `like_count` 使用原子 `+1` 操作
5. 取消点赞时 `like_count` 使用原子 `-1` 操作，并保护不低于 0
6. 提供以下接口（均需登录，放在 `/home/*` 路由组）：
   - `POST /home/trip_like/:tripId` — 点赞（幂等）
   - `DELETE /home/trip_like/:tripId` — 取消点赞（幂等）
   - `GET /home/trip_like/:tripId/status` — 查询当前用户是否已点赞该行程
   - `GET /home/trip_like/list` — 获取当前用户点赞过的行程列表（分页，按点赞时间倒序）
7. 点赞前校验：行程必须存在、未删除、已上架广场
8. 点赞和取消点赞接口接入速率限制：每用户每行程 60 秒内最多 10 次操作（Key = 用户 ID + 行程 ID + 请求路径）

**验收标准**:

- [ ] 首次点赞成功，`like_count` +1
- [ ] 重复点赞幂等处理（不报错，不重复计数）
- [ ] 取消点赞成功，`like_count` -1
- [ ] 未点赞时取消点赞，幂等处理（不报错）
- [ ] `like_count` 不会变为负数
- [ ] 查询点赞状态返回 `{ liked: true/false }`
- [ ] 未登录时调用点赞接口返回 401
- [ ] `GET /home/trip_like/list` 返回当前用户点赞的行程列表，含 `liked_at` 点赞时间
- [ ] 点赞/取消超过频率限制时返回 429 及错误码 `rate_limit_exceeded`

### REQ-5: 广场搜索复合排序规则

**业务目标**: 让广场列表搜索结果按「相关度优先 → 点赞量 → 发布时间」的复合规则排序，提升搜索体验。

**功能需求**:

1. 搜索框行为：前端只有一个搜索输入框，对应 `keyword` 参数，输入的关键词**同时**匹配行程标题和目的地城市，不区分大小写，包含即算命中。

2. 无关键词时的排序（默认排序）：
   - 第一排序：点赞量多的在前
   - 第二排序：点赞量相同时，发布时间晚的在前

3. 有关键词时的排序（按命中情况打分）：

   | 命中情况 | 得分 | 说明 |
   |---------|:----:|------|
   | 标题和城市都命中 | 3 分 | 相关度最高，排最前 |
   | 只有城市命中 | 2 分 | 城市优先级高于标题 |
   | 只有标题命中 | 1 分 | 排在城市命中之后 |
   | 两者都未命中 | — | 不出现在结果中 |

   同分情况下的次级排序：点赞量多的在前 → 发布时间晚的在前

4. 搜索安全：keyword 参数写入数据库前须做正则特殊字符转义，防止注入

**验收标准**:

- [ ] 无关键词时按点赞量降序 → 时间降序排列
- [ ] 有关键词时，城市命中的结果排在纯标题命中之前
- [ ] 同时命中城市和标题的结果排在最前
- [ ] 同一命中等级内，按点赞量降序 → 时间降序排列
- [ ] 有关键词时不返回标题和城市均未命中的结果
- [ ] keyword 包含正则特殊字符时不报错，正常返回结果
