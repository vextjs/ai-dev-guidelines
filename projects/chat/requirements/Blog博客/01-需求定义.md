# Blog 博客功能需求定义

> **任务类型**: 需求开发（01-requirement-dev）— 完整模式
> **需求来源**: 用户提出在 chat 服务中新增博客功能（增删查改）
> **涉及项目**: chat 服务（Egg.js + TypeScript + MongoDB）

---

## 一、业务背景

chat 服务当前主要承载 AI 对话、行程规划等核心功能，缺少内容创作与发布能力。本次在 chat 服务内新增「Blog 博客」模块，支持用户创作和发布博客文章，提供完整的增删查改（CRUD）接口，为平台内容生态奠定基础。

## 三、逐项需求定义

### REQ-1: 博客文章创建

**业务目标**: 允许已登录用户创作博客文章，支持草稿保存和直接发布两种模式。

**功能需求**:

1. 接口：`POST /home/blog`，需登录（`userAuth` 鉴权）
2. 支持字段：
   - `title`（必填）：文章标题，1-200 字符（对应当前语言版本）
   - `content`（必填）：文章正文，无长度限制（对应当前语言版本）
   - `summary`（可选）：文章摘要，0-500 字符，若不填则自动取正文**第一段落**（按 `\n\n` 分割取首段，最多 500 字符）
   - `cover_image`（可选）：封面图 URL；若不填则自动从正文提取**第一张图片** URL（优先 Markdown `![](url)` 语法，其次 HTML `<img src>` 标签），均无图时为空字符串
   - `tags`（可选）：标签数组，每个标签 1-20 字符，最多 10 个
   - `status`（可选）：`draft`（草稿，默认）或 `published`（发布）
3. 创建时系统自动写入：
   - `lang`：从请求头 `language`（`zh`/`en`/`hk`）自动读取，**无需前端传入**；不同语言各为一条独立记录
   - `author_id`：当前登录用户 ID
   - `view_count`：初始值 0
   - `del_flag`：初始值 0
   - `created_at` / `updated_at`：当前时间
4. 当 `status=published` 时，系统自动记录 `published_at` 时间戳

**验收标准**:

- [ ] 已登录用户可成功创建草稿（`status=draft`）
- [ ] 已登录用户可直接创建并发布文章（`status=published`）
- [ ] `title` 为空时返回参数校验错误
- [ ] `content` 为空时返回参数校验错误
- [ ] `tags` 超过 10 个时返回业务错误
- [ ] `summary` 不填时自动取正文第一段落（按 `\n\n` 分割取首段，最多 500 字符）
- [ ] `cover_image` 不填时自动从正文提取第一张图片 URL（Markdown `![](url)` 或 HTML `<img src>`），无图时为空字符串
- [ ] 创建成功返回完整文章对象（含 `_id`）
- [ ] 未登录时调用返回 401

### REQ-2: 博客文章列表查询（含搜索与分页）

**业务目标**: 提供公开的博客列表接口，无需登录即可访问，支持关键词搜索、标签筛选和分页。

**功能需求**:

1. 接口：`GET /open/blog`，无需登录（`open` 分组）
2. 固定过滤条件：
   - `del_flag=0`（未删除）
   - `status=published`（已发布）
3. 可选查询参数：
   - `keyword`（可选）：同时模糊匹配 `title`（标题）和 `summary`（摘要），不区分大小写；特殊字符需转义，防止正则注入
   - `tag`（可选）：按标签精确过滤
   - `author_id`（可选）：按作者 ID 过滤，查看指定用户的文章
   - `limit`（可选，默认 20，最小 1，最大 50）：每页返回条数
   - `after`（可选）：游标值，获取该游标之后的数据（用于"下一页"）
   - `before`（可选）：游标值，获取该游标之前的数据（用于"上一页"）
4. **分页模式**：使用 monSQLize `findPage` **游标分页**（cursor-based pagination），通过 `after`/`before` 游标翻页，性能为 O(1)，不受数据量影响
5. 排序规则（固定）：按 `published_at` 降序（最新发布排前）
6. 返回字段：`_id`, `lang`, `title`, `summary`, `cover_image`, `tags`, `status`, `view_count`, `published_at`, `created_at`, `author_id`
   > `cover_image` 在创建时若未填写则由系统自动从正文提取，列表始终返回此字段（可能为空字符串）
7. 响应结构遵循 monSQLize `findPage` 原生格式：`{ items, pageInfo: { hasNext, hasPrev, startCursor, endCursor }, totals }`
8. **查询缓存**：结果启用 monSQLize 查询缓存，TTL 60 秒；写操作（创建/更新/删除）后自动失效

**验收标准**:

- [ ] 无需登录即可获取已发布文章列表
- [ ] `keyword` 参数能同时匹配标题和摘要（不区分大小写）
- [ ] `keyword` 包含正则特殊字符时不报错，正常返回结果
- [ ] `tag` 参数能精确过滤指定标签的文章
- [ ] 草稿（`status=draft`）和已删除（`del_flag=1`）文章不出现在列表中
- [ ] 游标分页正常工作，响应包含 `items`（数据列表）、`pageInfo`（`hasNext`/`hasPrev`/`startCursor`/`endCursor`）
- [ ] 首次请求不传 `after`/`before` 时返回第一页数据
- [ ] 使用 `after=endCursor` 可正确获取下一页
- [ ] 使用 `before=startCursor` 可正确获取上一页
- [ ] 最后一页 `pageInfo.hasNext` 为 `false`
- [ ] 空数据时返回空 `items` 数组而非报错

### REQ-3: 博客文章详情查询

**业务目标**: 获取单篇文章完整内容，同时记录浏览量。

**功能需求**:

1. 接口：`GET /open/blog/:blogId`，无需登录（`open` 分组）
2. 只返回已发布且未删除的文章（`status=published` + `del_flag=0`），否则返回 404
3. 返回字段：REQ-2 返回字段的超集，额外包含 `content`（正文全文）和 `updated_at`
4. 每次访问自动对 `view_count` 原子 `+1`（使用 MongoDB `$inc`，无需用户登录）
5. 文章作者自己访问时，`status=draft` 的草稿也可正常访问（需登录且 `author_id` 匹配当前用户）
6. **查询缓存**：详情结果启用 monSQLize 查询缓存，TTL 120 秒；文章更新/删除后自动失效

**验收标准**:

- [ ] 无需登录即可获取已发布文章详情（包含完整 `content`）
- [ ] 访问不存在的 `blogId` 返回 404 错误码 `BLOG_NOT_FOUND`
- [ ] 已删除文章返回 404
- [ ] 每次访问 `view_count` 自动 +1（原子操作）
- [ ] 文章作者可访问自己的草稿
- [ ] 非作者访问草稿返回 404

### REQ-4: 博客文章更新

**业务目标**: 允许作者修改自己的文章内容，支持草稿发布、已发布文章重新编辑及下架。

**功能需求**:

1. 接口：`PUT /home/blog/:blogId`，需登录（`userAuth` 鉴权）
2. 仅允许文章作者（`author_id === ctx.state.user._id`）修改，否则返回 403
3. 可更新字段（所有字段均为可选，仅更新传入的字段，未传字段保持不变）：
   - `title`, `content`, `summary`, `cover_image`, `tags`, `status`
4. 状态变更规则：
   - `draft → published`：若 `published_at` 为空则自动写入当前时间
   - `published → draft`：允许（文章下架），`published_at` 保留不清除
5. 每次更新自动刷新 `updated_at`
6. 更新成功后自动失效该文章的查询缓存

**验收标准**:

- [ ] 文章作者可成功更新文章内容
- [ ] 非作者调用返回 403 错误码 `BLOG_FORBIDDEN`
- [ ] 未登录调用返回 401
- [ ] 草稿首次发布时记录 `published_at`
- [ ] 仅传入部分字段时，其他字段不受影响
- [ ] `tags` 超过 10 个时返回业务错误
- [ ] 更新成功返回最新文章对象

---

### REQ-5: 博客文章删除（软删除）

**业务目标**: 允许作者删除自己的文章，采用软删除方式保留数据。

**功能需求**:

1. 接口：`DELETE /home/blog/:blogId`，需登录（`userAuth` 鉴权）
2. 仅允许文章作者（`author_id === ctx.state.user._id`）删除，否则返回 403
3. 软删除：将 `del_flag` 设为 `1`，`deleted_at` 记录删除时间，不物理删除数据
4. 删除后文章从公开列表和详情接口中消失
5. 删除成功后自动失效该文章的查询缓存

**验收标准**:

- [ ] 文章作者可成功软删除文章（`del_flag=1`）
- [ ] 非作者调用返回 403 错误码 `BLOG_FORBIDDEN`
- [ ] 未登录调用返回 401
- [ ] 删除后文章不再出现在公开列表和详情接口中
- [ ] 删除成功返回 `{ success: true }`
- [ ] 对已删除文章重复调用，幂等处理（不报错）

### REQ-6: 管理后台文章管理

**业务目标**: 管理员可查看全量文章（含草稿和已删除），并对文章进行强制上下架操作。

**功能需求**:

1. 管理列表：`GET /admin/blog`，需管理员鉴权
   - 支持按 `status`、`del_flag`、`author_id`、`keyword` 过滤
   - 返回所有状态文章（含草稿、已删除），不受 `del_flag` 和 `status` 限制
   - 使用 monSQLize `findPage` 游标分页（`limit` 最大 100，`after`/`before` 翻页）
2. 强制上下架：`PUT /admin/blog/:blogId/status`，需管理员鉴权
   - 支持将任意文章的 `status` 修改为 `published` 或 `draft`
   - 不受作者限制，管理员可操作任意文章

**验收标准**:

- [ ] 管理员可查看全量文章列表（含草稿、已删除）
- [ ] 按 `status` 和 `del_flag` 过滤正常工作
- [ ] 游标分页正常工作，响应包含 `items`、`pageInfo`、`totals`
- [ ] 管理员可强制将文章上架（`published`）或下架（`draft`）
- [ ] 非管理员调用管理接口返回 403

### REQ-7: 多语言内容支持

**业务目标**: 博客文章内容（标题、正文、摘要）支持简体中文、英文、繁体中文三种语言，根据请求语言返回对应语言版本的文章。

**功能需求**:

1. 语言控制：通过自定义请求头 `language`（值为 `zh`/`en`/`hk`）控制，不传默认 `en`
2. 多语言模式：采用**独立记录模式**，同一篇文章的不同语言版本在 `blogs` 集合中各为一条独立记录，通过 `lang` 字段区分
3. `lang` 字段由 Service 层从 `ctx.i18n.getLanguage()` 自动读取并写入，前端**无需**在请求 body 中传入
4. 创建文章：携带 `language: zh` 请求头发起 POST，创建一条 `lang=zh` 的简体中文记录；同理创建其他语言版本
5. 查询文章：列表和详情接口自动按请求头语言过滤，只返回对应语言的记录
6. 管理端：支持按 `lang` query 参数过滤查看各语言文章（不传则查全部语言）
7. 各语言版本的 `status`、`del_flag`、`view_count` 互相独立，可单独上下架、单独计数
8. 与项目既有模式一致（`feature_modules`/`trip_destination` 等均采用此方案）

**验收标准**:

- [ ] 携带 `language: zh` 请求头，列表接口只返回 `lang=zh` 的文章
- [ ] 携带 `language: en` 请求头，列表接口只返回 `lang=en` 的文章
- [ ] 携带 `language: hk` 请求头，列表接口只返回 `lang=hk` 的文章
- [ ] 不传 `language` 时，默认返回 `lang=en` 的文章
- [ ] POST 创建文章时，`lang` 字段由后端从请求头自动写入，无需前端传入
- [ ] 详情接口根据请求语言过滤，请求语言与记录 lang 不匹配时返回 `BLOG_NOT_FOUND`
- [ ] 管理端列表支持 `?lang=zh/en/hk` 参数过滤，不传则返回全部语言
- [ ] 错误消息也随 `language` 请求头切换语言

## 五、数据模型变更摘要

| 集合 | 操作 | 变更内容 |
|------|:----:|---------|
| `blogs` | **新建** | 博客文章主集合（标题、正文、状态、软删除、统计字段） |

### blogs 集合核心字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `_id` | ObjectId | 主键 |
| `lang` | String | 语言标识：`zh`=简体中文 / `en`=英文（默认） / `hk`=繁体中文。同一文章的不同语言版本为独立记录，由 Service 层自动写入，前端无需传入 |
| `title` | String | 文章标题（必填，1-200 字符，对应 lang 语言） |
| `content` | String | 文章正文（必填，无长度限制，对应 lang 语言） |
| `summary` | String | 摘要（可选，最多 500 字符，对应 lang 语言） |
| `cover_image` | String | 封面图 URL（可选，各语言版本共用同一封面） |
| `tags` | String[] | 标签数组（最多 10 个） |
| `status` | String | `draft` / `published`（各语言版本独立维护） |
| `author_id` | ObjectId | 作者用户 ID |
| `view_count` | Number | 浏览量（默认 0，原子自增；各语言版本独立计数） |
| `del_flag` | Number | 软删除标记（0=正常，1=已删除；各语言版本独立维护） |
| `published_at` | Date | 首次发布时间 |
| `deleted_at` | Date | 软删除时间 |
| `created_at` | Date | 创建时间 |
| `updated_at` | Date | 最后更新时间 |

> **多语言模式说明**：与项目 `feature_modules`、`trip_destination` 等模块保持一致，采用**独立记录模式（方案 B2）**：
> 同一篇文章的 zh/en/hk 三语言版本各为一条独立的 `blogs` 文档，通过 `lang` 字段区分。
> 查询时 Service 层自动从请求头 `language` 读取语言并追加 `lang` 过滤条件，前端无需额外处理。
