## 四、权限控制边界

> ⚠️ **重要**：使用 permia 前必须理解其权限控制的边界

### 4.1 permia 是什么

- **应用层 RBAC 权限判断引擎** — 管理角色、规则、用户绑定关系，提供 `can` / `assert` / `filterFields` 等 API 进行权限判断
- **通用库** — 框架无关，不绑定任何特定数据库或 ORM
- **接口权限控制** — 通过 HTTP 中间件调用 `assert()` 实现自动拦截（方案有完整示例）
- **数据库/字段权限判断** — 通过在 Service/DAO 层调用 `can()` 和 `filterFields()` 实现业务级权限控制

### 4.2 permia 不是什么

- **不是数据库代理** — 不会自动拦截数据库的查询、插入、更新、删除操作
- **不是 ORM 插件** — 不依赖 Mongoose / Sequelize / monSQLize 等 ORM；不自动 hook 数据库操作
- **不是 UI 组件库** — 不直接控制前端渲染；前端按钮/栏目显隐由前端根据权限数据自行判断
- **不是数据库原生权限系统** — 与 MongoDB 内置 RBAC / PostgreSQL Row-Level Security 等无关

### 4.3 两类权限的执行方式

| 权限类型 | 执行位置 | 执行方式 | 说明 |
|---------|---------|---------|------|
| **接口权限**（`<METHOD>:<path>`） | HTTP 中间件层 | `assert(userId, 'invoke', resource)` | 中间件自动拦截，无权限返回 403 |
| **数据库集合权限**（`db:<collection>`） | Service / DAO 层 | `can(userId, 'read', 'db:articles')` 或 `assert(...)` | 业务代码主动调用，阻止无权操作 |
| **字段级权限**（`db:<collection>:<field>`） | Service / DAO 层 | `filterFields(userId, 'read', 'db:articles', data)` | 查询后过滤返回数据；写入前过滤提交字段 |

> 💡 **如需自动拦截数据库操作**：后期可出 ORM 插件（如 `permia-mongoose-plugin`、`permia-monsqlize-plugin`），在 ORM 的 hook 层自动调用 `can()` / `filterFields()`。但核心库不强制依赖任何 ORM。

## 五、资源路径模型

> ⚠️ 详细规范见 [02-技术方案/01-资源路径模型.md](./02-技术方案/01-资源路径模型.md)

```
两类资源格式:

  数据库资源:  db:<collection>[:<field>]
    'db:users'                   集合整体的增删改查权限
    'db:users:email'             单字段读写权限
    'db:users:*'                 该集合所有字段
    'db:*'                       所有集合

  接口资源:    <METHOD>:<path>（METHOD 必须大写，与 req.method 一致）
    'GET:/api/users'             查询用户列表
    'POST:/api/users'            新增用户
    'DELETE:/api/users/:id'      删除用户（:id 匹配任意单段）
    '*:/api/users/*'             整个用户管理栏目（所有方法 + 所有子路径）
    '*:*'                        所有接口
    '*'                          全局超管（含 db + 接口）

操作类型(action):
  db: 资源 → read | write | create | update | delete | *
    write = create + update（写入类操作的快捷方式，matchAction 自动展开）
  接口资源 → invoke | *
  通用:      *

说明:
  - "栏目"和"按钮"权限本质是接口（API）级鉴权
  - 前端按钮/栏目是否显示由前端根据接口权限自行判断，permia 不处理 UI 渲染
  - 中间件构造: `${req.method}:${req.path}` → 'DELETE:/api/users/123'
  - userId=null/undefined：视为匿名用户，鉴权结果为 false
```

## 六、核心 API 设计

### 6.1 主类 Permia

```typescript
import { Permia } from 'permia';

const permia = new Permia({
  // storage: new FileAdapter(),       // 默认 FileAdapter（磁盘 JSON 持久化）
  cache: { enabled: true, ttl: 300000 },
  strict: true                         // deny 规则严格优先（默认 true）
});
await permia.init();                    // 🔴 必须调用，否则抛 NOT_INITIALIZED

// 角色管理
await permia.roles.create('admin',  { label: '管理员' });
await permia.roles.create('editor', { label: '编辑', parent: 'viewer' });

// 权限规则
await permia.roles.allow('admin',  '*',                    '*');
await permia.roles.allow('editor', ['read','write'],       'db:articles');
await permia.roles.allow('editor', 'read',                 'db:users:email');
await permia.roles.deny( 'editor', 'delete',               'db:users');
await permia.roles.allow('editor', 'invoke',               'POST:/api/articles');

// 用户绑定
await permia.users.assign('uid-1', 'editor');
await permia.users.revoke('uid-1', 'editor');

// 鉴权
const ok = await permia.can('uid-1', 'write', 'db:articles');   // → boolean
await permia.assert('uid-1', 'delete', 'db:users');              // 失败抛 PermiaError

// 字段过滤
const safe = await permia.filterFields('uid-1', 'read', 'db:users', record);

// 链式 API
const ctx = permia.for('uid-1');
await ctx.can('read', 'db:articles');
await ctx.filterFields('read', 'db:users', record);
```

### 6.2 存储适配器接口

```typescript
// 用户实现此接口可对接任意存储后端
interface StorageAdapter {
  init(): Promise<void>;                           // 异步初始化
  close(): Promise<void>;                          // 关闭清理
  getRoles(): Promise<Map<string, RoleData>>;      // → 全量角色
  getRole(id: string): Promise<RoleData | null>;   // → 单条
  setRole(id: string, data: RoleData): Promise<void>;
  deleteRole(id: string): Promise<void>;
  getUserRoles(userId: string): Promise<string[]>;        // → 角色 ID 列表
  setUserRoles(userId: string, roleIds: string[]): Promise<void>;
  getUsersByRole(roleId: string): Promise<string[]>;      // → 用户 ID 列表
  getRules(roleId: string): Promise<PermissionRule[]>;    // → 权限规则
  setRules(roleId: string, rules: PermissionRule[]): Promise<void>;
  deleteRules(roleId: string): Promise<void>;             // delete 角色时清规则
}

// 内置：FileAdapter（默认，磁盘 JSON 持久化）、MemoryAdapter（开发/测试用）
// 官方可选：MonSQLizeAdapter（独立包，基于 monsqlize，适合多实例部署）
```

## 八、存储适配器策略

| 适配器 | 包 | 默认 | 说明 |
|--------|-----|:---:|------|
| `FileAdapter` | 内置（零依赖） | ✅ | 默认适配器，磁盘 JSON 持久化，单实例生产可用 |
| `MemoryAdapter` | 内置（零依赖） | — | 开发/测试用，进程重启数据清空 |
| `MonSQLizeAdapter` | `permia-adapter-monsqlize`（独立包，后期） | — | 多实例部署推荐，基于 monSQLize 持久化 |
| 自定义 | 用户实现 `StorageAdapter` 接口 | — | 任意存储后端 |

> **设计原则**: permia 核心包零强依赖；monsqlize 为可选适配器，独立包发布，避免强耦合。
> **⚠️ FileAdapter 生产注意**: 容器化部署需挂载数据卷；Serverless 环境不适用；多进程部署各进程独立副本，需改用 MonSQLizeAdapter。
