## 一、资源路径格式规范

### 1.1 数据库资源：`db:<collection>[:<field>]`

| 路径 | 含义 |
|------|------|
| `db:users` | users 集合整体 |
| `db:users:email` | users.email 顶层字段 |
| `db:users:*` | users 所有顶层字段 |
| `db:*` | 所有集合 |

> ⚠️ v1.0.0 边界：不支持嵌套字段（`address.city`），不支持行级（Row-level）权限

### 1.2 接口资源：`<METHOD>:<path>`

| 路径 | 含义 |
|------|------|
| `GET:/api/users` | 查询用户列表 |
| `POST:/api/users` | 新增用户 |
| `PUT:/api/users/:id` | 编辑用户（`:id` 匹配任意单段） |
| `DELETE:/api/users/:id` | 删除用户 |
| `GET:/api/users/export` | 导出 |
| `*:/api/users/*` | 整个用户管理栏目（所有子路径） |
| `GET:*` | 所有 GET 接口 |
| `*:*` | 所有接口 |
| `*` | 全局超管（含 db + 接口） |

### 1.3 操作类型（action）

| 资源类型 | 可用 action |
|---------|-------------|
| `db:` 资源 | `read` / `write` / `create` / `update` / `delete` / `*` |
| 接口资源 | `invoke` / `*` |
| `*` 全局 | `*` |

### 1.4 关键约束

- **METHOD 必须大写**（与 `req.method` 一致）
- 权限规则路径风格必须与中间件构造方式保持一致
- 中间件构造：`` `${req.method}:${req.path}` `` → `'DELETE:/api/users/123'`
- `userId=null/undefined`：视为匿名用户，鉴权结果为 `false`

## 二、通配符规则

### 2.1 接口路径通配

| pattern | 匹配示例 | 说明 |
|---------|---------|------|
| `*:/users/:id` | `GET:/users/123` ✓ | Method 通配 + 路径参数 |
| `GET:*` | `GET:/users` ✓ | 全路径通配 |
| `*:*` | `DELETE:/orders/9` ✓ | 全通配 |
| `*:/users/*` | `GET:/users/123` ✓ / `GET:/users` ✗ | 子路径通配（至少一段） |
| `*:/users*` | `GET:/users` ✓ / `GET:/users/123` ✓ | 前缀通配（含自身） |

**末段 `*` 说明**：匹配剩余所有段（至少 1 段），`*:/users/*` 不匹配 `GET:/users`

### 2.2 db 资源通配

| pattern | 匹配示例 | 说明 |
|---------|---------|------|
| `db:users:*` | `db:users:email` ✓ | 末段通配 |
| `db:*` | `db:users` ✓ / `db:users:email` ✗ | 单段通配（段数必须一致） |

## 三、两种优先级的区分

> ⚠️ **重要**：本方案存在两个独立的"优先级"概念，不可混淆

### 3.1 匹配精确度（wildcard 内部）

描述 pattern 的命中范围，**不影响规则权重**：

```
精确 > 路径参数(:id) > 末段通配(*) > Method通配(*:) > 全通配(*:*) > 全局(*)
```

### 3.2 规则权重（checker 内部）

决定鉴权最终结果：

| 模式 | 行为 | 说明 |
|------|------|------|
| `strict=true`（默认）| 任意 deny 命中 → `false` | deny 无条件全局优先 |
| `strict=false` | 最后命中的规则胜出 | 规则顺序决定优先级 |

**示例**：
```
规则: allow GET:/api/users  +  deny *:/api/users
strict=true  → deny 命中 → false（即使 allow 更精确，deny 始终优先）
strict=false → 取决于规则在 mergeRules 中的排列顺序
```

## 四、核心数据结构

```javascript
// 权限规则（存储 + 运行时通用）
PermissionRule = {
  type:     'allow' | 'deny',        // 规则类型
  action:   string,                   // 'read' / 'invoke' / '*'
  resource: string,                   // 'db:users' / 'GET:/api/users' / '*'
}

// 角色数据
RoleData = {
  id:          string,                // 角色唯一 ID（如 'admin', 'editor'）
  label:       string,                // 显示名称（如 '管理员', '编辑'）
  parent:      string | null,         // 父角色 ID（单继承，父角色须先创建）
  description: string,                // 描述
  createdAt:   number,                // 创建时间 timestamp ms
  updatedAt:   number,                // 更新时间 timestamp ms
}

// 缓存条目（内部使用）
CacheEntry = {
  rules:     PermissionRule[],        // 已展开继承链的完整规则列表
  expiredAt: number,                  // Date.now() + ttl
}
```
