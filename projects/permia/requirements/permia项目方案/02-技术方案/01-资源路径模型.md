## 一、资源路径格式规范

### 1.1 数据库资源：`db:<collection>[:<field>]`

| 路径 | 含义 |
|------|------|
| `db:users` | users 集合整体 |
| `db:users:email` | users.email 顶层字段 |
| `db:users:*` | users 所有顶层字段 |
| `db:*` | 所有集合 |

> ⚠️ v1.0.0 边界：不支持嵌套字段（`address.city`），不支持行级（Row-level）权限，不支持多数据库区分（`db:users` 隐含"当前数据库"）

### 1.2 接口资源：`<METHOD>:<path>`

| 路径 | 含义 |
|------|------|
| `GET:/api/users` | 查询用户列表 |
| `POST:/api/users` | 新增用户 |
| `PUT:/api/users/:id` | 编辑用户（`:id` 匹配任意单段） |
| `DELETE:/api/users/:id` | 删除用户 |
| `GET:/api/users/export` | 导出 |
| `*:/api/users/*` | 整个用户管理栏目（所有子路径） |
| `GET:*` | 所有 GET 接口 |
| `*:*` | 所有接口 |
| `*` | 全局超管（含 db + 接口） |

### 1.3 操作类型（action）

| 资源类型 | 可用 action |
|---------|-------------|
| `db:` 资源 | `read` / `write` / `create` / `update` / `delete` / `*` |
| 接口资源 | `invoke` / `*` |
| `*` 全局 | `*` |

### 1.4 关键约束

- **METHOD 必须大写**（与 `req.method` 一致）
- 权限规则路径风格必须与中间件构造方式保持一致
- 中间件构造：`` `${req.method}:${req.path}` `` → `'DELETE:/api/users/123'`
- `userId=null/undefined`：视为匿名用户，鉴权结果为 `false`

## 二、通配符规则

### 2.1 接口路径通配

| pattern | 匹配示例 | 说明 |
|---------|---------|------|
| `*:/users/:id` | `GET:/users/123` ✓ | Method 通配 + 路径参数 |
| `GET:*` | `GET:/users` ✓ | 全路径通配 |
| `*:*` | `DELETE:/orders/9` ✓ | 全通配 |
| `*:/users/*` | `GET:/users/123` ✓ / `GET:/users` ✗ | 子路径通配（至少一段） |

**末段 `*` 说明**：匹配剩余所有段（至少 1 段），`*:/users/*` 不匹配 `GET:/users`

> ⚠️ **重要 — 容易踩坑**：如果你想用 `*:/api/users/*` 来覆盖整个"用户管理"栏目，请注意它**不会**匹配 `GET:/api/users`（用户列表页）！必须同时配置两条规则：
> ```
> allow('admin', 'invoke', '*:/api/users')     ← 匹配列表页 GET:/api/users
> allow('admin', 'invoke', '*:/api/users/*')   ← 匹配子路径 GET:/api/users/123、DELETE:/api/users/456 等
> ```
> 或者使用全局通配 `*:/api/*`（匹配 `/api/` 下所有至少一段的子路径）。

> ⚠️ v1.0.0 不支持前缀通配（如 `*:/users*`），仅支持按 `/` 分段的末段 `*` 通配。前缀通配可在后续版本考虑。

### 2.2 db 资源通配

| pattern | 匹配示例 | 说明 |
|---------|---------|------|
| `db:users:*` | `db:users:email` ✓ | 末段通配（匹配该集合所有字段） |
| `db:*` | `db:users` ✓ / `db:users:email` ✗ | 单段通配（段数必须一致，只匹配集合级） |
| `db:*:*` | `db:users:email` ✓ / `db:users` ✗ | 双段通配（段数必须一致，只匹配字段级） |

> 💡 **段数一致规则**：`db:` 资源的 `*` 通配仅匹配同一段位的任意值，不会跨段匹配。`db:*` 匹配所有集合（2 段），`db:*:*` 匹配所有集合的所有字段（3 段），两者不互相覆盖。如需同时覆盖集合级和字段级，请使用全局通配 `*`。

## 三、两种优先级的区分

> ⚠️ **重要**：本方案存在两个独立的"优先级"概念，不可混淆

### 3.1 匹配精确度（wildcard 内部）

描述 pattern 的命中范围，**不影响规则权重**：

```
精确 > 路径参数(:id) > 末段通配(*) > Method通配(*:) > 全通配(*:*) > 全局(*)
```

### 3.2 规则权重（checker 内部）

决定鉴权最终结果：

| 模式 | 行为 | 说明 |
|------|------|------|
| `strict=true`（默认）| 任意 deny 命中 → `false` | deny 无条件全局优先 |
| `strict=false` | 最后命中的规则胜出 | 规则顺序决定优先级 |

**示例**：
```
规则: allow GET:/api/users  +  deny *:/api/users
strict=true  → deny 命中 → false（即使 allow 更精确，deny 始终优先）
strict=false → 取决于规则在 mergeRules 中的排列顺序
```

## 四、action 设计理据

> 💡 本节说明 permia 为什么选择 `read/write/create/update/delete/invoke/*` 作为 action 体系

### 4.1 设计原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | **通用性** | action 不绑定任何特定数据库（MongoDB/MySQL/PostgreSQL）或框架，任何 Node.js 项目都能理解 |
| 2 | **简洁性** | 覆盖应用层 95% 的权限场景，不引入数据库运维级操作（如索引管理、数据压缩） |
| 3 | **语义清晰** | CRUD 语义（`create/read/update/delete`）对所有开发者直觉友好，无学习成本 |
| 4 | **关注点分离** | permia 只判断"有没有权限"，不关心底层数据库如何执行操作 |

### 4.2 为什么不用 MongoDB 原生 action 体系

MongoDB 内置 RBAC 有 30+ 种 privilege action（`find`、`insert`、`remove`、`createIndex`、`compact`、`changeStream`、`dbStats`...），它们是**数据库运维级**操作，与 permia 的定位不匹配：

| 原因 | 说明 |
|------|------|
| **permia 是通用库** | 需要适配 MongoDB、MySQL、PostgreSQL、Redis 等任何后端。用 MongoDB 专属 action（如 `find`、`remove`、`changeStream`）会导致其他数据库用户困惑 |
| **语义层级不同** | MongoDB 的 `createIndex`、`compact`、`dbStats` 是数据库运维操作，应用层权限管理不需要这些 |
| **permia 不直接操作数据库** | permia 不会替用户执行 `createIndex()`，它只判断"有没有权限做这件事"。应用层用 `read/create/update/delete` 语义更清晰 |
| **MongoDB action 数量过多** | 30+ 种 action 中大部分在应用层权限管理中无意义，引入会增加复杂度 |

### 4.3 与行业方案 action 对照表

| permia | MongoDB | CASL | Casbin | 说明 |
|:------:|:-------:|:----:|:------:|------|
| `read` | `find` | `read` | `read` | 查询/读取 |
| `create` | `insert` | `create` | `write` | 新建记录 |
| `update` | `update` | `update` | `write` | 修改现有记录 |
| `delete` | `remove` | `delete` | `write` | 删除记录 |
| `write` | — | — | `write` | `create + update` 快捷方式（permia 特有语义） |
| `invoke` | — | — | — | HTTP 接口调用（MongoDB/CASL/Casbin 无此概念） |
| `*` | (all) | `manage` | `*` | 所有操作 |
| — | `createIndex` | — | — | permia 不涉及数据库运维操作 |
| — | `compact` | — | — | permia 不涉及数据库运维操作 |
| — | `changeStream` | — | — | permia 不涉及数据库运维操作 |
| — | `dbStats` | — | — | permia 不涉及数据库运维操作 |

### 4.4 `invoke` action 的设计理由

接口资源的 action 只有 `invoke` 和 `*` 两个选项。虽然 HTTP 方法已在资源路径中（`POST:/api/articles`），`invoke` 看似冗余，但它的存在有以下作用：

| 作用 | 说明 |
|------|------|
| **`validateAction` 校验点** | 接口资源需要一个明确的 action 值供校验函数判断合法性，否则无法区分"接口资源用了 db action（如 `read`）"的错误用法 |
| **语义统一** | 所有 `can(userId, action, resource)` 调用都需要传 action，`invoke` 明确表达"调用此接口"的语义 |
| **与 db action 正交** | `invoke` 保证接口权限和 db 权限的 action 空间不重叠，避免混淆 |

> 💡 后续版本可考虑：允许接口资源使用 `read`/`write` 等 action（将 `invoke` 作为默认值而非强制值），但 v1.0.0 保持简洁。

### 4.5 后续扩展方向（v2.0 考虑）

- **自定义 action 注册**：`Permia` 构造器增加 `customActions?: string[]` 配置项，`validateAction` 放行已注册的自定义 action（如 `export`、`approve`、`audit`）
- **action 分组 / 别名映射**：类似 `write = create + update`，允许用户定义自己的 action 组合（如 `manage = create + update + delete`）

## 五、核心数据结构

```typescript
/** 权限规则（存储 + 运行时通用） */
interface PermissionRule {
  type:     'allow' | 'deny';        // 规则类型
  action:   string;                   // 'read' / 'invoke' / '*'
  resource: string;                   // 'db:users' / 'GET:/api/users' / '*'
}

/** 角色数据 */
interface RoleData {
  id:          string;                // 角色唯一 ID（如 'admin', 'editor'）
  label:       string;                // 显示名称（如 '管理员', '编辑'）
  parent:      string | null;        // 父角色 ID（单继承，父角色须先创建）
  description: string;                // 描述
  createdAt:   number;                // 创建时间 timestamp ms
  updatedAt:   number;                // 更新时间 timestamp ms
}

/** 缓存条目（内部使用） */
interface CacheEntry {
  rules:     PermissionRule[];        // 已展开继承链的完整规则列表
  expiredAt: number;                  // performance.now() + ttl（单调时钟）
}

/** Permia 构造器选项 */
interface PermiaOptions {
  storage?:  StorageAdapter;          // 默认 new FileAdapter()
  cache?:    CacheOptions;            // 默认 { enabled: true, ttl: 300000 }
  strict?:   boolean;                 // 默认 true（deny 全局优先）
}

/** 缓存配置 */
interface CacheOptions {
  enabled:   boolean;                 // 默认 true
  ttl:       number;                  // 默认 300000（5 分钟，单位 ms）
}
```
