### tsup.config.ts 构建配置（🆕 v2.4 / 更新 v2.5）

> 💡 `tsup` 是唯一构建工具，负责 JS 产物 + `.d.ts` 生成。
> 🔴 **v2.5 修复**：tsup 默认将所有格式输出到同一目录，但 `package.json` 的 `exports` 指向 `dist/esm/` 和 `dist/cjs/` 子目录。
> 因此使用两次构建分别输出到 `dist/esm/` 和 `dist/cjs/`，通过 `defineConfig` 数组配置实现。

```typescript
import { defineConfig } from 'tsup';

const shared = {
  entry: ['src/index.ts', 'src/check/wildcard.ts'],   // 主入口 + permia/match 子路径
  dts: true,                                            // 自动生成 .d.ts 类型声明
  splitting: false,
  sourcemap: true,
  clean: true,                                          // 构建前清理输出目录
  target: 'node18' as const,
};

export default defineConfig([
  {
    ...shared,
    format: ['esm'],                                    // ESM 格式（.js + .d.ts）
    outDir: 'dist/esm',                                 // 🔴 输出到 dist/esm/（匹配 package.json exports）
    outExtension: () => ({ js: '.js' }),
  },
  {
    ...shared,
    format: ['cjs'],                                    // CJS 格式（.cjs + .d.cts）
    outDir: 'dist/cjs',                                 // 🔴 输出到 dist/cjs/（匹配 package.json exports）
    outExtension: () => ({ js: '.cjs' }),
    clean: false,                                       // clean 只清理当前 outDir，设 true 也安全；保持 false 更防御性
  },
]);
```

> 💡 **`npm run build`** 执行 `tsup`，tsup 读取上述数组配置，依次构建 ESM 和 CJS 两个格式到各自子目录。
> 构建后 `dist/` 结构与 `package.json` 的 `exports` 完全对齐：
> - `dist/esm/index.js` ← `"import": "./dist/esm/index.js"`
> - `dist/cjs/index.cjs` ← `"require": "./dist/cjs/index.cjs"`
> - `dist/esm/check/wildcard.js` ← `"./match"` 子路径 ESM 入口
> - `dist/cjs/check/wildcard.cjs` ← `"./match"` 子路径 CJS 入口

## 四、注意事项与边界说明

### 核心设计约束

1. **两种优先级不可混淆**：通配符匹配精确度（`matchResource` 内部）与规则权重（`checker` 逻辑）完全独立，详见 [资源路径模型 §三](./01-资源路径模型.md)
2. **🔴 必须 `await init()`**：所有适配器（含 MemoryAdapter）都必须先调用 `init()`。`Permia` 主类在 `init()` 中设置 `_initialized=true`，所有公共 API 入口检查此标志，未初始化时抛 `PermiaError(NOT_INITIALIZED)`，防止忘记 init 导致所有鉴权静默返回 false。CJS 项目需将 `await permia.init()` 包裹在 `async function main() { ... }` 中
3. **规则变更 → `invalidateAll`**：任何角色规则修改都触发全量缓存失效（v1.0.0 安全取舍，后续版本可精准失效）
4. **`setUserRoles` 全量校验**：先校验所有 roleId 存在，再写入，防止半写状态
5. **`mergeRules` 去重**：多角色共同继承同一父角色时，该父角色规则只追加一次
6. **`filterFields` 仅顶层字段**：v1.0.0 不支持嵌套字段（`address.city`），嵌套对象整体控制

### action 语义

7. **`write` action 双向语义**（🆕 v2.3）：
   - **规则侧（OR）**：`matchAction('write', 'create')` → `true`，`matchAction('write', 'delete')` → `false`。规则存储层仍存储原始 action 值，展开仅在匹配时发生
   - **请求侧（AND）**：`can(userId, 'write', resource)` 在 `checker.can()` 层面展开为 `can('create') && can('update')`。确保"分别配置 create + update"和"配置 write"语义一致
   - 详见 [API设计 §八 write 双向语义](./03-API设计.md)
8. **action 设计基于通用 CRUD 语义**：`read/write/create/update/delete/invoke/*`，不绑定任何特定数据库。详见 [资源路径模型 §四 action 设计理据](./01-资源路径模型.md)

### getResources / getPermissions

9. **`getResources` 语义与限制**：`action='invoke'` 只含接口资源；`action='read'` 只含 `db:` 资源；`action='*'` 混合返回（不推荐）。⚠️ 存在假阳性：当 deny 精确匹配而 allow 为通配符时，`getResources` 无法排除被 deny 的具体路径。前端最终权限判断应以 `can()` 为准
10. **`getResources` 与 JWT 大小**：规则多时返回的资源路径数组可能很大。JWT Token 通常受 HTTP header 大小限制（~8KB），建议只写入通配符模式而非展开后的完整列表
11. **`getPermissions` 不适合写 Token**：规则多时 Token 过大，推荐用 `getResources`（返回路径列表）

### 鉴权行为

12. **`userId=null/undefined`**：视为匿名，鉴权始终为 `false`（不抛异常，符合预期）
13. **`matchResource` 仅模块导出**：通过 `import { matchResource } from 'permia'` 使用，不是实例方法。前端场景推荐使用 `import { matchResource } from 'permia/match'` 子路径导入，避免打入 Node.js 专属代码
14. **METHOD 必须大写**：与 Node.js `req.method` 一致，`delete` 不会命中 `DELETE:` 规则

### 角色管理

15. **父角色先创建**：`parent` 引用的角色必须已存在，否则抛 `ROLE_NOT_FOUND`
16. **`roles.delete` 级联清理**：删除角色时完整流程：检查无子角色 → 清理规则（`deleteRules`）→ 从所有已绑定用户中移除该角色（`getUsersByRole` + 逐用户 `revoke`）→ 删除角色数据 → `invalidateAll()`。`getUsersByRole` 只查直接绑定的用户，通过继承链间接拥有该角色的用户不受影响
17. **`strict=false` 为 experimental**：`strict=false` 模式下规则顺序决定结果，但多角色合并时顺序取决于 `roleIds` 遍历和继承链 BFS 顺序，用户无法控制。**强烈建议使用 `strict=true`**（默认），v1.0.0 中 `strict=false` 标记为实验性
18. **orphan roleId 容错**：角色已删除但用户绑定未清理时，`resolveRoleChain` 跳过该角色，不抛异常

### 存储与缓存

19. **FileAdapter 生产注意**：默认适配器，磁盘 JSON 持久化。⚠️ 容器化部署需挂载数据卷；Serverless 环境不适用；多进程部署各进程独立内存副本存在数据竞争，需改用 MonSQLizeAdapter
20. **缓存 TTL 使用单调时钟**：`PermissionCache` 使用 `performance.now()` 而非 `Date.now()` 判断过期，避免系统时钟跳变（NTP 同步、VM 快照恢复等）影响

### 权限控制边界

21. **permia 是应用层权限判断引擎**（🆕 v2.3）：不是数据库代理，不会自动拦截数据库操作。接口权限通过 HTTP 中间件自动拦截；db 权限需要在 Service/DAO 层主动调用 `can()` / `assert()` / `filterFields()`。详见 [需求定义 §四 权限控制边界](../01-需求定义.md)、[跨框架接入 §五 db 权限使用流程](./06-跨框架接入.md)
22. **`filterFields` 签名变更**（🆕 v2.3）：`action` 参数从最后一个可选参数调整为第二个必填参数 — `filterFields(userId, action, resource, data)`，与 `can(userId, action, resource)` 保持一致。写入过滤场景需要明确传 `'update'` 或 `'create'`

### TypeScript 与构建

23. **TypeScript 源码即类型**（🆕 v2.3）：源码使用 TypeScript 编写，`tsup --dts` 自动生成 `.d.ts`，无需独立 `types/` 目录维护。所有公共类型集中在 `src/types/index.ts` 定义，由 `src/index.ts` 统一 re-export
24. **ESM + CJS 双格式构建**（🆕 v2.3 / 更新 v2.4）：使用 `tsup` 构建（唯一构建工具），输出 ESM（`.js` + `.d.ts`）和 CJS（`.cjs` + `.d.cts`）。`tsc --noEmit` 仅用于类型检查。`package.json` 的 `exports` 配置双入口，支持 `import` 和 `require`
25. **`permia/match` 子路径导出**（🆕 v2.3）：`package.json` 的 `exports` 配置 `"./match"` 子路径，指向 `check/wildcard`。前端场景推荐使用此子路径导入 `matchResource`，避免 tree-shake 不完全时打入 Node.js 专属代码（FileAdapter、`node:fs` 等）
26. **Node.js >= 18.0.0**（🆕 v2.3）：Node 16 已 EOL（2023-09-11），Node 18 对 ESM 和 TypeScript 工具链支持更好。CI 矩阵测试 Node 18 / 20 / 22

### 演进预留

27. **v2.0 演进预留**：单继承（`parent: string | null`）为 v1.0.0 设计，v2.0 考虑引入多继承或角色组（role group）机制，数据结构可扩展为 `parents: string[]`
28. **自定义 action 扩展**（v2.0 考虑）：`Permia` 构造器增加 `customActions?: string[]` 配置项，`validateAction` 放行已注册的自定义 action（如 `export`、`approve`、`audit`）
29. **db 权限自动拦截**（后期独立包）：`permia-monsqlize-adapter`（🔮 首选外部适配器）、`permia-mongoose-plugin` 等 ORM 插件，在 ORM hook 层自动调用 `can()` / `filterFields()`。核心库不包含，不强制依赖任何 ORM
30. **FileAdapter 并发写入保护**（🆕 v2.4）：`_writing` 锁 + `_pendingWrite` 重调度机制，防止 debounce 期间 `fs.writeFile` 并发执行导致文件损坏。详见 [引擎与存储设计 §2.2](./05-引擎与存储设计.md)
