## 一、鉴权引擎（lib/check/）

### 1.1 checker.js — 主流程

**构造器**：`Checker(storage, cache, strict)`

**`can` 核心流程**：
```
async can(userId, action, resource):
  rules = await _getRules(userId)
  // userId=null/undefined → getUserRoles([]) → rules=[] → return false（匿名用户）

  if strict:                                   // 默认 true
    for rule of rules where type==='deny':
      if matchRule(rule, action, resource): return false   // deny 命中立即拒绝
    for rule of rules where type==='allow':
      if matchRule(rule, action, resource): return true    // allow 命中通过
    return false                               // 无 allow 命中 → 默认拒绝（白名单）

  else (strict=false):
    result = false
    for rule of rules:
      if matchRule(rule, action, resource):
        result = (rule.type === 'allow')       // 最后命中的规则胜出
    return result

async _getRules(userId):
  cached = cache.get(userId)
  if cached: return cached
  roleIds = await storage.getUserRoles(userId)
  rules = await resolver.mergeRules(roleIds, storage, strict)
  cache.set(userId, rules)
  return rules
```

**`filterFields` 流程**：
```
async filterFields(userId, resource, data, action='read'):
  ① 校验 resource 以 'db:' 开头，否则抛 INVALID_RESOURCE_PATH
  ② 校验 data 为非 null 普通对象，否则抛 INVALID_ARGUMENT
  ③ clone = Object.assign({}, data)    // 浅克隆（v1.0.0 只处理顶层字段）
  ④ for key of Object.keys(clone):
       fieldResource = resource + ':' + key   // 'db:users:email'
       if !await can(userId, action, fieldResource): delete clone[key]
  ⑤ return clone
```

**`getResources` 流程**（deny 过滤使用通配匹配）：
```
async getResources(userId, action='invoke'):
  rules = await _getRules(userId)
  allows = rules.filter(r => r.type==='allow' && matchAction(r.action, action))
  denies = rules.filter(r => r.type==='deny'  && matchAction(r.action, action))
  result = []
  for allowRule of allows:
    // strict=true: 检查是否被任意 deny 通配覆盖
    covered = strict && denies.some(d => matchResource(d.resource, allowRule.resource))
    if !covered: result.push(allowRule.resource)
  return [...new Set(result)]
  // action='invoke' → 只含接口资源（db: 规则 action 是 read 等，matchAction 不命中）
  // action='read'   → 只含 db: 资源
  // action='*'      → 混合返回（不推荐）
```

### 1.2 resolver.js — 继承链解析

```
// BFS 展开单个角色继承链
async resolveRoleChain(roleId, storage):
  visited = new Set(), queue = [roleId], result = []
  while queue.length > 0:
    current = queue.shift()
    if visited.has(current): throw PermiaError(CIRCULAR_INHERITANCE)  // 检测循环
    visited.add(current)
    result.push(current)
    role = await storage.getRole(current)
    if !role: continue                   // 容错：已删除角色直接跳过
    if role.parent:
      if visited.has(role.parent): throw PermiaError(CIRCULAR_INHERITANCE)  // 提前检测
      queue.push(role.parent)
  return result   // [self, parent, grandparent, ...]  子→父顺序

// 合并多个角色的规则
async mergeRules(roleIds, storage, strict):
  allRules = [], processedIds = new Set()
  for roleId of roleIds:
    chain = await resolveRoleChain(roleId, storage)
    for id of chain:
      if processedIds.has(id): continue  // 去重：多角色共同继承同一父角色时，规则只追加一次
      processedIds.add(id)
      allRules.push(...await storage.getRules(id))
  if strict:
    // deny 前置，确保 can() 遍历 deny 时在 allow 之前
    return [...allRules.filter(r=>r.type==='deny'), ...allRules.filter(r=>r.type==='allow')]
  return allRules
```

### 1.3 wildcard.js — 路径匹配（同时对外导出）

```javascript
// 模块导出
module.exports = { matchResource, matchAction, matchRule }

// 匹配资源路径
matchResource(pattern, resource):
  if pattern === '*': return true          // 全局通配
  if pattern === resource: return true     // 精确匹配（快速路径）

  接口资源（含 ':/'）:
    拆分 [method, path]，method 精确或 *，path 走 pathMatch

  db 资源（'db:' 开头）:
    按 ':' 拆段，长度必须相同（除非末段是 *）

// 路径段匹配
pathMatch(patternPath, resourcePath):
  按 '/' 拆段（过滤空串）逐段对比:
    ':param' 或任意 ':' 开头 → 匹配任意单段（路径参数）
    末段 '*'                  → 匹配剩余所有段（至少 1 段）
    其他                      → 精确匹配（大小写敏感）

// action 匹配
matchAction(pattern, action):
  return pattern === '*' || pattern === action

// 规则匹配（组合）
matchRule(rule, action, resource):
  return matchAction(rule.action, action) && matchResource(rule.resource, resource)
```

## 二、存储适配器（lib/storage/）

### 2.1 抽象基类 StorageAdapter

```javascript
class StorageAdapter {
  async init()                          // 异步初始化（FileAdapter 读磁盘）
  async close()                         // 关闭清理（FileAdapter flush）

  // 角色
  async getRoles()                      // → Map<string, RoleData>   全量
  async getRole(id)                     // → RoleData | null          单条
  async setRole(id, roleData)           // → void
  async deleteRole(id)                  // → void

  // 用户角色绑定
  async getUserRoles(userId)            // → string[]（不存在返回 []）
  async setUserRoles(userId, roleIds)   // → void
  async getUsersByRole(roleId)          // → string[]（反查直接绑定该角色的用户）

  // 权限规则
  async getRules(roleId)                // → PermissionRule[]（不存在返回 []）
  async setRules(roleId, rules)         // → void
  async deleteRules(roleId)             // → void（delete 角色时清规则）
}
// 所有方法默认 throw new Error('Not implemented')
```

### 2.2 FileAdapter（默认适配器）

```
FileAdapter({ path = './permia-data.json' })

内存结构:
  _data     = { roles:{}, userRoles:{}, rules:{} }   ← 全量数据
  _roleUsers = new Map()   // 反向索引 Map<roleId, Set<userId>>，加速 getUsersByRole
  _debounceTimer = null

init():
  → fs.promises.readFile(path) 异步读
  → JSON.parse（失败抛 STORAGE_ERROR，提示文件路径）
  → 文件不存在 → 使用空结构（自动创建）
  → 遍历 userRoles 构建 _roleUsers 反向索引

close():
  → clearTimeout(_debounceTimer)
  → await _writeToDisk()（确保进程退出前数据落盘）

写操作（所有写方法统一流程）:
  1. 更新 _data 内存副本（同步）
  2. 如涉及 userRoles 变更，同步更新 _roleUsers 反向索引（diff 算法）
  3. _scheduleWrite()：debounce 100ms 触发 _writeToDisk()

setUserRoles 反向索引 diff 算法:
  oldIds = _data.userRoles[userId] || []
  removed = oldIds.filter(r => !roleIds.includes(r))
  added   = roleIds.filter(r => !oldIds.includes(r))
  for r of removed: _roleUsers.get(r)?.delete(userId)
  for r of added:   if !_roleUsers.has(r) _roleUsers.set(r, new Set())
                    _roleUsers.get(r).add(userId)
  _data.userRoles[userId] = roleIds

并发安全:
  Node.js 单线程事件循环，内存操作无竞态
  多进程部署 → 各进程独立内存副本，数据不一致 → 改用 MonSQLizeAdapter
```

### 2.3 MemoryAdapter（测试用）

```
init() / close() → no-op
基于 Object + Map 实现所有接口（含反向索引维护）
进程重启数据清空，仅用于单元测试
```

### 2.4 适配器策略对比

| 适配器 | 默认 | 持久化 | 需 init() | 多进程 | 适用场景 |
|--------|------|--------|----------|--------|---------|
| `FileAdapter` | ✅ | ✅ 磁盘 JSON | 是 | ❌ | 单实例生产 |
| `MemoryAdapter` | — | ❌ | no-op | — | 单元测试 |
| `MonSQLizeAdapter` | — | ✅ MongoDB | 是 | ✅ | 多实例部署（后期独立包） |

## 四、错误体系（lib/core/errors.js）

```javascript
class PermiaError extends Error {
  constructor(code, message, data?)
  code: string     // 错误码（见下表）
  data: any        // 附加上下文（userId / roleId / resource 等）
}
```

| 错误码 | 触发场景 |
|--------|---------|
| `PERMISSION_DENIED` | `assert()` 鉴权失败 |
| `ROLE_NOT_FOUND` | `get/allow/deny/assign` 引用不存在的角色 |
| `ROLE_ALREADY_EXISTS` | `create` 时 roleId 重复 |
| `CIRCULAR_INHERITANCE` | `create/update` 时 parent 形成环 |
| `INVALID_RESOURCE_PATH` | 资源路径格式不合法 |
| `INVALID_ACTION` | action 与资源类型不匹配 |
| `INVALID_ARGUMENT` | 入参非法（如 `filterFields(data=null)`） |
| `STORAGE_ERROR` | 存储层错误（如 FileAdapter JSON 解析失败） |

### 5.2 validation.js

```
validateResource(resource):
  合法格式:
    'db:<name>'  /  'db:<name>:<field>'  /  'db:*'  /  'db:<name>:*'
    '<METHOD>:<path>'  — METHOD 在 HTTP_METHODS 中，path 以 / 开头
    '*'
  否则抛 INVALID_RESOURCE_PATH

validateAction(action, resource):
  resource 以 'db:' → action 须在 DB_ACTIONS
  resource 含 ':/'  → action 须在 API_ACTIONS
  resource === '*'  → action 任意
  否则抛 INVALID_ACTION
```

deduplicate(rules):  去除 type+action+resource 三者完全相同的重复规则
