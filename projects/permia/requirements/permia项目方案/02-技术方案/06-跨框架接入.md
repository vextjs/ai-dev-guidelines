## 一、接入模式总览

| 模式 | 时机 | 核心 API | 适用场景 |
|------|------|---------|---------|
| **实时鉴权** | 每次 HTTP 请求 | `assert(userId, 'invoke', resource)` | 中间件拦截，每请求验证 |
| **权限拉取** | 用户登录时 | `getResources(userId, 'invoke')` | 写入 JWT Token / 前端初始化菜单 |
| **db 权限检查** | Service/DAO 层 | `can(userId, action, resource)` / `assert(...)` | 数据库集合级读写控制 |
| **字段过滤** | 查询数据后 / 写入前 | `filterFields(userId, action, resource, data)` | 敏感字段脱敏返回 / 写入字段过滤 |
| **前端判断** | 页面渲染时 | `matchResource(pattern, path)` | 按钮/栏目显隐控制 |

> ⚠️ **权限控制边界**：permia 是应用层权限判断引擎，接口权限通过 HTTP 中间件自动拦截，db 权限需要在 Service/DAO 层主动调用。permia **不会**自动拦截数据库操作。详见 [需求定义 §四 权限控制边界](../01-需求定义.md)。

**接入流程**：
```
其他框架 userId → permia.users.assign(userId, roleId)
用户登录 → getResources → 写 JWT
每次请求 → assert(userId, 'invoke', METHOD:path) → 通过/403
Service 层 → assert(userId, 'read', 'db:articles') → 通过/拒绝
查询后   → filterFields(userId, 'read', 'db:collection', data) → 脱敏数据
写入前   → filterFields(userId, 'update', 'db:collection', payload) → 安全载荷
```

## 二、Express 完整接入示例

> ⚠️ **ESM 注意**：以下示例使用 top-level `await`，需要 ESM 环境（`package.json` 中 `"type": "module"`）。
> 如果你的项目使用 CJS，请将初始化代码包裹在 `async function main() { ... }` 中：
> ```typescript
> async function main() {
>   const permia = new Permia();
>   await permia.init();
>   // ... 其余代码 ...
>   app.listen(3000);
> }
> main();
> ```

### TypeScript 类型扩展（`req.userId`）

Express 的 `Request` 对象默认没有 `userId` 属性，需要在项目中声明类型扩展：

```typescript
// src/types/express.d.ts（或任意 .d.ts 文件，确保被 tsconfig 包含）
declare namespace Express {
  interface Request {
    userId: string;   // 由认证中间件（如 JWT 验证）挂载
  }
}
```

### 完整示例

```typescript
import express from 'express';
import jwt from 'jsonwebtoken';
import { Permia, PermiaError, PermiaErrorCode, matchResource } from 'permia';

const app = express();
app.use(express.json());
const SECRET = process.env.JWT_SECRET!;

// ① 初始化 permia
const permia = new Permia();
await permia.init();   // 🔴 ESM top-level await；CJS 请包裹在 async function 中

// ② 预置权限（通常由管理后台动态配置，此处为示例）
await permia.roles.create('admin', { label: '管理员' });
await permia.roles.allow('admin', 'invoke', '*:/api/*');
await permia.roles.allow('admin', '*', 'db:*');
await permia.users.assign('user-001', 'admin');

// ③ 公共接口白名单（无需登录即可访问）
const PUBLIC_PATHS = ['/api/auth/login', '/api/auth/register', '/health'];

// ④ 认证中间件（JWT 解析 → 挂载 req.userId）
// 🔴 必须在鉴权中间件之前注册，否则 req.userId 为 undefined
app.use((req, res, next) => {
  // 公共接口跳过认证
  if (PUBLIC_PATHS.includes(req.path)) return next();

  const token = req.headers.authorization?.replace('Bearer ', '');
  if (!token) return res.status(401).json({ message: 'Missing token' });

  try {
    const payload = jwt.verify(token, SECRET) as { userId: string };
    req.userId = payload.userId;
    next();
  } catch {
    return res.status(401).json({ message: 'Invalid token' });
  }
});

// ⑤ 鉴权中间件（每次请求执行 — 接口级权限）
app.use(async (req, res, next) => {
  // 公共接口跳过鉴权（与认证白名单保持一致）
  if (PUBLIC_PATHS.includes(req.path)) return next();

  const resource = `${req.method}:${req.path}`;  // 'DELETE:/api/users/123'
  try {
    await permia.assert(req.userId, 'invoke', resource);
    next();
  } catch (e) {
    if (e instanceof PermiaError && e.code === PermiaErrorCode.PERMISSION_DENIED) {
      return res.status(403).json({ code: e.code, message: e.message });
    }
    next(e);   // STORAGE_ERROR 等透传给错误中间件
  }
});

// ⑥ 登录接口（在白名单中，不经过认证/鉴权中间件）
app.post('/api/auth/login', async (req, res) => {
  const userId = await authenticate(req.body);   // 业务层认证
  const resources: string[] = await permia.getResources(userId, 'invoke');
  // resources = ['*:/api/users/*', 'GET:/api/articles', ...]
  const token = jwt.sign({ userId, resources }, SECRET, { expiresIn: '8h' });
  res.json({ token });
  // 前端拿到 resources 渲染菜单和按钮
});

// ⑦ 查询后字段过滤（db 权限 — 在路由处理器中主动调用）
app.get('/api/users/:id', async (req, res) => {
  const user = await db.findUser(req.params.id);
  const safe = await permia.filterFields(req.userId, 'read', 'db:users', user);
  res.json(safe);   // 仅含当前用户有权查看的字段
});

// ⑧ 优雅关闭
process.on('SIGTERM', async () => {
  await permia.close();
  process.exit(0);
});

app.listen(3000);
```

> 💡 **中间件执行顺序**：认证中间件（④）→ 鉴权中间件（⑤）→ 路由处理器。认证中间件解析 JWT 并挂载 `req.userId`，鉴权中间件使用 `req.userId` 做权限判断。公共接口（登录/注册/健康检查）在两个中间件中均跳过。
>
> ⚠️ **白名单注意**：`PUBLIC_PATHS` 使用精确路径匹配（`req.path`）。生产环境可改用前缀匹配（如 `req.path.startsWith('/api/auth/')`）或路由分组（将公共路由注册在中间件之前）。

## 三、Koa 接入示例

```typescript
import Koa from 'koa';
import { Permia, PermiaError, PermiaErrorCode } from 'permia';

const app = new Koa();
const permia = new Permia();
await permia.init();

// 鉴权中间件（接口级权限）
app.use(async (ctx, next) => {
  const resource = `${ctx.method}:${ctx.path}`;  // Koa 用 ctx.method / ctx.path
  try {
    await permia.assert(ctx.state.userId, 'invoke', resource);
    await next();
  } catch (e) {
    if (e instanceof PermiaError && e.code === PermiaErrorCode.PERMISSION_DENIED) {
      ctx.status = 403;
      ctx.body = { code: e.code, message: e.message };
    } else {
      throw e;
    }
  }
});
```

## 五、db 权限完整使用流程（Service/DAO 层）

> ⚠️ **重要**：permia 是**应用层权限判断引擎**，不会自动拦截数据库操作。db 权限需要在 Service/DAO 层**主动调用** `can()` / `assert()` / `filterFields()`。
>
> 接口权限通过 HTTP 中间件自动拦截（见 §二~§四），db 权限则需要在业务逻辑中手动集成。

### 5.1 完整 Service 层示例

```typescript
import { Permia, PermiaError, PermiaErrorCode } from 'permia';

class ArticleService {
  constructor(
    private permia: Permia,
    private db: any,  // 数据库客户端（MongoDB / MySQL 等，permia 不关心具体类型）
  ) {}

  /**
   * 读取文章 — 集合级权限 + 字段级过滤
   */
  async getArticle(userId: string, articleId: string): Promise<Record<string, unknown>> {
    // ① 检查集合级读权限（无权限抛 PERMISSION_DENIED）
    await this.permia.assert(userId, 'read', 'db:articles');

    // ② 查询数据库（permia 不参与此步骤）
    const article = await this.db.collection('articles').findOne({ _id: articleId });
    if (!article) throw new Error('Article not found');

    // ③ 字段级过滤 — 只返回用户有权读取的字段
    return this.permia.filterFields(userId, 'read', 'db:articles', article);
  }

  /**
   * 读取文章列表 — 批量字段过滤
   */
  async listArticles(userId: string): Promise<Record<string, unknown>[]> {
    // ① 检查集合级读权限
    await this.permia.assert(userId, 'read', 'db:articles');

    // ② 查询数据库
    const articles: Record<string, unknown>[] = await this.db.collection('articles').find().toArray();

    // ③ 逐条字段过滤（v1.0.0 不支持批量 filterFields，需循环处理）
    return Promise.all(
      articles.map(article =>
        this.permia.filterFields(userId, 'read', 'db:articles', article)
      )
    );
  }

  /**
   * 更新文章 — 写入前字段过滤
   */
  async updateArticle(
    userId: string,
    articleId: string,
    payload: Record<string, unknown>,
  ): Promise<void> {
    // ① 检查集合级更新权限
    await this.permia.assert(userId, 'update', 'db:articles');

    // ② 过滤用户无权修改的字段（如 editor 不能改 status、pinned 等）
    const safePayload = await this.permia.filterFields(userId, 'update', 'db:articles', payload);

    // ③ 安全检查：过滤后为空则无需执行更新
    if (Object.keys(safePayload).length === 0) {
      throw new PermiaError(
        PermiaErrorCode.PERMISSION_DENIED,
        `User ${userId} has no writable fields on db:articles`,
      );
    }

    // ④ 执行更新（permia 不参与此步骤）
    await this.db.collection('articles').updateOne(
      { _id: articleId },
      { $set: safePayload },
    );
  }

  /**
   * 创建文章 — 写入前字段过滤
   */
  async createArticle(userId: string, data: Record<string, unknown>): Promise<string> {
    // ① 检查集合级创建权限
    await this.permia.assert(userId, 'create', 'db:articles');

    // ② 过滤无权提交的字段
    const safeData = await this.permia.filterFields(userId, 'create', 'db:articles', data);

    // ③ 插入数据库
    const result = await this.db.collection('articles').insertOne(safeData);
    return result.insertedId;
  }

  /**
   * 删除文章
   */
  async deleteArticle(userId: string, articleId: string): Promise<void> {
    // 检查删除权限（无需字段过滤）
    await this.permia.assert(userId, 'delete', 'db:articles');
    await this.db.collection('articles').deleteOne({ _id: articleId });
  }

  /**
   * 使用 can() 做条件判断（而非 assert 抛异常）
   */
  async getArticleWithOptionalSecret(userId: string, articleId: string): Promise<Record<string, unknown>> {
    await this.permia.assert(userId, 'read', 'db:articles');
    const article = await this.db.collection('articles').findOne({ _id: articleId });
    if (!article) throw new Error('Article not found');

    // 使用 can() 做条件判断 — 不抛异常
    const canReadSecret = await this.permia.can(userId, 'read', 'db:articles:secretField');
    if (!canReadSecret) {
      delete article.secretField;
    }
    return article;
  }
}
```

### 5.2 在路由中使用 Service

```typescript
import express from 'express';

const app = express();
const permia = new Permia();
await permia.init();

const articleService = new ArticleService(permia, db);

// 路由处理器 — 中间件已处理接口级权限（§二），此处 Service 层处理 db 权限
app.get('/api/articles/:id', async (req, res, next) => {
  try {
    const article = await articleService.getArticle(req.userId, req.params.id);
    res.json(article);
  } catch (e) {
    if (e instanceof PermiaError && e.code === PermiaErrorCode.PERMISSION_DENIED) {
      return res.status(403).json({ code: e.code, message: e.message });
    }
    next(e);
  }
});

app.put('/api/articles/:id', async (req, res, next) => {
  try {
    await articleService.updateArticle(req.userId, req.params.id, req.body);
    res.json({ ok: true });
  } catch (e) {
    if (e instanceof PermiaError && e.code === PermiaErrorCode.PERMISSION_DENIED) {
      return res.status(403).json({ code: e.code, message: e.message });
    }
    next(e);
  }
});
```

### 5.3 db 权限与接口权限的关系

| 场景 | 接口权限（中间件） | db 权限（Service 层） | 说明 |
|------|:---:|:---:|------|
| 只需接口级拦截 | ✅ `assert('invoke', 'POST:/api/articles')` | — | 大部分 CRUD 接口只需接口权限 |
| 只需 db 级控制 | — | ✅ `assert('read', 'db:articles')` | 不走 HTTP 的内部调用（如定时任务、队列消费者） |
| 两者同时使用 | ✅ 中间件拦截无权接口 | ✅ Service 层字段过滤 | 接口权限做粗粒度门禁，db 权限做细粒度字段控制 |

> 💡 **推荐模式**：接口权限（中间件）负责"能不能访问这个 API"，db 权限（Service 层）负责"能不能读写这个集合/字段"。两者正交，按需组合。
>
> 如果同一操作同时配置了两种权限（如 `allow('editor', 'invoke', 'POST:/api/articles')` + `allow('editor', 'create', 'db:articles')`），两道检查都会执行 — 接口权限在中间件层拦截，db 权限在 Service 层拦截。

### 5.4 后期：ORM 自动拦截插件（v1.0.0 不包含）

如需**自动拦截**数据库操作（无需在每个 Service 方法中手动调用 `can()` / `filterFields()`），可通过 ORM/ODM 的 hook 机制实现：

```typescript
// 🔮 后期规划 — permia-mongoose-plugin（独立包，非 v1.0.0 范围）
import { permiaPlugin } from 'permia-mongoose-plugin';

const ArticleSchema = new Schema({ title: String, content: String, status: String });

// 注册插件 — 自动在 find/save/updateOne 等操作前调用 permia.assert()
ArticleSchema.plugin(permiaPlugin, {
  permia,
  resource: 'db:articles',
  // 自动映射：find → 'read', save → 'create', updateOne → 'update', deleteOne → 'delete'
});

// 使用时无需手动调用 permia — 插件在 mongoose middleware 中自动处理
const article = await Article.findById(articleId);  // 自动检查 read 权限
await Article.updateOne({ _id: articleId }, { $set: payload }); // 自动检查 update 权限 + 字段过滤
```

> ⚠️ 这是后期规划，v1.0.0 核心库不包含 ORM 插件。当前请在 Service/DAO 层手动调用 permia API。

## 六、前端菜单 / 按钮权限判断

登录时服务端返回 `resources`（通过 `getResources` 获取），前端存入 Token 或状态管理，渲染时使用 `matchResource` 判断。

### 6.1 前端打包说明

```typescript
// 🔴 推荐：使用子路径导入，避免打入 Node.js 专属代码（FileAdapter 等）
import { matchResource } from 'permia/match';

// ⚠️ 不推荐：直接从主入口导入，可能打入 FileAdapter、fs 等 Node.js 模块
// import { matchResource } from 'permia';
```

> `package.json` 的 `exports` 已配置 `"./match"` 子路径，指向 `dist/esm/check/wildcard.js`，仅包含纯 JavaScript 匹配逻辑，无 Node.js 专属依赖。
>
> 如果项目不使用 npm 包管理（如 CDN 场景），可以直接复制 `matchResource` 函数源码到前端项目中 — 该函数是纯函数，无外部依赖。

### 6.2 按钮级判断

```typescript
import { matchResource } from 'permia/match';

// 从 JWT payload 或状态管理中取出 resources
const resources: string[] = useAuth().resources;
// resources = ['*:/api/users/*', 'GET:/api/articles', 'POST:/api/articles']

// 按钮级判断
const canDeleteUser = resources.some(p => matchResource(p, 'DELETE:/api/users/123'));
// → true（*:/api/users/* 匹配）

const canCreateArticle = resources.some(p => matchResource(p, 'POST:/api/articles'));
// → true

const canDeleteArticle = resources.some(p => matchResource(p, 'DELETE:/api/articles/1'));
// → false（无匹配规则）
```

### 6.3 菜单栏目级判断

```typescript
interface MenuItem {
  name: string;
  testPath: string;
}

const menus: MenuItem[] = [
  { name: '用户管理', testPath: 'GET:/api/users' },
  { name: '文章管理', testPath: 'GET:/api/articles' },
  { name: '系统设置', testPath: 'GET:/api/settings' },
];

const visibleMenus = menus.filter(m =>
  resources.some(p => matchResource(p, m.testPath))
);
// → [{ name: '用户管理', ... }, { name: '文章管理', ... }]
// 系统设置无权限不显示
```

### 6.4 路由守卫

```typescript
// Vue Router 示例
router.beforeEach((to) => {
  const requiredResource = to.meta?.requiredResource as string | undefined;
  if (!requiredResource) return true;   // 无限制的路由直接放行

  const resources: string[] = useAuth().resources;
  const allowed = resources.some(p => matchResource(p, requiredResource));
  if (!allowed) return { path: '/403' };
});
```

## 七、跨服务接入（微服务场景）

```yaml
场景: 服务A 有自己的用户体系，通过 permia 管理权限

架构:
  服务A 的 userId（如 MongoDB ObjectId）→ 直接作为 permia 的 userId 使用
  permia 不关心 userId 的格式，任何 string 均可

接入步骤:
  1. 用户在服务A 注册后: permia.users.assign(userId, defaultRole)
  2. 用户登录时: permia.getResources(userId, 'invoke') → 写入 Token
  3. 每次 API 请求: 鉴权中间件 permia.assert(userId, 'invoke', resource)
  4. Service 层: permia.assert(userId, 'read', 'db:collection') + filterFields
  5. 权限变更时: permia.roles.allow/deny 修改角色规则 → 缓存自动失效
```

## 八、接入模式决策指南

```
用户请求 → 需要权限控制吗？
  │
  ├─ 只需接口级拦截（谁能调哪个 API）
  │   └─ 使用接口权限中间件（§二~§四）
  │
  ├─ 需要控制数据库读写（谁能读写哪个集合）
  │   └─ 使用 db 权限（§五 — Service 层 can/assert）
  │
  ├─ 需要字段级控制（谁能看/改哪些字段）
  │   └─ 使用 db 字段权限（§五 — filterFields）
  │
  ├─ 前端菜单/按钮显隐
  │   └─ 使用 getResources + matchResource（§六）
  │
  └─ 以上都需要
      └─ 接口权限（中间件门禁）+ db 权限（Service 层细粒度）+ 前端渲染
```
