## 🚫 架构约束（强制）

### 1. 禁止跳过预检查

每次任务开始前必须输出预检查结果（5 项必做），不得看到需求就直接执行。

### 2. 禁止使用 glob/find_path 扫描隐藏目录

`.ai-memory/` 等以 `.` 开头的隐藏目录必须使用 `list_directory` 逐层进入，glob 引擎会跳过隐藏目录导致误判。

### 3. 禁止未经确认修改/删除文件

涉及文件修改或删除操作时，必须先输出变更计划并等待用户确认（纯分析、只读、报告、记忆写入除外）。

### 4. 禁止凭想象设计方案

未实际读取文件就输出分析结论或设计方案。所有判断必须基于实际文件内容。

### 5. 禁止报告基于推断

报告中列出的问题必须经过实际文件读取验证，不得基于推断或猜测，问题须按 🔴/🟡/💡/❌ 分类。

### 6. 禁止日期字段留占位符

所有输出文件的日期字段必须替换为真实日期（通过 `now()` 工具获取），不得保留 `YYYY-MM-DD` 占位符。

### 7. 禁止文件名不带日期和序号

报告文件存放于 `reports/<子目录>/<agent>/YYYYMMDD/` 目录下，文件名为 `NN-<类型>-<简述>.md`（NN 为当日递增序号，仅扫描该目录）。记忆文件格式为 `YYYYMMDD.md`（每天一文件，会话内以 `## 会话 NN` 分段）。

### 8. 禁止删除 STATUS.md / CHANGELOG.md

任何情况下不得删除项目状态和变更日志文件。

### 9. 禁止跨 Agent 写入记忆

记忆文件必须写入 `clients/<当前agent>/` 目录，不得写入其他 Agent 的目录。

### 10. 禁止修改规范后不做交叉验证

修改规范文件后，必须逐个检查所有引用该规范的文件是否需要同步更新。

## 📐 结构约束

### 1. 单一职责

每个工作流目录（`core/workflows/XX-*/`）只负责一种任务类型，不得混合多种任务流程。

### 2. 模板分层

模板严格按层级组织：`lite/`（小需求精简）、`core/`（大需求完整）、`extended/`（扩展场景）、`common/`（通用组件）。新模板必须归入正确层级。

### 3. 项目规范模块化

每个项目的 `profile/` 目录必须包含模块化文件（01-项目信息、02-架构约束、03-代码风格），不得使用单文件合体式。

### 4. 报告按项目存储

报告文件只能存放在 `projects/<project>/reports/` 下，不得在顶层创建 `reports/` 目录。

### 5. 记忆按 Agent 隔离

记忆文件按客户端目录隔离：`.ai-memory/clients/<agent>/tasks/`，全局摘要放在 `.ai-memory/SUMMARY.md`。

---

## ⚠️ 例外情况

所有例外必须在相关文件中添加注释说明原因，并在报告中记录。
