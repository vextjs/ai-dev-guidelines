## 一、统一错误码体系

### 1.1 错误码格式规范

每个错误码**必须**以对象形式定义，包含 `code`（数字）和 `message`（字符串）两个字段：

```typescript
// ✅ 正确格式：对象格式（必须）
'USER_NOT_FOUND': {
    code: 40001,
    message: '用户不存在'
}

// ❌ 错误格式：简单字符串（已废弃，禁止新增）
'USER_NOT_FOUND': '用户不存在'
```

| 字段 | 类型 | 要求 | 说明 |
|------|------|------|------|
| `code` | `number` | 全局唯一 | 按模块分段，见第二节 |
| `message` | `string` | 每语言不同 | 面向用户的错误描述 |

### 1.2 唯一配置位置（🔴 强制）

所有错误码**只能**在 `config/dsl/` 目录配置，**禁止**在其他位置定义：

```
config/dsl/                ← ✅ 唯一合法位置
├── index.ts               # 导出 dslLocales，供 app.ts 加载
├── zh.ts                  # 简体中文（必须）
├── en.ts                  # 英文（必须）
└── hk.ts                  # 繁体中文（必须）
```

> ❌ 禁止在 `config/err_codes.ts` 中添加新错误码（历史遗留，仅保留不再修改）  
> ❌ 禁止在代码中硬编码错误消息字符串  
> ❌ 禁止在 Service / Controller 文件中定义错误码

### 1.3 app.ts 加载方式

```typescript
// app.ts
const { dsl } = require('schema-dsl');
const { dslLocales } = require('./config/dsl');

dsl.config({
    i18n: dslLocales
});
```

```typescript
// config/dsl/index.ts
import zh from './zh';
import en from './en';
import hk from './hk';

export const dslLocales = { zh, en, hk };
```

## 二、错误码分类与分段规则

### 2.1 分段规则

| 段 | 数值范围 | 分类 | 说明 |
|----|---------|------|------|
| `4xxxx` | 40000 - 49999 | **业务错误** | 客户端操作引起，如资源不存在、权限不足 |
| `5xxxx` | 50000 - 59999 | **系统错误** | 服务端内部错误，如数据库故障、第三方 API 失败 |
| `8xxxx` | 80000 - 89999 | **参数验证错误** | 请求参数格式不符合要求 |

### 2.2 详细分段建议

#### 4xxxx — 业务错误

| 子段 | 范围 | 用途 |
|------|------|------|
| 用户相关 | 40001 - 40099 | 用户不存在、邮箱已注册、密码错误等 |
| 权限相关 | 40100 - 40199 | 无权限、Token 过期等 |
| 资源相关 | 40200 - 40299 | 资源不存在、资源已删除等 |
| 业务约束 | 40300 - 40399 | 库存不足、余额不足、超出限制等 |

#### 5xxxx — 系统错误

| 子段 | 范围 | 用途 |
|------|------|------|
| 数据库错误 | 50001 - 50099 | 查询失败、写入失败等 |
| 第三方 API | 50100 - 50199 | 支付 API 失败、短信发送失败等 |
| 内部服务 | 50200 - 50299 | 服务间调用失败等 |

#### 8xxxx — 参数验证错误

| 子段 | 范围 | 用途 |
|------|------|------|
| 通用参数错误 | 80001 - 80099 | 参数格式不正确、必填参数缺失等 |
| 限流错误 | 82001 - 82010 | 请求过于频繁、IP 白名单拦截等 |

### 2.3 已使用错误码示例

```typescript
// config/dsl/zh.ts（示例）
export default {
    // 用户相关（40001 - 40099）
    'USER_NOT_FOUND':          { code: 40001, message: '用户不存在' },
    'EMAIL_ALREADY_EXISTS':    { code: 40002, message: '邮箱已被占用' },
    'USER_DISABLED':           { code: 40003, message: '用户已被禁用' },
    'PASSWORD_INCORRECT':      { code: 40004, message: '密码错误' },

    // 系统错误（50001 - 50099）
    'QUERY_FAILED':            { code: 50001, message: '查询失败，请稍后重试' },
    'USER_CREATE_FAILED':      { code: 50002, message: '用户创建失败' },
    'USER_UPDATE_FAILED':      { code: 50003, message: '用户更新失败' },

    // 参数验证（80001 - 80099）
    'PARAM_INVALID':           { code: 80001, message: '参数格式不正确' },

    // 限流（82001 - 82010）
    'RATE_LIMIT_EXCEEDED':     { code: 82001, message: '请求过于频繁，请稍后再试' },
    'RATE_LIMIT_IP_EXCEEDED':  { code: 82002, message: 'IP 请求过于频繁，请稍后再试' },
};
```

### 3.2 新增错误码操作步骤

1. 确认错误码所属分类（业务/系统/参数），选择对应数值段
2. 在 `config/dsl/zh.ts` 中添加简体中文配置
3. 在 `config/dsl/en.ts` 中添加英文配置（`code` 值必须与 zh.ts 一致）
4. 在 `config/dsl/hk.ts` 中添加繁体中文配置（`code` 值必须与 zh.ts 一致）
5. 在 Service 中使用错误码

> ⚠️ 三个文件中同一错误码的 `code` 数值必须完全一致，`message` 为对应语言的翻译

## 四、使用方式

### 4.1 两种抛出方式的区别

| 方式 | 方法 | 使用场景 | 说明 |
|------|------|---------|------|
| `ctx.dsl.error.throw(code)` | 直接抛出 | **try 块中**的业务错误 | 直接中断流程，不返回值 |
| `throw ctx.dsl.error.create(code)` | 创建并抛出 | **catch 块中**包装系统错误 | 创建错误对象后抛出 |

### 4.2 使用示例

```typescript
async getUserById(userId: string) {
    const { ctx } = this;
    try {
        const user = await ctx.msq.collection('users').findOne({ _id: userId, del_flag: 0 });

        // ✅ try 块中：用 ctx.dsl.error.throw（业务逻辑判断）
        if (!user) {
            ctx.dsl.error.throw('USER_NOT_FOUND');
        }
        if (user.status === 0) {
            ctx.dsl.error.throw('USER_DISABLED');
        }

        return user;

    } catch (error: any) {
        ctx.logger.error('[UserService.getUserById] 查询失败', {
            userId, error: error.message,
        });

        // ✅ catch 块中：有 code 属性说明是业务错误，直接重新抛出
        if (error.code) throw error;

        // ✅ catch 块中：无 code 属性说明是系统错误，用 ctx.dsl.error.create 包装
        throw ctx.dsl.error.create('QUERY_FAILED');
    }
}
```

### 4.3 参数校验错误（自动处理）

`ctx.validateDsl()` 校验失败时，会自动使用 `PARAM_INVALID` 错误码响应，**无需开发者手动处理**：

```typescript
// 定义规则
const schema = ctx.dsl({ email: 'email!', password: 'string:8-32!' });
// 校验失败时自动返回错误响应，代码不会继续往下执行
await ctx.validateDsl(schema, 'body');
```

## 五、完整使用示例

以下展示从参数校验到业务错误处理的完整链路：

```typescript
// ===== Step 1: 定义错误码（config/dsl/zh.ts）=====
export default {
    'EMAIL_ALREADY_EXISTS': { code: 40002, message: '邮箱已被占用' },
    'USER_CREATE_FAILED':   { code: 50002, message: '用户创建失败' },
};

// ===== Step 2: Controller 层（参数校验）=====
// app/controller/open/UserController.ts
export default class UserController extends Controller {
    /**
     * 用户注册
     * POST /open/users
     */
    async create() {
        const { ctx } = this;
        const schema = ctx.dsl({
            email:    'email!',
            password: 'string:8-32!',
        });
        await ctx.validateDsl(schema, 'body');  // 格式错误 → 自动返回 PARAM_INVALID
        const { email, password } = ctx.request.body;
        const user = await ctx.service.userService.createUser({ email, password });
        ctx.success(user);
        // Controller 无需 try-catch，exceptions 中间件统一处理
    }
}

// ===== Step 3: Service 层（业务逻辑 + 错误处理）=====
// app/service/UserService.ts
export default class UserService extends Service {
    async createUser(data: { email: string; password: string }) {
        const { ctx } = this;
        try {
            const existing = await ctx.msq
                .collection('users')
                .findOne({ email: data.email, del_flag: 0 });

            if (existing) {
                ctx.dsl.error.throw('EMAIL_ALREADY_EXISTS');  // 业务错误 → 40002
            }

            const user = await ctx.model.User.create({
                account:  data.email,
                email:    data.email,
                del_flag: 0,
                status:   1,
            });

            const result = user.toObject();
            delete result.password;
            return result;

        } catch (error: any) {
            ctx.logger.error('[UserService.createUser] 创建失败', {
                email: data.email,
                error: error.message,
            });
            if (error.code) throw error;
            throw ctx.dsl.error.create('USER_CREATE_FAILED');  // 系统错误 → 50002
        }
    }
}

// ===== Step 4: exceptions 中间件自动格式化错误响应 =====
// 响应示例（邮箱已存在时）：
// HTTP 200
// { "code": 40002, "message": "邮箱已被占用", "data": null }
```

## 六、禁止事项与常见错误

### 🔴 禁止事项

| 禁止行为 | 正确做法 |
|---------|---------|
| `throw new Error('User not found')` | `ctx.dsl.error.throw('USER_NOT_FOUND')` |
| `ctx.body = { code: -1, message: '错误' }` | 在 Service 层抛出，由 exceptions 处理 |
| 在 `err_codes.ts` 新增错误码 | 在 `config/dsl/zh.ts` 等三个文件中新增 |
| 只配置一种语言 | zh / en / hk 三个文件必须同步配置 |
| 字符串格式 `'USER_NOT_FOUND': '用户不存在'` | 对象格式 `{ code: 40001, message: '用户不存在' }` |
| 在 Controller 层 catch 后手动返回错误 | 不加 try-catch，由 exceptions 统一处理 |

### ⚠️ 常见错误

**错误1：在 catch 块忘记判断 error.code**
```typescript
// ❌ 错误：把业务错误当系统错误包装，丢失原始错误码
} catch (error: any) {
    throw ctx.dsl.error.create('QUERY_FAILED');  // 业务错误也被覆盖！
}

// ✅ 正确：先判断是否为业务错误
} catch (error: any) {
    if (error.code) throw error;  // 业务错误直接抛出
    throw ctx.dsl.error.create('QUERY_FAILED');  // 只包装系统错误
}
```

**错误2：三语言 code 值不一致**
```typescript
// ❌ 错误
// zh.ts: { code: 40001, message: '用户不存在' }
// en.ts: { code: 40002, message: 'User not found' }  ← code 不一致！

// ✅ 正确：三语言 code 值必须完全相同，只有 message 不同
```

**错误3：参数校验后仍手动判断**
```typescript
// ❌ 错误：多此一举
await ctx.validateDsl(schema, 'body');
if (!ctx.request.body.email) {  // validateDsl 已保证 email 存在
    ctx.dsl.error.throw('PARAM_INVALID');
}

// ✅ 正确：validateDsl 校验后直接使用参数
await ctx.validateDsl(schema, 'body');
const { email } = ctx.request.body;
```
