## 一、中间件总览

### 1.1 中间件盘点

| # | 中间件名 | 文件 | 注入内容 | 用途 |
|---|---------|------|---------|------|
| 1 | responseHelper | `middleware/responseHelper.ts` | `ctx.success()` | 统一成功响应格式 |
| 2 | schemaDslValidator | `middleware/schemaDslValidator.ts` | `ctx.dsl()` / `ctx.validateDsl()` / `ctx.dsl.error` | 参数验证与错误处理 |
| 3 | httpHelper | `middleware/httpHelper.ts` | `ctx.http` | HTTP 外部请求工具 |
| 4 | userAuth | `middleware/userAuth.ts` | `ctx.state.user` | JWT + 单点登录鉴权 |
| 5 | exceptions | `middleware/exceptions.ts` | 无（捕获器） | 统一异常捕获与格式化 |
| 6 | flexRateLimit | `middleware/flexRateLimit.ts` | 无（拦截器） | 全局/路由级别限流 |

> **已废弃**（不再使用）：`language.ts`、`crudHelper.ts`、`rate_limit.ts`

### 1.2 执行顺序

```
请求到达
  ↓
1. responseHelper      （注入 ctx.success）
  ↓
2. schemaDslValidator  （注入 ctx.dsl / ctx.dsl.error）
  ↓
3. httpHelper          （注入 ctx.http）
  ↓
4. flexRateLimit       （全局限流检查，可选）
  ↓
5. userAuth            （路由级，按需配置）
  ↓
6. flexRateLimit       （路由级限流，可选）
  ↓
7. Controller 业务逻辑
  ↓（如果抛出错误）
8. exceptions          （必须放最后，捕获所有错误）
  ↓
响应返回
```

> ⚠️ `exceptions` 必须是最后一个中间件，才能捕获所有其他中间件和 Controller/Service 的错误。  
> 🗑️ 已废弃中间件（不得再使用）：`language.ts`、`crudHelper.ts`、`rate_limit.ts`

### 1.3 config.default.ts 中间件配置

```typescript
// config/config.default.ts
config.middleware = [
    'responseHelper',       // 必须第一
    'schemaDslValidator',
    'httpHelper',
    'flexRateLimit',        // 全局限流（可选）
    'exceptions',           // 必须最后
];
```

> ❌ 禁止在 `config.middleware` 中添加 `language`、`crudHelper`、`rate_limit`（已废弃）

## 二、responseHelper — 统一响应格式

### 2.1 用途

注入 `ctx.success()` 方法，统一所有成功响应的 JSON 格式。

### 2.2 使用方式

```typescript
// Controller 层
export default class UserController extends Controller {
    async show() {
        const { ctx } = this;
        const user = await ctx.service.userService.getUserById(ctx.params.id);
        ctx.success(user);         // 有数据时传入 data
    }

    async destroy() {
        const { ctx } = this;
        await ctx.service.userService.deleteUser(ctx.params.id);
        ctx.success(null);         // 无需返回数据时传 null
        // 或
        ctx.success({ deleted: true });
    }
}
```

### 2.3 响应格式

```json
{
    "code": 0,
    "message": "操作成功",
    "data": { "_id": "abc123", "nick_name": "张三" },
    "timestamp": "2026-03-02T10:30:00.000Z"
}
```

### 2.4 注意事项

- ❌ 禁止在 Controller 中使用 `ctx.body = { ... }` 直接赋值（绕过 responseHelper）
- ❌ 禁止在 Controller 中使用 `ctx.status = 200` 手动设置状态码
- ✅ 所有成功响应统一使用 `ctx.success(data)`

## 三、schemaDslValidator — 参数验证与错误处理

### 3.1 用途

注入以下三个核心方法：
- `ctx.dsl(rules)` — 定义验证规则
- `ctx.validateDsl(schema, source)` — 执行验证
- `ctx.dsl.error.throw(code)` / `ctx.dsl.error.create(code)` — 业务错误处理

### 3.2 DSL 语法速查

| 规则示例 | 含义 |
|---------|------|
| `'string!'` | 必填字符串 |
| `'string:2-50'` | 可选字符串，长度 2-50 |
| `'string:2-50!'` | 必填字符串，长度 2-50 |
| `'int:1-100!'` | 必填整数，范围 1-100 |
| `'int:1-'` | 可选整数，最小值 1 |
| `'email!'` | 必填邮箱格式 |
| `'url'` | 可选 URL 格式 |
| `'objectId!'` | 必填 MongoDB ObjectId |
| `'boolean!'` | 必填布尔值 |
| `'array'` | 可选数组 |

### 3.3 三种参数来源

```typescript
await ctx.validateDsl(schema, 'body');    // POST/PUT body 参数
await ctx.validateDsl(schema, 'query');   // GET query 参数
await ctx.validateDsl(schema, 'params');  // Path 参数（:id 等）
```

### 3.4 配置示例（完整流程）

> ⚠️ **注意**：`ctx.validateDsl()` **不返回验证后的参数**，只做校验（失败时自动抛错）。  
> 参数校验通过后，仍需从原始来源取值：body → `ctx.request.body`，query → `ctx.request.query`，path → `ctx.params`。  
> **query 参数全为字符串类型**，数字类型字段需手动用 `Number()` 或 `parseInt()` 转换。

```typescript
// Controller 层 —— body 参数（POST/PUT，无需类型转换）
async create() {
    const { ctx } = this;
    const schema = ctx.dsl({
        email:     'email!',
        password:  'string:8-32!',
        nick_name: 'string:2-50',
        age:       'int:1-150',
    });
    await ctx.validateDsl(schema, 'body');
    // ✅ body 参数直接解构使用（schema-dsl 已做类型校验）
    const { email, password, nick_name, age } = ctx.request.body;
    const result = await ctx.service.userService.createUser({ email, password, nick_name, age });
    ctx.success(result);
}

// Controller 层 —— query 参数（GET，字符串需手动转换数字）
async list() {
    const { ctx } = this;
    const schema = ctx.dsl({
        page:     'int:1-',
        pageSize: 'int:1-100',
        keyword:  'string:0-50',
    });
    await ctx.validateDsl(schema, 'query');
    // ✅ query 参数均为字符串，数字字段需用 parseInt 转换
    const { page = 1, pageSize = 20, keyword } = ctx.request.query;
    const result = await ctx.service.userService.getUserList({
        page:     parseInt(page as string),
        pageSize: parseInt(pageSize as string),
        keyword:  keyword as string | undefined,
    });
    ctx.success(result);
}
```

### 3.5 错误处理方法对比

| 方法 | 使用位置 | 说明 |
|------|---------|------|
| `ctx.dsl.error.throw('CODE')` | Service try 块中 | 直接抛出业务错误，中断流程 |
| `throw ctx.dsl.error.create('CODE')` | Service catch 块中 | 创建错误对象后抛出，包装系统错误 |

### 3.6 常见问题

**Q: `validateDsl` 验证失败后，代码还会继续执行吗？**  
A: 不会。验证失败会自动抛出错误，由 `exceptions` 中间件捕获并返回错误响应，后续代码不会执行。

**Q: 错误码必须配置在哪里？**  
A: 必须配置在 `config/dsl/zh.ts`、`config/dsl/en.ts`、`config/dsl/hk.ts` 三个文件中，缺一不可。

## 四、httpHelper — HTTP 请求工具

### 4.1 用途

注入 `ctx.http` 对象，封装外部 HTTP 请求，提供自动重试、超时控制、链路追踪、日志脱敏等能力。

### 4.2 核心方法

| 方法 | 签名 | 说明 |
|------|------|------|
| `postJSON` | `(tag, url, body, init?)` | POST JSON 数据（自动 JSON.stringify） |
| `getJSON` | `(tag, url, init?)` | GET 并自动解析 JSON |
| `fetchJSON` | `(tag, url, init?)` | 通用请求（推荐，可指定 method） |
| `postForm` | `(tag, url, formData, init?)` | POST 表单数据 |
| `getText` | `(tag, url, init?)` | 获取文本响应 |
| `postText` | `(tag, url, body, init?)` | POST 文本数据 |
| `getStream` | `(tag, url, init?)` | 获取流式响应 |
| `download` | `(tag, url, init?)` | 下载文件（流式） |

> 🔴 第一个参数 `tag` 为日志标识，格式为 `ClassName.methodName`，**必填，不可省略**。

### 4.3 配置选项

| 选项 | 类型 | 默认值 | 说明 |
|------|------|-------|------|
| `timeoutMs` | `number` | `15000` | 超时时间（毫秒） |
| `retries` | `number` | `1` | 重试次数 |
| `backoff.baseMs` | `number` | `300` | 重试基础延迟（毫秒） |
| `backoff.factor` | `number` | `2` | 重试延迟倍数 |
| `backoff.jitter` | `boolean` | `true` | 是否添加随机抖动 |
| `parseJson` | `boolean \| 'auto'` | `'auto'` | 是否自动解析 JSON 响应（`'auto'` 时根据 Content-Type 判断） |
| `expectedStatuses` | `number[]` | — | 额外允许的成功状态码（如 `[201, 202]`） |
| `map4xxToFail` | `boolean` | `true` | 4xx 响应是否映射为业务错误（映射后不会触发重试） |
| `idempotencyKey` | `string` | — | 幂等键，添加到请求头防止重复操作 |
| `redactHeaders` | `string[]` | `['authorization','cookie']` | 日志中脱敏的请求头（统一用小写，HTTP header 不区分大小写） |
| `maxResponseBytes` | `number` | `0`（不限） | 响应最大字节数 |
| `headers` | `object` | — | 自定义请求头 |

### 4.4 使用示例

```typescript
export default class NotificationService extends Service {
    /**
     * 发送短信验证码
     */
    async sendSmsCode(phone: string, code: string) {
        const { ctx } = this;
        try {
            const result = await ctx.http.postJSON(
                'NotificationService.sendSmsCode',    // tag（必填）
                'https://api.sms.com/v1/send',
                { phone, template: 'verify_code', params: { code } },
                {
                    headers: { 'X-API-Key': ctx.config.smsApiKey },
                    timeoutMs: 5000,
                    retries: 1,
                    redactHeaders: ['x-api-key'],     // 脱敏 API 密钥
                }
            );
            ctx.logger.info('[NotificationService.sendSmsCode] 短信发送成功', {
                phone: phone.replace(/(\d{3})\d{4}(\d{4})/, '$1****$2'),
            });
            return result;
        } catch (error: any) {
            ctx.logger.error('[NotificationService.sendSmsCode] 短信发送失败', {
                error: error.message,
            });
            if (error.code) throw error;  // 业务错误直接抛出，不覆盖原始错误码
            throw ctx.dsl.error.create('SMS_SEND_FAILED');
        }
    }

    /**
     * 调用支付 API（幂等操作）
     */
    async createPaymentOrder(orderId: string, amount: number) {
        const { ctx } = this;
        try {
            return await ctx.http.postJSON(
                'NotificationService.createPaymentOrder',
                ctx.config.paymentApiUrl,
                { orderId, amount },
                {
                    timeoutMs: 15000,
                    retries: 2,
                    idempotencyKey: `order-${orderId}`,  // 防止重复下单
                    redactHeaders: ['authorization'],     // 统一用小写
                }
            );
        } catch (error: any) {
            ctx.logger.error('[NotificationService.createPaymentOrder] 支付订单创建失败', {
                orderId, amount, error: error.message,
            });
            if (error.code) throw error;  // 业务错误直接抛出，不覆盖原始错误码
            throw ctx.dsl.error.create('PAYMENT_ORDER_CREATE_FAILED');
        }
    }
}
```

### 4.5 注意事项

- ✅ `tag` 参数必须填写，格式为 `ClassName.methodName`
- ✅ 根据接口特性设置合适的 `timeoutMs`（支付类通常需要 10-15s）
- ✅ 关键操作（支付、下单）使用 `idempotencyKey` 防止重复
- ✅ 敏感请求头（Authorization、X-API-Key）必须配置 `redactHeaders`
- ❌ 禁止省略 tag 参数
- ❌ 禁止直接使用 `fetch` 或 `axios` 发起外部请求

## 五、userAuth — JWT 鉴权

### 5.1 用途

验证请求携带的 JWT Token，解析用户信息并注入到 `ctx.state.user`。

### 5.2 注入内容

鉴权成功后，`ctx.state.user` 包含解析自 JWT Token 的用户信息（具体字段取决于签发 Token 时写入的 payload，常见字段如下）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `_id` | `string` | 用户 ID |
| `email` | `string` | 用户邮箱 |
| `status` | `number` | 用户状态 |

> ⚠️ 实际字段以各项目签发 Token 时的 payload 为准，使用前通过 MCP 确认。

### 5.3 路由配置示例

```typescript
// app/routes/home/RESTful.ts
import { Application } from 'egg';

export default (app: Application) => {
    const { router, controller, middleware } = app;
    const userAuth = middleware.userAuth();  // 实例化中间件

    // 需要登录的接口：加上 userAuth
    router.get('/home/users/:id', userAuth, controller.home.user.show);
    router.put('/home/users/:id', userAuth, controller.home.user.update);

    // 不需要登录的接口：不加 userAuth
    router.post('/open/users', controller.open.user.create);
    router.post('/open/sessions', controller.open.session.create);
};
```

### 5.4 Controller 中使用

```typescript
export default class UserController extends Controller {
    /**
     * 获取当前登录用户信息
     * GET /home/user/profile
     * 🔐 需要用户登录
     */
    async getProfile() {
        const { ctx } = this;
        // userAuth 注入的用户信息
        const userId = ctx.state.user._id;
        const user = await ctx.service.userService.getUserById(userId);
        ctx.success(user);
    }
}
```

### 5.5 注意事项

- ✅ 所有 `/home/*` 路由通常需要 `userAuth`；`/admin/*` 路由使用 `adminAuth`；`/internal/*` 使用 `serviceAuth` 或 IP 白名单
- ✅ `/open/*` 路由（登录、注册等）不需要任何鉴权中间件
- ✅ 使用 `UserId` 类型限流时，必须先执行 `userAuth`（顺序：`userAuth` → `flexRateLimit`）
- ❌ 禁止在 `/internal/*` 路由使用 `userAuth`（内部服务用 IP 白名单）

## 六、exceptions — 统一异常处理

### 6.1 用途

捕获所有未处理的错误，统一格式化为 JSON 错误响应。**必须作为最后一个中间件**。

### 6.2 工作机制

```
任意层（Controller / Service / 其他中间件）抛出错误
  ↓
exceptions 捕获
  ↓
判断 error.code 是否存在
  ├── 有 code → 业务错误 → 使用 error.code 和 error.message 响应
  └── 无 code → 系统错误 → HTTP 500，返回通用错误消息
  ↓
记录错误日志
  ↓
返回统一格式的 JSON 响应
```

### 6.3 错误响应格式

**业务错误（有 code）**：
```json
{
    "code": 40001,
    "message": "用户不存在",
    "data": null,
    "timestamp": "2026-03-02T10:30:00.000Z"
}
```

**系统错误（无 code）**：
```json
{
    "code": 50000,
    "message": "服务器内部错误，请稍后重试",
    "data": null,
    "timestamp": "2026-03-02T10:30:00.000Z"
}
```

### 6.4 注意事项

- ✅ `exceptions` 在 `config.middleware` 数组中**必须排最后**
- ✅ Controller 层不需要也不应该有 try-catch（交给 exceptions 处理）
- ✅ exceptions 会自动记录错误日志，Service 层只需记录带业务上下文的额外信息

## 七、flexRateLimit — 限流

### 7.1 用途

防止恶意请求和接口滥用，支持全局配置和路由级别配置。

### 7.2 限流维度

| 维度 | 枚举值 | 说明 | 适用场景 |
|------|-------|------|---------|
| IP | `LimitOptionsType.Ip` | 基于客户端 IP | 登录、注册、公开 API |
| 用户 | `LimitOptionsType.UserId` | 基于用户 ID（需先鉴权） | 已登录接口 |
| 邮箱 | `LimitOptionsType.Email` | 基于 body.email 字段 | 注册、找回密码 |
| 手机号 | `LimitOptionsType.PhoneNumber` | 基于 body.phone_number 字段 | 发送短信验证码 |
| 自定义 | `LimitOptionsType.Custom` | 自定义 key 限流，**必须额外配置 `key` 字段** | 特殊业务场景 |

> ⚠️ `LimitOptionsType.Custom` 与其他维度不同，必须传 `key` 字段，否则限流不生效：
> ```typescript
> // ✅ 正确：Custom 必须指定 key
> { type: LimitOptionsType.Custom, key: 'payment-callback', expire: 60, count: 10 }
> 
> // ❌ 错误：缺少 key 字段
> { type: LimitOptionsType.Custom, expire: 60, count: 10 }
> ```

### 7.3 限流算法

| 算法 | 配置值 | 特点 | 推荐场景 |
|------|--------|------|---------|
| 滑动窗口 | `sliding-window` | 精确，无边界问题（**推荐**） | 大多数场景 |
| 固定窗口 | `fixed-window` | 简单高效，有边界问题 | 低频接口 |
| 令牌桶 | `token-bucket` | 支持突发流量 | 高频 API |
| 漏桶 | `leaky-bucket` | 流量平滑 | 消息队列 |

### 7.4 全局配置示例

```typescript
// config/config.default.ts
config.flexRateLimit = {
    defaultAlgorithm: 'sliding-window',
    errorCode: 'RATE_LIMIT_EXCEEDED',
    enableHeaders: true,

    // 全局限流规则（应用到 match 的所有路由）
    rules: [
        { type: 'ip', expire: 60, count: 100 },     // 1分钟 100 次
        { type: 'ip', expire: 3600, count: 1000 },  // 1小时 1000 次
    ],

    // 只对这些路由生效（/internal/ 通过 ipWhitelist 控制，不放这里）
    match: ['/api/', '/home/', '/admin/'],

    // 跳过限流的路由（优先级最高，完全跳过所有检查包括白名单）
    ignore: ['/health', '/'],

    // 内部路由通过 IP 白名单控制，外网 IP 直接返回 403
    ipWhitelist: {
        routes: [{
            routes: '/internal/',
            ip: ['127.0.0.1', '::1', '192.168.*', '172.16.*', '10.*'],
        }],
    },
};
```

> ⚠️ **`ignore` 与 `ipWhitelist` 的区别**：
> - `ignore`：**完全跳过**所有限流检查，包括 IP 白名单验证，外网 IP 也能访问
> - `ipWhitelist`：经过白名单验证，外网 IP 返回 403，内网 IP 放行
> - 内部路由（`/internal/`）**必须用 `ipWhitelist`**，不能用 `ignore`，否则外网可直接访问

### 7.5 路由级别配置示例

```typescript
// app/routes/open/RESTful.ts
import { Application } from 'egg';
import { LimitOptionsType } from 'enum';

export default (app: Application) => {
    const { router, controller, middleware } = app;

    // 登录：IP 限流（防暴力破解）
    const loginLimit = middleware.flexRateLimit([
        { type: LimitOptionsType.Ip, expire: 60, count: 5 },
        { type: LimitOptionsType.Ip, expire: 3600, count: 20 },
    ]);
    router.post('/open/sessions', loginLimit, controller.open.session.create);

    // 注册：邮箱 + IP 双重限流
    const registerLimit = middleware.flexRateLimit([
        { type: LimitOptionsType.Email, expire: 3600, count: 3 },
        { type: LimitOptionsType.Ip, expire: 3600, count: 10 },
    ]);
    router.post('/open/users', registerLimit, controller.open.user.create);

    // 发送验证码：用户 + 手机号 + IP 三重限流（需先鉴权）
    const userAuth = middleware.userAuth();
    const sendCodeLimit = middleware.flexRateLimit([
        { type: LimitOptionsType.UserId, expire: 60, count: 1 },
        { type: LimitOptionsType.PhoneNumber, expire: 60, count: 1 },
        { type: LimitOptionsType.Ip, expire: 3600, count: 20 },
    ]);
    router.post('/home/verification-codes', userAuth, sendCodeLimit, controller.home.verificationCode.create);
};
```

### 7.6 响应头

触发限流时返回 HTTP 429，并附带响应头：

```http
HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1740902400000
Retry-After: 60

{ "code": 82001, "message": "请求过于频繁，请稍后再试", "data": null }
```

### 7.7 重要注意事项

**① 路由级别配置会完全覆盖全局配置（不叠加）**

```typescript
// 全局：1分钟 100 次 + 1小时 1000 次
// 路由级别：1分钟 5 次
// 结果：该路由只执行 1分钟 5 次，1小时 1000 次的规则被覆盖

// 如需保留长期限流，路由级别需手动添加：
const loginLimit = middleware.flexRateLimit([
    { type: LimitOptionsType.Ip, expire: 60, count: 5 },
    { type: LimitOptionsType.Ip, expire: 3600, count: 20 },  // 手动补充
]);
```

**② UserId 限流必须在 userAuth 之后**

```typescript
// ✅ 正确顺序
router.post('/home/data', userAuth, rateLimitByUser, controller.home.data.create);

// ❌ 错误顺序（UserId 限流会被跳过）
router.post('/home/data', rateLimitByUser, userAuth, controller.home.data.create);
```

**③ Email / PhoneNumber 限流依赖请求参数**

```
Email 限流    → body 中必须有 email 字段
PhoneNumber 限流 → body 中必须有 phone_number 字段
```

**④ IP 白名单与限流独立配置**

```typescript
config.flexRateLimit = {
    match: ['/api/', '/home/'],         // 公开路由：只限流
    ipWhitelist: {
        routes: [{
            routes: '/internal/',        // 内部路由：只做 IP 白名单
            ip: ['127.0.0.1', '192.168.*', '172.16.*', '10.*'],
        }],
    },
};
```

## 八、最佳实践与常见问题

### 8.1 最佳实践

| 场景 | 推荐方案 |
|------|---------|
| 登录/注册等公开敏感接口 | 路由级别 IP 限流（`expire: 60, count: 5`） |
| 发送验证码 | 用户 + 手机号 + IP 三重限流 |
| 普通业务 API | 全局限流（`expire: 60, count: 100`）覆盖即可 |
| 内部服务路由 | IP 白名单（不限流） |
| 调用第三方 API | httpHelper + `idempotencyKey`（关键操作） |
| 错误码新增 | zh / en / hk 三文件同步，code 全局唯一 |

### 8.2 常见问题

**Q1: 为什么 Controller 层不需要 try-catch？**  
A: `exceptions` 中间件会自动捕获所有未处理的错误并格式化响应。Controller 加 try-catch 反而可能导致错误被错误地处理或丢失。

**Q2: `ctx.success()` 和 `ctx.body =` 有什么区别？**  
A: `ctx.success()` 由 `responseHelper` 注入，自动包装为统一的 `{ code: 0, message, data, timestamp }` 格式。`ctx.body =` 直接赋值会绕过统一格式化，**禁止使用**。

**Q3: flexRateLimit 全局配置和路由级别配置可以叠加吗？**  
A: 不能叠加。路由级别配置会**完全替换**全局配置。如需保留长期限流规则，需在路由级别手动重复配置。

**Q4: 如何确认限流是否生效？**  
A: 正常请求的响应头中会有 `X-RateLimit-Limit` 和 `X-RateLimit-Remaining` 字段（需开启 `enableHeaders: true`）。

**Q5: 内部服务路由需要做限流吗？**  
A: 不建议对内部路由做限流，推荐使用 **IP 白名单**（`ipWhitelist`）保护。  
⚠️ 不要将 `/internal/` 加入 `ignore`（会让外网也能访问），应配置 `ipWhitelist.routes`，外网 IP 访问返回 403，内网 IP 正常放行。
