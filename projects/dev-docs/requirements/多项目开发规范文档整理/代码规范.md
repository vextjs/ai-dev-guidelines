## 二、注释规范

### 2.1 Controller 方法注释

**注释格式说明**：

Controller 方法本身**没有函数参数**（请求参数通过 `ctx` 取得），因此：

| 标签 | 是否标准 JSDoc | 正确用法 |
|------|-------------|------|
| `@returns` | ✅ 标准 | `@returns {Promise<IUser>} 说明` |
| `@throws` | ✅ 标准，附加约定 | `@throws {Error} code:40001 USER_NOT_FOUND - 用户不存在`，`{}` 内写类型 `Error`，code 数值写在描述中 |
| `@param` | ✅ 标准，**不适用 Controller** | Controller 方法签名为 `async show() {}`，无函数入参，强行用 `@param` 描述 `ctx.params.id` 是对规范的滥用，JSDoc 工具会报 warning |
| `@body` / `@query` / `@params` | ✅ HTTP 接口约定标签 | Controller 层**没有函数入参**，HTTP 框架（NestJS / Midway / Egg.js）社区的惯用做法：用语义化自定义标签描述请求参数来源，结构清晰，可配合文档工具映射 |

> **为什么 Controller 不用标准 `@param`？**  
> 标准 JSDoc `@param` 描述的是**函数签名里的入参**。Controller 方法签名是 `async show() {}`（零个参数），请求参数通过 `ctx.request.body/query` 在函数体内取得，不是函数入参。  
> 用 `@param {string} ctx.params.id` 会造成 JSDoc 工具误解，且表达不了参数来源（body / query / path 的区别）。  
> `@body` / `@query` / `@params` 是 HTTP 框架文档的**业界惯用约定**，语义更精确，后续引入文档工具时直接配置标签映射即可。

```typescript
export default class UserController extends Controller {
    /**
     * 获取用户详情
     *
     * GET /home/users/:id
     * 🔐 userAuth（普通用户鉴权）
     *
     * @params id {string} 用户 ID（ObjectId，必填）
     *
     * @returns {Promise<IUser>} 用户信息对象
     * @throws {Error} code:40001 USER_NOT_FOUND - 用户不存在
     */
    async show() { /* ... */ }

    /**
     * 用户注册
     *
     * POST /open/users
     *
     * @body email {string} 邮箱（必填）
     * @body password {string} 密码，8-32 位（必填）
     * @body nick_name {string?} 昵称，2-50 位（可选）
     *
     * @returns {Promise<{ user: IUser, token: string }>} 用户信息与 Token
     * @throws {Error} code:40002 EMAIL_ALREADY_EXISTS - 邮箱已被占用
     * @throws {Error} code:50002 USER_CREATE_FAILED - 用户创建失败
     */
    async create() { /* ... */ }

    /**
     * 获取用户列表
     *
     * GET /home/users
     * 🔐 userAuth（普通用户鉴权）
     *
     * @query page {number?} 页码，默认 1，最小值 1
     * @query pageSize {number?} 每页数量，默认 20，范围 1-100
     * @query keyword {string?} 搜索关键词（可选）
     *
     * @returns {Promise<{ list: IUser[], pagination: IPagination }>} 用户列表与分页信息
     * @throws {Error} code:50001 QUERY_FAILED - 查询失败，请稍后重试
     */
    async list() { /* ... */ }

    /**
     * 禁用用户（管理员操作）
     *
     * POST /admin/users/:id/disable
     * 🔐 adminAuth（管理员鉴权）
     *
     * @params id {string} 用户 ID（ObjectId，必填）
     *
     * @returns {Promise<{ disabled: true }>}
     * @throws {Error} code:40001 USER_NOT_FOUND - 用户不存在
     */
    async disable() { /* ... */ }
}
```

**注释规范要点**：

| 要素 | 规则 |
|------|------|
| 接口路径 | 写在描述段落下方，格式：`HTTP方法 /路径` |
| 鉴权标识 | 接口路径下一行，格式：`🔐 中间件名（说明）` |
| 请求参数 | `@body` / `@query` / `@params` + `字段名 {类型} 说明`，可选字段类型加 `?`：`{string?}` |
| 成功响应 | `@returns {Promise<Type>} 说明` |
| **失败响应** | `@throws {Error} code:数值 ERROR_KEY - 说明`，**code 数值必填写在描述中** |

### 2.1.1 鉴权模式标注规范

项目有三种鉴权中间件，注释中**必须明确标注使用哪一种**：

| 鉴权中间件 | 标注写法 | 适用模块 | 说明 |
|-----------|---------|---------|------|
| `userAuth` | `🔐 userAuth（普通用户鉴权）` | `/home/*` | 验证用户 JWT Token，注入 `ctx.state.user` |
| `adminAuth` | `🔐 adminAuth（管理员鉴权）` | `/admin/*` | 验证管理员权限 |
| `serviceAuth` | `🔐 serviceAuth（服务间鉴权）` | `/internal/*` | 验证服务间调用权限（或使用 IP 白名单） |
| 无鉴权 | 不写 🔐 行 | `/open/*` | 公开接口，无需鉴权 |

```typescript

import { Service } from "egg"  // 双引号、缺分号

export default class UserService extends Service {
```

```typescript
// ✅ 正确：async/await

    const existing = await ctx.msq.collection('users').findOne({ email: data.email });
    if (existing) {
        ctx.dsl.error.throw('EMAIL_ALREADY_EXISTS');

        });
}
```

## 五、最佳实践

### 5.1 分层职责（🔴 强制，8 条禁止项）

```typescript
// ✅ 正确：Controller 直接用 ctx.dsl() 定义校验规则，不单独创建 Validator 文件
export default class UserController extends Controller {
    async show() {
        const { ctx } = this;
        // 参数校验直接写在方法内
        const schema = ctx.dsl({ id: 'objectId!' });
        await ctx.validateDsl(schema, 'params');
        const user = await ctx.service.userService.getUserById(ctx.params.id);
        ctx.success(user);  // 统一使用 ctx.success
    }

    async create() {
        const { ctx } = this;
        const schema = ctx.dsl({
            email:    'email!',
            password: 'string:8-32!',
            nick_name: 'string:2-50',
        });
        await ctx.validateDsl(schema, 'body');
        const { email, password, nick_name } = ctx.request.body;
        const user = await ctx.service.userService.createUser({ email, password, nick_name });
        ctx.success(user);
    }
}

// ❌ 禁止1：Controller 直接操作数据库
async show() {
    const user = await ctx.model.User.findById(ctx.params.id);  // 禁止！
    ctx.body = user;  // 禁止！绕过 responseHelper
}

// ❌ 禁止2：Service 使用 throw new Error()
async getUser(id: string) {
    throw new Error('User not found');  // 禁止！必须使用 ctx.dsl.error.throw
}

// ❌ 禁止3：Model 层编写业务逻辑
UserSchema.methods.validateAge = function() { /* 禁止！*/ };
```

> **关于 `validator/` 目录**：校验逻辑原则上**直接写在 Controller 方法内**（使用 `ctx.dsl()`）。  
> `validator/` 目录仅用于极少数场景：多个接口共用**完全相同**的复杂校验规则，且抽取后能明显减少重复代码时，才考虑单独提取。日常开发无需创建 Validator 文件。

### 5.2 错误处理规范（🔴 强制）

```typescript
// ✅ 正确的 Service 层错误处理模式
async getUserById(userId: string) {
    const { ctx } = this;
    try {
        const user = await ctx.msq.collection('users').findOne({ _id: userId });
        if (!user) {
            ctx.dsl.error.throw('USER_NOT_FOUND');  // try 块中：业务错误
        }
        return user;
    } catch (error: any) {
        ctx.logger.error('[UserService.getUserById] 查询失败', {
            userId,
            error: error.message,
        });
        if (error.code) throw error;  // 已是业务错误，直接抛出
        throw ctx.dsl.error.create('QUERY_FAILED');  // catch 块中：包装系统错误
    }
}
```

### 5.4 Service 模块拆分规范

单个 Service 文件满足以下任一条件时**必须拆分**：

| 触发条件 | 说明 |
|---------|------|
| 文件超过 **500 行** | 按业务模块拆分 |
| 公共方法超过 **20 个** | 按职责拆分 |
| 包含多个业务领域逻辑 | 按业务域拆分 |

推荐拆分方式：

```
service/
├── UserService.ts          # 用户基础信息（CRUD、资料）
├── UserAuthService.ts      # 用户认证（登录、注册、密码）
├── UserRelationService.ts  # 用户关系（关注、粉丝）
└── PointsService.ts        # 积分系统（增减、兑换）
```

### 5.5 数据库操作规范（🔴 强制）

| 操作类型 | 使用工具 | 原因 |
|---------|---------|------|
| 查询（SELECT） | `ctx.msq`（monSQLize） | 性能优化、链式 API |
| 写入（INSERT / UPDATE / DELETE） | `ctx.model`（mongoose） | 数据安全、事务支持 |

```typescript
// ✅ 正确
const user = await ctx.msq.collection('users').findOne({ _id: userId, del_flag: 0 });
const newUser = await ctx.model.User.create({ email, nick_name, del_flag: 0 });
await ctx.model.User.findByIdAndUpdate(userId, { $set: { nick_name } }, { new: true });

// 🔴 禁止凭记忆猜测字段名，必须通过 MCP 查询实际 Schema
```

### 5.6 TypeScript 类型文件组织

| 类型分类 | 存放位置 | 举例 |
|---------|---------|------|
| 全局工具类型 | `typings/index.d.ts` | `Nullable<T>`, `DeepPartial<T>` |
| Egg.js 框架扩展 | `typings/app/` | `EggAppConfig` 扩展 |
| 中间件类型 | `typings/{name}.d.ts` | `typings/flexRateLimit.d.ts` |
| 数据库实体 | `typings/entity/` | `typings/entity/User.d.ts` |
| 业务接口 | `typings/interface/` | `typings/interface/IUser.ts` |

> ❌ 禁止在业务代码（`app/` 目录）中定义全局类型文件
