## 一、日志级别定义

Egg.js 框架提供四个日志级别，按严重程度从高到低排列：

| 级别 | 方法 | 使用场景 | 示例 |
|------|------|---------|------|
| **ERROR** | `ctx.logger.error()` | 系统异常、数据库故障、第三方 API 调用失败、需要立即告警的问题 | 数据库连接断开、支付 API 超时 |
| **WARN** | `ctx.logger.warn()` | 可能影响功能但不中断服务的问题、即将达到限制、配置缺失 | 限流接近上限、缓存命中率低 |
| **INFO** | `ctx.logger.info()` | 关键业务操作成功完成、服务启动/停止、重要状态变化 | 用户注册成功、订单创建成功 |
| **DEBUG** | `ctx.logger.debug()` | 开发调试信息，仅在开发/测试环境输出，生产环境不记录 | 中间查询结果、临时变量值 |

### 1.1 级别选择判断标准

```
发生了异常/错误？
├── 是，且影响核心功能或需要立即处理 → ERROR
├── 是，但服务仍可继续运行 → WARN
└── 否（正常业务流程）
    ├── 关键业务操作完成 → INFO
    └── 调试信息 → DEBUG（仅开发环境）
```

```typescript
// ERROR：第三方 API 调用失败，影响核心功能
ctx.logger.error('[PaymentService.createOrder] 支付 API 调用失败', {
    orderId,
    error: error.message,
});

// WARN：限流接近阈值，服务仍可运行
ctx.logger.warn('[FlexRateLimit] IP 请求接近限额', {
    ip,
    remaining: 5,
    limit: 100,
});

// INFO：关键业务成功完成
ctx.logger.info('[OrderService.createOrder] 订单创建成功', {
    orderId: order._id,
    userId,
    totalAmount,
});

// DEBUG：开发调试
ctx.logger.debug('[UserService.getUserList] 查询条件', { query, skip, limit });
```

## 二、日志格式规范

### 2.1 标签前缀规范（🔴 强制）

所有日志**必须**以 `[ClassName.methodName]` 格式作为开头标签，便于快速定位日志来源：

```
[ClassName.methodName] 描述信息
```

```typescript
// ✅ 正确：包含类名和方法名
ctx.logger.error('[UserService.getUserById] 查询失败', { userId, error: error.message });
ctx.logger.info('[OrderService.createOrder] 订单创建成功', { orderId });
ctx.logger.warn('[UserAuthService.login] 登录失败次数过多', { email, attempts });

// ❌ 错误：没有标签，无法定位来源
ctx.logger.error('查询失败');
ctx.logger.info('success');
```

### 2.2 结构化日志

日志的第二个参数**必须**是结构化对象，而非拼接字符串，便于日志系统解析和检索：

```typescript
// ✅ 正确：结构化对象
ctx.logger.error('[UserService.getUserById] 查询失败', {
    userId,
    collection: 'users',
    error: error.message,
    stack: error.stack,  // error 级别可附加 stack
});

ctx.logger.info('[OrderService.createOrder] 订单创建成功', {
    orderId: order._id,
    userId,
    productId,
    totalAmount,
    createdAt: new Date().toISOString(),
});

// ❌ 错误：字符串拼接，难以解析
ctx.logger.error(`[UserService] 查询用户 ${userId} 失败：${error.message}`);
```

### 2.3 HTTP 请求日志格式

通过 `httpHelper` 发起外部请求时，日志的 `tag` 参数即为日志标签，格式同上：

```typescript
const result = await ctx.http.postJSON(
    'PaymentService.createOrder',   // tag → 日志标签 [PaymentService.createOrder]
    'https://api.payment.com/orders',
    orderData,
    { timeoutMs: 10000, retries: 2 }
);
// httpHelper 自动记录请求耗时、状态码、错误信息
```

## 三、关键信息记录要求

### 3.1 Service 层必须记录的场景

| 场景 | 级别 | 必须包含的字段 |
|------|------|-------------|
| 外部 API 调用成功 | INFO | tag、关键业务 ID、响应关键字段 |
| 外部 API 调用失败 | ERROR | tag、请求参数（脱敏）、error.message |
| 关键写入操作成功 | INFO | 操作类型、相关 ID（用户ID/订单ID等） |
| 数据库查询/写入失败 | ERROR | tag、查询条件（脱敏）、error.message |
| 业务规则不满足 | 不需要记录 | exceptions 中间件已统一记录 |

### 3.2 错误日志必要字段

ERROR 级别日志**必须**包含以下信息：

```typescript
ctx.logger.error('[ServiceName.methodName] 操作描述', {
    // 1. 关键业务 ID（用于问题追踪）
    userId,          // 或 orderId / chatId / paymentId
    // 2. 触发操作的关键参数（脱敏后）
    email: data.email,
    // 3. 错误信息
    error: error.message,
    // 4. 可选：错误堆栈（严重错误时附加）
    // stack: error.stack,
});
```

### 3.3 INFO 日志精简原则

INFO 日志用于记录关键业务里程碑，**不记录**每一步的中间结果：

```typescript
// ✅ 正确：只记录关键成功节点
ctx.logger.info('[SmsService.sendVerificationCode] 短信发送成功', {
    phone,
    // 不记录 code 本身（敏感信息）
});

// ❌ 错误：记录太多中间步骤，噪音过多
ctx.logger.info('[SmsService] 开始发送短信');
ctx.logger.info('[SmsService] 调用短信 API');
ctx.logger.info('[SmsService] 短信 API 返回');
ctx.logger.info('[SmsService] 发送完成');
```

### 3.4 服务启动/停止日志

在 `app.ts` 中记录服务启动关键步骤：

```typescript
// app.ts
async willReady() {
    try {
        await this.msq.connect();
        this.app.logger.info('[App] MonSQLize 连接池初始化成功');
    } catch (err: any) {
        this.app.logger.error('[App] MonSQLize 连接池初始化失败', { error: err.message });
        throw err;
    }
}

async beforeClose() {
    await this.msq.close();
    this.app.logger.info('[App] MonSQLize 连接池已关闭');
}
```

## 四、敏感信息处理

### 4.1 禁止记录的字段（🔴 强制）

以下字段**绝对禁止**出现在任何级别的日志中：

| 字段类型 | 示例 |
|---------|------|
| 密码 | `password`, `pwd`, `secret` |
| Token | `token`, `access_token`, `refresh_token` |
| 短信/邮箱验证码 | `code`, `verify_code`, `sms_code` |
| 支付敏感信息 | 完整卡号、CVV、银行账号 |
| API 密钥 | `api_key`, `app_secret` |
| 个人隐私 | 完整身份证号、完整手机号（建议脱敏后记录） |

```typescript
// ❌ 错误：记录了敏感信息
ctx.logger.info('[UserAuthService.login] 登录请求', {
    email,
    password,   // 禁止！
    token,      // 禁止！
});

// ✅ 正确：只记录必要的非敏感信息
ctx.logger.info('[UserAuthService.login] 登录成功', {
    userId: user._id,
    email,
    loginAt: new Date().toISOString(),
});
```

### 4.2 手机号脱敏记录

```typescript
// ✅ 正确：手机号脱敏后记录（保留前3后4）
const maskedPhone = phone.replace(/(\d{3})\d{4}(\d{4})/, '$1****$2');
ctx.logger.info('[SmsService.sendCode] 短信发送成功', {
    phone: maskedPhone,  // 138****1234
});
```

### 4.3 httpHelper 请求头脱敏

通过 `httpHelper` 发起外部请求时，使用 `redactHeaders` 配置需要在日志中脱敏的请求头：

```typescript
const result = await ctx.http.postJSON(
    'PaymentService.createOrder',
    'https://api.payment.com/orders',
    orderData,
    {
        headers: {
            'Authorization': `Bearer ${apiKey}`,
            'X-API-Key': secretKey,
        },
        // ✅ 脱敏敏感请求头，日志中不会显示其值
        redactHeaders: ['authorization', 'x-api-key'],
    }
);
```

## 五、最佳实践

### 5.1 禁用 console（🔴 强制）

```typescript
// ❌ 禁止：console 日志不受 Egg.js 日志系统管理
console.log('用户创建成功');
console.error('查询失败:', error);

// ✅ 正确：统一使用 ctx.logger
ctx.logger.info('[UserService.createUser] 用户创建成功', { userId: user._id });
ctx.logger.error('[UserService.getUserById] 查询失败', { userId, error: error.message });
```

### 5.2 错误日志完整性要求

```typescript
// ✅ 完整的错误日志示例
} catch (error: any) {
    ctx.logger.error('[OrderService.createOrder] 创建订单失败', {
        // 关键 ID
        userId: data.userId,
        productId: data.productId,
        // 脱敏后的关键参数
        quantity: data.quantity,
        // 错误信息
        error: error.message,
        // 错误类型（帮助区分数据库错误/业务错误/网络错误）
        errorType: error.constructor.name,
    });
    if (error.code) throw error;
    throw ctx.dsl.error.create('ORDER_CREATE_FAILED');
}
```

### 5.3 避免重复日志

exceptions 中间件已统一记录所有未捕获的错误，Service 层**只需记录**需要补充上下文的错误：

```typescript
// ✅ Service 层日志：补充业务上下文（参数信息）
ctx.logger.error('[UserService.createUser] 创建失败', {
    email: data.email,  // 补充了"哪个邮箱触发的"这一上下文
    error: error.message,
});

// ❌ 不要在 Controller 层再记录一次相同的错误
// Controller 层无需日志，exceptions 会统一处理
```

### 5.4 日志分级示例对照

```typescript
export default class PaymentService extends Service {
    async processPayment(orderId: string, amount: number) {
        const { ctx } = this;

        // INFO：记录关键操作开始（仅重要操作）
        ctx.logger.info('[PaymentService.processPayment] 开始处理支付', {
            orderId,
            amount,
        });

        try {
            const result = await ctx.http.postJSON(
                'PaymentService.processPayment',
                ctx.config.paymentApiUrl,
                { orderId, amount },
                { timeoutMs: 15000, retries: 2 }
            );

            // INFO：关键业务成功
            ctx.logger.info('[PaymentService.processPayment] 支付成功', {
                orderId,
                paymentId: result.paymentId,
                status: result.status,
            });

            return result;

        } catch (error: any) {
            // ERROR：关键业务失败
            ctx.logger.error('[PaymentService.processPayment] 支付处理失败', {
                orderId,
                amount,
                error: error.message,
            });

            if (error.code) throw error;
            throw ctx.dsl.error.create('PAYMENT_PROCESS_FAILED');
        }
    }
}
```
