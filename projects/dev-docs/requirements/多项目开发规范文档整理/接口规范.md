## 一、RESTful 设计原则

### 1.1 路由风格

所有新接口**必须**使用 RESTful 风格，禁止使用旧分组方式（`router.group`）。

```typescript
// ✅ 正确：RESTful 风格（强制使用）
router.get('/home/users', userAuth, controller.home.user.list);
router.get('/home/users/:id', userAuth, controller.home.user.show);
router.post('/home/users', userAuth, controller.home.user.create);
router.put('/home/users/:id', userAuth, controller.home.user.update);
router.delete('/home/users/:id', userAuth, controller.home.user.destroy);

// ❌ 禁止：旧分组方式
router.group('/home', (router) => {
    router.get('/user/list', controller.home.user.list);  // 禁止！
});
```

### 1.2 URL 命名规范

| 规则 | 正确 ✅ | 错误 ❌ |
|------|--------|--------|
| 路径全小写 | `/home/users` | `/Home/Users` |
| 多单词用连字符 | `/user-profile` | `/userProfile` |
| 资源名复数 | `/users` | `/user` |
| 非标准操作用动词后缀 | `/users/:id/disable` | `/disableUser` |
| 不在 URL 中携带版本号 | `/home/users` | `/v1/home/users` |

### 1.3 HTTP 方法语义

| HTTP 方法 | 操作 | Controller 方法 | 示例 |
|-----------|------|----------------|------|
| `GET` | 查询列表 | `list` | `GET /home/users` |
| `GET` | 查询详情 | `show` | `GET /home/users/:id` |
| `POST` | 创建资源 | `create` | `POST /home/users` |
| `PUT` | 全量更新 | `update` | `PUT /home/users/:id` |
| `PATCH` | 部分更新 | `update` | `PATCH /home/users/:id` |
| `DELETE` | 删除资源 | `destroy` | `DELETE /home/users/:id` |

### 1.4 模块划分与路由前缀

| 模块 | 路由前缀 | 鉴权要求 | 说明 |
|------|---------|---------|------|
| 前台 | `/home/*` | 用户 JWT 鉴权 | 普通用户接口 |
| 开放 | `/open/*` | 无需鉴权 | 登录、注册、公开查询 |
| 内部 | `/internal/*` | 服务间鉴权 / IP 白名单 | 服务间调用 |
| 后台 | `/admin/*` | 管理员鉴权 | 管理后台接口 |

### 1.5 路由文件规范（🔴 强制）

新路由**必须**写入对应模块的 `RESTful.ts` 文件：

```
app/routes/
├── home/RESTful.ts       # 前台路由
├── open/RESTful.ts       # 公开路由
├── internal/RESTful.ts   # 内部路由
└── admin/RESTful.ts      # 后台路由
```

```typescript
// app/routes/home/RESTful.ts
import { Application } from 'egg';

export default (app: Application) => {
    const { router, controller, middleware } = app;
    const userAuth = middleware.userAuth();

    /**
     * ========== 用户模块 ==========
     */
    // GET /home/users
    router.get('/home/users', userAuth, controller.home.user.list);
    // GET /home/users/:id
    router.get('/home/users/:id', userAuth, controller.home.user.show);
    // POST /home/users
    router.post('/home/users', userAuth, controller.home.user.create);
    // PUT /home/users/:id
    router.put('/home/users/:id', userAuth, controller.home.user.update);
    // DELETE /home/users/:id
    router.delete('/home/users/:id', userAuth, controller.home.user.destroy);
};
```

**步骤4（🔴 强制）：在 `app/router.ts` 中注册新路由文件**

```typescript
// app/router.ts
import { Application } from 'egg';
import homeRESTful from './routes/home/RESTful';
import openRESTful  from './routes/open/RESTful';

export default (app: Application) => {
    // 注册新规范路由
    homeRESTful(app);
    openRESTful(app);

    // 旧路由文件保持不变，禁止修改
    require('./routes/home')(app);
    require('./routes/open')(app);
};
```

## 二、请求/响应格式

### 2.1 统一成功响应格式

所有成功响应**必须**通过 `ctx.success()` 返回，格式由 `responseHelper` 中间件统一生成：

```typescript
// Controller 层
ctx.success(data);  // 正确
ctx.body = { ... }; // ❌ 禁止！绕过 responseHelper
```

**响应结构**：

```json
{
    "code": 0,
    "message": "操作成功",
    "data": { ... },
    "timestamp": "2026-03-02T10:30:00.000Z"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | `number` | 0 表示成功 |
| `message` | `string` | 操作描述 |
| `data` | `any` | 业务数据，无数据时为 `null` |
| `timestamp` | `string` | ISO 8601 时间戳 |

### 2.2 列表响应格式

```json
{
    "code": 0,
    "message": "操作成功",
    "data": {
        "list": [ { ... }, { ... } ],
        "pagination": {
            "page": 1,
            "pageSize": 20,
            "total": 100,
            "totalPages": 5
        }
    }
}
```

### 2.3 错误响应格式

由 `exceptions` 中间件统一生成，**开发人员无需手动构造**：

```json
{
    "code": 40001,
    "message": "用户不存在",
    "data": null,
    "timestamp": "2026-03-02T10:30:00.000Z"
}
```

> ⚠️ **HTTP 状态码说明**：业务错误（如用户不存在、参数校验失败）统一返回 **HTTP 200**，通过响应体中的 `code` 字段区分成功/失败。非 200 状态码仅用于框架/中间件层面的特殊场景：

| HTTP 状态码 | 场景 |
|-----------|------|
| `200` | 成功，以及所有业务错误（通过 `code` 字段区分） |
| `401` | 未鉴权（userAuth 中间件处理，Token 缺失/过期） |
| `403` | IP 白名单拦截（flexRateLimit ipWhitelist） |
| `429` | 限流触发（flexRateLimit 处理） |
| `500` | 未捕获的系统异常 |

## 三、参数校验

### 3.1 schema-dsl 语法速查

所有请求参数**必须**通过 `schema-dsl` 校验，不得直接使用未校验的参数。校验规则**直接写在 Controller 方法内**，无需单独创建 Validator 文件。

**基本格式**：`type:min-max|default[!]`

**类型（type）**：

| 类型 | 说明 | 示例规则 |
|------|------|---------|
| `string` | 字符串 | `string:2-50` |
| `int` | 整数 | `int:1-100` |
| `float` | 浮点数 | `float:0-` |
| `email` | 邮箱格式 | `email!` |
| `url` | URL 格式 | `url` |
| `objectId` | MongoDB ObjectId | `objectId!` |
| `array` | 数组 | `array` |
| `boolean` | 布尔值 | `boolean` |

**范围（min-max）**：

| 格式 | 含义 | 示例 |
|------|------|------|
| `min-max` | 最小到最大 | `int:1-100`（1 到 100） |
| `min-` | 最小值，无上限 | `int:1-`（≥1） |
| `-max` | 最大值，无下限 | `int:-100`（≤100） |
| 省略 | 不限 | `string` |

**默认值（\|default）**：

| 格式 | 含义 | 示例 |
|------|------|------|
| `\|值` | 字段缺失时使用默认值 | `int:1-\|1`（默认值 1） |
| 省略 | 无默认值 | `int:1-` |

**必填标记（!）**：字段名后加 `!` 表示必填，省略则为可选。

### 3.2 Controller 层校验示例

参数校验**直接写在 Controller 方法内**，使用 `ctx.dsl()` 定义规则，`ctx.validateDsl()` 执行校验。校验失败时自动响应错误，后续代码不会执行。

```typescript
export default class UserController extends Controller {

    /**
     * 查询用户列表
     *
     * GET /home/users
     * 🔐 userAuth（普通用户鉴权）
     *
     * @query page {number?} 页码，默认 1，最小值 1
     * @query pageSize {number?} 每页数量，默认 20，范围 1-100
     * @query keyword {string?} 搜索关键词（可选）
     *
     * @returns {Promise<{ list: IUser[], pagination: IPagination }>}
     * @throws {Error} code:50001 QUERY_FAILED - 查询失败，请稍后重试
     */
    async list() {
        const { ctx } = this;
        // 校验直接写在方法内
        const schema = ctx.dsl({
            page:     'int:1-',         // 可选，最小值 1
            pageSize: 'int:1-100',      // 可选，1-100
            keyword:  'string:0-50',    // 可选，0-50 字符
        });
        await ctx.validateDsl(schema, 'query');  // 校验失败自动响应，代码不继续执行

        // ✅ validateDsl 不返回参数，仍需从 ctx.request.query 取值
        // ✅ query 参数全为字符串，数字字段需用 parseInt 转换
        const { page = 1, pageSize = 20, keyword } = ctx.request.query;
        const result = await ctx.service.userService.getUserList({
            page:     parseInt(page as string),
            pageSize: parseInt(pageSize as string),
            keyword:  keyword as string | undefined,
        });
        ctx.success(result);
    }

    /**
     * 创建用户
     *
     * POST /open/users
     *
     * @body email {string} 邮箱（必填）
     * @body password {string} 密码，8-32 位（必填）
     * @body nick_name {string?} 昵称，2-50 位（可选）
     *
     * @returns {Promise<IUser>}
     * @throws {Error} code:40002 EMAIL_ALREADY_EXISTS - 邮箱已被占用
     * @throws {Error} code:50002 USER_CREATE_FAILED - 用户创建失败
     */
    async create() {
        const { ctx } = this;
        const schema = ctx.dsl({
            email:     'email!',           // 必填邮箱
            password:  'string:8-32!',     // 必填，8-32 位
            nick_name: 'string:2-50',      // 可选，2-50 位
        });
        await ctx.validateDsl(schema, 'body');

        const { email, password, nick_name } = ctx.request.body;
        const user = await ctx.service.userService.createUser({ email, password, nick_name });
        ctx.success(user);
    }

    /**
     * 获取用户详情
     *
     * GET /home/users/:id
     * 🔐 userAuth（普通用户鉴权）
     *
     * @params id {string} 用户 ID（ObjectId，必填）
     *
     * @returns {Promise<IUser>}
     * @throws {Error} code:40001 USER_NOT_FOUND - 用户不存在
     * @throws {Error} code:50001 QUERY_FAILED - 查询失败，请稍后重试
     */
    async show() {
        const { ctx } = this;
        const schema = ctx.dsl({ id: 'objectId!' });
        await ctx.validateDsl(schema, 'params');

        const user = await ctx.service.userService.getUserById(ctx.params.id);
        ctx.success(user);
    }
}
```

### 3.3 三种参数来源

| 来源 | validateDsl 第二参数 | 取值方式 | 适用场景 |
|------|---------------------|---------|---------|
| Request Body | `'body'` | `ctx.request.body` | POST / PUT 请求 |
| Query String | `'query'` | `ctx.request.query` | GET 请求过滤/分页 |
| Path 参数 | `'params'` | `ctx.params` | `:id` 等路径参数 |

### 3.4 常用校验场景

```typescript
// 注册/登录
const registerSchema = ctx.dsl({
    email:       'email!',
    password:    'string:8-32!',
    nick_name:   'string:2-50',
    agree_terms: 'boolean!',
});

// 分页查询
const pageSchema = ctx.dsl({
    page:     'int:1-',
    pageSize: 'int:1-100',
    sort:     'string',
    order:    'string',
});

// ID 操作

// 搜索过滤

    status:     'int:0-2',
    start_date: 'string',

    tag_ids:    'array',
});
```

## 四、版本管理

### 4.1 版本策略

项目通过**路由前缀（模块划分）**而非 URL 版本号来管理接口演进：

- 不在 URL 中嵌入 `/v1/`、`/v2/` 等版本号
- 通过添加新路由（保留旧路由）的方式向后兼容
- 废弃接口在文档中标注 `@deprecated`，给出替代接口

### 4.2 路由文件演进规则

```typescript
// 新增接口：直接在 RESTful.ts 中添加
router.post('/home/users/:id/verify-email', userAuth, controller.home.user.verifyEmail);

// 废弃接口：保留旧路由，注释标注，不删除
// @deprecated 请使用 POST /home/users/:id/verify-email
router.post('/home/user/verify', userAuth, controller.home.user.verify);
```

### 4.3 响应格式扩展规则

- **新增字段**：允许直接在 `data` 中增加字段（向后兼容）
- **删除/改名字段**：需要在废弃期保留旧字段，新旧字段并存，并在文档标注
- **修改字段类型**：视为破坏性变更，需充分评估影响

## 五、错误处理

### 5.1 分层处理职责

| 层级 | 职责 | 工具 |
|------|------|------|
| **Controller** | 参数格式校验（直接内嵌，无需 Validator 文件） | `ctx.validateDsl()`（失败自动响应） |
| **Service** | 业务错误抛出 | `ctx.dsl.error.throw(errorCode)` |
| **Service** | 系统错误包装 | `throw ctx.dsl.error.create(errorCode)` |
| **exceptions 中间件** | 统一捕获并格式化响应 | 自动触发，无需手动调用 |

### 5.2 Controller 层 — 无需 try-catch，校验直接内嵌

Controller 层**不需要**也**不应该**包含 try-catch，错误由 `exceptions` 中间件统一捕获。  
参数校验**直接使用 `ctx.dsl()` 写在方法内**，不单独创建 Validator 文件（除非多个接口复用完全相同的复杂校验规则）：

```typescript
// ✅ 正确：校验内嵌 + 无 try-catch
async create() {
    const { ctx } = this;
    const schema = ctx.dsl({ email: 'email!', password: 'string:8-32!' });
    await ctx.validateDsl(schema, 'body');  // 校验失败自动响应，不继续执行
    const { email, password } = ctx.request.body;
    const user = await ctx.service.userService.createUser({ email, password });
    ctx.success(user);
}

// ❌ 错误：Controller 手动捕获错误
async create() {
    try {
        // ...
    } catch (error) {
        ctx.body = { code: -1, message: error.message };  // 禁止！
    }
}
```

### 5.3 Service 层 — 标准错误处理模式

```typescript
async getUserById(userId: string) {
    const { ctx } = this;
    try {
        const user = await ctx.msq.collection('users').findOne({ _id: userId, del_flag: 0 });

        if (!user) {
            // ✅ try 块中的业务错误：使用 ctx.dsl.error.throw
            ctx.dsl.error.throw('USER_NOT_FOUND');
        }

        return user;

    } catch (error: any) {
        ctx.logger.error('[UserService.getUserById] 查询失败', {
            userId,
            error: error.message,
        });

        // ✅ 已是业务错误（有 code 属性），直接抛出
        if (error.code) throw error;

        // ✅ catch 块中的系统错误：使用 ctx.dsl.error.create 包装后抛出
        throw ctx.dsl.error.create('QUERY_FAILED');
    }
}
```

### 5.4 exceptions 中间件工作机制

`exceptions.ts` 必须作为最后一个中间件，自动捕获所有未处理的错误并格式化响应：

```
请求 → 各中间件 → Controller → Service 抛出错误
                                        ↓
                              exceptions 捕获
                                        ↓
                              判断 error.code 是否存在
                              有 → 业务错误，原样返回
                              无 → 系统错误，返回 500
                                        ↓
                              统一格式化 JSON 响应
```
