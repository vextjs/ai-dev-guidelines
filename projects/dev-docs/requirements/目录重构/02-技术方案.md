## 一、需求背景

将 ai-dev-guidelines 当前 9 个平级子目录收拢为 `core/`（AI 执行内核）+ `docs/`（文档区）+ `projects/`（不变）+ `changelogs/`（不变）的四层结构。

**核心目标**:
- 减少顶级目录数量（9 → 4），提升语义清晰度
- 按受众分离：core/ = AI 机器执行，docs/ = 人类阅读 + AI 参考
- 简化 spec-self-fix → self-fix 命名

---

## 二、技术方案

### 2.1 目录移动（8 个 git mv 操作）

**路径映射规则**:

| # | 源路径 | 目标路径 | 操作 |
|---|--------|---------|------|
| 1 | `workflows/` | `core/workflows/` | 整体移入 |
| 2 | `standards/` | `core/standards/` | 整体移入 |
| 3 | `templates/` | `core/templates/` | 整体移入 |
| 4 | `tools/` | `core/tools/` | 整体移入 |
| 5 | `spec-self-fix/` | `core/self-fix/` | 移入+重命名 |
| 6 | `docs/*` (所有内容) | `docs/guides/` | 内容下沉一级 |
| 7 | `best-practices/` | `docs/best-practices/` | 整体移入 |
| 8 | `examples/` | `docs/examples/` | 整体移入 |

**执行顺序**:
1. 先创建 `core/` 和 `docs/guides/` 目录
2. 对原 `docs/` 内容先移到 `docs/guides/`（因为 `docs/` 目录本身要复用）
3. 其余按顺序 git mv

### 2.2 路径引用批量更新

**替换规则**（7 条映射）:

```
workflows/          → core/workflows/
standards/          → core/standards/
templates/          → core/templates/
tools/              → core/tools/
spec-self-fix/      → core/self-fix/
best-practices/     → docs/best-practices/
examples/           → docs/examples/
```

**特殊处理 — docs/ 引用**:
- 原 `docs/getting-started.md` → `docs/guides/getting-started.md`
- 原 `docs/execution-workflow-guide/` → `docs/guides/execution-workflow-guide/`
- 原 `docs/spec-self-fix-guide/` → `docs/guides/spec-self-fix-guide/`
- 原 `docs/DESIGN-PHILOSOPHY.md` → `docs/guides/DESIGN-PHILOSOPHY.md`
- 原 `docs/README.md` → `docs/guides/README.md`
- 需注意区分"引用原 docs/ 子内容"和"引用新 docs/ 目录"

**文件分层**:

| 层级 | 文件范围 | 修改策略 |
|------|---------|---------|
| P0 入口文件 | `.github/copilot-instructions.md`、`README.md`、`QUICK-REFERENCE.md`、`ONBOARDING.md`、`CONSTRAINTS.md`、`META.yaml` | 手动精确修改 |
| P1 内部文件 | core/ 下所有 .md + .js + .yaml 文件、docs/ 下所有文件、projects/*/profile/ | 脚本批量替换 + 手动复核 |
| P2 历史文件 | changelogs/、reports/、.ai-memory/ | **不修改** |

### 2.3 工具脚本路径修复

JS 文件中使用 `path.join(__dirname, '..')` 获取项目根目录。移入 `core/tools/` 后，`__dirname` 变为 `core/tools/`，`..` 指向 `core/` 而非项目根。

**修复方式**: 将所有 `path.join(__dirname, '..')` 改为 `path.join(__dirname, '../..')`

涉及文件:
- `validate-structure.js` — ROOT_DIR + EXPECTED_STRUCTURE
- `bump-version.js` — 文件路径清单
- `validate-links.js` — 根目录
- `doc-health-check.js` — 根目录
- `spec-health-check.js` — 根目录
- `update-task-index.js` — 根目录
- `add-constraint.js` — 根目录

### 2.4 docs/ 移动的顺序问题

`docs/` 目录本身会保留（作为新的文档区顶级目录），但其**内容**需要下沉到 `docs/guides/`。然后 `best-practices/` 和 `examples/` 移入 `docs/`。

执行顺序：
1. 在 `docs/` 下创建 `guides/` 子目录
2. 将 `docs/` 下的所有文件和子目录（除 `guides/`）移入 `docs/guides/`
3. 将 `best-practices/` 移入 `docs/`
4. 将 `examples/` 移入 `docs/`

## 四、风险评估

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| 路径引用遗漏 | 中 | 脚本批量替换 + grep 验证无残留 |
| docs/ 移动顺序错误 | 低 | 先移内容到 guides/，再移入其他目录 |
| JS 脚本根路径失效 | 低 | `__dirname + '../..'` 统一修复 |
| 回滚 | 低 | git reset --hard |
