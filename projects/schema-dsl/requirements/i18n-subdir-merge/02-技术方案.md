## 一、需求背景与目标

多人协作时，语言文件平铺在单文件导致 code 冲突和 Git merge 冲突，需支持**按子目录拆分语言文件、启动时合并**。

**核心目标**：
- 递归扫描子目录，同语言文件自动合并
- 同名 key 冲突可检测、可告警、可严格阻断
- **100% 兼容现有所有用法（零破坏性变更）**

## 二、现有 `dsl.config` 用法全景（兼容边界）

> ⚠️ 技术方案必须兼容以下所有现有用法，不能改变任何参数的语义

### 用法 1：字符串路径（当前只扫描一层）
```javascript
dsl.config({ i18n: '/path/to/locales' })
// 现有行为：readdirSync 一层，加载 .js/.json 文件
// 增强后：递归扫描子目录，其余行为不变
```

### 用法 2：对象直接传语言包（不涉及目录）
```javascript
dsl.config({ i18n: { 'zh-CN': { key: '值' }, 'en-US': { key: 'val' } } })
// 现有行为：Object.keys 遍历，Locale.addLocale
// 增强后：不变
```

### 用法 3：对象带 `localesPath`（文档记载用法）
```javascript
dsl.config({ i18n: { localesPath: './i18n/labels' } })
// 现有行为：当前 index.js 代码实测：此用法 localesPath 字段被忽略（Object.keys 只看语言 key）
// 增强后：需要兼容，如遇 localesPath 字段则走目录扫描
```

### 用法 4：不存在的路径（不应抛错）
```javascript
dsl.config({ i18n: '/non/existent/path' })
// 现有行为：console.warn，不抛错
// 增强后：不变
```

## 三、技术方案

### 3.1 F1：递归子目录扫描

**改动位置**：`index.js` — `dsl.config` 函数内，`options.i18n` 为字符串路径分支

**核心思路**：将现有 `readdirSync` 单层扫描提取为内部函数 `loadLocalesFromDir`，支持递归。

```
loadLocalesFromDir(dirPath, options)
  ├── readdirSync(dirPath, { withFileTypes: true })
  ├── 遍历每个 entry：
  │   ├── entry.isDirectory() → 递归调用 loadLocalesFromDir(subDir, options)
  │   └── entry.isFile() + (.js|.json) → 校验文件名 → require → Locale.addLocale（带冲突检测）
  └── 文件名格式校验：/^[a-z]{2}(-[A-Z]{2})?$/.test(localeName) 才加载，否则 SKIP
```

**语言文件名格式正则**：`/^[a-z]{2}(-[A-Z]{2})?$/`
- ✅ 通过：`zh-CN`、`en-US`、`ja-JP`、`zh`、`en`
- ❌ 跳过：`index`、`utils`、`CODE-SEGMENTS`、`README`

**加载顺序**：按 `readdirSync` 返回的文件系统顺序（字母序），保证跨平台一致性。

**与现有字符串路径分支的关系**：
- 原有代码逻辑被完整替换为 `loadLocalesFromDir` 调用
- 原来只扫描一层 `.js/.json` 的文件（不做文件名格式校验）→ 新逻辑同样扫描一层 + 额外递归子目录 + 文件名格式校验
- **SKIP 的文件仅打 `debug` 级别日志，不影响静默使用**

---

### 3.2 F2：同名 key 冲突检测

**改动位置**：`loadLocalesFromDir` 内部，调用 `Locale.addLocale` 之前

**冲突检测逻辑**：
```
合并前：检查 messages 的每个 key 是否已存在于 Locale.locales[locale]
  → 存在：
      strict 模式  → throw Error（含冲突 key、原文件路径、新文件路径）
      默认模式     → console.warn（含冲突 key 和文件路径）
  → 不存在：直接合并
```

**错误/警告格式**：
```
[schema-dsl] WARN: i18n key 冲突 'account.notFound' in locale 'zh-CN'
             已定义于: /path/locales/core/zh-CN.js
             被覆盖于: /path/locales/account/zh-CN.js
```

**`strict` 选项位置**：挂在 `options.i18n` 同级，即 `options.strict`（不放在 `i18n` 内部，保持顶层配置简洁）：
```javascript
dsl.config({
  i18n: '/path/to/locales',
  strict: true   // 新增顶层选项，默认 false
});
```

> 为什么不放 `i18n.strict`？  
> 因为 `i18n` 已有字符串/对象两种形态，增加第三种对象结构（带 strict）会导致歧义。  
> 顶层 `strict` 语义清晰，且后续其他配置项也可复用。

---

### 3.3 F3：语言文件名格式校验

- 正则：`/^[a-z]{2}(-[A-Z]{2})?$/` 校验文件名（去掉扩展名后）
- 不通过的文件：仅打 `debug` 日志（`[schema-dsl] SKIP: ...`），不 warn、不报错
- 现有用法的文件（如直接命名为 `zh-CN.js`）完全通过校验，行为不变

---

### 3.4 对象用法中 `localesPath` 的兼容处理

文档记载了 `dsl.config({ i18n: { localesPath: './i18n/labels' } })` 的用法，但现有代码实际上将 `localesPath` 当成语言 key 处理（`Object.keys` 遍历对象），会静默失败。

**增强策略**：在对象分支入口处，优先检测 `localesPath` 字段：
```javascript
if (typeof options.i18n === 'object') {
  if (options.i18n.localesPath) {
    // 走目录扫描逻辑（复用 loadLocalesFromDir）
    loadLocalesFromDir(options.i18n.localesPath, options);
  } else {
    // 原有逻辑：Object.keys 遍历语言包
    Object.keys(options.i18n).forEach(locale => {
      Locale.addLocale(locale, options.i18n[locale]);
    });
  }
}
```

这样让文档中写的 `localesPath` 用法真正生效，同时不破坏现有对象传语言包的用法。

## 六、注意事项

1. **`require` 缓存**：Node.js `require` 自带模块缓存，重复 `require` 同一文件返回缓存，无性能问题
2. **循环依赖**：目录扫描仅处理 `.js/.json` 叶子文件，不会触发循环引用
3. **`readdirSync` 返回顺序**：POSIX 文件系统字母序，Windows 也基本一致；加载顺序文档说明清楚即可
4. **`withFileTypes: true`**：Node.js 12+ 支持，符合项目最低版本要求
