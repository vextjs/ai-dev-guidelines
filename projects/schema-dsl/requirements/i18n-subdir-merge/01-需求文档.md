## 一、需求背景

多人协作开发时，所有业务模块的多语言 key（包括 code 码）都平铺在同一个语言文件中（如 `zh-CN.js`）。随着团队扩大，存在以下核心问题：

1. **code 数值冲突**：不同模块开发者各自分配 code 号，容易出现重复（如 A 和 B 都使用了 `code: 10001`），前端 switch-case 逻辑出错
2. **Git merge 冲突**：多人同时修改同一文件，导致频繁 merge 冲突
3. **所有权混乱**：任何人都可以修改任意 key，无法隔离职责边界

---

## 二、需求内容

### 功能需求

- [ ] **F1：`dsl.config({ i18n: path })` 支持递归子目录扫描**
  - 当 `i18n` 为目录路径时，自动扫描所有子目录中的语言文件
  - 子目录名作为模块组织层，不影响最终语言 key 的命名
  - 同语言文件按加载顺序深度合并（后者不覆盖前者，或可配置）

- [ ] **F2：同名 key 冲突告警**
  - 合并时检测到同名 key 重复定义时，输出 `WARN` 日志，提示冲突来源文件
  - 支持 `strict: true` 严格模式，冲突时直接抛出错误（阻断启动）

- [ ] **F3：语言文件名格式校验**
  - 只加载符合语言代码格式（`xx-XX.js` 或 `xx.js`）的文件
  - 跳过工具文件（如 `index.js`、`utils.js`、`CODE-SEGMENTS.js`）并输出 SKIP 日志

### 非功能需求

- **兼容性**：100% 向后兼容现有用法，现有单层目录、对象方式不受影响
- **性能**：递归扫描仅在应用启动时执行一次，运行时零开销
- **Node 版本**：兼容 Node.js >= 12（现有最低要求不变）

## 三、目标使用方式（用户视角）

### 目录结构

```
project/
├── locales/
│   ├── core/               # 公共 code 段：1000-1999（框架层维护）
│   │   ├── zh-CN.js
│   │   └── en-US.js
│   ├── account/            # 账户模块 code 段：10000-10999（开发者A）
│   │   ├── zh-CN.js
│   │   └── en-US.js
│   ├── order/              # 订单模块 code 段：20000-20999（开发者B）
│   │   ├── zh-CN.js
│   │   └── en-US.js
│   └── payment/            # 支付模块 code 段：30000-30999（开发者C）
│       ├── zh-CN.js
│       └── en-US.js
└── app.js
```

### API 调用方式（不变）

```javascript
// app.js — 应用启动时一次性加载
dsl.config({
  i18n: path.join(__dirname, 'locales')  // 自动递归扫描所有子目录
});

// 运行时切换语言（不变）
validate(schema, data, { locale: 'zh-CN' });
```

### 每个模块语言文件示例

```javascript
// locales/account/zh-CN.js（开发者A 独立维护）
module.exports = {
  'account.notFound': { code: 10001, message: '账户不存在' },
  'account.locked':   { code: 10002, message: '账户已锁定' },
};

// locales/order/zh-CN.js（开发者B 独立维护）
module.exports = {
  'order.notFound': { code: 20001, message: '订单不存在' },
  'order.notPaid':  { code: 20002, message: '订单未支付' },
};
```

## 四、验收标准

| # | 验收项 | 标准 |
|---|-------|------|
| V1 | 递归扫描 | `dsl.config({ i18n: 'locales' })` 能正确加载子目录下所有语言文件 |
| V2 | 合并正确性 | 同语言的多个子目录文件合并后，所有 key 均可通过 `validate` 和 `dsl.error` 正常使用 |
| V3 | 冲突告警 | 同名 key 重复定义时，控制台输出包含来源文件路径的 WARN 日志 |
| V4 | 严格模式 | `strict: true` 时，同名 key 冲突抛出 Error |
| V5 | 文件名过滤 | `index.js`、`utils.js` 等非语言文件不被加载 |
| V6 | 向后兼容 | 现有单层目录 / 对象方式的用法测试全部通过（无回归） |
| V7 | 测试覆盖 | 新增测试用例全部通过 |
