## 🏗️ 核心架构概念

### 路由三段式

```typescript
// path: 路由路径
// options: 声明式配置对象（中间件、参数校验、文档元信息等）
// handler: 路由处理函数
route('POST /users', {
  middleware: [auth],
  validate: { body: userSchema },
  docs: { summary: '创建用户' }
}, async (req, res) => {
  const user = await app.services.user.create(req.body);
  res.json(user);
});
```

### 核心对象

| 对象 | 用途 | 访问方式 |
|------|------|---------|
| `app` | 应用实例，挂载全局资源 | `app.db`、`app.config`、`app.services.xxx` |
| `req` | 请求上下文（封装底层框架的请求） | handler 参数 |
| `res` | 响应对象 | `res.json()`，抛出异常自动捕获并返回错误响应 |

### 服务层

```typescript
// 所有 service 文件注入到 app 对象
// 通过 app.services.<name> 方式访问
class UserService {
  async create(data: CreateUserDTO) {
    return app.db.users.insert(data);
  }
}
// 访问: app.services.user.create(...)
```

### 插件机制

```typescript
// 插件可以在 app 对象上注册方法、属性、路由
const myPlugin = {
  name: 'my-plugin',
  register(app) {
    app.decorate('myUtil', () => { /* ... */ });
    app.addRoute('GET /plugin-route', {}, handler);
  }
};
```

## 🔗 关联项目

### vext-test — 功能验证项目

| 项目 | 说明 |
|------|------|
| **仓库** | `vextjs/vext-test`（GitHub） |
| **本地路径** | `vext-test/`（工作区根目录下，与 `vext/` 同级） |
| **用途** | 作为 vext 框架的**端到端功能验证应用**，运行真实的 vext 服务并执行 HTTP 级验证 |
| **验证脚本** | `verify.mjs` — 自动启动服务 → 执行全部验证 → 汇总结果 → 关闭服务 |
| **当前覆盖** | 46 项功能验证（A~L 共 12 个分类） |

#### 验证覆盖分类

| 分类 | 编号 | 覆盖范围 |
|------|------|---------|
| A. 路由 | #1-#6 | 文件映射、三段式/两段式、HTTP 方法、路径参数、路由级中间件 |
| B. 校验 | #7-#12 | body/query/param/header 校验、422 错误、`req.valid()` |
| C. 服务 | #13-#16 | 类导出+DI、`app.services`、child logger、服务间依赖 |
| D. 中间件 | #17-#19 | `defineMiddleware`、路由级引用、洋葱模型 |
| E. 插件 | #20-#26 | `definePlugin`、`app.extend`、onReady/onClose、dependencies |
| F. i18n | #27-#30, #44-#45 | 多目录语言包、i18n key、模板变量、Accept-Language、字符串快捷方式 |
| G. fetch | #31-#32, #46 | 自引用调用、requestId 传播、propagateHeaders 端到端 |
| H. 上下文 | #33-#34 | requestId 注入、`requestContext.getStore()` |
| I. OpenAPI | #35-#36 | `/openapi.json`、路由 paths 完整 |
| J. 错误 | #37-#41 | 统一包装、`app.throw`、string 业务码、404 兜底、409 冲突 |
| K. 配置 | #42 | `config/default.ts` 各项配置生效 |
| L. 生命周期 | #43 | onReady 状态 |

#### 🔴 强制要求

1. **所有 vext 框架新功能必须在 vext-test 中添加对应验证项**，确保端到端行为与文档/单元测试一致
2. 新增验证项必须在 `verify.mjs` 中注册，并更新文件头注释中的覆盖清单和总项数
3. `vext-test/README.md` 的验证项清单必须与 `verify.mjs` 保持同步
4. vext 发版前必须运行 `node verify.mjs` 确认所有验证项通过
5. 当 vext 出现破坏性变更时，vext-test 必须同步更新

> ⚠️ **不适合在 vext-test 中验证的功能**（需在 vext 框架内部单元测试覆盖）：
> - 进程级行为（如 `onFatalError` 钩子 — 需触发 uncaughtException）
> - 构建产物验证（`vext build` 输出结构）
> - Cluster 多进程管理（需 fork worker）
