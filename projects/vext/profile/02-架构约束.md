# vext 架构约束

> **项目**: vext  
> **优先级**: P0（强制规范，不可违反）  
> **最后更新**: 2026-02-27

---

## 🚫 架构禁止项

### 1. 禁止绕过 req/res 封装直接使用底层框架对象

```yaml
规则: 路由 handler 中只允许使用框架封装的 req 和 res 对象
原因: vext 设计目标是底层框架可替换（当前 Hono，未来自研），直接使用底层对象会导致强耦合

❌ 禁止:
  - 直接访问 Hono 的 c (Context) 对象
  - 使用 c.req.raw、c.env 等 Hono 特有 API
  - 在 handler 中 import hono 的类型或方法

✅ 正确:
  - 使用 req.params、req.query、req.body 等统一封装
  - 使用 res.json()、res.error() 等统一响应方法
  - 通过 app.config、app.db 等标准入口访问基础设施
```

### 2. 禁止在路由 handler 中直接访问数据库

```yaml
规则: 路由 handler 必须通过 service 层访问数据库，不得直接调用 app.db
原因: 分层架构要求业务逻辑集中在 service 层，handler 仅负责参数提取和响应返回

❌ 禁止:
  - handler 中直接 app.db.collection('users').find()
  - handler 中直接 import 数据库驱动

✅ 正确:
  - handler 调用 app.services.user.findById(id)
  - service 内部访问数据库
```

### 3. 禁止在 service 中处理 HTTP 请求/响应

```yaml
规则: service 层不得引用 req、res 对象或 HTTP 状态码
原因: service 是纯业务逻辑层，应与传输协议解耦，便于复用和测试

❌ 禁止:
  - service 方法接收 req 参数
  - service 中调用 res.json()
  - service 中 throw new HttpError(404, ...)

✅ 正确:
  - service 方法接收具体业务参数（id, name, options）
  - service 抛出业务错误，由框架错误中间件统一转换为 HTTP 响应
  - service 返回纯数据对象
```

### 4. 禁止绕过 schema-dsl 进行参数校验

```yaml
规则: 所有接口参数校验必须使用 schema-dsl 模块，不得自行编写校验逻辑
原因: 统一校验方式，确保校验规则可被文档生成工具消费

❌ 禁止:
  - handler 中手动 if (!params.id) throw ...
  - 使用 joi、zod 等第三方校验库
  - 不做任何校验直接使用参数

✅ 正确:
  - 在路由 options 中声明 schema-dsl 校验规则
  - 框架自动在 handler 执行前完成校验
  - 校验失败由框架统一返回错误响应
```

### 5. 禁止不通过路由三段式注册路由

```yaml
规则: 所有路由必须使用 path + options + handler 三段式注册
原因: options 是声明式配置对象，描述中间件、参数校验等元信息，需要暴露给文档生成工具

❌ 禁止:
  - 直接 app.get('/path', handler)（缺少 options）
  - 绕过路由注册机制手动挂载中间件
  - 在 handler 内部做应该在 options 中声明的事（如手动鉴权）

✅ 正确:
  - app.get('/users/:id', { auth: true, schema: { params: idSchema } }, handler)
  - 中间件、校验、鉴权等通过 options 声明
```

### 6. 禁止在非插件代码中直接修改 app 对象

```yaml
规则: 只有插件（plugin）可以在 app 对象上注册方法或属性
原因: 插件机制是框架扩展的唯一入口，随意修改 app 会导致命名冲突和不可追踪的副作用

❌ 禁止:
  - 在 service 中 app.customMethod = ...
  - 在 middleware 中 app.newProp = ...
  - 在启动脚本中随意挂载到 app

✅ 正确:
  - 通过插件接口注册: plugin.register(app => { app.decorate('myMethod', fn) })
  - 插件在启动阶段统一注册，运行时 app 对象冻结
```

### 7. 禁止 service 之间形成循环依赖

```yaml
规则: service 之间的调用关系必须是单向的，不得形成 A→B→A 循环
原因: 循环依赖导致初始化顺序不确定、内存泄漏、难以测试

❌ 禁止:
  - userService 调用 orderService，orderService 又调用 userService
  - 通过 app.services 间接形成循环引用

✅ 正确:
  - 提取公共逻辑到独立的 service 或 util
  - 通过事件机制解耦双向依赖
  - 保持清晰的依赖方向（上层 → 下层）
```

### 8. 禁止硬编码配置值

```yaml
规则: 所有配置项必须通过 app.config 访问，不得在代码中硬编码
原因: 配置需要按环境切换（dev/staging/prod），硬编码导致部署困难

❌ 禁止:
  - const port = 3000
  - const dbUrl = 'mongodb://localhost/vext'
  - const secret = 'my-jwt-secret'

✅ 正确:
  - const port = app.config.port
  - const dbUrl = app.config.db.url
  - 敏感信息通过环境变量注入到 config
```

### 9. 所有新功能必须在 vext-test 中添加端到端验证

```yaml
规则: 每个新增或修改的框架功能，必须在 vext-test 项目中添加对应的验证项
原因: 单元测试验证内部逻辑正确性，端到端验证确保功能在真实应用场景中可用；两者互补，缺一不可

❌ 禁止:
  - 新增框架功能但不在 vext-test 中添加验证项
  - 仅依赖单元测试声称"已覆盖"，跳过端到端验证
  - vext-test verify.mjs 的验证项清单与实际代码不同步
  - 发版前未运行 verify.mjs 确认全部通过

✅ 正确:
  - 新功能同步在 vext-test 中添加路由端点 + verify.mjs 验证项
  - verify.mjs 文件头注释、README.md 验证清单与代码保持同步
  - 发版前执行 node verify.mjs 确认所有验证项通过
  - 不适合端到端验证的功能（进程级行为、构建产物、Cluster）在框架内部单元测试覆盖
```
