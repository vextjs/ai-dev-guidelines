# 三轮验证机制详细清单

> 23 项验证清单，确保 AI 输出质量

---

## 📋 验证原则

```yaml
原则 1 - 必须验证:
  - 每个关键步骤后必须执行验证
  - 不得跳过任何验证项

原则 2 - 自动修复:
  - 可自动修复的问题直接处理
  - 记录修复过程

原则 3 - 透明报告:
  - 验证结果必须向用户报告
  - 失败项必须说明原因

原则 4 - 阻断机制:
  - 关键验证失败阻断流程
  - 用户确认后方可继续

原则 5 - 进度可见:
  - 方案验证和代码验证必须输出进度
  - 必须标注每个验证项结果
```

## 🔴 第一轮：逻辑验证（6 项）

### 1. 需求覆盖

```yaml
检查内容: 所有需求点是否有对应实现

量化标准: 
  - 对照需求列表，每项需有函数/类/模块实现
  - 覆盖率 = 已实现需求数 / 总需求数

🔴 反向验证检查点（必须执行）:
  用户说"只XXX" → 方案是否包含其他？
    - 示例：用户说"只用分钟"，检查是否还有 days/hours 参数
    - 发现违背 → ❌ 阻断，立即修正
  
  用户说"所有XXX" → 方案是否有遗漏？
    - 示例：用户说"所有过期时间"，检查全局搜索结果是否完整
    - 发现遗漏 → ❌ 阻断，补充清单
  
  用户说"不要XXX" → 方案是否违背？
    - 示例：用户说"不要改定时任务"，检查定时任务是否被修改
    - 发现违背 → ❌ 阻断，移除修改

失败处理: 列出未覆盖的需求，报告用户
自动修复: ❌ 否
```

### 2. 边界处理

```yaml
检查内容: 参数验证是否处理边界情况

检查边界:
  - null
  - undefined
  - 空字符串 ''
  - 数字 0
  - 空数组 []
  - 空对象 {}
  - 极大值（如 Number.MAX_SAFE_INTEGER）
  - 极小值（如 Number.MIN_SAFE_INTEGER）
  - 负数（当期望正数时）

失败处理: 列出未处理的边界情况
自动修复: ✅ 是 - 自动添加边界检查代码

修复示例:
  // 原代码
  function getUser(id) { return users[id]; }
  
  // 自动修复后
  function getUser(id) {
    if (id == null || id === '') return null;
    return users[id];
  }
```

### 3. 错误处理

```yaml
检查内容: 所有 async 函数是否有 try-catch

检查规则:
  - async 函数必须有错误处理
  - Promise 链必须有 .catch()
  - 数据库操作必须有错误处理
  - 网络请求必须有错误处理

失败处理: 列出缺少错误处理的函数
自动修复: ✅ 是 - 自动添加 try-catch

修复示例:
  // 原代码
  async function fetchUser(id) {
    const user = await db.users.findById(id);
    return user;
  }
  
  // 自动修复后
  async function fetchUser(id) {
    try {
      const user = await db.users.findById(id);
      return user;
    } catch (error) {
      logger.error('fetchUser failed', { id, error: error.message });
      throw error;
    }
  }
```

### 4. 逻辑完整

```yaml
检查内容: 条件分支是否完整

检查规则:
  - if 是否有对应的 else
  - switch 是否有 default
  - 三元表达式是否覆盖所有情况
  - 循环是否有退出条件

失败处理: 列出不完整的条件分支
自动修复: ✅ 是 - 自动补充 else/default

修复示例:
  // 原代码
  if (status === 'active') { activate(); }
  
  // 自动修复后
  if (status === 'active') {
    activate();
  } else {
    // 其他状态，保持不变或记录日志
    logger.debug('Skipping non-active status', { status });
  }
```

### 5. 流程正确

```yaml
检查内容: 业务流程是否符合需求

检查规则:
  - 流程顺序是否正确
  - 是否有遗漏步骤
  - 是否有多余步骤
  - 状态转换是否正确

失败处理: 绘制流程对比图，说明差异
自动修复: ❌ 否 - 需要人工确认业务逻辑
```

### 6. 返回值

```yaml
检查内容: 返回值类型和结构是否正确

检查规则:
  - 返回类型是否与声明一致
  - 返回结构是否符合接口定义
  - 是否所有路径都有返回值
  - 是否正确处理 void 返回

失败处理: 列出返回值问题
自动修复: ❌ 否 - 需要人工确认返回值设计
```
