## 📝 响应流程

### 标准响应流程（模式 1~4）

```yaml
Step 1: 识别用户意图
  - 提取关键信息
  - 分类意图类型（纠正/质疑/重复/混淆）

Step 2: 致歉并说明
  "非常抱歉！我会立即检查..."

Step 3: 定位相关规范
  - 根据对话内容定位规范文件
  - 如不确定，向用户确认

Step 4: 执行检测
  - 自我检查是否遵守规范
  - 检查规范是否清晰/完整

Step 5: 分析根因
  - 为什么产生偏差？
  - 规范问题 or AI 问题？

Step 6: 生成修复建议
  - 具体的修复方案
  - 预期效果

Step 7: 询问用户
  "是否记录此问题并修复规范？[Y/N]"

Step 8: 执行修复（如用户同意）
  - 应用修复
  - 生成修复记录
  - 更新 records/summary.md
```

---

### 升级响应流程（模式 5 — 🆕 v2.0）

```yaml
Step 1: 识别反复问题
  - 提取问题类型
  - 扫描 core/self-fix/records/ 历史
  - 确认出现次数和升级等级

Step 2: 承认问题并说明升级
  "您说得对，这类问题不应该反复出现。
   我检测到这是第 N 次出现，已触发升级分析。"

Step 3: 断层检测
  逐项检查上次修复是否完整:
  □ 执行流程入口文件已修复？
  □ spec-self-fix 知识库已回写？
  □ 检测规则已更新？
  □ 自动触发场景已新增？
  □ 修复模式已更新？
  □ 修复记录已更新？

Step 4: 5-Why 根因分析
  - 不停留在表面原因
  - 深入到流程/机制层面的根因

Step 5: 制定全链路修复方案
  - 列出所有需要修改的文件
  - 说明每处修改的目的
  - 标注升级措施（建议→强制、被动→主动、单点→全量）

Step 6: 输出变更计划等待用户确认
  "以下是全链路修复的变更计划，共 N 个文件：
   | # | 文件 | 操作 | 内容 |
   ...
   是否确认执行？"

Step 7: 执行全链路修复（用户确认后）
  - 按计划修改所有文件
  - 每个文件修改后验证

Step 8: 回写完整性验证
  修复完成后逐项确认:
  □ 执行流程入口 ✅
  □ 知识库检测模块 ✅
  □ 知识库触发模块 ✅
  □ 知识库修复模块 ✅
  □ 修复记录和统计 ✅
  → 全部 ✅ → 修复完整，复现风险消除

Step 9: 生成修复记录
  - 写入 core/self-fix/records/
  - 更新 records/summary.md
  - 更新 .ai-memory 记忆文件
```

## 📊 修复记录格式

每次触发都应记录到 `../../records/YYYYMMDD-user-feedback-xxx.md`：

### 标准修复记录（模式 1~4）

```markdown
# 用户反馈触发的规范修复

**日期**: 2026-02-12 15:30  
**触发模式**: 理解偏差纠正  
**问题类型**: 需求理解不清

## 对话记录

**用户**: "我明明说的是添加限流，你怎么做成了防火墙？"

**AI 识别**:
- 关键词: "明明"、"不是"
- 意图: 纠正理解偏差
- 期望: 添加限流
- 实际: 防火墙

## 规范检查

**相关规范**: `core/workflows/01-requirement-dev/README.md`

**问题定位**: 阶段 1 需求理解步骤

**当前表述**:

### 阶段 1: 理解需求
1. 阅读用户需求描述
2. 分析涉及的模块和功能
3. 列出关键改动点

**问题**: 缺少"复述核心需求"步骤

## 修复方案

**方案**: 在阶段 1 增加第 3 步

**修改后**:

### 阶段 1: 理解需求
1. 阅读用户需求描述
2. 分析涉及的模块和功能
3. **用一句话复述核心需求**  ← 新增
4. 列出关键改动点

## 修复结果

✅ 已应用修复  
✅ 已更新规范文件  
✅ 已向用户确认

## 后续跟踪

- [ ] 观察后续任务是否仍有类似问题
- [ ] 1 周后验证修复效果
```

## ⚡ 快速参考

### 用户说什么会触发？

| 用户表达 | 触发类型 | 优先级 | 模式 |
|---------|---------|--------|:----:|
| "我明明说的是..." | 理解偏差 | P0 | 1 |
| "你为什么不按规范..." | 规范执行质疑 | P0 | 2 |
| 重复 3 次同一需求 | 隐式反馈 | P1 | 3 |
| "这个步骤有必要吗？" | 混淆质疑 | P2 | 4 |
| "这问题又出现了" / "每次都重复" | 🔴 反复问题升级 | P0 | 5（🆕） |
| "修了等于没修" / "后续都会复现" | 🔴 修复断层 | P0 | 5（🆕） |
| "规范中没有定义..." / "不合理" | 规范缺失/不合理 | P1 | 6（🆕） |

### AI 应该怎么做？

```yaml
模式 1~4（标准流程）:
  1. 立即致歉
  2. 说明会检查规范
  3. 执行检测和分析
  4. 生成修复建议
  5. 询问用户是否修复
  6. 记录到 records/

模式 5（升级流程 — 🆕 v2.0）:
  1. 承认反复问题
  2. 扫描修复历史，确认出现次数
  3. 执行断层检测（执行流程 + 知识库 + 记录）
  4. 执行 5-Why 根因分析
  5. 制定全链路修复方案
  6. 输出变更计划，等待用户确认
  7. 执行修复（执行流程入口 + spec-self-fix 知识库）
  8. 回写完整性验证（逐项确认 6 个检查点）
  9. 生成修复记录 + 更新统计

模式 6（规范补充流程 — 🆕 v2.1）:
  1. 识别反馈类型（缺失/不合理/模糊/新增）
  2. 审查现有规范（grep + read_file）
  3. 评估合理性（约束 #12）
  4. 生成修复方案
  5. 等待用户确认
  6. 执行修复 + 交叉验证（约束 #14）
  7. 记录到 records/
```

## 📋 变更日志

```yaml
v2.1 (2026-02-27):
  新增:
    - 模式 6: 规范缺失/不合理反馈（规范补充触发）
      - 4 种子类型: 缺失型 / 不合理型 / 模糊型 / 新增型
      - 与模式 2（执行质疑）、模式 4（混淆质疑）的区分说明
      - 7 步识别流程: 判断类型 → 审查现有规范 → 评估合理性 → 生成方案 → 确认 → 交叉验证 → 记录
      - 2 个实际案例（文件拆分规范缺失 + SUMMARY 日期精度不够）
    - 算法 1 新增: 规范补充触发关键词（"没有定义"、"缺失"、"不合理"、"需要新增"等）
    - 算法 2 新增: 建设性反馈语气特征（建议语气、条件句、分析语气）
  根因:
    - §会话07 用户提出 3 个审计问题，其中"规范中没有定义文件过大需要拆分"和
      "SUMMARY 日期精度不够"均属于规范缺失/不合理场景。
      原有模式 1~5 无法精确覆盖此类"用户指出规范本身需要补充或修改"的意图。
    - 模式 6 的核心价值: 将用户的建设性反馈转化为规范改进，
      与模式 2（AI没按规范执行）和模式 5（问题反复出现）形成互补。

v2.0 (2026-02-27):
  新增:
    - 模式 5: 反复出现同类问题升级触发
      - 升级等级体系（🟢常规/🟡关注/🔴严重）
      - 断层检测清单（执行流程 + 知识库 + 记录 共 6 项检查点）
      - 全链路修复流程（从单点修复升级为全链路修复）
      - 回写完整性验证（防止"修了但没真修"）
      - 实际案例（版本号遗漏 BUG-023~031 + 知识库断层 §会话06）
    - 算法 4: 修复历史匹配（扫描 records/ + 断层检测）
    - 升级触发关键词（"又出现了"、"每次都"、"重复"等）
    - 升级响应流程（9 步，含断层检测和回写验证）
    - 升级修复记录格式（含历史表、断层检测表）
    - 机制定位新增第 5 点: 升级驱动
  根因:
    - §会话06 用户指出: 上次修复只写入了执行流程入口文件（QUICK-REFERENCE/CONSTRAINTS/task-memory），
      但 spec-self-fix 知识库本身没有同步更新，6 个子模块仍停留在 v1.0。
      这意味着修复存在"断层"— 未来健康检查不会使用最新检测规则，问题可能复现。
    - 模式 5 的核心价值: 当用户指出同类问题反复出现时，不能再做单点修复，
      必须执行全链路修复（执行流程 + 知识库 + 记录），并验证回写完整性。
  关联修复记录:
    - core/self-fix/records/2026-02-27-version-sync-gap-and-stage0-timing.md
    - core/self-fix/records/2026-02-27-selffix-knowledge-base-writeback.md

v1.0 (2026-02-12):
  初始版本: 模式 1~4（理解偏差、规范执行质疑、重复需求、混淆质疑）
```
