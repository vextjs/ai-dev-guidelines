## 🎯 触发场景

### 场景 1: 预检查阶段

```yaml
触发时机: 工作流执行前（core/workflows/00-pre-check/）

检测内容:
  - 工作流文件是否完整
  - 工作流是否有验证清单
  - 模板文件是否存在

执行方式: 自动检测，发现问题立即提示

示例:
  AI 收到: "添加限流功能"
  预检查: 读取 core/workflows/01-requirement-dev/README.md
  检测: 文件是否包含必要章节
  如缺失: "检测到工作流文件缺少验证清单，是否补充？"
```

```yaml
触发时机: 执行任务时发现矛盾规范

检测内容:

执行方式: 立即暂停，提示用户
```

### 场景 5: 版本号全量同步检测（🆕 v2.0）

```yaml
触发时机:
  - 🔴 发布新版本时（任何涉及版本号变更的操作）
  - 🔴 执行规范健康检查时
  - 修改任何规范文件后（约束 #14 交叉验证）

检测内容:
  🔴 必须逐个 read_file 以下 8 个文件，确认版本号全部一致（禁止推断）:

  | # | 文件路径                           | 版本号位置                                                          |
  |:-:|-----------------------------------|-------------------------------------------------------------------|
  | 1 | .github/copilot-instructions.md   | L1 `**版本**: vX.Y.Z`                                              |
  | 2 | README.md                         | L3 `> **版本**: vX.Y.Z`                                            |
  | 3 | QUICK-REFERENCE.md                | L5 `**版本**: vX.Y.Z`                                              |
  | 4 | CONSTRAINTS.md                    | L6 `**版本**: vX.Y.Z`                                              |
  | 5 | STATUS.md                         | L5 `**当前版本**: vX.Y.Z`                                          |
  | 6 | CHANGELOG.md                      | 版本概览表新增行                                                     |
  | 7 | core/workflows/decision-tree.yaml      | L1 注释 + L4 `version` 字段 + L82 注释 + L87 `mandatory_precheck.version` |
  | 8 | core/workflows/00-pre-check/README.md  | L3 `> **版本**: vX.Y.Z` + 文件末尾 `**版本**: vX.Y.Z`（两处）         |

  ⚠️ 此清单与 QUICK-REFERENCE.md §版本号文件清单 同源，新增文件时两处必须同步更新。
  ⚠️ 非入口文件（如 task-memory.md v1.7、temp-reports.md v1.5）使用独立版本号，不在此清单中。

执行方式: 🔴 自动执行，发现不一致立即报告

检查方式（两步）:
  1. 被动检查（原有）: 本次修改的文件已更新版本号和最后更新日期？
  2. 🔴 主动全量检查（v2.0 新增）: 8 个文件的版本号是否全部一致？
     - 必须逐个 read_file 确认，禁止基于"上次检查过"或"应该没问题"推断
     - 任一文件版本号不一致 → 🔴 P0 问题，立即修复

示例:
  AI 修改: QUICK-REFERENCE.md 版本号 v2.10.0 → v2.11.0
  自动触发: 逐个 read_file 检查 8 个文件
  发现: decision-tree.yaml 仍为 v2.10.0
  报告: "🔴 版本号不一致: decision-tree.yaml (v2.10.0) ≠ 其余文件 (v2.11.0)"

根因背景:
  此场景源于版本号遗漏反复发生 3 次（BUG-023~031），根因是只有被动检查
  （"改了的文件版本号对吗？"）而缺少主动全量检查（"所有含版本号的文件都一致吗？"）。
  v2.0 新增此自动触发场景，确保每次涉及版本变更时自动执行全量检查。

关联:
  - detection/conflict-detection.md §规则 1（8 文件清单的检测逻辑）
  - QUICK-REFERENCE.md §版本号文件清单（权威清单来源）
  - CONSTRAINTS.md 约束 #14（交叉验证要求）
  - repair/repair-patterns.md §模式 5（版本号批量更新修复模式）
```

### 场景 6: 阶段 0 时序合规检测（🆕 v2.0）

```yaml
触发时机:
  - 🔴 每次会话初始化时（阶段 0 执行后自检）
  - 修改 task-memory.md 或 00-pre-check/README.md 后
  - 执行规范健康检查时
  - 用户指出阶段 0 执行偏差时

检测内容:
  1. 阶段 0 时序强制规则是否在两处文件中都存在:
     - core/workflows/common/task-memory.md §阶段 0
     - core/workflows/00-pre-check/README.md §阶段 0
  2. 两处的时序规则内容是否一致
  3. 阶段 0 自检清单首项是否为时序检查:
     "🔴 记忆写入是否在分析用户问题之前完成？（不是分析完才补写）"
  4. AI 实际执行顺序是否符合时序规则:
     ✅ 正确: 预检查 → 写入记忆(§会话NN) → 输出"📝 记忆已更新" → 开始分析
     ❌ 错误: 预检查 → 分析问题 → 输出结论 → 补写记忆

执行方式: 🔴 自动执行，发现违规立即纠正

检查要点:
  - task-memory.md 是阶段 0 时序规则的权威来源
  - 00-pre-check/README.md 应与其保持完全同步
  - 如两处不一致 → 以 task-memory.md 为准同步

示例:
  AI 收到用户消息后:
    Step 1: 执行预检查 5 项 ✅
    Step 2: 写入记忆 §会话 NN ✅
    Step 3: 输出 "📝 记忆已更新" ✅
    Step 4: 自检 → 记忆写入在分析之前完成 ✅
    Step 5: 开始分析用户问题

  ❌ 违规示例（已发生事故：§会话05）:
    Step 1: 执行预检查 5 项 ✅
    Step 2: 开始分析用户问题 ❌（跳过了记忆写入）
    Step 3: 输出分析结论
    Step 4: 补写记忆 ❌（延迟写入）
    结果: 记忆写入可能被遗忘或时间线混乱

根因背景:
  此场景源于 §会话05 的阶段 0 时序违规事故 —— AI 先分析问题再补写记忆，
  违反了消息驱动 5 阶段的时序要求。修复后在 task-memory.md 和 00-pre-check
  两处都加入了 🔴 NO EXCEPTIONS 时序强制规则。本自动触发场景确保:
  1) 规则定义不会在后续修改中产生偏差
  2) AI 实际执行时能即时自检

关联:
  - detection/conflict-detection.md §规则 5（流程时序合规性检测逻辑）
  - detection/obsolete-detection.md §规则 5（流程时序过时检测）
  - repair/repair-patterns.md §模式 9（时序违规修复模式）
  - core/workflows/common/task-memory.md §阶段 0 时序强制规则
  - core/workflows/00-pre-check/README.md §阶段 0 时序强制规则
```

### 场景 7: 反复问题自动升级（🆕 v2.0）

```yaml
触发时机:
  - 同一类问题出现 2 次及以上时
  - 用户明确指出"又出现了"、"重复了"、"每次都…"

检测内容:
  1. 回顾 core/self-fix/records/ 目录下的修复记录
  2. 提取问题类型标签（版本号不一致、时序违规、约束遗漏...）
  3. 统计同类问题出现次数
  4. 如果同类问题 ≥ 2 次 → 自动触发升级分析

升级分析流程:
  1. 执行 5-Why 根因分析（深入到根因，不停留在表面）
  2. 评估之前的修复是否真正生效:
     a. 修复是否只写入了"记录"而没有嵌入"执行流程"？
     b. 修复是否只改了一处而遗漏了其他引用处？
     c. 修复是否只是"建议"而没有变成"强制规则"？
  3. 制定防复现方案:
     a. 被动措施 → 升级为主动措施（如: 被动版本检查 → 主动全量检查）
     b. 建议性规则 → 升级为 🔴 NO EXCEPTIONS 强制规则
     c. 单点修复 → 扩展为全链路修复（检测 + 触发 + 修复 + 验证）
  4. 🔴 确认修复已回写到 spec-self-fix 知识库本身:
     - detection/ 模块是否更新了检测规则？
     - triggers/ 模块是否新增了自动触发场景？
     - repair/ 模块是否新增了修复模式？
     - records/summary.md 是否更新了统计？

执行方式: 半自动（分析自动执行，修复方案需用户确认）

升级等级:
  | 出现次数 | 升级等级 | 动作 |
  |:--------:|:-------:|------|
  | 1 次     | 🟢 常规 | 单点修复 + 记录 |
  | 2 次     | 🟡 关注 | 5-Why 分析 + 加强检测规则 |
  | 3 次+    | 🔴 严重 | 全链路修复 + 强制规则 + 知识库回写 + 自动化工具建设 |

示例:
  版本号遗漏:
    第 1 次 (§会话03): BUG-023~027 → 单点修复 → 🟢
    第 2 次 (§会话04): BUG-028~029 → 修复 + 关注 → 🟡
    第 3 次 (§会话05): BUG-030~031 → 触发升级 → 🔴
      → 5-Why 分析 → 根因: 缺少明确文件清单 + 只有被动检查
      → 全链路修复:
        ✅ QUICK-REFERENCE 新增 8 文件清单 + 主动全量检查
        ✅ CONSTRAINTS #14 新增版本号全量同步行
        ✅ detection/conflict-detection.md 嵌入 8 文件清单
        ✅ triggers/auto-triggers.md 新增场景 5
        ✅ repair/repair-patterns.md 模式 5 升级
        ✅ records/summary.md 更新统计
      → 建议: P1 建设 core/tools/bump-version.js 自动化脚本

🔴 关键断层检测（防止"修了但没真修"）:
  修复完成后，必须逐项确认以下检查清单:
    □ 修复是否写入了执行流程入口文件？（QUICK-REFERENCE / CONSTRAINTS / task-memory / 00-pre-check）
    □ 修复是否回写到 spec-self-fix 知识库？（detection / triggers / repair）
    □ 修复是否有对应的检测规则？（能自动发现同类问题）
    □ 修复是否有对应的自动触发场景？（能在正确时机触发检测）
    □ 修复是否有对应的修复模式？（能指导如何修复）
    □ 修复记录是否更新到 records/summary.md？

  如果任一项为"否" → 修复不完整，存在复现风险

根因背景:
  此场景源于 §会话06 用户指出的核心问题 —— 上次修复只写入了执行流程入口
  （QUICK-REFERENCE / CONSTRAINTS / task-memory），但 spec-self-fix 知识库本身
  没有同步更新，导致自我修复机制的检测/触发/修复模块仍停留在 v1.0。
  这意味着未来的健康检查不会使用最新的检测规则，修复可能被遗忘、复现。

关联:
  - core/self-fix/records/summary.md（修复历史统计）
  - triggers/user-intent-detection.md §模式 5（用户触发的反复问题识别）
```

```yaml
  3. 任一文件数值不一致 → 🔴 P0 问题，立即修复

  此场景源于约束 #20 新增后只更新了 CONSTRAINTS.md 和 QUICK-REFERENCE.md，

关联:
  - QUICK-REFERENCE.md §约束条数引用清单（权威清单来源）
  - detection/conflict-detection.md §规则 6（约束条数一致性检测逻辑）
  - CONSTRAINTS.md 约束 #14（交叉验证要求）
```

### 场景 9: 预检查缺少记忆写入确认行检测（🆕 v2.2）

```yaml
触发时机:
  - 🔴 每次会话初始化时（阶段 0 执行后自检 — 与场景 6 联动）
  - 修改 00-pre-check/README.md、QUICK-REFERENCE.md、memory-and-rules.md、copilot-instructions.md 后
  - 执行规范健康检查时
  - 用户指出阶段 0 记忆写入被跳过时

检测内容:
  1. 预检查第 6 行（📝 记忆已创建）是否在以下 4 处文件中都存在且格式一致:
     a. core/workflows/00-pre-check/README.md §检查清单 + §输出格式（标准/扩展）
     b. QUICK-REFERENCE.md §预检查模板
     c. core/workflows/00-pre-check/memory-and-rules.md §阶段 0 输出
     d. .github/copilot-instructions.md 预检查描述（包含 "第 6 行阶段0 记忆写入硬性阻塞"）
  2. 第 6 行格式是否统一为:
     "6. 📝 记忆已创建: .ai-memory/clients/<agent>/tasks/YYYYMMDD.md §会话NN (🔄)"
  3. 第 6 行是否标注为 "🔴 阶段0 硬性阻塞"（防止被误认为可选项）
  4. task-memory.md §阶段 0 自检清单首项是否包含预检查第 6 行输出检查
  5. AI 实际输出的预检查结果是否包含第 6 行（运行时自检）

执行方式: 🔴 自动执行，发现缺失立即补齐

示例:
  AI 输出预检查:
    1. 工作区: E:\MySelf ✅
    2. 任务类型: 深度分析 ✅
    3. 输出位置: projects/vext/reports/analysis/zed-copilot/20260228/ ✅
    4. Agent: zed-copilot ✅
    5. 上次记忆: ⚠️ 无 ✅
    6. 📝 记忆已创建: .ai-memory/clients/zed-copilot/tasks/20260228.md §会话01 (🔄) ✅
    → 阶段 0 完成，可以开始分析用户问题

  ❌ 违规示例:
    1. 工作区: E:\MySelf ✅
    2. 任务类型: 深度分析 ✅
    3. 输出位置: projects/vext/... ✅
    4. Agent: zed-copilot ✅
    5. 上次记忆: ⚠️ 无 ✅
    （缺少第 6 行 — 直接开始分析）
    → 🔴 预检查未完成，阶段 0 记忆写入被跳过

  ❌ 规范定义缺失示例:
    QUICK-REFERENCE.md §预检查模板只有 5 行（无第 6 行）
    → 🔴 硬性阻塞定义被移除，阶段 0 时序保护失效
    → 触发修复: repair/repair-patterns.md §模式 11

根因背景:
  此场景源于 2 次阶段 0 时序违规事故（§会话05 + vscode-copilot §会话01）。
  AI 在预检查 5 项输出后认为"预检查完成了"，直接切换到任务执行模式，
  跳过记忆写入。修复方案为将记忆写入嵌入预检查输出的第 6 行（soft gate → hard gate 升级）。
  本触发场景确保:
    1) 第 6 行定义在 4 处文件中一致存在（定义层面）
    2) AI 实际输出的预检查包含第 6 行（执行层面）
  与场景 6（时序合规检测）互为补充：场景 6 检测时序规则定义一致性，
  场景 9 检测硬性阻塞机制（第 6 行）的存在性。

关联:
```
