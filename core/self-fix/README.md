## 🎯 机制概述

规范自我修复机制是一套完整的系统，让 AI 能够：

1. **自动检测**规范问题（冲突、过时、冗余、缺失、时序违规）
2. **智能修复**问题（自动/半自动/辅助）
3. **记录追溯**所有修复历史
4. **响应用户**反馈并改进规范
5. **升级驱动**反复出现的同类问题自动升级修复力度（🆕 v2.3）
6. **闭环验证**修复后验证知识库回写完整性，防止断层（🆕 v2.3）
7. **规范补充**响应用户指出的规范缺失/不合理，主动审查并补充（🆕 v2.4）

## 🚀 快速开始

### 场景 1: 用户指出规范问题

```yaml
用户说: "我明明说的是添加限流，你怎么做成了防火墙？"

AI 自动:
  1. 识别用户意图：指出理解偏差
  2. 检查相关规范：core/workflows/01-requirement-dev/README.md
  3. 分析根因：需求理解步骤不够清晰
  4. 生成修复建议
  5. 询问用户是否修复规范

详见: triggers/user-intent-detection.md
```

### 场景 2: 用户抱怨违反规范

```yaml
用户说: "你为什么不按规范来执行预检查？"

AI 自动:
  1. 识别关键词："不按规范"
  2. 定位相关规范：core/workflows/00-pre-check/README.md
  3. 检查是否遵守了规范
  4. 如违反：分析原因，修复规范表述
  5. 如遵守：规范表述不清晰，增强说明

详见: triggers/user-intent-detection.md
```

### 场景 3: 定期健康检查

```yaml
用户说: "运行规范健康检查"

AI 执行:
  1. 遍历所有规范文件
  2. 执行 4 类检测
  3. 生成问题报告
  4. 批量修复或逐个确认

详见: detection/ 目录
```

## 🎯 触发机制（核心）

### 1. 用户意图触发 ⭐⭐⭐⭐⭐ 最重要

**文件**: `triggers/user-intent-detection.md`

用户通过**抱怨/纠正/质疑/建议**来触发规范检查：

```yaml
触发模式:
  - "我明明说的是 xxx，你怎么..."（模式 1~3）
  - "你为什么不按规范..."（模式 2）
  - "这不符合 xxx 要求"（模式 2）
  - 用户重复同一个需求（模式 3）
  - "规范中没有定义..."（模式 6 — 🆕 v2.4）
  - "这个规范不合理"（模式 6 — 🆕 v2.4）
  - "需要新增一条规范"（模式 6 — 🆕 v2.4）

AI 响应:
  1. 识别用户意图
  2. 定位相关规范
  3. 检查是否遵守/规范是否清晰/规范是否缺失
  4. 提出修复/补充建议
  5. 记录到 records/
```

### 2. 自动触发

**文件**: `triggers/auto-triggers.md`

AI 在执行任务时自动检测：

```yaml
触发场景:
  - 发现规范冲突
  - 任务无法执行（规范问题导致）
  - 链接失效
  - 预检查阶段
```

### 3. 手动触发

用户主动要求：

```yaml
触发指令:
  - "运行规范健康检查"
  - "检查规范是否有问题"
  - "验证规范一致性"
```

## 🆘 常见问题

**Q: 什么时候触发规范检查？**
A: 三种情况：1) 用户抱怨/纠正 2) AI 发现冲突 3) 定期检查

**Q: 会自动修改规范吗？**
A: 安全修复（如补充链接）可自动，重要修复需用户确认

**Q: 修复记录有什么用？**
A: 追溯历史，分析规范演进，避免重复问题

**Q: 如何防止修复后问题再次复现？**
A: v2.3 新增了"修复回写完整性验证"（6 项检查点），确保修复同时写入执行流程入口和 spec-self-fix 知识库。同时，反复问题自动升级机制（场景 7 / 模式 5）会在同类问题 ≥2 次时触发升级分析。

**Q: 什么是知识库断层？**
A: 当修复只写入了执行流程入口文件（QUICK-REFERENCE / CONSTRAINTS / task-memory），但未回写到 spec-self-fix 知识库（detection / triggers / repair）时，就产生了断层。未来的健康检查和自动检测将无法使用最新规则，导致问题可能复现。模式 10 专门解决此问题。
