# 工作流共用框架

> 所有工作流的共用结构和步骤间数据传递规范

---

## 🎯 工作流通用结构

所有工作流都遵循以下基本结构：

```
预检查 → 阶段 1 → 确认点 → 阶段 2 → 确认点 → ... → 完成
```

### 通用阶段

| 阶段 | 名称 | 输入 | 输出 | 确认点 |
|-----|------|------|------|--------|
| 0 | 预检查 | 用户请求 | 任务上下文 | - |
| 1 | 理解/分析 | 用户需求 | 需求摘要 | CP1 |
| 2 | 设计/方案 | 需求摘要 | 技术方案 | CP2 |
| 3 | 执行/实现 | 技术方案 | 代码变更 | CP3 |
| 4 | 验证/测试 | 代码变更 | 测试结果 | CP4 |
| 5 | 文档/交付 | 测试结果 | 文档输出 | CP5 |

---

## 📋 步骤间数据传递

### 数据传递规范

每个阶段完成后，输出结构化的"阶段摘要"供下一阶段使用：

```yaml
# 阶段摘要格式
阶段: <阶段名称>
状态: 完成/待确认
输出:
  核心结论: <一句话总结>
  关键数据:
    - <数据项1>
    - <数据项2>
  下一步: <下一阶段需要做什么>
```

### 各阶段输出示例

#### 阶段 1 输出（需求理解）

```yaml
阶段: 需求理解
状态: 待确认
输出:
  核心需求: 在 user-service 添加 API 限流功能
  关键要求:
    - 限流规则: 100次/分钟/IP
    - 存储: Redis
    - 响应: 429 + Retry-After
  涉及模块:
    - middleware/
    - config/
  下一步: 设计技术方案
```

#### 阶段 2 输出（技术方案）

```yaml
阶段: 技术方案
状态: 待确认
输出:
  方案概述: 使用 sliding-window 算法 + Redis 存储
  修改文件:
    - src/middleware/rate-limit.ts (新增)
    - src/config/index.ts (修改)
    - src/app.ts (修改)
  新增依赖:
    - ioredis@^5.0.0
  下一步: 实现代码
```

#### 阶段 3 输出（代码实现）

```yaml
阶段: 代码实现
状态: 待确认
输出:
  变更统计:
    新增: 2 文件
    修改: 3 文件
    删除: 0 文件
  关键实现:
    - rate-limit.ts: 限流中间件核心逻辑
    - config: 添加 Redis 配置项
  下一步: 执行测试
```

## 🔄 统一执行流程（7 阶段）

> 🔴 v2.13.0 起移除快速/完整双模式（FIX-016），所有需求开发统一为 7 阶段流程。

```
预检查 → 理解需求 → CP1 → 技术方案 → CP2 → 实施方案 → CP3 → 执行 → 报告 → 归档文档 → 验证检查
```

- 核心 3 个确认点（CP1/CP2/CP3）强制
- CP4（测试）、CP5（文档）按需
- 所有需求均须生成归档文档文件（小需求内容可精简，但文件必须存在）
- 模板选择：小需求（< 5 文件）用 `lite/`，大需求（>= 5 文件）用 `core/`

## 📝 工作流专属验证

不同工作流有专属的验证清单：

### 需求开发验证

```yaml
必查项:
  ✅ 需求点覆盖: 所有需求有对应实现
  ✅ 接口契约: 入参/出参/错误码定义完整
  ✅ 配置完整: 新增配置项有默认值和说明
  ✅ 文档同步: README/CHANGELOG 已更新
```

### Bug 修复验证

```yaml
必查项:
  ✅ 问题复现: 修复前能复现问题
  ✅ 修复有效: 修复后问题不再出现
  ✅ 无副作用: 相关功能未受影响
  ✅ 回归测试: 关键路径测试通过
```

### 性能优化验证

```yaml
必查项:
  ✅ 基线数据: 有优化前的性能数据
  ✅ 对比数据: 有优化后的性能数据
  ✅ 提升幅度: 达到预期目标
  ✅ 稳定性: 多次测试结果一致
```
