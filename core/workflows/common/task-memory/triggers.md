# 触发时机（v1.9 消息驱动 5+1 阶段）

**最后更新**: 2026-03-02

> 本文件从 [task-memory.md](../task-memory.md) 拆分，定义记忆写入的触发时机与各阶段详细规则。

---

## ⚡ 核心原则

> **原则 1 — 消息驱动（🆕 v1.6）**: 触发器以**对话中的消息事件**为锚点
> （用户发消息 / AI 回复 / AI 执行完毕），而非 AI 内部工作流阶段。
>
> **原则 2 — 自动输出（v1.5）**: 报告和记忆的写入是 AI 的自动行为，绝不询问用户。
> 每次会话都必须输出报告文件+更新记忆，无一例外。
>
> **原则 3 — 先写后干**: AI 无法感知自身剩余 token，输出中途截断时没有机会执行任何收尾动作。
> 开始执行前先创建初始记忆，过程中在每个消息事件增量更新+写入报告，
> 确保 token 耗尽时至少有最近一次快照可用。

## 5+1 阶段总览（🆕 v1.9）

```yaml
阶段 0:   会话初始化（AI 收到首条消息时 — 预检查 + 🔴 第 6 行记忆写入硬性阻塞）  ← 入口门禁
阶段 1:   用户发消息时（每条有实质内容的用户消息 — 捕获用户输入）
阶段 2:   AI 回复时（每次 AI 输出结果 — 写报告 + 更新记忆）
阶段 3:   AI 执行完毕时（单轮工具调用全部完成 — 记录变更清单）
阶段 4.5: 执行合规性回溯检查（任务结束前 — 🔴 出口门禁，验证流程合规性）  ← 🆕 v1.9
阶段 4:   任务结束时（整个任务完成 — 最终状态更新 ✅）

执行顺序:
  首轮:   0 → 1 → 2+3 →
  后续轮: 1 → 2+3 → 1 → 2+3 → ... →
  结束:   4.5 → 4

典型会话时间线:
  用户消息1 → [阶段0+1] → AI分析执行 → AI回复1 → [阶段2+3]
  用户消息2 → [阶段1]   → AI分析执行 → AI回复2 → [阶段2+3]
  用户确认  →                          → [阶段4.5] → [阶段4]

入口-出口闭环:
  阶段 0（入口门禁）确保会话开始时预检查+记忆写入不被跳过
  阶段 4.5（出口门禁）确保会话结束前流程合规性不被遗忘
  二者形成对称闭环，覆盖 AI 执行的起点和终点

v1.7 → v1.9 的核心变化
  - 🆕 新增阶段 4.5（执行合规性回溯检查 — 出口门禁）
  - 执行顺序结束段从 "→ 4" 变为 "→ 4.5 → 4"
  - 与阶段 0（入口门禁）形成对称闭环
  - 联动自修复场景 10（合规检查未通过时自动触发）

v1.5 → v1.7 的核心变化
  - 新增阶段 0（会话初始化）和阶段 1（用户发消息）
  - 阶段编号与实际执行顺序一致（不再是 1→3→2→4 的混乱顺序）
  - 阶段 3 触发条件从主观的"关键节点"改为客观的"执行完毕"
  - 记忆模板新增 §📨 对话记录（用户消息+AI回复摘要表格）
```

## 阶段 0：会话初始化（🔴 嵌入预检查第 6 行 — 硬性阻塞）

```yaml
触发条件: AI 收到本次会话的第一条用户消息，预检查第 5 项完成后
触发方式: 🔴 自动执行，禁止询问用户
目的: 创建/追加记忆文件 + 作为预检查第 6 行输出（硬性阻塞）

🔴 时序强制规则（NO EXCEPTIONS）:
  阶段 0 的记忆写入已嵌入预检查输出的第 6 行，与预检查 1-5 项合并为不可分割的原子操作。
  第 6 行未输出 = 预检查未完成 = 禁止开始分析用户问题。
  禁止"先分析问题、再补写记忆"— 这会导致记忆写入被遗忘或延迟。
  正确顺序: 预检查 1-5 项 → 写入记忆 → 输出第 6 行"📝 记忆已创建" → 然后才开始分析用户问题
  错误顺序: 预检查 1-5 项 → 分析问题 → 输出结论 → 最后才补写记忆 ❌（已发生 2 次事故：§会话05 + vscode-copilot §会话01）

🔴 硬性阻塞机制（soft gate → hard gate 升级）:
  之前的失败模式: 预检查 5 项输出后 AI 认为"预检查完成了"，切换到任务执行模式，跳过记忆写入
  修复原理: 将记忆写入作为预检查输出的第 6 行，不写第 6 行 = 预检查本身不完整
  等价于: "soft gate（文字规则）" 升级为 "hard gate（输出格式强制）"

🔴 tool call 顺序约束（NO EXCEPTIONS）:
  会话的前三个 tool call 必须是以下顺序，完成前禁止执行任何分析性文件读取:
    1. now()             — 获取真实日期
    2. list_directory()  — 扫描 .ai-memory/clients/<agent>/tasks/
    3. edit_file()       — 创建/追加今日记忆文件
  为什么: 防止"先读目标文件分析问题、再补写记忆"的绕过模式（已发生 3 次事故）

AI 执行:
  1. 执行预检查 5 项必做（见 00-pre-check/README.md）
  2. 确定当前 Agent 标识（预检查第 4 项）
  3. 获取当前真实日期和时间（使用 now() 工具）
  4. 确保 .ai-memory/clients/<agent>/tasks/ 目录存在（不存在则创建）
  5. 确定今天的记忆文件（🆕 v1.7 简化）:
     a. 文件路径 = tasks/YYYYMMDD.md（纯日期，无序号）
     b. 必须使用 list_directory 确认目录内容（禁止 find_path / glob）
        ⚠️ glob 引擎默认跳过 "." 开头的隐藏目录（如 .ai-memory/）
     c. 文件已存在 → 读取文件，统计已有 ## 会话 NN 的最大 NN
     d. 文件不存在 → 创建新文件（写入头部信息）
     e. 💡 预检查 Step 5 已完成此扫描，直接复用其结果即可
  6. 追加新会话段落:
     - 文件已存在 → 追加 ## 会话 NN+1（NN 为已有最大值）
     - 文件新建 → 写入 ## 会话 01
     - 状态标记为 🔄 进行中
  7. 写入已知信息:
     - 任务目标（来自用户首条消息）
     - 📨 对话记录表格 → 记录首条用户消息摘要（轮次 1, 👤→, 含关联引用）
  8. 更新 .ai-memory/clients/<agent>/SUMMARY.md → 追加到最近任务（状态 🔄）
     🔴 此步骤不可跳过：SUMMARY.md 是跨会话恢复的入口索引，
        缺失会导致后续会话无法发现本次任务
  9. 更新 .ai-memory/SUMMARY.md（全局摘要）→ 追加关键决策记录（如有）
  10. 🔴 输出预检查第 6 行（硬性阻塞确认）:
      "6. 📝 记忆已创建: .ai-memory/clients/<agent>/tasks/YYYYMMDD.md §会话NN (🔄)"
      此行未输出 = 阶段 0 未完成 = 禁止进入后续阶段

记忆文件命名（🆕 v1.7）:
  <YYYYMMDD>.md
    YYYYMMDD : 8 位日期（强制，无分隔符）

  示例:
    20260226.md   # 2026-02-26 的所有会话（内含 ## 会话 01 ~ ## 会话 N）
    20260227.md   # 2026-02-27 的所有会话

输出（嵌入预检查第 6 行）:
  "6. 📝 记忆已创建: .ai-memory/clients/<agent>/tasks/YYYYMMDD.md §会话NN (🔄)"

🔴 阶段 0 自检清单（创建/追加记忆后逐项确认）:
  - [ ] 🔴 预检查第 6 行是否已输出？（"6. 📝 记忆已创建: ..."）— 未输出 = 预检查未完成
  - [ ] 🔴 记忆写入是否在分析用户问题之前完成？（不是分析完才补写）
  - [ ] 使用了 list_directory（非 find_path/glob）确认 tasks/ 目录？
  - [ ] 记忆文件名为纯日期格式 YYYYMMDD.md（非旧格式 YYYYMMDD-NN-TYPE-id.md）？
  - [ ] 会话编号基于文件内已有 ## 会话 NN 的 max+1（新文件为 01）？
  - [ ] 当前会话状态为 🔄（非 ✅）？
  - [ ] 📨 对话记录已记录用户首条消息摘要（含关联引用列）？
  - [ ] clients/<agent>/SUMMARY.md 已追加本次任务条目？
  - [ ] .ai-memory/SUMMARY.md（全局）已更新？
  如果任一项为"否" → 立即补齐，不得继续后续阶段

本质: 合并了 v1.5 的"预检查"和"阶段 1（任务开始时）"
与 v1.6 区别: 不再需要计算 NN 序号，直接用日期定位文件；追加模式而非创建新文件
```

## 阶段 1：用户发消息时（捕获用户输入）🆕 v1.7

```yaml
触发条件: 用户发送包含新意图/新需求/方向修正的消息
触发方式: 🔴 自动执行，禁止询问用户
目的: 第一时间捕获用户输入，防止追加需求丢失

AI 执行:
  1. 记录用户消息的核心意图到 📨 对话记录（1 行摘要 + 关联引用，不是全文复制）
  2. 如果用户修正了方向 → 更新记忆中的 🎯 任务摘要
  3. 如果用户补充了新需求 → 追加到记忆的 ⚠️ 待跟进

写入内容:
  - 📨 对话记录追加: | N | 👤→ | [用户消息摘要] |
  - 如有方向修正 → 更新 🎯 任务摘要
  - 如有新需求 → 追加到 ⚠️ 待跟进

⚠️ 重要: 这一步发生在 AI 开始分析之前
  - 极轻量 — 只记录摘要，不做分析，预估消耗 ~100 token
  - 确保即使 AI 在后续分析中 token 耗尽，用户说了什么至少有记录
  - 首条消息的记录已在阶段 0 完成，阶段 1 主要覆盖后续追加消息

🔴 过滤规则 — 以下消息不触发阶段 1:
  - 纯确认性消息: "好的" / "Y" / "确认" / "可以"
  - 流程控制: "继续" / "接着" / "下一步"
  - 情绪表达: "好的谢谢" / "明白了"
  判断标准: 用户消息包含新的意图/需求/约束/方向修正时才写入

为什么需要这个阶段（v1.5 缺失的问题）:
  - v1.5 中用户消息不是触发条件，追加需求没有记忆写入点
  - 用户消息是最有价值的原始输入，不应等 AI 理解后才记录转述版
  - 多轮对话中用户可能多次修正方向，每次修正都应被捕获
```

## 阶段 2：AI 回复时（写报告 + 更新记忆）

```yaml
触发条件: AI 准备向用户输出分析结果/回复
触发方式: 🔴 自动执行，禁止询问用户
目的: 记录 AI 的分析结论 + 写入报告

AI 执行:
  1. 确定报告类型和目标子目录（analysis/ / bugs/ / diagnostics/ 等）
  2. 确定目标目录: reports/<子目录>/<agent>/YYYYMMDD/（如不存在则创建）
  3. 扫描该目录，确定序号 NN（max+1 或 01）
  4. 写入报告文件到 reports/<子目录>/<agent>/YYYYMMDD/NN-<类型>-<简述>.md
     报告包含完整分析内容（规范详见 temp-reports.md）
  5. 更新记忆文件:
     - 📄 关联报告 → 追加报告链接
     - 📨 对话记录 → 追加: | N | 🤖← | [AI 回复摘要 + 报告路径] |
  6. 对话中只输出结论摘要 + 报告文件路径

对话输出格式:
  AI: "[结论摘要 — 2-5 句话]

       📄 完整报告: reports/<子目录>/<agent>/YYYYMMDD/NN-<类型>-<简述>.md"

与 v1.5 区别:
  - 触发器从"分析完成"改为"AI 回复"（可观测事件，更清晰）
  - 新增 📨 对话记录追加
```

## 阶段 3：AI 执行完毕时（记录变更清单）

```yaml
触发条件: AI 在一轮消息中的所有工具调用/文件操作完成
触发方式: 🔴 自动执行，禁止询问用户
目的: 记录实际执行的变更

AI 执行:
  1. 更新记忆文件 → 追加本轮实际修改的文件列表和关键决策
  2. 🔴 实时更新报告文件中已完成的问题项状态（🆕 v1.8）:
     - 报告中列出的待修复问题/待办项，每完成一个立即在报告中标记 ✅
     - 不要等全部完成再批量更新 — 中途中断时报告本身就是进度的实时快照
     - 示例: 报告列出 5 个 Bug，修完第 3 个后报告中 Bug 1~3 已标 ✅，Bug 4~5 仍为待办
  3. 状态保持 🔄 进行中

写入内容:
  - 💡 关键决策 → 追加本轮决策
  - 如修改了文件 → 在 ⚠️ 待跟进 中更新已完成/新增项
  - 🔴 如本轮完成了报告中的某个问题项 → 立即更新报告对应条目（🆕 v1.8）

注意: 增量更新应简洁高效，不要花大量 token 写冗长记录

与 v1.5 的区别:
  - 触发器从主观的"关键节点"改为客观的"执行完毕"
  - 不再需要判断"是否重要" — 每次有工具调用的轮次结束后都更新
  - 如果 AI 纯文字回复（无工具调用），阶段 3 不触发，只触发阶段 2

⚠️ 实操简化: 阶段 2 和阶段 3 通常在同一时刻写入
  （AI 准备回复用户时，同时完成写报告 + 记录变更）
  逻辑上保持两个概念的区分（报告/结论 vs 变更清单/决策），
  但实际可以合并为一次记忆文件更新操作。
```

## 阶段 4.5：执行合规性回溯检查（🆕 v1.9 — 出口门禁）

```yaml
触发条件: AI 准备进入阶段 4（任务结束）之前，自动执行
触发方式: 🔴 自动执行，禁止询问用户
目的: 回溯检查本次会话是否按流程执行，与阶段 0（入口门禁）形成闭环

🔴 核心检查清单（3 项必做 + 2 项推荐）:

  必做检查:
    1. 预检查完整性:
       - [ ] 第 6 行"📝 记忆已创建"是否已输出？
       - [ ] 记忆文件（.ai-memory/clients/<agent>/tasks/YYYYMMDD.md）是否已创建/追加？
       - [ ] 记忆写入是否在分析问题之前完成？（时序合规）

    2. 确认点遵守:
       - [ ] 涉及文件修改时，是否按 CP1→CP2→CP3 顺序执行？
       - [ ] 是否存在跳过确认点直接执行的情况？
       - [ ] 纯分析任务（无文件修改）可跳过此项

    3. 记忆/报告写入:
       - [ ] 本会话是否已写入至少一份报告文件？
       - [ ] 记忆文件是否包含报告链接（📄 关联报告）？
       - [ ] 记忆文件是否包含对话记录（📨 对话记录）？

  推荐检查:
    4. 任务类型映射:
       - [ ] 是否使用了正确的工作流？（如需求开发用 01-requirement-dev）
       - [ ] 是否存在混合意图误判？

    5. 变更计划确认:
       - [ ] 涉及文件修改/删除时，是否先输出变更计划并获得用户确认？
       - [ ] 报告/记忆的自动写入除外（无需确认）

AI 执行:
  1. 逐项检查上述清单
  2. 全部通过 → 输出 "✅ 阶段 4.5 合规检查通过" → 进入阶段 4
  3. 存在未通过项 → 分类处理:
     a. 可立即修复（如记忆未写入、报告未生成）→ 立即修复 → 重新检查
     b. 无法修复（如确认点已跳过）→ 记录到记忆文件 ⚠️ 待跟进 + 触发自修复场景 10
  4. 🔴 所有可修复项修复完成后，才进入阶段 4

输出格式:
  "✅ 阶段 4.5 合规检查:
    - 预检查完整性: ✅ 第 6 行已输出，记忆已创建
    - 确认点遵守: ✅ CP1→CP2→CP3 按序执行（或 N/A 无文件修改）
    - 记忆/报告写入: ✅ 报告 NN-xxx.md 已写入，记忆已含链接+对话记录
    - 任务类型映射: ✅ 正确（推荐）
    - 变更计划确认: ✅ 已确认（推荐）"

违规时输出格式:
  "⚠️ 阶段 4.5 合规检查发现问题:
    - 记忆/报告写入: ❌ 报告未生成 → 立即修复中...
    - [修复完成后] ✅ 已修复，报告 NN-xxx.md 已写入"

与阶段 0 的对称关系:
  阶段 0（入口）: 确保"先写记忆后执行"— 防止开始时遗漏
  阶段 4.5（出口）: 确保"执行完检查合规"— 防止结束时遗漏
  二者共同构成流程合规的完整闭环

联动自修复:
  合规检查未通过且无法立即修复时 → 触发 auto-triggers.md 场景 10（执行合规性回溯检查联动）
  场景 10 会记录违规到 records/，并在同类违规 ≥ 2 次时触发场景 7（反复问题升级）

根因背景:
  此阶段源于阶段 0 时序违规已发生 3 次（§会话05 + vscode-copilot §会话01 + 第 3 次），
  根因是流程只有入口检查（阶段 0）而无出口检查，AI 跳过步骤后无人发现。
  阶段 4.5 作为出口门禁，即使入口被跳过，出口也能发现并记录/修复。
```

## 阶段 4：任务结束时（最终状态更新）

```yaml
触发条件: 用户确认任务完成 / AI 判断任务已完成（🔴 前置条件：阶段 4.5 已通过）
触发方式: 🔴 自动执行，禁止询问用户
目的: 最终状态更新

AI 执行:
  1. 执行完成验证（见下方 §任务完成验证）
  2. 🔴 更新关联报告文件头部状态 → 📝 进行中 改为 ✅ 已完成（🆕 v1.8）
     - 确认报告中所有问题项/待办项已在阶段 3 逐个标记完成
     - 如仍有未完成项 → 在报告中明确标注遗留项，状态保持 📝 进行中
  3. 最终更新记忆文件 → 状态改为 ✅ 完成
  4. 更新 clients/<agent>/SUMMARY.md → 更新状态 + 补充"关键决策"
  5. 同步关键决策到全局 .ai-memory/SUMMARY.md
  6. 更新 TASK-INDEX.md → 追加任务记录行（已有机制）

与 v1.8 区别: 新增前置条件（阶段 4.5 必须已通过）（🆕 v1.9）
与 v1.7 区别: 新增报告头部状态更新（步骤 2）
```

## 🔴 任务完成验证（声称完成前必须执行）

```yaml
触发条件: AI 准备声称"任务完成"之前

验证清单:
  1. 输出文件存在性:
     - [ ] 所有声称创建的文件确实存在？（用工具验证，不要假设）
     - [ ] 文件路径正确？（在预期的目录下）
     - [ ] 文件名符合命名规范？（NN-<类型>-<简述>.md，存放于 <agent>/YYYYMMDD/ 目录）

  2. 报告文件完整性:
     - [ ] 本次会话已输出至少一个报告文件？
     - [ ] 报告头部包含所有必填字段？（日期、Agent、状态）
     - [ ] 报告头部状态已更新？（📝 进行中 → ✅ 已完成，或遗留项已标注）（🆕 v1.8）
     - [ ] 报告中各问题项/待办项状态已逐个更新？（🆕 v1.8）
     - [ ] 日期字段是真实日期，非 YYYY-MM-DD 占位符？
     - [ ] 报告有完整的结构（标题→正文→结论）？
     - [ ] 没有中途截断的内容？

  3. 记忆文件完整性:
     - [ ] .ai-memory 中的任务记忆已更新？
     - [ ] 记忆文件包含报告链接？
     - [ ] SUMMARY.md 已追加/更新对应条目？

  4. 关联文件一致性（如修改了规范文件）:
     - [ ] 已执行交叉验证清单？
     - [ ] 所有引用处已同步更新？

输出格式:
  AI: "✅ 完成验证:
        - 报告文件: reports/analysis/20260227-01-xxx.md（已确认存在）
        - 记忆文件: 已更新，包含报告链接
        - SUMMARY: 已同步
        - 关联检查: 已同步 2 个文件"

🔴 未通过验证项必须修复后才能声称完成。
```

## ⚠️ 新会话继续任务（含 Token 耗尽恢复）

> 当用户在新会话中表达"继续"意图时，预检查流程自动识别并进入恢复模式。
> 完整恢复流程定义在 `core/workflows/00-pre-check/README.md §继续任务`。

```yaml
触发语句（任一匹配即触发）:
  - "继续" / "接着" / "继续上次任务"
  - "恢复记忆" / "恢复上下文"
  - "上次卡住了" / "上次断了"
  - "token 不够了继续" / "刚才没完成"

恢复流程摘要:
  1. 扫描 projects/*/.ai-memory/clients/<agent>/SUMMARY.md → 找状态 🔄 的任务
  2. 0个=没有未完成任务；1个=直接恢复；多个=让用户选
  3. 读取 clients/<agent>/tasks/<id>.md → 提取目标、报告链接、待办
  4. 读取 📨 对话记录 → 快速回忆对话脉络（🆕 v1.6）
  5. 读取关联的报告文件 → 恢复完整分析上下文
  6. 向用户输出摘要并确认
  7. 从断点继续执行，同时更新记忆文件

因为采用了"报告为主体+消息驱动+先写后干"策略，即使 token 耗尽，也有:
  - 📨 对话记录中的用户消息摘要（阶段 0+1 写入的）  ← 🆕 v1.6
  - 记忆文件中的任务目标和报告链接（阶段 0 写入的）
  - 报告文件中的完整分析内容（阶段 2 写入的）
  - 已更新的变更和决策（阶段 3 写入的）
```
