## 📋 SUMMARY.md 模板

```markdown
# AI 任务记忆 - [项目名]

> **最后更新**: YYYY-MM-DD HH:MM ← 🔴 必须替换为真实日期时间
> **总任务数**: N

## 🔑 关键决策记录

| 时间 | Agent | 决策 | 原因 | 影响范围 |
|------|-------|------|------|---------|
| YYYY-MM-DD HH:MM | zed-copilot | [决策内容] | [原因] | [文件/模块] |

## 📋 最近任务（最新 10 条）

| 时间 | Agent | 类型 | 标题 | 关键变更 | 详情 |
|------|-------|------|------|---------|------|
| YYYY-MM-DD HH:MM | zed-copilot | REQ | [标题] | [一句话总结] | [tasks/xxx.md](./tasks/xxx.md) |
| YYYY-MM-DD HH:MM | webstorm-copilot | ANALYSIS | [标题] | [一句话总结] | [tasks/xxx.md](./tasks/xxx.md) |

## ⚠️ 未完成 / 待跟进

| 时间 | Agent | 任务 | 待办 | 原因 |
|------|-------|------|------|------|
| YYYY-MM-DD HH:MM | zed-copilot | [任务名] | [待办事项] | [为什么没完成] |
```

## 📋 每日记忆模板（🆕 v1.7）

> **v1.7 核心变化**: 记忆从"每会话一文件"改为"每天一文件"。
> 文件命名 `YYYYMMDD.md`，不再使用 NN 序号。
> 每个新会话以 `## 会话 NN` 分段追加到同一天的文件中。
> 对话记录新增"关联引用"列（4 列表格）。
> 完整分析内容仍在报告文件中。

```markdown
# YYYY-MM-DD 任务记忆

> **项目**: [项目名]
> **Agent**: [zed-copilot / webstorm-copilot / cursor / ...] ← 🔴 必须标识
> **会话数**: N ← 每新增会话时递增
> **最后更新**: YYYY-MM-DD HH:MM ← 🔴 必须替换为真实日期时间

---

## 会话 01: [标题]（HH:MM）

- **类型**: 需求开发 / Bug修复 / 优化 / 深度分析 / ...
- **状态**: ✅ 完成 / 🔄 进行中

**任务摘要**: [用 2-3 句话描述这次任务做了什么，为什么做，结果如何]

| 轮次 | 方向 | 摘要 | 关联引用 |
|:----:|:----:|------|----------|
| 1 | 👤→ | [用户首条消息的核心意图，1 行] | 🧠 YYYYMMDD.md, 📋 profile/README.md |
| 1 | 🤖← | [AI 回复摘要] | 📄 NN-类型-简述 |
| 2 | 👤→ | [用户追加需求/追问/修正方向] | |
| 2 | 🤖← | [AI 回复摘要] | 📄 NN-类型-简述 |

**关键决策**: [1-2 行，如有]
**待跟进**: [如有未完成事项]

📦 **编码检查点**（仅编码任务且计划变更 ≥ 3 个文件时出现）:

| # | 时间 | 完成项 | 文件 |
|---|------|--------|------|
| CP1 | HH:MM | [任务编号或 N/M 文件完成] | `path/file.ts` (新建/修改) |

---

## 会话 02: [标题]（HH:MM）

...（每个新会话追加一段，格式同上）

---

## 📄 全天关联报告汇总

| 报告文件 | 类型 | 对应会话 |
|----------|------|----------|
| reports/analysis/<agent>/YYYYMMDD/01-analysis-xxx.md | 深度分析 | §会话01 |
| reports/bugs/<agent>/YYYYMMDD/02-bug-xxx.md | Bug分析 | §会话02 |

## ⚠️ 全天待跟进汇总

- [ ] [跨会话的未完成事项]
```

### 📦 编码检查点区块说明（🆕 v2.11.0）

> 编码检查点是编码实施类任务专用的增量进度记录区块，解决多文件变更中途中断后进度丢失问题。
> 详见 `triggers.md §编码检查点`。

```yaml
触发条件: 编码任务 + 计划变更文件数 >= 3

出现位置: 当前会话段落内，⚠️ 待跟进 之后、📨 对话记录 之前

格式:
  ### 📦 编码检查点
  | # | 时间 | 完成项 | 文件 |
  |---|------|--------|------|
  | CP1 | HH:MM | 任务 1.0 schema-adapter | `lib/schema-adapter.ts` (新建) |
  | CP2 | HH:MM | 任务 1.1 config-loader | `lib/config-loader.ts` (新建) |

  旧项目无 IMPLEMENTATION-PLAN 时（兜底向后兼容）:
  | CP1 | HH:MM | 3/7 文件完成 | `types/export.ts` (新建), `services/export-service.ts` (新建), `routes/export.ts` (新建) |

🔴 IMPLEMENTATION-PLAN.md 为强制产物（阶段 3 生成，详见 01-requirement-dev §阶段 3）

写入时机:
  默认（参考 IMPLEMENTATION-PLAN 任务编号）: 每完成一个任务编号 → 追加 1 行
  兜底（旧项目无 PLAN 时向后兼容）: 每完成 3 个文件变更 → 追加 1 行

同步操作（检查点写入时必须一并完成）:
  1. 追加 📦 编码检查点表格行
  2. 更新 ⚠️ 待跟进中对应条目标记 [x]
  3. 更新 IMPLEMENTATION-PLAN.md 中该任务状态 🔲→✅（旧项目无 PLAN 时跳过此步）

不触发的场景:
  - 计划变更文件数 < 3（沿用阶段 3 默认机制）
  - 非编码任务（分析/文档/规范修改等）
```

> 💡 **对话记录表格说明**:
> - 4 列: 轮次 / 方向 / 摘要 / 关联引用
> - 关联引用使用轻量标记: 📋（项目规范）📄（报告）🧠（记忆）📑（规范文件）
> - 纯确认性消息（"好的"/"Y"/"继续"）不记录
> - 摘要列不再写报告完整路径（由关联引用列承担）
> - §📄 全天关联报告汇总 = 汇总视角；对话记录关联引用列 = 时间线视角，互补不重复

### 🔴 文件名命名规范（🆕 v1.7 更新）

```yaml
格式: <YYYYMMDD>.md（在各 Agent 目录内）

🔴 每天一个记忆文件，以纯日期命名（不再使用 NN 序号）
🔴 Agent 由目录隔离保证（clients/<agent>/），不写入文件名
🔴 会话内以 ## 会话 NN 分段追加（NN 为当天第几个会话）

✅ 正确示例（文件位于 clients/zed-copilot/tasks/ 下）:
  20260226.md    # 2026-02-26 的所有会话记忆
  20260227.md    # 2026-02-27 的所有会话记忆

❌ 错误示例:
  20260226-01-ANALYSIS-v3-deep-review.md  # v1.6 旧格式（已废弃）
  20260226-zed-copilot-ANALYSIS-xxx.md    # Agent 不应在文件名中（v1.2 已废弃）
  ANALYSIS-v3-deep-review.md              # 缺少日期

创建 vs 追加:
  - tasks/ 下不存在 YYYYMMDD.md → 创建新文件
  - tasks/ 下已存在 YYYYMMDD.md → 读取已有会话数 → 追加新会话段落（## 会话 NN+1）

向后兼容:
  - 旧格式文件（YYYYMMDD-NN-TYPE-id.md）仍可读取
  - 新会话一律使用新格式（YYYYMMDD.md）
  - 预检查扫描时同时兼容两种格式
```

### 🔴 报告文件序号独立规则（🆕 v1.7）

```yaml
报告存放: reports/<子目录>/<agent>/YYYYMMDD/（🆕 temp-reports v1.6 按 Agent+日期目录隔离）
报告命名: NN-<类型>-<简述>.md（日期由目录层级表达，文件名不再包含 YYYYMMDD-）
NN 来源: 仅扫描 reports/<子目录>/<agent>/YYYYMMDD/ 目录
         与记忆文件完全解耦

❌ 禁止: 报告 NN 和记忆共享同一个全局序号池
         （v1.6 及之前的设计缺陷，已在 v1.7 修正）

报告详细规范见: core/workflows/common/temp-reports.md
```

## 🔍 使用方式

### 预检查时输出上次记忆（v1.4）

> **设计目的**: 用户在每次新会话中立即知道当前对话基于哪条记忆延续，无需手动查找。

```yaml
预检查第 5 项 — "上次记忆":
  触发时机: 预检查阶段（每次必做，非按需）— 属于阶段 0 的一部分

  扫描逻辑:
    1. 确定当前 Agent 标识（预检查第 4 项已获取）
    2. 用 list_directory 逐层进入 projects/*/.ai-memory/clients/<agent>/tasks/ 目录
       🔴 禁止使用 find_path/glob 扫描（glob 引擎默认跳过 . 开头的隐藏目录）
    3. 按文件名日期排序（YYYYMMDD 自然排序即可）
    4. 取最新日期的文件
    5. 读取该文件末尾最新会话段落，提取状态字段（🔄 进行中 / ✅ 完成）

  输出格式:
    找到记忆: "5. 上次记忆: projects/<project>/.ai-memory/clients/<agent>/tasks/20260227.md §会话03 (✅ 完成)"
    无记忆:   "5. 上次记忆: ⚠️ 无（首次任务 / 未找到记忆文件）"
    目录不存在: "5. 上次记忆: ⚠️ 无（.ai-memory 目录不存在）"

  💡 价值:
    - 用户立即知道当前对话基于哪条记忆延续
    - 快速跳转查看上次做了什么
    - 读取 📨 对话记录 可快速回忆上次对话的完整脉络（🆕 v1.6）
    - 在多 Agent 场景下确认记忆归属

  🔴 禁止行为:
    - 上次记忆字段留空或不扫描就输出
    - 未实际扫描目录就声称"无记忆"
    - 使用 find_path/glob 扫描 .ai-memory 目录（会漏掉隐藏目录）
```

### 新会话开始时（阶段 0 的完整流程）

```yaml
AI 收到任务请求后，执行阶段 0:
  1. 输出上次记忆路径（预检查第 5 项）
  2. 读取 .ai-memory/SUMMARY.md（全局关键决策）
  3. 读取 .ai-memory/clients/<agent>/SUMMARY.md（本 Agent 任务列表）
  4. 如涉及之前任务，读取对应 tasks/<id>.md:
     - 📨 对话记录 → 快速回忆对话脉络（🆕 v1.6）
     - 📄 关联报告 → 获取报告链接
  5. 按需读取关联报告文件恢复详细上下文
  6. 创建新的记忆文件（阶段 0）
  7. 记录用户首条消息摘要到 📨 对话记录（阶段 0+1 合并）
```

### 用户问"之前做了什么"

```yaml
AI 执行:
  1. 读取 .ai-memory/SUMMARY.md
  2. 列出最近 N 条任务记录
  3. 用户可以追问某条任务的详情 → 读取 tasks/<id>.md → 读取关联报告
```

## ⚙️ 维护

```yaml
清理策略:
  - 各 Agent 的 SUMMARY.md 保留最近 20 条，更早的只保留关键决策
  - clients/<agent>/tasks/ 下超过 30 天的文件可以清理（按日期文件名判断）
  - reports/ 下超过 30 天的文件可以清理
  - 用户可以说"清理旧记忆"触发
  - 🆕 v1.7: 清理粒度为整个日期文件（删除 YYYYMMDD.md 即清除该天所有会话记忆）

🆕 归档策略（v1.8）:
  触发条件（满足任一即触发）:
    - 单日记忆文件超过 8 个会话（## 会话 NN 中 NN ≥ 9）
    - 单日记忆文件超过 400 行

  归档操作:
    1. 创建归档文件: YYYYMMDD-archive.md（与当日文件同目录）
    2. 将最早的会话（会话 01 ~ 会话 NN-3）整体移入归档文件
    3. 当日文件只保留:
       - 头部元信息（# 标题 + Agent/日期/状态）
       - ## 📦 归档说明（指向归档文件 + 归档会话范围）
       - 最近 3 个会话（完整内容）
       - ## 📄 全天关联报告汇总（保留完整）
       - ## ⚠️ 全天待跟进汇总（保留完整）
    4. 归档文件结构:
       - 头部标注: "此文件为 YYYYMMDD.md 的归档，包含会话 01~NN-3"
       - 完整保留被归档会话的所有内容（对话记录、关联报告、决策等）

  归档文件命名:
    ✅ 正确: 20260227-archive.md（与 20260227.md 同目录）
    ❌ 错误: 20260227-archive-01.md（不支持多次归档分片，归档文件只有一个，追加模式）

  二次归档:
    - 如归档后当日文件再次超限 → 将新的早期会话追加到已有 YYYYMMDD-archive.md 末尾
    - 当日文件始终只保留最近 3 个会话

  恢复任务时的归档处理:
    - 正常恢复只读当日文件（最近 3 个会话 + 待跟进汇总，足够恢复上下文）
    - 需要更早上下文时 → 读取 YYYYMMDD-archive.md 中对应会话段落
    - 归档文件中的报告链接仍然有效（报告文件不受归档影响）

按 Agent 查看:
  - 用户可以说"只看 zed 的记忆" → 只读 clients/zed-copilot/SUMMARY.md
  - 用户可以说"清理 webstorm 的旧记忆" → 清理 clients/webstorm-copilot/tasks/
```
