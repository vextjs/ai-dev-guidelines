# 规范自我审查工作流

> 定义 AI 对规范体系自身执行全量深度审查的标准化流程

**最后更新**: 2026-03-03  
**适用**: 用户说"自我审查"、"规范审查"、"spec audit"、"全面体检"、"规范健康检查"，或修改 `core/` 下规范文件后自动追加

---

## 🔴 与现有机制的边界（必读）

```yaml
本工作流 ≠ self-fix（自修复）:
  self-fix:   反应式 — 发现问题 → 修复（面向单个问题）
  self-audit: 主动式 — 定期/事件驱动全面体检（面向整个体系）

本工作流 ≠ 10-analysis（深度分析）:
  10-analysis: 通用分析工作流，面向任何目标
  self-audit:  规范体系专用审查，有固定的 15 个维度和标准

本工作流 ≠ CROSS-VALIDATION（交叉验证）:
  CROSS-VALIDATION: 文件级数值交叉验证清单（版本号/约束条数/日期）
  self-audit:       在其之上增加设计审查、有效性审查、架构审查等深层维度

协作关系:
  CROSS-VALIDATION.md  ←── 维度 Ⅳ 直接委托调用
  self-fix/             ←── 发现问题后交给 self-fix 走修复流程
  10-analysis/          ←── 复用其报告输出规范
  auto-triggers.md      ←── 场景 11 触发本工作流
```

```yaml
前置条件: 约束 #14 交叉验证已通过
```

### 2. 消息触发（用户主动）

```yaml
触发关键词:
  - "自我审查" / "规范审查" / "spec audit"
  - "全面体检" / "规范健康检查"
  - "审查规范体系" / "规范自检"

🔴 不分级: 每次触发都执行全部 15 个维度，不省略、不跳过
🔴 不在乎 token: 所有维度做到最深，不因 token 成本而简化
```

### 3. 防递归悖论

```yaml

  - 仍触发全量 15 维度审查

  - 维度 Ⅶ~Ⅸ 对自审文件的结论需人工二次确认
```

## 📋 15 维度全量深度审查体系

### 总览

```yaml
分组 A — 定位与哲学 (3 维度):
  Ⅰ.  项目定位符合性
  Ⅱ.  设计原则遵守度
  Ⅲ.  架构层次完整性

分组 B — 数据与引用 (3 维度):
  Ⅳ.  数值一致性
  Ⅴ.  引用完整性
  Ⅵ.  语义一致性

分组 C — 规则与约束 (3 维度):
  Ⅶ.  约束体系合理性
  Ⅷ.  工作流内部逻辑完整性
  Ⅸ.  环境假设有效性

分组 D — 运行与实效 (3 维度):
  Ⅹ.  机制有效性
  Ⅺ.  防复现完备性
  Ⅻ.  层间同步一致性

分组 E — 演进与生态 (3 维度):
  ⅩⅢ. 复杂度健康度
  ⅩⅣ. 演进路线一致性
  ⅩⅤ. 工具链与文档覆盖度
```

### 分组 A — 定位与哲学

#### 维度 Ⅰ：项目定位符合性

```yaml
审查目标: 每个规范/机制/约束是否服务于"AI 工作操作系统"定位

审查标准依据:
  - docs/DESIGN-PHILOSOPHY.md §一、核心定位
  - docs/DESIGN-PHILOSOPHY.md 操作系统类比表（6 项映射）

检查项:
  1. 每条新增约束是否能映射到操作系统类比表中的某个角色？
     - 文件系统（报告+记忆）
     - 内存管理（消息驱动 5+1 阶段）
     - 进程调度（11 种工作流 + 决策树）
     - 权限控制（约束 + 确认点）
     - 多进程（多 Agent 目录隔离）
     - 自我修复（spec-self-fix 模块）
     - 驱动程序（模板体系）
  2. 是否存在偏离定位的设计？（如引入了"项目管理"功能、"编码规范强制器"等越界设计）
  3. 新增的工作流/模板/工具是否在"AI 工作操作系统"的职责范围内？
  4. core/standards/ 中的编码规范（9 个文件）是否属于"驱动程序"角色？是否有越界到"应用程序"层？

检查方法:
  1. 读取 docs/DESIGN-PHILOSOPHY.md §一、核心定位
  2. 列出所有 core/ 下最近修改的文件
  3. 逐个验证修改内容是否服务于 6 项操作系统映射中的某一项
  4. 对无法映射的内容标记为 🟡 偏离嫌疑

输出格式:
  ✅ 符合定位: <文件> — 服务于<映射角色>
  🟡 偏离嫌疑: <文件> — 无法映射到任何角色，原因: <分析>
  🔴 定位违反: <文件> — 明确属于"不是什么"中列出的类别
```

#### 维度 Ⅱ：设计原则遵守度

```yaml
审查目标: 5 条设计原则是否被所有规范文件贯穿？有无违反？

审查标准依据:
  - docs/DESIGN-PHILOSOPHY.md §二、设计原则（5 条）

检查项:
  原则 1 — AI 是第一用户:
    - 新增规范是否对 AI 友好？（用 emoji 强信号、表格索引、正反对照示例）
    - 有无"人类散文式"描述导致 AI 难以解析？
    - 数值类规则是否在多处冗余引用以减少 AI 跳转？
    - 🔴 是否存在隐式逻辑需要 AI "推断"而非"查表"的情况？

  原则 2 — 三层递进架构:
    - 入口层（copilot-instructions.md）是否仍在 ~40 行以内？
    - 速查层（QUICK-REFERENCE.md）是否仍控制在合理行数？
    - 新增内容是否放入了正确的层级？（入口层禁止放细节，速查层禁止放完整定义）
    - 是否有信息放错层级的情况？（如细节漏到了入口层、概述重复到了详情层）

  原则 3 — 有状态 > 无状态:
    - 消息驱动 5+1 阶段是否被所有工作流正确引用？
    - 是否存在某个工作流绕过了记忆写入？
    - 新会话恢复流程是否在所有触发点一致？

  原则 4 — 防复现 > 事后修复:
    - META.yaml 是否仍是单一真相源？有无新的"绕过 META 直接修改"的路径？
    - bump-version.js 的同步范围是否覆盖了所有引用？
    - 新增的约束/工作流/机制是否注册到了 CROSS-VALIDATION.md 的检查清单？

  原则 5 — 约束是护栏非枷锁:
    - 约束是否存在死锁场景？（两条约束互相矛盾导致 AI 无法执行）
    - 约束是否过度限制了 AI 的灵活性？
    - 约束的豁免条件是否明确？（如约束 #16 豁免了报告/记忆写入）

检查方法:
  1. 读取 docs/DESIGN-PHILOSOPHY.md §二
  2. 逐条原则，抽取 core/ 下 3~5 个代表性文件进行验证
  3. 重点检查最近修改的文件是否遵守

输出格式:
  各原则逐条列出 ✅ 遵守 / 🟡 部分偏离 / 🔴 违反 + 证据
```

#### 维度 Ⅲ：架构层次完整性

```yaml
审查目标: 6 层架构各层数据是否与实际一致

审查标准依据:
  - docs/DESIGN-PHILOSOPHY.md §三、架构全景（ASCII 图）

检查项:
  1. 入口层: copilot-instructions.md 实际行数 vs 全景图描述（~40 行）
  2. 速查层: QUICK-REFERENCE.md 实际行数 vs 全景图描述（~420 行）
  3. 执行层:
     - 预检查实际项数 vs 描述（5项+第6行）
     - 工作流实际数量 vs 描述（当前应为 11 种，含本工作流）
     - 模板实际数量 vs 描述（29 个）
  4. 约束层: 约束实际条数 vs 描述（20 条）
  5. 持久化层: reports/ 和 .ai-memory/ 结构是否与描述一致
  6. 自维护层: 检测模块/修复模块/防复现模块是否都存在

检查方法:
  1. 读取 docs/DESIGN-PHILOSOPHY.md §三
  2. 逐层 list_directory + read_file 验证实际数据
  3. 对比全景图中的数值标注 vs 实际计数

输出格式:
  各层逐项: 描述值 → 实际值 → ✅ 一致 / 🔴 不一致（需更新全景图或修复实际结构）
```

### 分组 B — 数据与引用

#### 维度 Ⅳ：数值一致性

```yaml
审查目标: 所有跨文件数值精确一致

审查标准依据:
  - core/CROSS-VALIDATION.md 全部清单
  - core/META.yaml 单一真相源

检查项:
  1. 版本号（8 文件）: 逐个 read_file 对比 META.yaml 中的 version 值
  2. 约束条数（11 文件）: 逐个 read_file 对比 META.yaml 中的 constraint_count 值
  3. 最后更新日期（7 文件）: 逐个 read_file 对比 META.yaml 中的 last_updated 值
  4. 预检查项数: copilot-instructions.md / 00-pre-check / QUICK-REFERENCE 三处对比
  5. 工作流数量: workflows/README.md / DESIGN-PHILOSOPHY.md / decision-tree.yaml 三处对比
  6. 模板数量: templates/README.md / DESIGN-PHILOSOPHY.md / STATUS.md 三处对比
  7. 自动触发场景数: auto-triggers.md §触发场景 实际条目数 vs README.md 统计值
  8. 修复模式数: repair-patterns.md §模式概览 实际条目数 vs README.md 统计值

检查方法:
  1. 优先运行 node core/tools/bump-version.js --check（覆盖版本号+约束条数+日期）
  2. 对工具未覆盖的数值（5~8），手动逐文件 read_file 对比
  3. 🔴 禁止推断，必须实际读取文件确认

输出格式:
  各数值项: 文件A=X / 文件B=X / 文件C=Y → ✅ 一致 / 🔴 不一致（列出偏差详情）
```

#### 维度 Ⅴ：引用完整性

```yaml
审查目标: 所有内部链接/锚点指向实际存在的文件和章节

审查标准依据:
  - self-fix/detection/conflict-detection.md §规则 4（路径引用一致性）
  - self-fix/detection/obsolete-detection.md §规则 1（断链检测）

检查项:
  1. Markdown 相对链接: [文字](./path/to/file.md) 中的路径是否存在？
  2. 锚点引用: [文字](./file.md#section-title) 中的锚点是否在目标文件中？
  3. 代码块内的路径引用: 如 "详见 core/workflows/..." 中的路径是否有效？
  4. 目录结构声明 vs 实际文件系统: README.md 中画的目录树与实际是否一致？
  5. 🔴 拆分后遗症: 文件拆分后，旧锚点引用是否全部更新到新文件？

检查方法:
  1. 可用时优先运行 node core/tools/validate-links.js
  2. 对核心文件（copilot-instructions / QUICK-REFERENCE / CONSTRAINTS / README）手动验证每个链接
  3. 重点检查最近修改/拆分/重命名的文件

输出格式:
  ✅ 有效: <源文件> → <目标文件> 存在
  🔴 断链: <源文件 L行号> → <目标路径> 不存在
  🟡 锚点失效: <源文件 L行号> → <目标文件#锚点> 锚点不存在
```

#### 维度 Ⅵ：语义一致性

```yaml
审查目标: 同一概念/术语/流程在不同文件中的描述语义一致

审查标准依据:
  - self-fix/detection/redundancy-detection.md §规则 1~4
  - self-fix/detection/conflict-detection.md §规则 3（确认点定义一致性）

检查项:
  1. 核心术语定义一致性:
     - "预检查" 在 copilot-instructions / QUICK-REFERENCE / 00-pre-check 三处的描述是否一致？
     - "消息驱动 5+1 阶段" 在 task-memory.md / triggers.md / CONSTRAINTS.md #18 三处的描述是否一致？
     - "报告为主体、记忆为索引" 在 task-memory.md / QUICK-REFERENCE / temp-reports.md 三处是否一致？
     - "确认点" 在 confirmation-points.md / decision-tree.yaml / 各工作流 三处定义是否一致？

  2. 流程步骤描述一致性:
     - 预检查的步骤描述（5 项 + 第 6 行）在各处是否完全相同？
     - 阶段 0 的执行步骤在 triggers.md 和 00-pre-check 中是否一致？
     - 新会话恢复流程在 triggers.md §新会话 和 QUICK-REFERENCE §恢复 中是否一致？

  3. 禁止行为清单一致性:
     - copilot-instructions.md 中的禁止行为 与 00-pre-check 中的禁止行为是否对齐？
     - 各约束的 ❌ 绝对禁止 列表在 CONSTRAINTS.md 和 QUICK-REFERENCE 中是否对齐？

  4. 术语使用统一性:
     - 同一概念是否在不同文件中用了不同的名称？（如 "spec-self-fix" vs "自修复" vs "自我修复"）
     - 版本标记方式是否统一？（如 "🆕 v1.7" vs "v1.7 新增" vs "(🆕 v1.7)"）

检查方法:
  1. 对每个核心术语，grep 搜索所有出现位置
  2. 对比各处的完整定义文本，不只是看是否"提到了"
  3. 对流程步骤类内容，逐步骤比对

输出格式:
  ✅ 一致: <术语> 在 N 处定义一致
  🟡 表述差异: <术语> 在 <文件A> 和 <文件B> 中措辞不同但语义等价 — 可接受/建议统一
  🔴 语义冲突: <术语> 在 <文件A> 说"X"，在 <文件B> 说"Y"，含义矛盾
```

### 分组 C — 规则与约束

#### 维度 Ⅶ：约束体系合理性

```yaml
审查目标: 20 条约束的逻辑合理性、优先级分配、覆盖完整性

审查标准依据:
  - core/CONSTRAINTS.md（20 条完整定义）
  - QUICK-REFERENCE.md §核心约束（速查表）

检查项:
  1. 逻辑矛盾检测:
     - 是否有两条约束的要求互相矛盾？
     - 是否存在死锁场景？（遵守约束 A 必然违反约束 B）
     - 示例: 约束 #16（修改需确认）是否与约束 #17（报告自动输出禁止询问）有清晰边界？

  2. 优先级合理性:
     - 🔴P0（9 条）/ 🟡P1（8 条）/ 💡P2（3 条）的分配是否合理？
     - QUICK-REFERENCE 速查表 与 CONSTRAINTS.md 中每条约束的优先级标签是否逐条一致？
     - 有无应升级为 P0 但标为 P1/P2 的约束？
     - 有无过度标为 P0 实际影响较低的约束？

  3. 功能重叠检测:
     - 约束 #1（删除需确认）与 #16（文件修改需确认）— 删除是修改的子集，是否需合并？
     - 约束 #11（修复需扫描）与 #13（自动关联文件检查）— 职责边界是否清晰？
     - 约束 #14（规范修改交叉验证）与 #11（修复需扫描）— 是否有重叠？

  4. 覆盖缺失检测:
     - 当前 20 条约束是否覆盖了所有已知风险场景？
     - 基于 self-fix/records/ 中的历史修复记录，是否有反复出现但无约束覆盖的问题？
     - 是否需要新增约束？（审慎评估，避免约束膨胀）

  5. 豁免条件明确性:
     - 每条约束的适用范围和豁免条件是否明确？
     - 约束 #16 的"纯分析/只读/报告/记忆写入除外"是否足够清晰？
     - 约束 #20（文件过大拆分）的"已有文件豁免"是否会导致老文件无限膨胀？

检查方法:
  1. 读取 CONSTRAINTS.md 全部 20 条
  2. 两两对比检测矛盾和重叠
  3. 读取 QUICK-REFERENCE.md 速查表逐条核对优先级标签
  4. 读取 self-fix/records/ 查找无约束覆盖的问题

输出格式:
  各条约束逐条评估: 编号 + 优先级 + 合理性结论 + 与其他约束的关系
  总结: 矛盾 N 对 / 重叠 N 对 / 缺失 N 条 / 优先级偏差 N 条
```

```yaml
审查目标: 各工作流之间的转入/转出规则、边界条件、互斥关系是否完整

审查标准依据:
  - core/workflows/ 下所有工作流 README.md

检查项:
  1. 关键词映射完备性:

     - 是否有常见请求无法匹配到任何工作流？
     - 各工作流的关键词权重是否合理？（如 analysis weight=7 是否太低？）

  2. 工作流转出规则:
     - 10-analysis 的变更意图检测（FIX-010）是否正确转出到 01-requirement-dev？

  3. 工作流互斥/并行关系:
     - 哪些工作流可以并行执行？哪些互斥？
     - 是否有明确的规则定义并行和互斥关系？

  4. 确认点覆盖:
     - 涉及代码修改的工作流（01/02/03/05/06/07）是否都有 CP1/CP2/CP3？
     - 纯分析工作流（04/08/10/11）是否正确跳过 CP？
     - 确认点的定义在 confirmation-points.md 与各工作流中是否一致？

  5. 前置检查覆盖:
     - 所有工作流是否都以 00-pre-check 为前置？
     - 00-pre-check 的第 6 行（阶段 0 记忆写入）是否被所有工作流正确执行？

检查方法:
  1. 读取 decision-tree.yaml §task_identification
  2. 逐个读取各工作流 README.md 的入口/出口规则
  3. 交叉验证转出/转入规则的对称性

输出格式:
  工作流映射表（11 种）: 关键词覆盖度 / 转出规则完整性 / 确认点覆盖 / 前置检查
  发现的缺口: <工作流> 缺少 <规则>
```

```yaml
  - 不跳过任何维度

  🔴 每个维度的输出必须包含（约束 #8 — 输出需验证）:
    1. 通过项 + 发现项 + 结论
    2. 三项验证标注（每个发现/建议逐条附带）:
       - 合理性: 该发现是否基于实际文件验证？列出读取的文件和行号作为证据
       - 可实施性: 修复方案是否已验证可落地？涉及的文件/工具/流程是否确认存在？
       - 收益/必要性: 修复后的具体收益是什么？（如"消除 N 处不一致" / "防止 X 类事故复现"）
    3. 如发现无法通过三项验证 → 标注为 ❌ 误报 并说明原因，不列入问题清单

  🔴 禁止行为（约束 #8 ❌ 绝对禁止）:
    - 输出未经验证的发现（未实际 read_file 就断言问题存在）
    - 输出无法落地的修复建议（依赖不存在的工具或假设不成立的前提）
    - 省略验证标注（每条发现必须显式标注三项验证结论，方便用户一眼确认已验证）

维度间依赖:
  - 维度 Ⅳ（数值一致性）的结果被维度 Ⅲ（架构完整性）引用
  - 维度 Ⅹ（机制有效性）的结果被维度 Ⅺ（防复现完备性）引用
  - 维度 Ⅻ（层间同步）的结果被维度 Ⅺ（防复现完备性）引用
  → 因此必须按顺序执行，不可并行
```

### 阶段 4：输出审查报告

```yaml
输入: 15 维度审查结论（含三项验证）
输出: 结构化审查报告

报告结构:
  1. 审查总览表（15 维度 × 状态 + 问题数 + 验证状态）
  2. 分组评估（A~E 各组总评）
  3. 各维度详细分析（发现 + 证据 + 结论 + 🔴 三项验证标注）
  4. 问题清单（按严重度排序: 🔴 → 🟡 → 💡 → ❌）
  5. 修复建议（每个问题附带修复方案 + 涉及文件 + 🔴 三项验证）
  6. 与上次审查的对比（如有历史审查报告）
  7. 综合健康度评分（百分制 + 各维度分项）

  🔴 报告中每个发现/建议必须附带验证标注（约束 #8）:
    格式:
      > **发现**: <描述>
      > - 合理性: ✅ 已验证 — 读取 <文件> L<行号> 确认 <证据>
      > - 可实施性: ✅ 已验证 — 修改 <文件> 即可修复 / ⚠️ 待验证 — <原因>
      > - 收益: <具体收益描述>

    为什么强制标注:
      - 用户无需额外发消息确认 AI 是否做过验证
      - 一眼可见每条结论的证据链
      - 防止 AI 基于推断而非实际文件验证输出问题

问题分级:
  🔴 必须立即修复 — 数据不一致/引用断链/约束矛盾/架构数据过时
  🟡 建议修复     — 语义偏差/术语不统一/覆盖缺口/文档过时
  💡 观察项       — 复杂度接近阈值/行数增长/可合并项
  ❌ 不合理       — 设计偏离定位/假设已失效/裸修复无防线

🔴 报告自动写入:
  路径: reports/analysis/<agent>/YYYYMMDD/NN-analysis-规范自我审查.md
  🔴 禁止询问用户是否写报告
```

### 阶段 4.5：审查→修复衔接（🆕 v2.11.0）

```yaml
输入: 审查报告中的问题清单（分级 🔴/🟡/💡/❌）
输出: 🔴 级问题已触发修复 或 已输出变更计划等待用户确认

设计意图:
  §边界 声明了"发现问题后交给 self-fix 走修复流程"，本阶段将这个衔接落实到执行步骤中。
  审查不能止步于"发现问题、记入待跟进" — 能修的当场修，不能修的明确标注原因。

步骤:
  0. 🆕 核对 issue.md 问题追踪表:
     a. 读取 issue.md，筛选所有 🔴 未解决 和 🟡 部分解决 的问题
     b. 对每条问题验证当前状态（是否已被修复/仍存在）
     c. 仍存在的问题 → 分析根因 → 按下方分级规则分流处理（与审查发现的问题合并）
     d. 已修复的问题 → 更新 issue.md 中该问题状态为 ✅ 已解决 + 填写解决日期
     e. 分析未解决问题的根因，评估是否需要新增防复现机制

  1. 遍历审查报告中的问题清单（含 issue.md 中未解决的问题），按分级分流处理

  🔴 必须立即修复（自动执行，无需用户确认）:
    a. 数值不一致类（版本号/约束条数/日期）→ 运行 bump-version.js --sync 自动修复
    b. 路径引用缺失/错误类 → 按 repair-patterns.md §模式 2（路径修正）执行
       🔴 必须 grep 全局搜索同类问题，禁止只修已发现的（约束 #11）
    c. 断链类 → 按 repair-patterns.md §模式 3（断链修复）执行
    d. 其他可自动修复类 → 匹配 repair-patterns.md 对应模式执行
    e. 🔴 每项修复后立即在报告中标记 ✅（阶段 3 实时更新规则）

  🟡 建议修复（需用户确认）:
    a. 输出变更计划（涉及文件 + 修改内容 + 三项验证标注）
    b. 等待用户确认后执行（约束 #16 — 文件修改需确认）
    c. 用户拒绝 → 标注"用户决定暂不修复"，记入待跟进

  💡 观察项（不执行修复）:
    a. 记录到记忆文件的 ⚠️ 待跟进表
    b. 在报告中保留观察项，标注当前值和阈值

  ❌ 误报（不执行修复）:
    a. 在报告中标注验证结论"验证后不存在"
    b. 不记入待跟进

  2. 修复完成后执行验证:
    a. 对每项修复 grep 全局搜索确认同类问题 = 0（约束 #11）
    b. 可用时运行自动化工具验证（bump-version.js --check / validate-links.js）
    c. 在报告尾部追加修复验证结果

  3. 更新审查报告:
    a. 问题清单中已修复项标记 ✅
    b. 追加"修复执行记录"章节（操作 + 文件 + 行号 + 状态）
    c. 更新综合健康度评分（修复后重新计算）

🔴 与 self-fix 的协作边界:
  - 本阶段直接复用 self-fix/repair/ 中的修复模式，不重新发明修复逻辑
  - 修复记录写入审查报告（不写入 self-fix/records/，因为审查报告本身就是记录）
  - 如修复过程中发现新问题（连锁问题）→ 追加到问题清单，重新分级分流
  - 如修复失败或无法自动修复 → 标注原因，降级为 🟡 等待用户决策
```

### 阶段 5：更新记忆

```yaml
输入: 审查报告（含阶段 4.5 修复结果）
输出: 记忆文件已更新

步骤:
  1. 追加会话记录到 .ai-memory/clients/<agent>/tasks/YYYYMMDD.md
  2. 内容: 审查结论摘要 + 报告链接 + 对话记录 + 待跟进
  3. 🔴 级问题已在阶段 4.5 修复 → 记录修复结果
  4. 🟡 级用户拒绝修复 / 💡 级观察项 → 记入 ⚠️ 待跟进

🔴 记忆自动写入，禁止询问用户
```
