# 工作流 0: 任务识别（AI 必读）

> **目的**: AI 收到用户请求后，首先判断属于哪种任务类型
> 
> **执行时机**: 每次收到新任务的第一步
> 
> **执行者**: AI Agent

---

## 🎯 识别流程

```
用户输入
    ↓
分析关键词
    ↓
匹配任务类型
    ↓
确定工作流文件
    ↓
报告给用户
    ↓
继续执行对应工作流
```

> ⚠️ **深度分析 vs 需求开发的意图判断**（详见 [rules.md §Layer 3 意图推断](./rules.md)）:
> - 🔴 不依赖关键词列表判定，而是分析用户**最终目的**（Layer 3 三问）
> - 最终目的是变更代码/文件 → **需求开发**（01-requirement-dev）；分析只是获得结论 → 深度分析（10-analysis）
> - 详见 rules.md §Layer 3 三个判断问题（Q1 最终目的 / Q2 分析是手段还是目的 / Q3 是否有具体事项）

## 🤖 AI 执行步骤

### Step 1: 提取关键词
```typescript
// 伪代码示例
function extractKeywords(userInput: string): string[] {
  // 转小写、分词、去停用词
  const words = tokenize(userInput.toLowerCase());
  return removeStopWords(words);
}

// 示例
输入: "在 user 项目集成 Redis 限流功能"
输出: ["user", "集成", "redis", "限流", "功能"]
```

### Step 2: 匹配任务类型
```typescript
function matchTaskType(keywords: string[]): TaskType {
  // 优先匹配 Tier 1
  for (const tier1Type of TIER1_TYPES) {
    if (hasAnyKeyword(keywords, tier1Type.keywords)) {
      return tier1Type;
    }
  }
  
  // 再匹配 Tier 2
  for (const tier2Type of TIER2_TYPES) {
    if (hasAnyKeyword(keywords, tier2Type.keywords)) {
      return tier2Type;
    }
  }
  
  // 默认: 需求开发
  return REQUIREMENT_DEV;
}

// 示例
关键词: ["集成", "限流", "功能"]
匹配: "集成" → 需求开发
```

### Step 3: 确认并报告
```markdown
AI 输出示例:

- 任务类型: 需求开发
- 工作流: core/workflows/01-requirement-dev/README.md

我将按照需求开发流程执行，预计生成 4 个文档：
1. 需求分析文档
2. 技术方案文档

4. 对接说明文档

现在开始执行...
```

## ⚠️ 特殊情况处理

### 情况 1: 无法明确识别
```
如果用户输入模糊，AI 应该:
1. 列出可能的任务类型（最多 3 个）
2. 向用户提问确认
3. 等待用户选择后再继续

示例:
用户: "处理一下用户模块"
AI: 我识别到可能是以下任务类型：
  1. 需求开发（添加新功能）
  2. Bug 修复（修复问题）
  3. 架构重构（重构代码）
  
  请问具体是哪种类型？
```

### 情况 2: 多任务混合
```
如果用户一次提出多个任务:
1. 识别并列出所有任务
2. 建议分开执行
3. 询问执行顺序

示例:
用户: "实现限流功能，并优化查询性能"
AI: 我识别到 2 个任务：
  任务 1: 需求开发 - 实现限流功能
  任务 2: 性能优化 - 优化查询性能
  
  建议分开执行，请问先执行哪个？
```

### 情况 3: 跨项目任务
```
如果涉及多个项目:
1. 为每个项目单独执行
2. 在各自的 projects/<project-name>/ 目录生成文档

示例:
用户: "在 user 和 order 项目都添加限流"
AI: 识别到跨项目任务，我将分别执行：
  1. user 项目 - requirements/用户限流功能/
  2. order 项目 - requirements/订单限流功能/
```

## ✅ 识别检查清单

AI 完成任务识别后，必须确认：

- [ ] Layer 2: 已提取关键词信号（候选类型列表）
- [ ] Layer 3: 已执行意图推断三问（Q1 最终目的 / Q2 分析是手段还是目的 / Q3 是否有具体事项）
- [ ] Layer 3: 已记录意图推断理由
- [ ] 已确定最终任务类型和工作流文件
- [ ] 已识别项目名称
- [ ] 已向用户报告识别结果（含意图推断理由）
- [ ] 如有疑问已询问用户

## 📊 识别决策树

```
用户输入
    ↓
Layer 1: 用户明确声明任务类型?
    ──YES──> 直接采用
    ↓ NO
Layer 2: 提取关键词信号 → 产出候选类型列表
    ↓
Layer 3: 意图推断三问（🔴 核心判断层）
    ↓
  Q1. 最终目的是变更代码/文件?
    ──YES──> 变更类（根据 Layer 2 候选选择：需求开发/Bug修复/重构/...）
    ↓ NO
  Q2. "分析/评估"是手段还是目的?
    ──手段──> 变更类（分析只是前置步骤）
    ↓ 目的
  Q3. 是否描述了具体执行事项?
    ──YES──> 变更类（有具体事项 = 有变更诉求）
    ↓ NO
  纯分析/调研/问答（根据 Layer 2 候选选择）
    ↓
Layer 4: 结合会话/项目上下文确认或修正
    ↓
最终判定
```

## 💡 识别最佳实践

### ✅ 好的做法
1. 优先匹配核心任务类型（Tier 1）
2. 有疑问立即询问用户
3. 向用户明确报告识别结果
4. 多任务建议分开执行

### ❌ 要避免的
1. 不要猜测用户意图
2. 不要跳过识别步骤
3. 不要默默选择任务类型
4. 不要一次执行多个任务
