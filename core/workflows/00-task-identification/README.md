# 工作流 0: 任务识别（AI 必读）

> **目的**: AI 收到用户请求后，首先判断属于哪种任务类型
> 
> **执行时机**: 每次收到新任务的第一步
> 
> **执行者**: AI Agent

---

## 🎯 识别流程

```
用户输入
    ↓
分析关键词
    ↓
匹配任务类型
    ↓
确定工作流文件
    ↓
报告给用户
    ↓
继续执行对应工作流
```

## 📋 任务类型识别表

### Tier 1: 核心任务类型（优先匹配）

| 任务类型 | 关键词 | 工作流文件 | 示例用户输入 |
|---------|--------|-----------|-------------|
| **需求开发** | 实现、开发、添加、集成、新功能、对接 | `01-requirement-dev/README.md` | "在 user 项目添加限流功能" |

### Tier 2: 扩展任务类型（次优先）

| 任务类型 | 关键词 | 工作流文件 | 示例用户输入 |
|---------|--------|-----------|-------------|
| **技术调研** | 调研、选型、对比、评估 | `04-research/README.md` | "调研缓存方案 Redis vs Memcached" |
| **架构重构** | 重构、重写、拆分、改造 | `05-refactoring/README.md` | "重构用户认证模块" |
| **数据库变更** | 迁移、Schema、表结构、索引 | `06-database/README.md` | "添加用户表的索引" |
| **安全修复** | 漏洞、安全、注入、XSS | `07-security/README.md` | "修复 SQL 注入漏洞" |
| **事故复盘** | 故障、宕机、复盘、事故 | `08-incident/README.md` | "分析昨晚数据库宕机事故" |
| **深度分析** | 分析、审查、评估、检查、对比、审计 | `10-analysis/README.md` | "分析这个项目的架构质量" |

> ⚠️ **深度分析 vs 需求开发的混合意图判断**（详见 [rules.md §混合意图优先级规则](./rules.md)）:
> - 用户请求同时包含"分析/审查"**和**"变更/修改/调整/重构"意图 → 判定为 **需求开发**（01-requirement-dev），不是深度分析
> - 只有纯分析、纯审查、纯评估（不涉及任何变更）→ 才走深度分析（10-analysis）

## 🤖 AI 执行步骤

### Step 1: 提取关键词
```typescript
// 伪代码示例
function extractKeywords(userInput: string): string[] {
  // 转小写、分词、去停用词
  const words = tokenize(userInput.toLowerCase());
  return removeStopWords(words);
}

// 示例
输入: "在 user 项目集成 Redis 限流功能"
输出: ["user", "集成", "redis", "限流", "功能"]
```

### Step 2: 匹配任务类型
```typescript
function matchTaskType(keywords: string[]): TaskType {
  // 优先匹配 Tier 1
  for (const tier1Type of TIER1_TYPES) {
    if (hasAnyKeyword(keywords, tier1Type.keywords)) {
      return tier1Type;
    }
  }
  
  // 再匹配 Tier 2
  for (const tier2Type of TIER2_TYPES) {
    if (hasAnyKeyword(keywords, tier2Type.keywords)) {
      return tier2Type;
    }
  }
  
  // 默认: 需求开发
  return REQUIREMENT_DEV;
}

// 示例
关键词: ["集成", "限流", "功能"]
匹配: "集成" → 需求开发
```

## ⚠️ 特殊情况处理

### 情况 1: 无法明确识别
```
如果用户输入模糊，AI 应该:
1. 列出可能的任务类型（最多 3 个）
2. 向用户提问确认
3. 等待用户选择后再继续

示例:
用户: "处理一下用户模块"
AI: 我识别到可能是以下任务类型：
  1. 需求开发（添加新功能）
  2. Bug 修复（修复问题）
  3. 架构重构（重构代码）
  
  请问具体是哪种类型？
```

### 情况 2: 多任务混合
```
如果用户一次提出多个任务:
1. 识别并列出所有任务
2. 建议分开执行
3. 询问执行顺序

示例:
用户: "实现限流功能，并优化查询性能"
AI: 我识别到 2 个任务：
  任务 1: 需求开发 - 实现限流功能
  任务 2: 性能优化 - 优化查询性能
  
  建议分开执行，请问先执行哪个？
```

### 情况 3: 跨项目任务
```
如果涉及多个项目:
1. 为每个项目单独执行
2. 在各自的 projects/<project-name>/ 目录生成文档

示例:
用户: "在 user 和 order 项目都添加限流"
AI: 识别到跨项目任务，我将分别执行：
  1. user 项目 - REQ-user-20260211-rate-limit
  2. order 项目 - REQ-order-20260211-rate-limit
```

## 📊 识别决策树

```
用户输入
    ↓
🔴 混合意图检测（最高优先级）:
  同时包含"分析/审查/评估"AND"变更/修改/调整/重构/新增/删除"?
    ──YES──> 需求开发（分析只是前置步骤，最终目的是变更）
    ↓ NO
包含"修复"/"Bug"? ──YES──> Bug 修复
    ↓ NO
包含"优化"/"慢"? ──YES──> 性能优化
    ↓ NO
包含"对接"/"API"? ──YES──> 系统对接
    ↓ NO
包含"调研"/"选型"? ──YES──> 技术调研
    ↓ NO
包含"重构"/"改造"? ──YES──> 架构重构
    ↓ NO
包含"迁移"/"Schema"? ──YES──> 数据库变更
    ↓ NO
包含"漏洞"/"安全"? ──YES──> 安全修复
    ↓ NO
包含"故障"/"复盘"? ──YES──> 事故复盘
    ↓ NO
包含"分析"/"审查"/"评估"/"检查"? ──YES──> 深度分析（纯分析，不含变更意图）
    ↓ NO
默认 ──> 需求开发
```

## 💡 识别最佳实践

### ✅ 好的做法
1. 优先匹配核心任务类型（Tier 1）
2. 有疑问立即询问用户
3. 向用户明确报告识别结果
4. 多任务建议分开执行

### ❌ 要避免的
1. 不要猜测用户意图
2. 不要跳过识别步骤
3. 不要默默选择任务类型
4. 不要一次执行多个任务
