# 意图矩阵表

> 完整的意图分类、风险等级和执行规则

---

## 📋 意图分类概述

### 什么是意图

```yaml
定义: AI 根据用户请求识别的任务类型
作用: 决定执行哪些步骤、加载哪些规范、采用什么风险级别
识别时机: 预检查阶段（检查项 3）
```

### 分类原则

```yaml
分类维度:
  - 输出类型: 代码 / 测试 / 文档 / 分析
  - 风险级别: P0 / P1 / P2
  - 复杂程度: 简单 / 中等 / 复杂
  - 确认需求: 需要 / 不需要
```

## 🎯 Tier 1: 核心意图（日常 80%）

这些是最常用的任务类型，AI 必须优先匹配。

| # | 意图名称 | 触发词 | 风险 | 代码 | 测试 | 文档 | 确认点 |
|---|---------|-------|------|------|------|------|-------|
| 01 | **需求开发** | 开发、实现、新增、集成、对接、添加 | P1 | ✅ | ✅ | ✅ | 4 |
| 02 | **Bug 修复** | 修复、解决、Bug、问题、报错、异常 | P1 | ✅ | ✅ | ✅ | 4 |
| 03 | **性能优化** | 优化、性能、加速、慢、提升 | P1 | ✅ | ✅ | ✅ | 4 |

### 需求开发详情

```yaml
典型场景:
  - "在 user 项目添加限流功能"
  - "实现用户认证模块"
  - "集成 Redis 缓存"

工作流: core/workflows/01-requirement-dev/
输出文档:
  - 01-requirement.md
  - 02-technical.md
  - 03-implementation/（实施方案目录）
  - 04-接口文档.md（涉及 API 接口新增/变更时，可选）或 04-对接文档.md（涉及第三方系统对接时，可选）

接口文档触发条件:
  必须生成: 新增 API 端点、修改接口入参/出参/错误码、第三方系统对接、接口行为变更
  不需要生成: 纯内部逻辑修改、纯前端修改、仅数据库变更（不影响接口响应）
  模板: api-doc-template.md（纯接口）或 integration-template.md（第三方对接）

确认点:
  CP1: 需求理解确认
  CP2: 技术方案确认
  CP3: 实施方案确认（含实施前检查）
  CP4: 文档内容确认
```

### Bug 修复详情

```yaml
典型场景:
  - "修复登录超时的问题"
  - "解决消息丢失 Bug"
  - "用户无法注册的问题"

工作流: core/workflows/02-bug-fix/
输出文档:
  - 01-analysis.md
  - 02-solution.md
  - 03-implementation/（实施方案，小Bug为单文件）
  - 04-接口变更.md（修复涉及接口契约变更时，可选）

接口变更文档触发条件:
  必须生成: 修复导致接口入参/出参变化、错误码新增/变更、接口行为变化、鉴权/权限变更
  不需要生成: 纯内部逻辑修复、纯数据修复、纯配置修复（接口契约无变化）
  模板: api-doc-template.md（精简使用，仅记录变更部分）

确认点:
  CP1: 问题分析确认
  CP2: 解决方案确认
  CP3: 实施方案确认（含实施前检查）
```

### 性能优化详情

```yaml
典型场景:
  - "优化查询接口的性能"
  - "加速首页加载"
  - "数据库查询太慢"

工作流: core/workflows/03-optimization/
输出文档:
  - 01-baseline.md
  - 02-analysis.md
  - 03-solution.md
  - 04-implementation/（实施方案，小优化为单文件）
  - 05-validation.md

确认点:
  CP1: 瓶颈分析确认
  CP2: 优化方案确认
  CP3: 实施方案确认（含实施前检查）
```

## 🔷 Tier 2: 扩展意图（日常 20%）

这些是特殊场景的任务类型。

| # | 意图名称 | 触发词 | 风险 | 代码 | 测试 | 文档 | 确认点 |
|---|---------|-------|------|------|------|------|-------|
| 04 | **技术调研** | 调研、选型、对比、评估 | P2 | ❌ | ❌ | ✅ | 2 |
| 05 | **架构重构** | 重构、重写、拆分、改造 | P0 | ✅ | ✅ | ✅ | 3+ |
| 06 | **数据库变更** | 迁移、Schema、索引、表结构 | P0 | ✅ | ✅ | ✅ | 3 |
| 07 | **安全修复** | 漏洞、安全、注入、XSS | P0 | ✅ | ✅ | ✅ | 3 |
| 08 | **事故复盘** | 故障、宕机、复盘、事故 | P1 | ❌ | ❌ | ✅ | 0 |

### 技术调研详情

```yaml
典型场景:
  - "调研缓存方案 Redis vs Memcached"
  - "评估 GraphQL 和 REST 的优劣"
  - "技术选型分析"

工作流: core/workflows/04-research/
输出文档:
  - 01-background.md
  - 02-comparison.md
  - 03-poc.md（可选）
  - 04-conclusion.md
```

### 架构重构详情（P0）

```yaml
典型场景:
  - "重构用户认证模块"
  - "微服务拆分"
  - "单体改造"

工作流: core/workflows/05-refactoring/
输出文档:
  - 01-current-state.md
  - 02-target-design.md
  - 03-plan.md
  - 04-implementation/（按阶段）
  - 05-summary.md

确认点:
  CP1: 现状分析确认
  CP2: 目标架构确认
  CP3: 每阶段实施方案确认（含实施前检查）

特殊要求:
  - 必须有回滚方案
  - 必须分阶段实施
  - 必须有灰度计划
```

### 数据库变更详情（P0）

```yaml
典型场景:
  - "添加用户表的索引"
  - "Schema 迁移"
  - "数据迁移"

工作流: core/workflows/06-database/
输出文档:
  - 01-design.md
  - scripts/*.sql（迁移+回滚+验证）
  - 02-validation.md
  - 03-execution.md

确认点:
  CP1: 变更设计确认
  CP2: 脚本确认
  CP3: 生产执行前确认

特殊要求:
  - 必须有回滚脚本
  - 必须在测试环境验证
  - 必须有数据备份
```

## ⚪ Tier 3: 特殊意图

| # | 意图名称 | 触发词 | 风险 | 特殊规则 |
|---|---------|-------|------|---------|
| 09 | **简单问答** | 什么是、解释、说明、怎么理解 | P2 | 直接回答，跳过完整流程 |
| 10 | **代码审查** | 审查、Review、检查代码 | P2 | 只分析不修改 |
| 11 | **仅测试** | 测试、验证、检查功能 | P2 | 只生成测试，不改代码 |
| 12 | **仅文档** | 文档、README、注释 | P2 | 只生成文档，不改代码 |
| 13 | **紧急修复** | 紧急、立即、线上问题 | P0 | 简化流程，必须有回滚 |

### 简单问答详情

```yaml
典型场景:
  - "这个函数是干什么的？"
  - "解释一下这段代码"
  - "MongoDB 的索引原理"

处理方式:
  - 执行预检查（简化版）
  - 直接回答问题
  - 不生成文档
```

### 紧急修复详情（P0）

```yaml
典型场景:
  - "线上支付失败，紧急修复"
  - "生产环境 Bug，立即处理"

特殊处理:
  - 跳过: 测试用例生成
  - 必须: 方案确认、代码确认、回滚方案
  - 加速: 简化文档
```

## 📊 意图识别规则

### 关键词匹配表

| 关键词组 | 匹配意图 |
|---------|---------|
| 开发、实现、新增、添加、创建、集成 | 需求开发 |
| 修复、解决、Bug、问题、错误、异常 | Bug 修复 |
| 优化、性能、加速、缓存、慢 | 性能优化 |
| 调研、选型、对比、评估、分析方案 | 技术调研 |
| 重构、重写、拆分、改造、升级 | 架构重构 |
| 数据库、Schema、迁移、索引、表 | 数据库变更 |
| 安全、漏洞、注入、XSS、加固 | 安全修复 |
| 故障、宕机、复盘、事故 | 事故复盘 |
| 什么是、解释、说明、怎么 | 简单问答 |
| 审查、Review、检查代码 | 代码审查 |
| 紧急、立即、线上、生产 | 紧急修复 |

### 优先级规则

```yaml
匹配顺序:
  1. 先匹配 Tier 1（核心意图）
  2. 再匹配 Tier 2（扩展意图）
  3. 最后匹配 Tier 3（特殊意图）
  4. 无法匹配时，默认为"需求开发"

多关键词处理:
  - 取风险等级最高的意图
  - 如"开发并部署" → 风险取 P0（部署）
```

### 不确定时的处理

```yaml
当无法明确识别时:
  1. 列出可能的意图（最多 3 个）
  2. 向用户提问确认
  3. 等待用户选择后继续

示例:
  用户: "处理一下用户模块"
  
  AI: "我识别到可能是以下任务类型：
        [A] 需求开发 - 添加新功能
        [B] Bug 修复 - 修复问题
        [C] 架构重构 - 重构代码
        
        请选择任务类型: [A/B/C]"
```

## 🔄 意图→步骤映射

> ⚠️ **以下步骤映射基于 v1.x 体系，已过时。v2.1 各工作流的实际步骤请以各自的 `README.md` 为准。**

### 需求开发步骤

```
预检查 → 需求确认 → 项目分析 → 技术方案 → [CP2] → 
代码实现 → [CP3] → 测试生成 → 测试执行 → [CP4] → 
文档生成 → [CP5] → 验证 → 完成
```

### Bug 修复步骤

```
预检查 → 问题复现 → 原因分析 → [CP1] → 
解决方案 → [CP2] → 代码修复 → [CP3] → 
测试验证 → [CP4] → 文档归档 → 完成
```

### 简单问答步骤

```
预检查（简化）→ 直接回答 → 完成
```
